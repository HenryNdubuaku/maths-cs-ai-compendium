# 为什么用 C++，以及 ML 框架如何工作

*本书中每一次 `jnp.matmul`、每一次 `torch.nn.Linear`、每一次 `np.dot` 调用，底层都在执行 C++ 和 CUDA 代码。本文件揭开这层帷幕：为什么 ML 框架要这样构建、给 Python 工程师的 C++ 速成、何时需要编写自定义 C++ kernel，以及如何将它们绑定到 Python——这是连接你所写代码与其运行硬件之间的桥梁。*

- 你已经花了 15 章写 Python。你导入过 JAX，调用过 `jax.grad`，跑过训练循环，搭过模型。这一切感觉都像是 Python。但事实是：**真正的计算几乎没有一项发生在 Python 里。**

- 当你在 PyTorch 中写 `output = model(input)`，或在 JAX 中写 `output = jnp.matmul(W, x)` 时，Python 几乎什么都没做。它只是构造了一段对计算的描述（一张操作图），然后把它交给 C++/CUDA 后端去做真正的工作。Python 是方向盘，C++ 是发动机。

## 为什么是 Python 前端、C++ 后端

- 这种双语言架构之所以存在，是因为 Python 和 C++ 擅长截然相反的事情：

| | Python | C++ |
|--|--------|-----|
| 开发速度 | 快（动态类型、REPL、无需编译） | 慢（静态类型、头文件、编译耗时） |
| 执行速度 | 比 C 慢约 100 倍（解释执行、GIL） | 接近硬件速度（编译执行、无开销） |
| 内存控制 | 自动（GC），无法控制布局 | 手动，对每个字节都能精确控制 |
| 硬件访问 | 无（没有 SIMD、没有 GPU、没有自定义内存） | 完整（intrinsics、CUDA、内联汇编） |
| 生态系统 | ML 方面丰富（notebook、可视化、数据） | 系统方面丰富（OS、驱动、引擎） |

- 关键洞察：**让每种语言做它擅长的事**。Python 负责人类生产力更重要的部分（实验设计、超参数调优、数据探索）。C++ 负责机器性能更重要的部分（矩阵乘法、卷积、attention kernel）。

- 一次矩阵乘法 `jnp.matmul(A, B)`，当 $A$ 是 $4096 \times 4096$ 时执行约 1370 亿次浮点运算。用纯 Python（嵌套循环）来跑，大约需要 30 分钟。用优化过、带 AVX-512 SIMD 和多线程的 C++，大约只要 10 毫秒。这是 **180,000 倍**的差距。Python 玩出多少花样都弥补不了这个鸿沟。

## ML 框架是如何组织起来的

- 每个主流 ML 框架都遵循同样的架构：

```
用户代码（Python）
    ↓
Python API 层（torch.nn、jax.numpy、numpy）
    ↓
Dispatch / JIT 编译器（torch.compile、XLA、NumPy dispatch）
    ↓
C++ kernel 库（ATen/PyTorch、XLA、BLAS/LAPACK）
    ↓
硬件专属后端（CUDA、cuDNN、MKL、oneDNN、Metal）
    ↓
硬件（CPU SIMD 单元、GPU 核心、TPU MXU）
```

### NumPy

- NumPy 的核心是用 C 写的。你调用 `np.dot(A, B)` 时，Python 调用一个 C 函数，该函数再调用 BLAS（Basic Linear Algebra Subprograms），通常是 Intel MKL 或 OpenBLAS。BLAS 是手工优化的 C 和 Fortran 代码，使用 SIMD 指令、cache 友好的内存访问模式和多线程。让矩阵乘法变快，背后是几十年的优化积累。

- NumPy 只支持 CPU，不使用 GPU。但在 CPU 上它极快，因为它委托给了当前可用的最佳 BLAS 实现。

### PyTorch

- PyTorch 的计算引擎是 **ATen**（A Tensor Library），用 C++ 编写。ATen 实现了约 2000 个 tensor 操作（add、matmul、conv2d、softmax……），每个都有 CPU 和 CUDA 后端。

- 你调用 `torch.matmul(A, B)` 时：
    1. Python 派发到 ATen 的 C++ 函数。
    2. ATen 检查 device（CPU 或 CUDA）和 dtype。
    3. CPU 上：调用 MKL/OpenBLAS。GPU 上：调用 cuBLAS（NVIDIA 的 GPU 优化版 BLAS）。
    4. 结果被包成一个 Python tensor 对象返回。

- **torch.compile**（PyTorch 2.0+）更进一步：它 trace 你的 Python 代码，构建计算图，并用 **Triton**（针对 GPU）或 **C++/OpenMP**（针对 CPU）来编译。编译后的代码会融合操作、消除 Python 开销，可以比 eager 模式快 2-5 倍。

### JAX

- JAX 把 Python 函数编译为 **XLA**（Accelerated Linear Algebra），这是 Google 为 ML 工作负载设计的编译器。你对一个函数 `jax.jit` 时：
    1. JAX trace 该函数，把操作捕获为一张 XLA 计算图（HLO——High Level Operations）。
    2. XLA 优化这张图：融合操作、消除冗余计算、优化内存布局。
    3. XLA 编译到目标后端：CPU（经 LLVM）、GPU（经 CUDA/PTX）或 TPU（经 TPU 专属指令）。
    4. 编译后的代码直接在硬件上运行，Python 完全不参与。

- 这就是为什么 `jax.jit` 如此重要：没有它，每个操作都是一次独立的 Python→C++ 往返。有了它，整个函数就是一个编译好的 kernel。

## 给 Python 工程师的 C++ 速成

- 你不需要成为 C++ 专家。你只需要懂到能读懂 kernel 代码、写简单的扩展、看懂性能讨论。下面是要点。

### 类型与变量

```cpp
// C++ 需要显式类型（与 Python 不同）
int count = 0;           // 32-bit integer
float loss = 0.5f;       // 32-bit float
double lr = 3e-4;        // 64-bit float
bool training = true;    // boolean

// 数组（固定大小，栈上分配）
float weights[1024];     // 1024 个 float，在内存中连续

// 指针：一个保存内存地址的变量
float* ptr = weights;    // ptr 指向 weights 的第一个元素
float val = ptr[42];     // 通过指针算术访问第 42 个元素
// ptr[42] 等价于 *(ptr + 42)
```

- **指针**是与 Python 最大的概念差异。在 Python 里一切都是引用，你从不关心内存地址。在 C++ 里，指针给你对内存的直接访问——强大但危险（悬垂指针、缓冲区溢出）。

### 函数

```cpp
// 函数声明：return_type name(param_type param_name)
float relu(float x) {
    return x > 0.0f ? x : 0.0f;
}

// 按引用传递（避免拷贝大对象）
void scale_vector(std::vector<float>& vec, float factor) {
    for (size_t i = 0; i < vec.size(); i++) {
        vec[i] *= factor;
    }
}

// const 引用：只读、无拷贝
float sum(const std::vector<float>& vec) {
    float total = 0.0f;
    for (float x : vec) {  // range-based for 循环（类似 Python 的 for x in vec）
        total += x;
    }
    return total;
}
```

### 内存：栈 vs 堆

```cpp
// 栈分配：快，生命周期自动（函数返回时释放）
float buffer[256];   // 栈上 256 个 float

// 堆分配：手动，生命周期可超出函数
float* data = new float[n];   // 在堆上分配 n 个 float
// ... 使用 data ...
delete[] data;                 // 你必须自己释放（没有垃圾回收器）

// 现代 C++：智能指针（自动清理，类似 Python 引用）
#include <memory>
auto data = std::make_unique<float[]>(n);  // 出作用域时自动释放
```

- **关键规则**：栈快但有限（通常 1-8 MB）。大数组（tensor、feature map）必须放在堆上。在 Python 里，一切都在堆上，GC 负责清理。在 C++ 里，你自己管理（或用智能指针）。

### 模板（泛型）

```cpp
// 一个适用于任意数值类型的函数
template <typename T>
T add(T a, T b) {
    return a + b;
}

add<float>(1.5f, 2.5f);   // 返回 4.0f
add<int>(3, 4);             // 返回 7
```

- 模板是 C++ 库（如 ATen）编写同时支持 float16、float32、float64 等类型代码的方式，而无需重复实现。

### 标准库要点

```cpp
#include <vector>      // 动态数组（类似 Python list）
#include <string>      // 字符串类型
#include <unordered_map>  // 哈希表（类似 Python dict）
#include <algorithm>   // sort、find、transform 等
#include <cmath>       // 数学函数

std::vector<float> vec = {1.0f, 2.0f, 3.0f};
vec.push_back(4.0f);            // 追加
float first = vec[0];           // 索引
size_t len = vec.size();        // 长度

std::unordered_map<std::string, int> counts;
counts["hello"] = 5;            // 插入
if (counts.count("hello")) { }  // 检查存在性
```

## 何时编写自定义 C++ kernel

- 大多数 ML 工程师永远不需要写 C++。框架的内置操作覆盖了 99% 的用例。你只在以下情况才应考虑自定义 C++：

1. **框架中没有你需要的操作**：一种新的激活函数、一种自定义 attention 模式、一种无法用现有 op 组合表达的专用 loss 函数。

2. **为性能融合操作**：你的模型执行 `relu(layernorm(matmul(x, W) + b))`。每个操作启动一个独立的 kernel，读写内存并同步。一个融合 kernel 一次做完所有事，避免内存往返。这可以快 2-5 倍。

3. **降低内存占用**：自定义 kernel 可以在不保存所有中间激活的情况下计算梯度（kernel 级的 gradient checkpointing）。

4. **面向新型硬件**：一种新加速器（如 Cerebras、Groq）可能没有框架支持。你直接写 kernel。

- 对于情形 1-2，**Triton**（第 16 章，文件 05）通常就足够了，且比直接写 CUDA C 容易得多。只有当 Triton 无法表达你需要的东西时，才下沉到 CUDA C。

## 如何将 C++ 绑定到 Python

- 写 C++ 只是一半工作。你还需要从 Python 调用它。

### pybind11（通用）

- pybind11 用极少的样板代码为 C++ 函数创建 Python 绑定：

```cpp
// my_ops.cpp
#include <pybind11/pybind11.h>
#include <pybind11/numpy.h>
namespace py = pybind11;

// 一个简单的自定义操作
py::array_t<float> custom_relu(py::array_t<float> input) {
    auto buf = input.request();
    float* ptr = static_cast<float*>(buf.ptr);
    size_t n = buf.size;

    auto result = py::array_t<float>(n);
    float* out = static_cast<float*>(result.request().ptr);

    for (size_t i = 0; i < n; i++) {
        out[i] = ptr[i] > 0 ? ptr[i] : 0;
    }
    return result;
}

PYBIND11_MODULE(my_ops, m) {
    m.def("custom_relu", &custom_relu, "Custom ReLU operation");
}
```

```bash
# 编译
pip install pybind11
c++ -O3 -shared -std=c++17 -fPIC $(python3 -m pybind11 --includes) my_ops.cpp -o my_ops$(python3-config --extension-suffix)
```

```python

```

### PyTorch C++ 扩展

- PyTorch 提供了一种精简方式来添加自定义 op：

```cpp
// custom_op.cpp
#include <torch/extension.h>

torch::Tensor custom_gelu(torch::Tensor x) {
    return x * 0.5 * (1.0 + torch::erf(x / std::sqrt(2.0)));
}

PYBIND11_MODULE(TORCH_EXTENSION_NAME, m) {
    m.def("custom_gelu", &custom_gelu, "Custom GELU activation");
}
```

```python
# 即时加载并编译
from torch.utils.cpp_extension import load

custom_ops = load(
    name="custom_ops",
    sources=["custom_op.cpp"],
    extra_cflags=["-O3"],
)

x = torch.randn(1000)
y = custom_ops.custom_gelu(x)
```

- `torch.utils.cpp_extension.load` 一次调用就编译 C++ 代码、创建共享库并作为 Python 模块加载。这是在 PyTorch 中试验自定义 C++ op 最简单的方式。

### JAX Custom Calls

- JAX 使用 XLA custom calls。流程更复杂（你要向 XLA 注册一个 C 函数），但概念相同：写 C/C++，绑定它，从 Python 调用。

- 对大多数 JAX 用户，**Pallas**（见文件 05）是更好的选择：它让你用类似 Python 的语法写 GPU kernel，由 XLA 编译，而不必离开 JAX 生态。

## 大局观

- 本文件解释了 Python 与硬件之间的这一层。本章其余文件深入更底层：
    - **文件 01**：硬件本身（CPU 架构、GPU 架构、内存系统）
    - **文件 02-03**：CPU 上的 SIMD 编程（ARM NEON、x86 AVX）——你写的 C++ 会用到 CPU 的向量单元
    - **文件 04**：用 CUDA 做 GPU 编程——你写的 C++ 跑在数千个 GPU 核心上
    - **文件 05**：Triton、Pallas 以及更高层的 GPU 编程——你写的 Python 编译成 GPU kernel

- 这一进程对应着抽象阶梯：C++ intrinsics（最低层、控制力最强）→ CUDA（GPU 专属）→ Triton/Pallas（Pythonic、编译式）→ JAX/PyTorch（最高层、自动化）。每一层都用控制力换取便利。理解较低层次，能让你成为较高层次更好的使用者。

## 编码任务（用 g++ 或 clang++ 编译）

1. 写你的第一个 C++ 程序。分配一个数组，填充它，计算求和，并测量耗时。这会介绍编译、数组、指针和计时。
```cpp
// task1_basics.cpp
// 编译：g++ -O3 -o task1 task1_basics.cpp
// 运行：./task1

#include <iostream>
#include <chrono>
#include <vector>

int main() {
    const int N = 10'000'000;  // C++ 允许用 ' 作为数字分隔符
    std::vector<float> data(N);

    // 填充数组
    for (int i = 0; i < N; i++) {
        data[i] = static_cast<float>(i) * 0.001f;
    }

    // 计算求和
    auto start = std::chrono::high_resolution_clock::now();
    float sum = 0.0f;
    for (int i = 0; i < N; i++) {
        sum += data[i];
    }
    auto end = std::chrono::high_resolution_clock::now();
    double elapsed = std::chrono::duration<double, std::milli>(end - start).count();

    std::cout << "Sum: " << sum << std::endl;
    std::cout << "Time: " << elapsed << " ms" << std::endl;
    std::cout << "Elements: " << N << std::endl;
    std::cout << "Throughput: " << (N * sizeof(float)) / elapsed / 1e6 << " GB/s" << std::endl;

    return 0;
}
```

2. 写一个对数组计算 ReLU 的 C++ 函数，然后用 pybind11 构建 Python 绑定。从 Python 调用它，并与 NumPy 比较速度。
```cpp
// task2_relu.cpp
// 编译：c++ -O3 -shared -std=c++17 -fPIC $(python3 -m pybind11 --includes) \
//          task2_relu.cpp -o my_relu$(python3-config --extension-suffix)

#include <pybind11/pybind11.h>
#include <pybind11/numpy.h>
namespace py = pybind11;

py::array_t<float> cpp_relu(py::array_t<float> input) {
    auto buf = input.request();
    float* ptr = static_cast<float*>(buf.ptr);
    int n = buf.size;

    auto result = py::array_t<float>(n);
    float* out = static_cast<float*>(result.request().ptr);

    for (int i = 0; i < n; i++) {
        out[i] = ptr[i] > 0.0f ? ptr[i] : 0.0f;
    }
    return result;
}

PYBIND11_MODULE(my_relu, m) {
    m.def("relu", &cpp_relu, "C++ ReLU");
}
```
```python
# test_relu.py — 编译完上面的 C++ 模块后运行
import numpy as np
import time
import my_relu  # 编译好的 C++ 模块

x = np.random.randn(10_000_000).astype(np.float32)

# C++ ReLU
start = time.time()
for _ in range(100):
    y_cpp = my_relu.relu(x)
cpp_time = (time.time() - start) / 100

# NumPy ReLU
start = time.time()
for _ in range(100):
    y_np = np.maximum(x, 0)

print(f"C++ ReLU:   {cpp_time*1000:.2f} ms")
print(f"NumPy ReLU: {np_time*1000:.2f} ms")
print(f"Match: {np.allclose(y_cpp, y_np)}")
```

3. 写一个 C++ 程序，演示为什么内存布局很重要。比较行主序与列主序访问模式，并测量性能差异。
```cpp
// task3_layout.cpp
// 编译：g++ -O3 -o task3 task3_layout.cpp

#include <iostream>
#include <chrono>
#include <vector>

int main() {
    const int N = 4096;
    std::vector<float> matrix(N * N, 1.0f);

    // 行主序访问：顺序内存地址（cache 友好）
    auto start = std::chrono::high_resolution_clock::now();
    float sum_row = 0.0f;
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < N; j++) {
            sum_row += matrix[i * N + j];  // stride-1 访问
        }
    }
    auto end = std::chrono::high_resolution_clock::now();
    double row_ms = std::chrono::duration<double, std::milli>(end - start).count();

    // 列主序访问：stride-N 访问（对 cache 不友好）
    start = std::chrono::high_resolution_clock::now();
    float sum_col = 0.0f;
    for (int j = 0; j < N; j++) {
        for (int i = 0; i < N; i++) {
            sum_col += matrix[i * N + j];  // stride-N 访问（cache miss 严重！）
        }
    }
    end = std::chrono::high_resolution_clock::now();
    double col_ms = std::chrono::duration<double, std::milli>(end - start).count();

    std::cout << "Row-major (cache-friendly): " << row_ms << " ms" << std::endl;
    std::cout << "Col-major (cache-hostile):  " << col_ms << " ms" << std::endl;
    std::cout << "Slowdown: " << col_ms / row_ms << "x" << std::endl;
    std::cout << "(Both sums: " << sum_row << ", " << sum_col << ")" << std::endl;

    return 0;
}
```
