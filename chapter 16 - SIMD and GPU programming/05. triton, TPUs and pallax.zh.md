# Triton 与 TPU

*CUDA C 功能强大但繁琐。Triton 让你用 Python 编写 GPU kernel。TPU 则以不同的取舍为 GPU 提供了替代方案。本文件涵盖 Triton kernel 编程、以 Flash Attention 为案例研究、TPU 架构与 JAX/Pallas，以及如何选择合适的工具。关于 Vulkan 和跨平台 GPU 计算，见文件 07。*

- 上一文件用 CUDA C 讲授 GPU 编程。本文件沿抽象阶梯向上爬升:Triton 让你用 Python 以 20% 的精力获得 CUDA 80% 的性能。TPU 和 Vulkan 则针对特定用例提供了替代的硬件目标。

## Triton:用 Python 编写 GPU Kernel

- **Triton**(OpenAI 出品)是一种基于 Python 的语言,用于编写 GPU kernel。与 CUDA 中关注单个 thread 不同,在 Triton 中你关注的是数据 **block**。Triton 的编译器会自动处理 thread 映射、memory coalescing、shared memory 管理以及许多优化。

- **Triton 之所以重要**:CUDA C 要求对 warp 调度、shared memory bank 冲突、register 压力和 coalescing 模式有深入理解。Triton 把大部分细节抽象掉,使只懂 Python 而不懂系统编程的 ML 研究者也能开发 GPU kernel。

### 你的第一个 Triton Kernel

```python
import triton
import triton.language as tl
import torch

@triton.jit
def add_kernel(
    x_ptr, y_ptr, output_ptr,
    n_elements,
    BLOCK_SIZE: tl.constexpr,  # compile-time constant
):
    # Each program instance processes one block of BLOCK_SIZE elements
    pid = tl.program_id(axis=0)  # which block am I?
    block_start = pid * BLOCK_SIZE

    # Offsets for this block
    offsets = block_start + tl.arange(0, BLOCK_SIZE)

    # Mask to handle the case where n_elements is not a multiple of BLOCK_SIZE
    mask = offsets < n_elements

    # Load data (masked: out-of-bounds reads return 0)
    x = tl.load(x_ptr + offsets, mask=mask)
    y = tl.load(y_ptr + offsets, mask=mask)

    # Compute
    output = x + y

    # Store result
    tl.store(output_ptr + offsets, output, mask=mask)


def add(x: torch.Tensor, y: torch.Tensor) -> torch.Tensor:
    output = torch.empty_like(x)
    n_elements = output.numel()

    # Launch: one program per block
    grid = lambda meta: (triton.cdiv(n_elements, meta['BLOCK_SIZE']),)
    add_kernel[grid](x, y, output, n_elements, BLOCK_SIZE=1024)

    return output


# Usage
x = torch.randn(1000000, device='cuda')
y = torch.randn(1000000, device='cuda')
z = add(x, y)
```

- **与 CUDA 的关键差异**:
    - 没有显式的 thread 管理。你按 **block**(program)而非 thread 思考。
    - `tl.arange(0, BLOCK_SIZE)` 为整个 block 创建一个偏移量向量。对该向量的所有操作都隐式地被向量化。
    - `mask` 处理边界条件(类似 AVX-512 的 mask register,见文件 03)。无需标量清理循环。
    - `tl.load` 和 `tl.store` 自动处理 coalesced 访问。
    - `@triton.jit` 在首次调用时将函数编译为 PTX(GPU 汇编),然后缓存已编译的 kernel。

### Triton Softmax Kernel

- Softmax 是一个绝佳的 Triton 示例,因为它需要对数据做多趟处理(max、减、exp、sum、除),并且能从将数据在多趟之间保留在 SRAM(shared memory)中受益:

```python
@triton.jit
def softmax_kernel(
    output_ptr, input_ptr, input_row_stride, output_row_stride, n_cols,
    BLOCK_SIZE: tl.constexpr,
):
    # Each program handles one row
    row_idx = tl.program_id(0)
    row_start = input_ptr + row_idx * input_row_stride

    # Load the row
    col_offsets = tl.arange(0, BLOCK_SIZE)
    mask = col_offsets < n_cols
    row = tl.load(row_start + col_offsets, mask=mask, other=-float('inf'))

    # Softmax: max for numerical stability, then exp, then normalise
    row_max = tl.max(row, axis=0)
    numerator = tl.exp(row - row_max)
    denominator = tl.sum(numerator, axis=0)
    softmax_output = numerator / denominator

    # Store result
    output_start = output_ptr + row_idx * output_row_stride
    tl.store(output_start + col_offsets, softmax_output, mask=mask)
```

- 在 PyTorch 中,`F.softmax(x, dim=-1)` 会启动 3 个独立的 kernel(max、exp-and-sum、divide),每个都要从 global memory 读取并写回。Triton 版本在一个 kernel 中完成所有操作,把数据保留在 register/SRAM 中。这种 **kernel 融合**(kernel fusion)正是自定义 Triton kernel 能比 PyTorch 内置算子快 2-4 倍的原因。

### Triton 自动调优

- Triton 支持**自动调优**(auto-tuning):尝试多种配置并选择最快的那个:

```python
@triton.autotune(
    configs=[
        triton.Config({'BLOCK_SIZE_M': 128, 'BLOCK_SIZE_N': 128, 'BLOCK_SIZE_K': 32}),
        triton.Config({'BLOCK_SIZE_M': 64, 'BLOCK_SIZE_N': 256, 'BLOCK_SIZE_K': 32}),
        triton.Config({'BLOCK_SIZE_M': 256, 'BLOCK_SIZE_N': 64, 'BLOCK_SIZE_K': 64}),
    ],
    key=['M', 'N', 'K'],  # re-tune when these change
)
@triton.jit
def matmul_kernel(a_ptr, b_ptr, c_ptr, M, N, K, ...):
    ...
```

- Triton 在真实硬件上对每种配置进行基准测试,并选出最快的。最优 tile 尺寸取决于 GPU 架构、矩阵维度和内存布局——自动调优无需手动实验就能找到它们。

### Triton vs CUDA:何时使用哪个

| | Triton | CUDA C |
|--|--------|--------|
| 语言 | Python | C/C++ |
| 抽象层级 | block 级 | thread 级 |
| 开发速度 | 快(每个 kernel 10-50 行) | 慢(100-500 行) |
| 性能上限 | 约为手工调优 CUDA 的 80-95% | 100%(完全硬件控制) |
| shared memory | 自动 | 手动 |
| coalescing | 自动 | 手动 |
| warp 级原语 | 有限 | 完整(shuffle、vote 等) |
| 硬件支持 | 仅 NVIDIA(AMD 为实验性) | 仅 NVIDIA |

- **使用 Triton** 的场景:融合 kernel、自定义 attention 模式、激活函数,以及大多数 ML 研究 kernel 需求。
- **使用 CUDA C** 的场景:追求极致性能(最后的 5-20%)、warp 级原语、复杂的数据相关并行,以及 Triton 无法表达的模式。

## 案例研究:Flash Attention

- **Flash Attention**(Dao 等,2022)是近年来 ML 领域最具影响力的自定义 kernel。它以 $O(n)$ 内存而非 $O(n^2)$ 内存计算 attention,从而支持更长的序列。

- **问题所在**:标准 attention 计算 $\text{softmax}(QK^T / \sqrt{d}) \cdot V$。$QK^T$ 矩阵为 $n \times n$,其中 $n$ 为序列长度。当 $n = 128K$ 时,该矩阵为 $128K \times 128K \times 4$ 字节 = 64 GB。它无法装进 GPU 内存。

- **关键洞察**:你不必物化整张 $n \times n$ 矩阵。以 **tile** 为单位计算 attention:加载一个 $Q$ 的 block、一个 $K$ 的 block,计算它们的部分 attention 分数,累加,然后移动到下一个 block。$n \times n$ 矩阵永远不会被完整物化——同一时刻只有一块 tile 存在于 SRAM 中。

- **在线 softmax**:棘手之处在于 softmax,它需要知道整行的最大值(用于数值稳定性)。Flash Attention 使用**在线 softmax** 技巧:维护一个运行最大值,并在发现新最大值时对先前计算的值进行重缩放。这样就可以一次一个 tile 地增量计算 softmax。

- 算法如下:

```
For each block of Q rows:
    For each block of K columns:
        1. Load Q_block from HBM to SRAM
        2. Load K_block from HBM to SRAM
        3. Compute S_block = Q_block @ K_block.T (in SRAM)
        4. Update running max, rescale previous results
        5. Compute exp(S_block - running_max)
        6. Update running sum and output accumulator
    Load V_block and compute final output
    Write output block back to HBM
```

- **为什么它快**:内层循环完全在 SRAM(shared memory)中执行。global memory(HBM)只在加载 Q、K、V 的 block 和写回最终输出时被访问。数据复用因子与 SRAM 大小成正比,而 SRAM 的访问速度约为 HBM 的 100 倍。

- Flash Attention 同时有 Triton 和 CUDA C 实现。CUDA 版本更快(效率约高 10%),但 Triton 版本可读性和可改性远更好,这对于新 attention 变体的研究至关重要。

## TPU 架构

- **TPU**(Tensor Processing Unit)是 Google 定制的 ML 加速器。它采用与 GPU 截然不同的方法:

- **脉动阵列**(systolic array):TPU 的核心计算单元是**矩阵乘单元**(Matrix Multiply Unit, MXU),一个 128×128 或 256×256 的脉动阵列,通过让数据流过乘加单元网格来计算矩阵乘法。数据从边缘进入并在阵列中传播,每个单元执行一次乘加并把结果传递给下一个。

- 与 GPU(调度数千个独立 thread)不同,脉动阵列是一个单一的、确定性的数据流。没有 thread 调度,没有 warp 分歧,没有分支预测。这种简洁性使 MXU 在矩阵乘法上极其节能。

- **HBM**:TPU 使用与 GPU 相同的高带宽内存(High Bandwidth Memory)。TPU v5e 每芯片有 16 GB HBM2e;TPU v5p 有 95 GB HBM2e。

- **ICI**(Inter-Chip Interconnect,片间互连):TPU pod 用定制的的高速网络连接数百个 TPU。跨 TPU pod 的数据并行和模型并行(第 6 章)由 JAX 原生支持。

- **BFloat16**:TPU 是首个使用 bfloat16 的硬件(第 13 章,文件 02)。BF16 与 float32 有相同的指数范围(防止训练期间溢出),但尾数精度更低。这种取舍非常适合 ML——梯度值分布范围很广,但不需要 23 位精度。

### 编程 TPU:JAX 与 Pallas

- TPU 通过 **JAX** 和 **XLA** 编程。你编写 Python/JAX 代码,`jax.jit` 将其编译为 XLA HLO,XLA 再把 HLO 编译为 TPU 专属指令。没有 CUDA,也没有 C++。

```python
import jax
import jax.numpy as jnp

@jax.jit
def matmul(a, b):
    return jnp.dot(a, b)

# This runs on CPU, GPU, or TPU depending on the device
a = jnp.ones((1024, 1024))
b = jnp.ones((1024, 1024))
c = matmul(a, b)
```

- **Pallas** 是 JAX 的 kernel 编写 API——相当于 JAX 版的 Triton。它让你编写低层 kernel,再由 XLA 编译到 GPU 或 TPU:

```python
from jax.experimental import pallas as pl
import jax.numpy as jnp

def add_kernel(x_ref, y_ref, o_ref):
    o_ref[...] = x_ref[...] + y_ref[...]

def add_pallas(x, y):
    return pl.pallas_call(
        add_kernel,
        out_shape=jax.ShapeDtypeStruct(x.shape, x.dtype),
        grid=(x.shape[0] // 128,),
        in_specs=[pl.BlockSpec((128,), lambda i: (i,)),
                  pl.BlockSpec((128,), lambda i: (i,))],
        out_specs=pl.BlockSpec((128,), lambda i: (i,)),
    )(x, y)
```

- Pallas 比 Triton 更新且不够成熟,但它是为 TPU 编写自定义 kernel 的唯一方式(因为 TPU 不支持 CUDA)。

### GPU vs TPU

| | GPU(NVIDIA) | TPU(Google) |
|--|-------------|--------------|
| 可获得性 | 任意云、本地机房 | 仅 Google Cloud |
| 编程 | CUDA C、Triton、PyTorch | JAX/XLA、Pallas |
| 灵活性 | 通用计算 | 为矩阵密集型 ML 优化 |
| 峰值 matmul FLOPS | 极高(Tensor Cores) | 极高(MXU) |
| 非 matmul 算子 | 表现良好 | 较慢(经向量单元路由,不走 MXU) |
| 多芯片扩展 | NVLink(8 GPU)、InfiniBand | ICI(数千 TPU,集成更紧密) |
| 成本效益 | 具竞争力 | 大规模训练往往更便宜 |
| 生态 | 最大(PyTorch、TensorFlow、JAX) | 以 JAX 为中心 |

- **使用 GPU** 的场景:大多数 ML 负载、基于 PyTorch 的研究、推理服务、含大量非 matmul 计算的负载。
- **使用 TPU** 的场景:大规模 JAX 训练(数千芯片)、Google Cloud 上对成本敏感的训练、以矩阵乘法为主的负载。

## 选择合适的工具

| 负载 | 最佳工具 | 原因 |
|----------|-----------|-----|
| ML 训练(PyTorch) | NVIDIA GPU + CUDA/Triton | 生态最大、工具链最佳 |
| ML 训练(JAX,大规模) | TPU 或 NVIDIA GPU | Google 规模下用 TPU 省成本,GPU 更灵活 |
| 自定义融合 kernel | Triton(Python)或 CUDA C | Triton 开发快,CUDA 性能极致 |
| JAX 自定义 kernel | Pallas | TPU 上唯一选择,也能用于 GPU |
| 跨平台推理 | Vulkan(文件 07)或 ONNX Runtime | 可运行于任意 GPU 厂商 |
| 移动/边缘推理 | Metal(Apple)、Vulkan(Android)、NNAPI | 平台专属加速器 |
| 浏览器推理 | WebGPU(文件 07) | 浏览器内唯一选择 |
| 仅 CPU 推理 | ONNX Runtime + AVX/NEON | 无需 GPU,使用 SIMD(文件 02-03) |
| 新型硬件 | 厂商专属 SDK | 每种加速器都有各自工具链 |

## 编码任务(使用带 GPU 运行时的 CoLab)

1. 编写并运行一个向量加法的 Triton kernel,与 PyTorch 内置加法做性能对比。
```python
import triton
import triton.language as tl
import torch
import time

@triton.jit
def add_kernel(x_ptr, y_ptr, out_ptr, n, BLOCK: tl.constexpr):
    pid = tl.program_id(0)
    offs = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offs < n
    x = tl.load(x_ptr + offs, mask=mask)
    y = tl.load(y_ptr + offs, mask=mask)
    tl.store(out_ptr + offs, x + y, mask=mask)

n = 10_000_000
x = torch.randn(n, device='cuda')
y = torch.randn(n, device='cuda')

# Triton
out_triton = torch.empty_like(x)
grid = lambda meta: (triton.cdiv(n, meta['BLOCK']),)
add_kernel[grid](x, y, out_triton, n, BLOCK=1024)

# PyTorch
out_torch = x + y

# Verify correctness
assert torch.allclose(out_triton, out_torch, atol=1e-5)

# Benchmark
torch.cuda.synchronize()
start = time.time()
for _ in range(1000):
    add_kernel[grid](x, y, out_triton, n, BLOCK=1024)
torch.cuda.synchronize()
triton_time = (time.time() - start) / 1000

start = time.time()
for _ in range(1000):
    out_torch = x + y
torch.cuda.synchronize()
torch_time = (time.time() - start) / 1000

print(f"Triton:  {triton_time*1000:.3f} ms")
print(f"PyTorch: {torch_time*1000:.3f} ms")
print(f"Ratio:   {torch_time/triton_time:.2f}x")
```

2. 编写一个 Triton 融合 kernel,在单趟中完成 multiply + add + ReLU,与三个独立的 PyTorch 算子做对比。
```python
import triton
import triton.language as tl
import torch
import time

@triton.jit
def fused_mul_add_relu_kernel(x_ptr, w_ptr, b_ptr, out_ptr, n, BLOCK: tl.constexpr):
    pid = tl.program_id(0)
    offs = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offs < n
    x = tl.load(x_ptr + offs, mask=mask)
    w = tl.load(w_ptr + offs, mask=mask)
    b = tl.load(b_ptr + offs, mask=mask)
    result = tl.maximum(x * w + b, 0.0)  # fused: mul + add + relu
    tl.store(out_ptr + offs, result, mask=mask)

n = 10_000_000
x = torch.randn(n, device='cuda')
w = torch.randn(n, device='cuda')
b = torch.randn(n, device='cuda')

# Fused (Triton)
out_fused = torch.empty_like(x)
grid = lambda meta: (triton.cdiv(n, meta['BLOCK']),)
fused_mul_add_relu_kernel[grid](x, w, b, out_fused, n, BLOCK=1024)

# Unfused (PyTorch)
out_unfused = torch.relu(x * w + b)

assert torch.allclose(out_fused, out_unfused, atol=1e-5)

# Benchmark
torch.cuda.synchronize()
start = time.time()
for _ in range(1000):
    fused_mul_add_relu_kernel[grid](x, w, b, out_fused, n, BLOCK=1024)
torch.cuda.synchronize()
fused_time = (time.time() - start) / 1000

start = time.time()
for _ in range(1000):
    out_unfused = torch.relu(x * w + b)
torch.cuda.synchronize()
unfused_time = (time.time() - start) / 1000

print(f"Fused (Triton):    {fused_time*1000:.3f} ms")
print(f"Unfused (PyTorch): {unfused_time*1000:.3f} ms")
print(f"Speedup:           {unfused_time/fused_time:.2f}x")
```

3. 测量 JAX 的 XLA 编译器如何自动融合算子。对比加 jit 与不加 jit 时一条算子链的表现。
```python
import jax
import jax.numpy as jnp
import time

def chain_ops(x):
    x = x * 2.0
    x = x + 1.0
    x = jnp.maximum(x, 0.0)  # ReLU
    x = x / jnp.sum(x)
    return x

chain_jit = jax.jit(chain_ops)
x = jax.random.normal(jax.random.PRNGKey(0), (10000, 1000))

# Warm up
_ = chain_jit(x)
jax.block_until_ready(_)

# Eager (each op is a separate kernel launch)
start = time.time()
for _ in range(100):
    y = chain_ops(x)
jax.block_until_ready(y)
eager_time = (time.time() - start) / 100

# JIT (XLA fuses operations)
start = time.time()
for _ in range(100):
    y = chain_jit(x)
jax.block_until_ready(y)
jit_time = (time.time() - start) / 100

print(f"Eager: {eager_time*1000:.2f} ms")
print(f"JIT:   {jit_time*1000:.2f} ms")
print(f"Speedup: {eager_time/jit_time:.1f}x (XLA fuses the 4 operations into 1 kernel)")
```
