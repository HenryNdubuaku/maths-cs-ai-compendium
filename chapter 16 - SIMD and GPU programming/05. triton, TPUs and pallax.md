## Triton: GPU-ядра на Python

- **Triton** (от OpenAI) — это язык на базе Python для написания GPU-ядер. Вместо того чтобы рассуждать об отдельных потоках (как в CUDA), вы рассуждаете о **блоках** данных. Компилятор Triton автоматически управляет отображением потоков, объединением обращений к памяти (memory coalescing), управлением общей памятью (shared memory) и множеством оптимизаций.

- **Почему Triton важен**: CUDA C требует глубоких знаний о планировании варпов (warp scheduling), конфликтах банков общей памяти, нагрузке на регистры и шаблонах объединения обращений. Triton абстрагирует большую часть этого, делая разработку GPU-ядер доступной для исследователей в области машинного обучения, которые знают Python, но не системное программирование.

### Ваше первое ядро Triton

```python
import triton
import triton.language as tl
import torch

@triton.jit
def add_kernel(
    x_ptr, y_ptr, output_ptr,
    n_elements,
    BLOCK_SIZE: tl.constexpr,  # compile-time constant
):
    # Each program instance processes one block of BLOCK_SIZE elements
    pid = tl.program_id(axis=0)  # which block am I?
    block_start = pid * BLOCK_SIZE

    # Offsets for this block
    offsets = block_start + tl.arange(0, BLOCK_SIZE)

    # Mask to handle the case where n_elements is not a multiple of BLOCK_SIZE
    mask = offsets < n_elements

    # Load data (masked: out-of-bounds reads return 0)
    x = tl.load(x_ptr + offsets, mask=mask)
    y = tl.load(y_ptr + offsets, mask=mask)

    # Compute
    output = x + y

    # Store result
    tl.store(output_ptr + offsets, output, mask=mask)


def add(x: torch.Tensor, y: torch.Tensor) -> torch.Tensor:
    output = torch.empty_like(x)
    n_elements = output.numel()

    # Launch: one program per block
    grid = lambda meta: (triton.cdiv(n_elements, meta['BLOCK_SIZE']),)
    add_kernel[grid](x, y, output, n_elements, BLOCK_SIZE=1024)

    return output


# Usage
x = torch.randn(1000000, device='cuda')
y = torch.randn(1000000, device='cuda')
z = add(x, y)
```

- **Ключевые отличия от CUDA**:
    - Нет явного управления потоками. Вы мыслите **блоками** (программами), а не потоками.
    - `tl.arange(0, BLOCK_SIZE)` создает вектор смещений для всего блока. Все операции над этим вектором неявно векторизуются.
    - `mask` обрабатывает граничные условия (подобно масочным регистрам AVX-512, файл 03). Скалярный цикл очистки не требуется.
    - `tl.load` и `tl.store` автоматически обрабатывают объединение обращений.
    - `@triton.jit` компилирует функцию в PTX (ассемблер GPU) при первом вызове, а затем кеширует скомпилированное ядро.

### Ядро Softmax на Triton

- Softmax — отличный пример для Triton, так как он требует нескольких проходов по данным (максимум, вычитание, экспонента, сумма, деление) и выигрывает от хранения данных в SRAM (общей памяти) между проходами:

```python
@triton.jit
def softmax_kernel(
    output_ptr, input_ptr, input_row_stride, output_row_stride, n_cols,
    BLOCK_SIZE: tl.constexpr,
):
    # Each program handles one row
    row_idx = tl.program_id(0)
    row_start = input_ptr + row_idx * input_row_stride

    # Load the row
    col_offsets = tl.arange(0, BLOCK_SIZE)
    mask = col_offsets < n_cols
    row = tl.load(row_start + col_offsets, mask=mask, other=-float('inf'))

    # Softmax: max for numerical stability, then exp, then normalise
    row_max = tl.max(row, axis=0)
    numerator = tl.exp(row - row_max)
    denominator = tl.sum(numerator, axis=0)
    softmax_output = numerator / denominator

    # Store result
    output_start = output_ptr + row_idx * output_row_stride
    tl.store(output_start + col_offsets, softmax_output, mask=mask)
```

- В PyTorch `F.softmax(x, dim=-1)` запускает 3 отдельных ядра (максимум, экспонента-и-сумма, деление), каждое из которых читает из глобальной памяти и записывает в нее. Версия на Triton выполняет всё в одном ядре, удерживая данные в регистрах/SRAM. Это **слияние ядер** (kernel fusion) — причина, по которой пользовательские ядра Triton могут быть в 2–4 раза быстрее встроенных операций PyTorch.

### Автонастройка Triton

- Triton поддерживает **автонастройку** (auto-tuning): пробуйте несколько конфигураций и выбирайте самую быструю:

```python
@triton.autotune(
    configs=[
        triton.Config({'BLOCK_SIZE_M': 128, 'BLOCK_SIZE_N': 128, 'BLOCK_SIZE_K': 32}),
        triton.Config({'BLOCK_SIZE_M': 64, 'BLOCK_SIZE_N': 256, 'BLOCK_SIZE_K': 32}),
        triton.Config({'BLOCK_SIZE_M': 256, 'BLOCK_SIZE_N': 64, 'BLOCK_SIZE_K': 64}),
    ],
    key=['M', 'N', 'K'],  # re-tune when these change
)
@triton.jit
def matmul_kernel(a_ptr, b_ptr, c_ptr, M, N, K, ...):
    ...
```

- Triton бенчмаркает каждую конфигурацию на реальном оборудовании и выбирает самую быструю. Оптимальные размеры тайлов зависят от архитектуры GPU, размерностей матрицы и компоновки памяти — автонастройка находит их без ручных экспериментов.

### Triton против CUDA: когда что использовать

| | Triton | CUDA C |
|--|--------|--------|
| Язык | Python | C/C++ |
| Абстракция | Уровень блоков | Уровень потоков |
| Скорость разработки | Быстро (10–50 строк на ядро) | Медленно (100–500 строк) |
| Потолок производительности | ~80–95% от оптимизированного вручную CUDA | 100% (полный контроль над железом) |
| Общая память | Автоматически | Вручную |
| Объединение обращений | Автоматически | Вручную |
| Примитивы уровня варпа | Ограниченно | Полностью (shuffle, vote и т.д.) |
| Поддержка оборудования | Только NVIDIA (AMD экспериментально) | Только NVIDIA |

- **Используйте Triton** для: слияния ядер, пользовательских шаблонов внимания, функций активации, большинства нужд исследовательских ядер в ML.
- **Используйте CUDA C** для: максимальной производительности (последние 5–20%), примитивов уровня варпа, сложного параллелизма, зависящего от данных, или когда Triton не может выразить ваш шаблон.

## Тематическое исследование: Flash Attention

- **Flash Attention** (Dao et al., 2022) — это наиболее значимое пользовательское ядро (custom kernel) в современном машинном обучении. Оно вычисляет внимание с использованием $O(n)$ памяти вместо $O(n^2)$, что позволяет работать с гораздо более длинными последовательностями.

- **Проблема**: стандартное внимание вычисляет $\text{softmax}(QK^T / \sqrt{d}) \cdot V$. Матрица $QK^T$ имеет размер $n \times n$, где $n$ — длина последовательности. Для $n = 128K$ эта матрица занимает $128K \times 128K \times 4$ байта = 64 ГБ. Она не помещается в память GPU.

- **Идея**: нет необходимости материализовать полную матрицу $n \times n$. Вычисляйте внимание **тайлами** (блоками): загрузите блок $Q$, блок $K$, вычислите их частичные оценки внимания, аккумулируйте результат и переходите к следующему блоку. Матрица $n \times n$ никогда не материализуется целиком — в SRAM одновременно находится только один тайл.

- **Онлайн-softmax**: самая сложная часть — это softmax, для которого необходимо знать максимум по всей строке (для численной стабильности). Flash Attention использует трюк с **онлайн-softmax**: поддерживается текущий максимум, и при обнаружении нового максимума ранее вычисленные значения перемасштабируются. Это позволяет вычислять softmax инкрементально, по одному тайлу за раз.

- Алгоритм:

```
For each block of Q rows:
    For each block of K columns:
        1. Load Q_block from HBM to SRAM
        2. Load K_block from HBM to SRAM
        3. Compute S_block = Q_block @ K_block.T (in SRAM)
        4. Update running max, rescale previous results
        5. Compute exp(S_block - running_max)
        6. Update running sum and output accumulator
    Load V_block and compute final output
    Write output block back to HBM
```

- **Почему это быстро**: внутренний цикл работает полностью в SRAM (разделяемой памяти). Обращение к глобальной памяти (HBM) происходит только для загрузки блоков Q, K, V и записи итогового результата. Коэффициент повторного использования данных пропорционален размеру SRAM, доступ к которой примерно в 100 раз быстрее, чем к HBM.

- Flash Attention реализован как на Triton, так и на CUDA C. Версия на CUDA быстрее (примерно на 10% эффективнее), но версия на Triton гораздо более читаема и модифицируема, что важно для исследований новых вариантов внимания.

## Архитектура TPU

- **TPU** (Tensor Processing Units) — это специализированные ускорители машинного обучения от Google. Они используют принципиально иной подход, чем GPU:

- **Систолические массивы**: основным вычислительным блоком TPU является **Matrix Multiply Unit (MXU)** — систолический массив размером 128×128 или 256×256, который вычисляет матричные произведения путем пропускания данных через сетку блоков умножения-сложения. Данные поступают с краев и распространяются через массив, при этом каждый блок выполняет одну операцию умножения-сложения и передает результат следующему.

- В отличие от GPU (которые планируют тысячи независимых потоков), систолический массив представляет собой единый детерминированный поток данных. Здесь нет планирования потоков, нет дивергенции варпов, нет предсказания переходов. Эта простота делает MXU чрезвычайно энергоэффективным для матричного умножения.

- **HBM**: TPU используют ту же память с высокой пропускной способностью (High Bandwidth Memory), что и GPU. TPU v5e имеет 16 ГБ HBM2e на чип; TPU v5p имеет 95 ГБ HBM2e.

- **ICI** (Inter-Chip Interconnect): поды TPU соединяют сотни TPU с помощью специализированной высокоскоростной сети. Параллелизм по данным и параллелизм по модели (глава 6) между подами TPU поддерживаются нативно в JAX.

- **BFloat16**: TPU стали первыми устройствами, использующими bfloat16 (глава 13, файл 02). BF16 имеет тот же диапазон экспоненты, что и float32 (предотвращая переполнение во время обучения), но с меньшей точностью мантиссы. Этот компромисс идеален для машинного обучения, где значения градиентов охватывают широкий диапазон, но не требуют 23 бит точности.

### Программирование TPU: JAX и Pallas

- TPU программируются через **JAX** и **XLA**. Вы пишете код на Python/JAX, `jax.jit` компилирует его в XLA HLO, а XLA компилирует HLO в инструкции, специфичные для TPU. Никакого CUDA, никакого C++.

```python
import jax
import jax.numpy as jnp

@jax.jit
def matmul(a, b):
    return jnp.dot(a, b)

# This runs on CPU, GPU, or TPU depending on the device
a = jnp.ones((1024, 1024))
b = jnp.ones((1024, 1024))
c = matmul(a, b)
```

- **Pallas** — это API для создания ядер в JAX, аналог Triton для JAX. Оно позволяет писать низкоуровневые ядра, которые XLA компилирует для GPU или TPU:

```python
from jax.experimental import pallas as pl
import jax.numpy as jnp

def add_kernel(x_ref, y_ref, o_ref):
    o_ref[...] = x_ref[...] + y_ref[...]

def add_pallas(x, y):
    return pl.pallas_call(
        add_kernel,
        out_shape=jax.ShapeDtypeStruct(x.shape, x.dtype),
        grid=(x.shape[0] // 128,),
        in_specs=[pl.BlockSpec((128,), lambda i: (i,)),
                  pl.BlockSpec((128,), lambda i: (i,))],
        out_specs=pl.BlockSpec((128,), lambda i: (i,)),
    )(x, y)
```

- Pallas новее и менее зрелый, чем Triton, но это единственный способ написания пользовательских ядер для TPU (поскольку TPU не поддерживают CUDA).

### GPU против TPU

| | GPU (NVIDIA) | TPU (Google) |
|--|-------------|--------------|
| Доступность | Любое облако, локально | Только Google Cloud |
| Программирование | CUDA C, Triton, PyTorch | JAX/XLA, Pallas |
| Гибкость | Вычисления общего назначения | Оптимизировано для ML с интенсивными матричными вычислениями |
| Пиковые FLOPS матричных умножений | Очень высокие (Tensor Cores) | Очень высокие (MXU) |
| Операции, отличные от матричных | Хорошо | Медленнее (обрабатываются через векторный блок, а не MXU) |
| Масштабирование между чипами | NVLink (8 GPU), InfiniBand | ICI (тысячи TPU, более тесная интеграция) |
| Экономическая эффективность | Конкурентоспособная | Часто дешевле для крупномасштабного обучения |
| Экосистема | Самая большая (PyTorch, TensorFlow, JAX) | Ориентирована на JAX |

- **Используйте GPU** для: большинства задач машинного обучения, исследований на базе PyTorch, инференса, задач со значительной долей вычислений, не связанных с матричными умножениями.
- **Используйте TPU** для: крупномасштабного обучения на JAX (тысячи чипов), обучения с жесткими требованиями к бюджету в Google Cloud, задач, в которых доминируют матричные умножения.

## Выбор подходящего инструмента

| Задача | Лучший инструмент | Почему |
|----------|-----------|-----|
| Обучение ML (PyTorch) | NVIDIA GPU + CUDA/Triton | Самая большая экосистема, лучшие инструменты |
| Обучение ML (JAX, крупный масштаб) | TPU или NVIDIA GPU | TPU для экономии в масштабах Google, GPU для гибкости |

| Новое оборудование | SDK от вендора | У каждого ускорителя свой набор инструментов |

## Задачи по программированию (используйте CoLab с GPU-рантаймом)

1. Напишите и запустите ядро Triton для сложения векторов. Сравните его производительность со встроенным сложением в PyTorch.
```python
import triton
import triton.language as tl
import torch
import time

@triton.jit
def add_kernel(x_ptr, y_ptr, out_ptr, n, BLOCK: tl.constexpr):
    pid = tl.program_id(0)
    offs = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offs < n
    x = tl.load(x_ptr + offs, mask=mask)
    y = tl.load(y_ptr + offs, mask=mask)
    tl.store(out_ptr + offs, x + y, mask=mask)

n = 10_000_000
x = torch.randn(n, device='cuda')
y = torch.randn(n, device='cuda')

# Triton
out_triton = torch.empty_like(x)
grid = lambda meta: (triton.cdiv(n, meta['BLOCK']),)
add_kernel[grid](x, y, out_triton, n, BLOCK=1024)

# PyTorch
out_torch = x + y

# Verify correctness
assert torch.allclose(out_triton, out_torch, atol=1e-5)

# Benchmark
torch.cuda.synchronize()
start = time.time()
for _ in range(1000):
    add_kernel[grid](x, y, out_triton, n, BLOCK=1024)
torch.cuda.synchronize()
triton_time = (time.time() - start) / 1000

start = time.time()
for _ in range(1000):
    out_torch = x + y
torch.cuda.synchronize()
torch_time = (time.time() - start) / 1000

print(f"Triton:  {triton_time*1000:.3f} ms")
print(f"PyTorch: {torch_time*1000:.3f} ms")
print(f"Ratio:   {torch_time/triton_time:.2f}x")
```

2. Напишите fused-ядро Triton, которое выполняет умножение + сложение + ReLU за один проход. Сравните с тремя отдельными операциями PyTorch.
```python
import triton
import triton.language as tl
import torch
import time

@triton.jit
def fused_mul_add_relu_kernel(x_ptr, w_ptr, b_ptr, out_ptr, n, BLOCK: tl.constexpr):
    pid = tl.program_id(0)
    offs = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offs < n
    x = tl.load(x_ptr + offs, mask=mask)
    w = tl.load(w_ptr + offs, mask=mask)
    b = tl.load(b_ptr + offs, mask=mask)
    result = tl.maximum(x * w + b, 0.0)  # fused: mul + add + relu
    tl.store(out_ptr + offs, result, mask=mask)

n = 10_000_000
x = torch.randn(n, device='cuda')
w = torch.randn(n, device='cuda')
b = torch.randn(n, device='cuda')

# Fused (Triton)
out_fused = torch.empty_like(x)
grid = lambda meta: (triton.cdiv(n, meta['BLOCK']),)
fused_mul_add_relu_kernel[grid](x, w, b, out_fused, n, BLOCK=1024)

# Unfused (PyTorch)
out_unfused = torch.relu(x * w + b)

assert torch.allclose(out_fused, out_unfused, atol=1e-5)

# Benchmark
torch.cuda.synchronize()
start = time.time()
for _ in range(1000):
    fused_mul_add_relu_kernel[grid](x, w, b, out_fused, n, BLOCK=1024)
torch.cuda.synchronize()
fused_time = (time.time() - start) / 1000

start = time.time()
for _ in range(1000):
    out_unfused = torch.relu(x * w + b)
torch.cuda.synchronize()
unfused_time = (time.time() - start) / 1000

print(f"Fused (Triton):    {fused_time*1000:.3f} ms")
print(f"Unfused (PyTorch): {unfused_time*1000:.3f} ms")
print(f"Speedup:           {unfused_time/fused_time:.2f}x")
```

3. Измерьте, как компилятор XLA в JAX автоматически выполняет слияние (fusion) операций. Сравните цепочку операций с использованием `jit` и без него.
```python
import jax
import jax.numpy as jnp
import time

def chain_ops(x):
    x = x * 2.0
    x = x + 1.0
    x = jnp.maximum(x, 0.0)  # ReLU
    x = x / jnp.sum(x)
    return x

chain_jit = jax.jit(chain_ops)
x = jax.random.normal(jax.random.PRNGKey(0), (10000, 1000))

# Warm up
_ = chain_jit(x)
jax.block_until_ready(_)

# Eager (each op is a separate kernel launch)
start = time.time()
for _ in range(100):
    y = chain_ops(x)
jax.block_until_ready(y)
eager_time = (time.time() - start) / 100

# JIT (XLA fuses operations)
start = time.time()
for _ in range(100):
    y = chain_jit(x)
jax.block_until_ready(y)
jit_time = (time.time() - start) / 100

print(f"Eager: {eager_time*1000:.2f} ms")
```
