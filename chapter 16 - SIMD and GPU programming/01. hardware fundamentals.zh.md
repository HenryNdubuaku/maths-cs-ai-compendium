# 硬件基础

*在写 SIMD 或 GPU 代码之前，你需要理解你所编程的硬件。本文件讲述为什么并行化取代了时钟频率、现代 CPU 如何执行指令、SIMD 是什么、用于推理性能的 roofline 模型，以及芯片架构的全景。*

- 几十年来，软件变快是免费的：买一块时钟频率更高的新 CPU，你的程序不改一行代码就跑得更快。这个时代在 2005 年前后结束。理解它为什么结束、又是什么取代了它，对任何想写出快速代码的人都至关重要。

## 免费性能的终结

- **摩尔定律**（1965）观察到，一块芯片上的晶体管数量大约每两年翻一倍。这一定律持续了 60 年。晶体管越多意味着晶体管越小，晶体管越小意味着时钟频率越高，频率越高意味着程序越快。

- 但在 2005 年前后，时钟频率撞上了 ~4 GHz 的墙。问题出在**功耗**。一块芯片的功耗大致为：

$$P \propto C \cdot V^2 \cdot f$$

- 其中 $C$ 是电容（与晶体管数量成正比），$V$ 是电压，$f$ 是时钟频率。要提高频率，就必须提高电压（以更快地切换晶体管）。但功耗随 $V^2 \cdot f$ 增长，因此频率的少量提升会带来功耗（和发热）的大幅上升。到 4 GHz 时，芯片已经冲到 100 瓦以上。要到 8 GHz 需要无法实现的散热。

- 解决办法：与其让一个核心更快，不如在同一块芯片上放**多个核心**。一块 3 GHz 的 4 核芯片功耗与一块 4.5 GHz 单核芯片相近，但能做 4 倍的并行工作。这就是为什么每块现代 CPU 都有多个核心，也是为什么并行化（SIMD、多线程、GPU 计算）成为获取更高性能的唯一途径。

- **对 ML 的启示**：一个在单核上要跑 10 分钟的训练步骤，无法靠买更快的 CPU 来加速。只能靠使用更多核心（数据并行，第 6 章）、更宽的 SIMD 单元（本章）或 GPU（数千核心）来加速。

## 现代 CPU 如何执行指令

- 现代的一个 CPU core 远比第 13 章那个简单的 fetch-decode-execute 模型复杂。它用若干技巧在每个周期内执行更多指令：

- **超标量执行（Superscalar execution）**：CPU 拥有多个执行单元（ALU、FPU、load/store 单元），可以同时执行多条独立的指令。一个现代 core 如果指令之间互不依赖，每个周期可能执行 4-6 条指令。

- **乱序执行（OoO）**：CPU 不按程序顺序执行指令。它向前查看指令流，找到输入已就绪的指令并立即执行，而不管它们的位置。这隐藏了延迟：当一条指令在等内存数据（100+ 周期）时，CPU 执行其他已就绪的指令。

- **分支预测**：条件分支（`if` 语句、循环条件）带来不确定性：在条件求值之前，CPU 不知道走哪条路径。CPU 不会停顿，而是**预测**结果并沿预测路径推测执行。如果预测正确（现代预测器 >95% 正确），就没有时间损失。如果预测错误，推测的工作被丢弃，正确路径被执行（约 15 周期惩罚）。

- **推测执行**：分支预测的延伸。CPU 执行那些*可能*并不需要的指令，赌它们会被需要。这填满流水线，让执行单元保持忙碌。

- 以上这些全部是自动的——CPU 无需程序员介入就完成它们。但它们只对**指令级并行**（ILP）有帮助：单一指令流内部的独立指令。对于**数据级并行**（对许多数据元素做同一操作），我们需要 SIMD。

## SIMD：单指令多数据

- **SIMD** 的思路是用一条指令同时处理多个数据元素。不是相加两个数，而是一次相加两个各含 4（或 8、或 16）个数的向量。

- 不用 SIMD（标量）：

```cpp
// 逐元素相加两个数组：4 条 add 指令
for (int i = 0; i < 4; i++) {
    c[i] = a[i] + b[i];  // 每次迭代一条 add
}
```

- 用 SIMD（向量化）：

```cpp
// 相加两个数组：1 条 SIMD 指令完成全部 4 次加法
#include <immintrin.h>  // x86 SIMD intrinsics

__m128 va = _mm_load_ps(a);    // 把 4 个 float 装入一个 128 位 register
__m128 vb = _mm_load_ps(b);    // 把 4 个 float 装入另一个 register
__m128 vc = _mm_add_ps(va, vb); // 同时相加全部 4 对
_mm_store_ps(c, vc);            // 存回 4 个结果
```

- SIMD 版本以 1/4 的指令数完成同样的工作。这是 4 倍的理论加速，靠的是每条指令处理 4 个 float 而不是 1 个。

### 向量 register

- SIMD 指令操作的是**向量 register**：容纳多个数据元素的宽 register。

| Register 宽度 | Float（32 位） | Double（64 位） | 名称 |
|----------------|-----------------|-------------------|------|
| 128 位 | 4 | 2 | SSE（x86）、NEON（ARM） |
| 256 位 | 8 | 4 | AVX/AVX2（x86） |
| 512 位 | 16 | 8 | AVX-512（x86） |
| 可变（128-2048） | 可变 | 可变 | SVE/SVE2（ARM） |

- register 越宽 = 并行度越高。一条 512 位的 AVX-512 指令一次处理 16 个 float，相比标量代码理论上有 16 倍加速。实际加速更低，因为受内存带宽限制（你的计算速度赶不上向 CPU 喂数据的速度）。

- 对 ML 而言：float32 值的矩阵乘法从 SIMD 获益极大。内层循环（两个向量的点积）直接映射到 SIMD 的 multiply-accumulate 指令。这就是为什么 BLAS 库（NumPy 和 PyTorch 调用的对象）用 SIMD 做了重度优化。

## Roofline 模型

- 怎么知道你的代码够快？**roofline 模型**提供了一个框架，用两个硬件极限来刻画性能：

1. **峰值算力**（FLOPS）：每秒最大浮点运算次数。对一块 4 GHz、带 256 位 AVX（每条指令 8 个 float）、有 2 个 FMA 单元的 CPU：$4 \times 10^9 \times 8 \times 2 = 64$ GFLOPS。

2. **峰值内存带宽**（字节/秒）：数据从内存搬到 CPU 的速度。一块现代 CPU 可能有 50 GB/s 的内存带宽。

- 你代码的**算术强度**（arithmetic intensity）是计算量与内存访问量的比值：

$$\text{Arithmetic Intensity} = \frac{\text{FLOPS}}{\text{Bytes transferred}}$$

- 如果算术强度低（每加载一个字节只做很少操作），你的代码是 **memory-bound**：大部分时间在等数据。让计算变快（更宽的 SIMD、更高时钟）无济于事。

- 如果算术强度高（每个字节做很多操作），你的代码是 **compute-bound**：大部分时间在计算。更快的内存无济于事。

- roofline：

$$\text{Achievable FLOPS} = \min\left(\text{Peak FLOPS}, \; \text{Bandwidth} \times \text{Arithmetic Intensity}\right)$$

- **矩阵乘法**算术强度高：$O(n^3)$ 次操作作用于 $O(n^2)$ 数据，强度约 $O(n)$。对大矩阵而言是 compute-bound。这就是为什么 GPU（高算力）在矩阵密集的 ML 工作负载上占据主导。

- **逐元素操作**（ReLU、add、multiply）算术强度低：每加载一个元素做 1 次操作。这些是 memory-bound。让 GPU 更快没用；你需要更快的内存（或把这些操作与算力密集的操作融合，以避免单独的内存往返）。

- roofline 模型解释了为什么 **kernel fusion** 如此重要：把一个 matmul 加上 bias 再加 ReLU 组合成单个 kernel，避免把中间结果写回内存再读回来，把三个 memory-bound 操作变成一个 compute-bound 操作。

## latency 与 throughput

- **latency** 是完成一次操作的时间。**throughput** 是单位时间内完成的操作数。

- 一个类比：一辆公交车 latency 高（每个站都要停）但 throughput 高（一次载 50 人）。一辆出租车 latency 低（直达目的地）但 throughput 低（载 1-4 人）。

- GPU 是公交车：单次操作 latency 高（每条指令要很多周期才能完成）但 throughput 极大（数千核心同时处理）。CPU 是出租车：latency 低（乱序执行、分支预测、深 cache 让延迟最小）但 throughput 有限（4-64 个核心）。

- 这就是为什么 GPU 更适合 ML 训练（throughput 重要：处理百万级样本），而 CPU 更适合 OS 任务（latency 重要：对一次按键立即响应）。

- **流水线**把 latency 转成 throughput。如果一条指令要 5 周期，但流水线每个周期启动一条新指令，throughput 就是每周期 1 条指令（尽管每条仍需 5 周期完成）。这与第 13 章的 CPU 流水线原理相同，但它适用于每一层：SIMD 单元、内存控制器、GPU 核心都是流水的。

## 芯片架构全景

- 你为之写代码的硬件决定了可用的 SIMD 指令：

### x86（Intel、AMD）

- 在桌面、笔记本和数据中心 CPU 中占主导。SIMD：SSE（128 位）、AVX/AVX2（256 位）、AVX-512（512 位）。Intel AMX 为 AI 工作负载提供专用的矩阵乘法单元。

- **优势**：单核性能最高、SIMD 最宽、软件生态成熟（MKL、oneDNN）。
- **劣势**：功耗高、指令集复杂、价格贵。

### ARM

- 在移动端占主导（每一部智能手机），在服务器（AWS Graviton、Ampere Altra）和笔记本（Apple M 系列）中增长。SIMD：NEON（128 位）、SVE/SVE2（可扩展，128-2048 位）。

- **优势**：极佳的能效（每瓦性能）、自定义核心（Apple M4 在单核性能上接近 Intel，功耗却只占一小部分）。
- **劣势**：SIMD 较窄（NEON 仅 128 位，不过 SVE 可以更宽）、HPC 软件生态较小。

### Apple Silicon（M1/M2/M3/M4）

- 基于 ARM 并有自定义扩展。包含 **AMX**（Apple Matrix eXtensions）——未公开文档的矩阵乘法单元，Accelerate 框架用它做 BLAS 运算。统一内存架构：CPU 和 GPU 共享同一物理内存，消除了 CPU↔GPU 拷贝瓶颈。

- **对 ML**：Apple 的 Neural Engine（16 核，专用 ML 加速器）和统一内存使 M 系列芯片在本地 ML 推理和小规模训练上出乎意料地能干。但没有 CUDA：你必须用 Metal（Apple 的 GPU API）或 MLX（Apple 的 ML 框架）。

### RISC-V

- 开源 ISA。无授权费（与 ARM 不同）。在嵌入式系统、IoT 和研究中增长。SIMD："V"（vector）扩展提供类似 ARM SVE 的可扩展向量处理。

- **对 ML**：在 ML 工作负载上尚不能与 x86/ARM 竞争，但值得关注。若干 AI 加速器初创公司使用 RISC-V 核心。

### GPU（NVIDIA、AMD、Intel）

- 在文件 04-05 中深入讨论。数千个简单核心，针对 throughput 优化。NVIDIA 靠 CUDA 在 ML 领域占主导；AMD 用 ROCm 竞争；Intel 用 Arc GPU 和 Gaudi 加速器入场。

### TPU（Google）

- 为 ML 专门设计的 ASIC。为矩阵乘法优化的 systolic array。在文件 05 中讨论。

## 散热与功耗约束

- 性能最终受功耗和散热限制：

- **TDP**（Thermal Design Power，热设计功耗）：一块芯片可持续消耗的最大功率。笔记本 CPU 的 TDP 可能是 15W；服务器 CPU 250W；数据中心 GPU 700W（NVIDIA B200）。

- **暗硅**（dark silicon）：任何时刻都有一部分晶体管必须断电，才能不超出散热预算。一块芯片理论上可以同时启用所有晶体管，但那样会烧毁。

- **能效**（FLOPS/瓦）越来越成为关键指标，而非原始 FLOPS。这就是为什么：
    - ARM 正在接管数据中心（FLOPS/瓦优于 x86）。
    - TPU 尽管峰值 FLOPS 更低，却能和 GPU 竞争（在 ML 工作负载上 FLOPS/瓦好得多）。
    - 量化（INT8、FP8）不仅关乎内存：它也降低了每次操作的功耗。

- 对大规模 ML：训练一个前沿 LLM 要消耗数兆瓦功率、持续数月。电费可能超过硬件成本。能效直接影响 AI 研究的经济学。

## 实践：在 C++ 中测量性能

- 要推理性能，你得先测量它。下面是一套最小的 C++ benchmark 设置：

```cpp
#include <iostream>
#include <chrono>
#include <vector>

// 标量加法
void add_scalar(const float* a, const float* b, float* c, int n) {
    for (int i = 0; i < n; i++) {
        c[i] = a[i] + b[i];
    }
}

int main() {
    const int N = 1 << 24;  // 约 1600 万元素
    std::vector<float> a(N, 1.0f), b(N, 2.0f), c(N);

    // 预热（填满 cache、触发频率调节）
    add_scalar(a.data(), b.data(), c.data(), N);

    // benchmark
    auto start = std::chrono::high_resolution_clock::now();

    for (int trial = 0; trial < 100; trial++) {
        add_scalar(a.data(), b.data(), c.data(), N);
    }

    auto end = std::chrono::high_resolution_clock::now();
    double elapsed = std::chrono::duration<double>(end - start).count();

    double total_bytes = 3.0 * N * sizeof(float) * 100;  // 读 a、读 b、写 c
    double bandwidth = total_bytes / elapsed / 1e9;        // GB/s

    std::cout << "Time: " << elapsed << " s\n";
    std::cout << "Bandwidth: " << bandwidth << " GB/s\n";

    return 0;
}
```

```bash
# 带优化编译
g++ -O3 -march=native -o bench bench.cpp
./bench
```

- **本代码中的 C++ 关键概念**：
    - `#include <vector>`：动态数组（`std::vector<float>`）——类似 Python 的 `list`，但有类型且在内存中连续。
    - `a.data()`：返回指向底层数组的裸指针（`float*`）——SIMD intrinsics 需要它。
    - `std::chrono`：用于 benchmark 的高分辨率计时器。
    - `-O3`：最高编译优化级别。编译器可能会自动向量化你的循环（自动使用 SIMD）。`-march=native` 启用你的 CPU 支持的所有 SIMD 指令。

- **为什么要预热**：第一次运行会填满 cache，并可能触发 CPU 频率调节（turbo boost）。后续运行更具代表性。

- **为什么要测带宽**：对 memory-bound 操作（如逐元素 add），有意义的指标是带宽（GB/s），而不是 FLOPS。如果你测得的带宽接近硬件极限（DDR5 约 50 GB/s），你就是 memory-bound，SIMD 帮助不大（瓶颈在内存，不在算力）。

1. 计算常见 ML 操作的算术强度，并把它们归类为 memory-bound 或 compute-bound。
```python
import jax.numpy as jnp

def arithmetic_intensity(flops, bytes_transferred):
    return flops / bytes_transferred

# 逐元素 ReLU：每个元素 1 次比较，读 + 写
n = 1024
relu_flops = n  # 每个元素 1 次操作
relu_bytes = 2 * n * 4  # 读输入 + 写输出（float32）
print(f"ReLU: {arithmetic_intensity(relu_flops, relu_bytes):.2f} FLOPS/byte → memory-bound")

# 矩阵乘法：2*n^3 次操作，读 2*n^2 + 写 n^2 个 float
matmul_flops = 2 * n**3
matmul_bytes = 3 * n**2 * 4  # 读 A + 读 B + 写 C
print(f"Matmul ({n}×{n}): {arithmetic_intensity(matmul_flops, matmul_bytes):.0f} FLOPS/byte → compute-bound")

# Layer norm：约 5n 次操作（mean、var、normalise），读 + 写
ln_flops = 5 * n
ln_bytes = 2 * n * 4
print(f"LayerNorm: {arithmetic_intensity(ln_flops, ln_bytes):.2f} FLOPS/byte → memory-bound")

# 3x3 卷积：2*9*C_in*C_out*H*W，读 kernel + feature map + 写输出
C_in, C_out, H, W = 64, 128, 32, 32
conv_flops = 2 * 9 * C_in * C_out * H * W
conv_bytes = (9 * C_in * C_out + C_in * H * W + C_out * H * W) * 4
print(f"Conv3x3: {arithmetic_intensity(conv_flops, conv_bytes):.0f} FLOPS/byte → compute-bound")
```

2. 演示为什么并行化很重要。比较串行与并行（NumPy）执行随数据规模增长的表现。
```python
import numpy as np
import time

for n in [1000, 10000, 100000, 1000000, 10000000]:
    a = np.random.randn(n).astype(np.float32)
    b = np.random.randn(n).astype(np.float32)

    # "串行"（Python 循环）
    start = time.time()
    c = [a[i] * b[i] for i in range(min(n, 100000))]  # 为省事限制在 10 万以内
    seq_time = time.time() - start
    if n > 100000:
        seq_time *= n / 100000  # 外推

    # "并行"（NumPy，内部用 SIMD + 多线程）
    start = time.time()
    c = a * b
    par_time = time.time() - start

    print(f"n={n:>10,}  sequential={seq_time:.4f}s  parallel={par_time:.6f}s  "
          f"speedup={seq_time/par_time:.0f}x")
```
