# RISC-V и встраиваемые системы

*RISC-V — это архитектура набора команд с открытым исходным кодом, меняющая индустрию микросхем. В этом файле рассматриваются философия RISC-V, векторное расширение V, инференс машинного обучения во встраиваемых системах, TinyML на микроконтроллерах, RISC-V в ускорителях ИИ и ограничения при развёртывании на периферии.*

- Каждая архитектура чипов, которую мы рассматривали до сих пор (x86, ARM), требует **лицензии**. Intel и AMD платят за x86. Apple, Qualcomm и каждый производитель смартфонов платят ARM миллиарды в год. **RISC-V** — другая: это открытый стандарт. Любой может проектировать, производить и продавать чипы RISC-V, не выплачивая никому роялти. Это меняет экономику проектирования чипов, особенно для ИИ.

## Философия RISC-V

- RISC-V (произносится «риск файв») была создана в Калифорнийском университете в Беркли в 2010 году как чистый, современный набор команд RISC. Ключевые принципы:

    - **Открытый стандарт**: спецификация ISA доступна бесплатно. Вы можете создать CPU RISC-V без лицензионных сборов, соглашений о неразглашении (NDA) или юридических договоров. Это похоже на то, чем является Linux для операционных систем — любой может использовать, изменять и строить на его основе.

    - **Модульный дизайн**: базовая ISA (RV32I или RV64I) минимальна — всего 47 инструкций. Всё остальное — это опциональные **расширения**: M (умножение/деление), A (атомарные операции), F/D (числа с плавающей запятой), C (сжатые инструкции), V (векторная обработка). Вы выбираете только то, что нужно, сохраняя чип компактным и эффективным.

    - **Отсутствие «наследия»**: x86 несет в себе 45 лет обратной совместимости. ARM — 35 лет. RISC-V начинает с чистого листа, учитывая уроки обеих архитектур. Никаких неясных инструкций, существующих только для совместимости с программным обеспечением 1980-х годов.

- **Кто использует RISC-V**: SiFive (ядра общего назначения), Alibaba (серверные ядра Xuantie), Western Digital (контроллеры накопителей, отгружены миллиарды), Espressif (ESP32-C3, популярный чип для IoT) и десятки стартапов в области ИИ-ускорителей, которые используют RISC-V в качестве управляющего процессора для своих специализированных вычислительных блоков.

## Базовая архитектура RISC-V

- Базовая целочисленная ISA (RV64I для 64-битной версии) имеет:
    - **32 регистра общего назначения** (x0-x31, каждый по 64 бита). x0 аппаратно привязан к нулю (полезно для реализации типичных паттернов без специальных инструкций).
    - **Фиксированную ширину инструкции 32 бита** (с расширением C, добавляющим 16-битные сжатые инструкции для плотности кода).
    - **Архитектуру load-store**: как и в ARM, арифметические операции выполняются только над регистрами. Доступ к памяти осуществляется через явные инструкции загрузки/сохранения (load/store).

```
# RISC-V assembly (for flavour — you will use C/C++)
add  x3, x1, x2      # x3 = x1 + x2
lw   x4, 0(x5)       # load word from address in x5
sw   x4, 8(x5)       # store word to address x5 + 8
beq  x1, x2, label   # branch if x1 == x2
```

- Простота ISA делает ядра RISC-V компактными и энергоэффективными. Минимальное ядро RV32I может быть реализовано примерно на 10 000 вентилях (ARM Cortex-M0 — около 12 000). Это важно для встраиваемых систем, где на счету каждый милливатт и каждый квадратный миллиметр кремния.

## Расширение V: векторная обработка в RISC-V

- **Расширение V** (RVV) добавляет масштабируемую векторную обработку в RISC-V, подобно ARM SVE. Векторные регистры имеют настраиваемую длину (VLEN), определяемую оборудованием (от 128 до 65 536 бит). Код пишется так, чтобы быть **независимым от длины вектора** (vector-length agnostic): он работает при любом VLEN без перекомпиляции.

```c
#include <riscv_vector.h>

// Vector addition using RVV intrinsics
void vadd_rvv(const float* a, const float* b, float* c, int n) {
    while (n > 0) {
        // vsetvl: set vector length — processes min(n, hardware_max) elements
        size_t vl = __riscv_vsetvl_e32m1(n);

        // Load vl elements
        vfloat32m1_t va = __riscv_vle32_v_f32m1(a, vl);
        vfloat32m1_t vb = __riscv_vle32_v_f32m1(b, vl);

        // Add
        vfloat32m1_t vc = __riscv_vfadd_vv_f32m1(va, vb, vl);

        // Store
        __riscv_vse32_v_f32m1(c, vc, vl);

        // Advance pointers
        a += vl; b += vl; c += vl; n -= vl;
    }
}
```

- **`vsetvl`** — ключевая инструкция. Она сообщает оборудованию: «Я хочу обработать столько-то элементов», а оборудование отвечает: «Я могу обработать столько-то» (с ограничением по VLEN). Цикл автоматически адаптируется к любой ширине вектора, без необходимости скалярной «очистки» (последняя итерация просто обрабатывает меньше элементов).

- **LMUL** (множитель длины): RVV может группировать несколько векторных регистров вместе (m1, m2, m4, m8) для обработки большего количества элементов за инструкцию ценой уменьшения количества доступных регистров. `m1` использует один регистр на векторный операнд; `m8` использует восемь, обрабатывая в 8 раз больше элементов, но оставляя доступными только 4 группы регистров.

- По сравнению с x86 AVX (фиксированные 256/512 бит) и ARM NEON (фиксированные 128 бит), масштабируемость RVV является важным преимуществом для разнообразного оборудования: один и тот же код работает как на крошечном встраиваемом ядре (VLEN=128), так и на высокопроизводительном серверном ядре (VLEN=1024+).

## Встраиваемое машинное обучение: TinyML

- **TinyML** — это машинное обучение на микроконтроллерах: устройствах с килобайтами оперативной памяти, процессорами мегагерцового класса и бюджетом мощности в милливатты. Представьте: датчик, распознающий ключевые слова («Привет, Siri»), акселерометр, классифицирующий жесты, или камера, считающая людей — всё это работает на чипе стоимостью $0.50 без подключения к интернету.

- Ограничения экстремальны:

| Ресурс | Серверный GPU | Смартфон | Микроконтроллер |
|----------|-----------|------------|-----------------|
| RAM | 80 GB | 6 GB | 256 KB |
| Память | TB | 128 GB | 1 MB |
| Вычисления | 1000 TFLOPS | 10 TFLOPS | 0.001 TFLOPS |
| Мощность | 700 W | 5 W | 0.001 W |
| Стоимость | $30,000 | $500 | $1 |

- Модель, которая помещается на серверном GPU ($O(10^{10})$ параметров), не поместится на микроконтроллере. Модели TinyML имеют $O(10^4)$–$O(10^6)$ параметров и используют INT8 или даже INT4 квантование.

### TensorFlow Lite Micro (TFLM)

- **TFLM** — это фреймворк Google для инференса на микроконтроллерах. Он запускает квантованные модели TensorFlow Lite без динамического выделения памяти, без ОС и с размером бинарного файла около 20 КБ.

```cpp
// TinyML inference on a microcontroller (simplified)
#include "tensorflow/lite/micro/micro_interpreter.h"
#include "tensorflow/lite/micro/micro_mutable_op_resolver.h"

// Model is compiled into a C array (const unsigned char model_data[])
const tflite::Model* model = tflite::GetModel(model_data);

// Allocate a fixed memory arena (no malloc!)
constexpr int kArenaSize = 10 * 1024;  // 10 KB
uint8_t tensor_arena[kArenaSize];

// Set up interpreter
tflite::MicroInterpreter interpreter(model, resolver, tensor_arena, kArenaSize);
interpreter.AllocateTensors();

// Set input
float* input = interpreter.input(0)->data.f;
input[0] = sensor_reading;

// Run inference
interpreter.Invoke();

// Read output
float* output = interpreter.output(0)->data.f;
if (output[0] > 0.8f) {
    trigger_alert();
}
```

- **Ключевые ограничения в этом коде**:
    - `tensor_arena` распределяется статически — никакого `malloc`, никакой кучи. Встраиваемые системы (embedded systems) часто не имеют динамического распределителя памяти.
    - Модель представляет собой `const` массив байтов, хранящийся во флеш-памяти (ROM), а не загружаемый из файловой системы.
    - Весь фреймворк + модель + среда выполнения занимают всего несколько десятков КБ.

### Оптимизация моделей для периферийных устройств (Edge)

- Чтобы запустить модель на микроконтроллере, требуется агрессивная оптимизация:

    - **Квантование** (глава 18): преобразование весов из формата float32 в INT8 (в 4 раза меньше, в 2–4 раза быстрее на оборудовании, работающем только с целыми числами). Посттренировочное квантование выполняется просто; квантование с учетом обучения (quantisation-aware training) позволяет сохранить больше точности.

    - **Прунинг**: удаление весов, близких к нулю. Структурный прунинг (удаление целых каналов/голов внимания) более удобен для аппаратного обеспечения, чем неструктурный прунинг (случайные нули), поскольку он сокращает реальные вычисления, а не только объем хранимых данных.

    - **Дистилляция знаний** (глава 6): обучение небольшой модели-"ученика" имитации большой модели-"учителя". Ученик достигает более высокой точности, чем при обучении с нуля, поскольку он учится на мягких предсказаниях учителя.

    - **Поиск нейронных архитектур (NAS)**: автоматический поиск эффективных архитектур, которые укладываются в аппаратный бюджет (задержка, память, энергопотребление). **MicroNets** и **MCUNet** находят архитектуры, оптимизированные для конкретных микроконтроллеров.

    - **Слияние операторов**: объединение свёртки + пакетной нормализации + ReLU в одну объединенную операцию, что исключает промежуточные записи в память (тот же принцип, что и слияние ядер GPU, но еще более критичный, когда у вас всего 256 КБ оперативной памяти).

## RISC-V в ускорителях ИИ

- Многие стартапы в области ИИ-ускорителей используют RISC-V не для прямого запуска моделей машинного обучения, а в качестве **процессора управления**, который управляет специализированными вычислительными блоками:

```
┌─────────────────────────────────────────┐
│              AI Accelerator             │
│                                         │
│  ┌──────────┐    ┌──────────────────┐   │
│  │  RISC-V  │───→│  Custom Matrix   │   │
│  │  Control │    │  Multiply Unit   │   │
│  │  Core    │    │  (systolic array,│   │
│  │          │    │  custom dataflow)│   │
│  └──────────┘    └──────────────────┘   │
│       │                    │            │
│       ▼                    ▼            │
│  ┌──────────┐    ┌──────────────────┐   │
│  │  Memory  │    │  On-chip SRAM    │   │
│  │  Control │    │  (activation     │   │
│  │          │    │   buffer)        │   │
│  └──────────┘    └──────────────────┘   │
└─────────────────────────────────────────┘
```

- Ядро RISC-V отвечает за: загрузку весов модели из внешней памяти, планирование выполнения слоев, управление потоком данных между вычислительными блоками и связь с хостом (через PCIe, USB или SPI). Тяжелые вычисления (матричные умножения, свёртки) выполняются специализированным оборудованием, а не ядром RISC-V.

- **Почему RISC-V для управления**: отсутствие лицензионных отчислений (критично для стартапов), возможность настройки (добавление специализированных инструкций), малый размер (управляющему ядру не нужна сложность x86), а открытая экосистема позволяет быстро создавать прототипы.

- **Примеры**: Esperanto Technologies (1000+ ядер RISC-V для ML), Tenstorrent (управление RISC-V + специализированные ядра tensix), SiFive (ядра RISC-V с векторными расширениями для периферийного ML).

## Ограничения при развертывании на периферии

- Развертывание машинного обучения на периферии (на устройстве, а не в облаке) вводит ограничения, с которыми не сталкивается облачное развертывание:

- **Энергопотребление**: устройство с питанием от батареи может иметь общий бюджет мощности 100 мВт. Запуск модели, потребляющей 50 мВт, оставляет всего 50 мВт для остальной части системы (датчики, радиомодуль, дисплей). Инференс с учетом энергопотребления планирует вычисления так, чтобы избежать теплового троттлинга и продлить срок службы батареи.

- **Задержка**: периферийный инференс часто должен быть в реальном времени. Детектор ключевых слов ("Hey Siri") должен реагировать в течение ~200 мс. Система восприятия автономного вождения (глава 11) должна обрабатывать кадры в течение ~30 мс. Сетевой запрос к облаку (50–200 мс) слишком медленный для таких задач.

- **Конфиденциальность**: обработка данных на устройстве означает, что конфиденциальные данные (медицинские изображения, записи голоса, личные фотографии) никогда не покидают устройство. Это является юридическим требованием в некоторых юрисдикциях (GDPR) и требованием доверия пользователей везде.

- **Связь**: периферийные устройства могут иметь прерывистое интернет-соединение или не иметь его вовсе. Модель, работающая на марсоходе (глава 11), подводной лодке или датчике на сельской ферме, должна работать полностью автономно.

- **Стоимость при масштабировании**: развертывание ML на миллиарде смартфонов стоит $0 на устройство (оборудование уже существует). Развертывание на миллиарде IoT-датчиков означает, что бюджет на ML-оборудование для каждого датчика составляет копейки. Нулевая стоимость лицензирования RISC-V имеет огромное значение в таких масштабах.

## Задачи по программированию (компилировать с помощью g++ или кросс-компилятора riscv64-gcc)

1. Напишите программу на C, которая имитирует пайплайн инференса TinyML: статически выделите буфер модели, выполните имитацию прямого прохода и измерьте использование ресурсов. Это учит учитывать встраиваемые ограничения (никакого malloc, фиксированная область памяти).
```cpp
// task1_tinyml_sim.cpp
// Compile: g++ -O2 -o task1 task1_tinyml_sim.cpp

#include <iostream>
#include <chrono>
#include <cmath>
#include <cstring>

// Simulate a microcontroller: fixed memory arena, no dynamic allocation
static constexpr int ARENA_SIZE = 32 * 1024;  // 32 KB total RAM budget
static uint8_t arena[ARENA_SIZE];

// Simple 2-layer MLP: 784 -> 64 -> 10 (MNIST-like, INT8 weights)
struct TinyModel {
    int8_t w1[784 * 64];      // layer 1 weights: 50,176 bytes
    int8_t b1[64];             // layer 1 biases
    int8_t w2[64 * 10];       // layer 2 weights: 640 bytes
    int8_t b2[10];             // layer 2 biases
    // Total: ~51 KB → must go in flash (ROM), not RAM
};

// Check if model fits in flash
void check_model_fit(int flash_kb) {
    int model_bytes = sizeof(TinyModel);
    std::cout << "Model size: " << model_bytes << " bytes ("
              << model_bytes / 1024 << " KB)\n";
    std::cout << "Flash: " << flash_kb << " KB → "
              << (model_bytes <= flash_kb * 1024 ? "FITS" : "TOO LARGE") << "\n";
}

// Mock inference using the fixed arena for activations
void mock_inference(const int8_t* input, int8_t* output) {
    // Activations go in the arena (RAM), not allocated dynamically
    int8_t* act1 = (int8_t*)arena;            // 64 bytes for layer 1 output
    int8_t* act2 = (int8_t*)(arena + 64);     // 10 bytes for layer 2 output

    // Layer 1: simplified matmul (not real quantised matmul, just structure demo)
    for (int j = 0; j < 64; j++) {
        int32_t sum = 0;  // accumulate in int32 to avoid overflow
        for (int i = 0; i < 784; i++) {
            sum += (int32_t)input[i] * 1;  // mock: weight = 1
        }
        act1[j] = (int8_t)std::max(-128, std::min(127, sum / 784));  // quantise back
        act1[j] = act1[j] > 0 ? act1[j] : 0;  // ReLU
    }

    // Layer 2
    for (int j = 0; j < 10; j++) {
        int32_t sum = 0;
        for (int i = 0; i < 64; i++) {
            sum += (int32_t)act1[i] * 1;
        }
        act2[j] = (int8_t)std::max(-128, std::min(127, sum / 64));
    }

    std::memcpy(output, act2, 10);
}

int main() {
    std::cout << "=== TinyML Resource Budget ===\n";
    std::cout << "Arena (RAM): " << ARENA_SIZE << " bytes ("
              << ARENA_SIZE / 1024 << " KB)\n";
    check_model_fit(256);  // typical MCU flash

    // Activation memory used
    int activation_bytes = 64 + 10;  // layer 1 + layer 2 outputs
    std::cout << "Activation memory: " << activation_bytes
              << " bytes / " << ARENA_SIZE << " available\n\n";

    // Benchmark inference
    int8_t input[784];
    int8_t output[10];
    std::memset(input, 1, 784);

    auto start = std::chrono::high_resolution_clock::now();
    for (int i = 0; i < 10000; i++) {
        mock_inference(input, output);
    }
    auto end = std::chrono::high_resolution_clock::now();
    double us = std::chrono::duration<double, std::micro>(end - start).count() / 10000;

    std::cout << "Inference latency: " << us << " us\n";
    std::cout << "At 160 MHz MCU (~6.25 ns/cycle): ~"
              << (int)(us * 160) << " cycles\n";

    std::cout << "Output logits: ";
    for (int i = 0; i < 10; i++) std::cout << (int)output[i] << " ";
    std::cout << "\n";

    return 0;
}
```

2. Напишите программу на C++, которая квантует веса `float32` в `INT8` и измеряет коэффициент сжатия и ошибку квантования.

```cpp
// task2_quantise.cpp
// Compile: g++ -O3 -o task2 task2_quantise.cpp

#include <iostream>
#include <vector>
#include <cmath>
#include <algorithm>
#include <numeric>

// Symmetric quantisation: map float range [-max, +max] to [-127, +127]
void quantise_symmetric(const float* input, int8_t* output, int n, float& scale) {
    float max_val = 0.0f;
    for (int i = 0; i < n; i++) {
        max_val = std::max(max_val, std::abs(input[i]));
    }
    scale = max_val / 127.0f;
    for (int i = 0; i < n; i++) {
        float scaled = input[i] / scale;
        output[i] = (int8_t)std::max(-127.0f, std::min(127.0f, std::round(scaled)));
    }
}

// Dequantise: INT8 back to float
void dequantise(const int8_t* input, float* output, int n, float scale) {
    for (int i = 0; i < n; i++) {
        output[i] = (float)input[i] * scale;
    }
}

int main() {
    const int N = 100000;

    // Simulate random weights (roughly normal distribution)
    std::vector<float> weights(N);
    for (int i = 0; i < N; i++) {
        // Simple pseudo-random normal-ish values
        float u1 = (float)(i * 7 % 997 + 1) / 998.0f;
        float u2 = (float)(i * 13 % 991 + 1) / 992.0f;
        weights[i] = std::sqrt(-2.0f * std::log(u1)) * std::cos(6.2832f * u2) * 0.1f;
    }

    // Quantise
    std::vector<int8_t> quantised(N);
    float scale;
    quantise_symmetric(weights.data(), quantised.data(), N, scale);

    // Dequantise and measure error
    std::vector<float> reconstructed(N);
    dequantise(quantised.data(), reconstructed.data(), N, scale);

    float max_error = 0.0f, total_error = 0.0f;
    for (int i = 0; i < N; i++) {
        float err = std::abs(weights[i] - reconstructed[i]);
        max_error = std::max(max_error, err);
        total_error += err;
    }

    std::cout << "=== Quantisation Results ===\n";
    std::cout << "Original:    " << N * 4 << " bytes (float32)\n";
    std::cout << "Quantised:   " << N * 1 << " bytes (int8) + 4 bytes (scale)\n";
    std::cout << "Compression: " << 4.0f << "x\n";
    std::cout << "Scale factor: " << scale << "\n";
    std::cout << "Mean abs error: " << total_error / N << "\n";
    std::cout << "Max abs error:  " << max_error << "\n";
    std::cout << "Max abs error / scale: " << max_error / scale
              << " (should be <= 0.5 quantisation levels)\n";

    return 0;
}
```

3. Напишите программу на C++, которая выполняет матричное умножение `INT8` с накоплением `INT32` — это реальные вычисления, выполняемые на встроенных ускорителях машинного обучения.

```cpp
// task3_int8_matmul.cpp
// Compile: g++ -O3 -o task3 task3_int8_matmul.cpp

#include <iostream>
#include <chrono>
#include <vector>
#include <cstdint>

// INT8 matmul with INT32 accumulation (what Tensor Cores and MCU accelerators do)
void matmul_int8(const int8_t* A, const int8_t* B, int32_t* C,
                 int M, int N, int K) {
    for (int i = 0; i < M; i++) {
        for (int j = 0; j < N; j++) {
            int32_t sum = 0;
            for (int k = 0; k < K; k++) {
                sum += (int32_t)A[i * K + k] * (int32_t)B[k * N + j];
            }
            C[i * N + j] = sum;
        }
    }
}

// Float32 matmul for comparison
void matmul_f32(const float* A, const float* B, float* C,
                int M, int N, int K) {
    for (int i = 0; i < M; i++) {
        for (int j = 0; j < N; j++) {
            float sum = 0.0f;
            for (int k = 0; k < K; k++) {
                sum += A[i * K + k] * B[k * N + j];
            }
            C[i * N + j] = sum;
        }
    }
}

int main() {
    const int M = 128, N = 128, K = 128;

    std::vector<int8_t> A_i8(M * K, 1), B_i8(K * N, 1);
    std::vector<int32_t> C_i32(M * N);

    std::vector<float> A_f32(M * K, 1.0f), B_f32(K * N, 1.0f);
    std::vector<float> C_f32(M * N);

    // Benchmark INT8
    auto start = std::chrono::high_resolution_clock::now();
    for (int t = 0; t < 100; t++) {
        matmul_int8(A_i8.data(), B_i8.data(), C_i32.data(), M, N, K);
    }
    auto end = std::chrono::high_resolution_clock::now();
    double i8_ms = std::chrono::duration<double, std::milli>(end - start).count() / 100;

    // Benchmark FP32
    start = std::chrono::high_resolution_clock::now();
    for (int t = 0; t < 100; t++) {
        matmul_f32(A_f32.data(), B_f32.data(), C_f32.data(), M, N, K);
    }
    end = std::chrono::high_resolution_clock::now();
    double f32_ms = std::chrono::duration<double, std::milli>(end - start).count() / 100;

    double gflops_i8 = 2.0 * M * N * K / i8_ms / 1e6;
    double gflops_f32 = 2.0 * M * N * K / f32_ms / 1e6;

    std::cout << "INT8 matmul:  " << i8_ms << " ms (" << gflops_i8 << " GOPS)\n";
    std::cout << "FP32 matmul:  " << f32_ms << " ms (" << gflops_f32 << " GFLOPS)\n";
    std::cout << "INT8 speedup: " << f32_ms / i8_ms << "x\n";
    std::cout << "Memory: INT8 = " << M*K + K*N << " bytes vs FP32 = "
              << (M*K + K*N) * 4 << " bytes (4x less)\n";

    return 0;
}
```
