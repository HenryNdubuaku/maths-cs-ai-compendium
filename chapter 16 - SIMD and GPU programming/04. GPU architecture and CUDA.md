# Архитектура GPU и CUDA

*GPU изменили сферу ИИ, предоставив тысячи ядер для массового параллелизма. В этом файле рассматриваются философия проектирования GPU в сравнении с CPU, иерархия памяти GPU, программирование на CUDA в C++, модель выполнения SIMT, шаблоны доступа к памяти, синхронизация, потоки, профилирование и поколения NVIDIA GPU — знания, необходимые для написания и понимания ядер (kernels) GPU.*

- Практические руководства по CUDA с полноценными рабочими примерами можно найти в сопутствующем репозитории: [github.com/HenryNdubuaku/cuda-tutorials](https://github.com/HenryNdubuaku/cuda-tutorials).

- Современный GPU NVIDIA имеет более 10 000 ядер CUDA. CPU имеет 4–128 ядер. Это преимущество в 100–1000 раз по количеству ядер — причина, по которой GPU доминируют в машинном обучении: обучение трансформера требует триллионов операций умножения-сложения, и GPU обрабатывают их параллельно в масштабах, недоступных для CPU.

- Даже если вы никогда не будете писать ядра CUDA самостоятельно, понимание архитектуры GPU объясняет: почему важен размер батча (нужно достаточно работы, чтобы загрузить GPU), почему память обычно является «узким местом» (а не вычисления) и почему определенные операции (разброс данных, условные переходы) работают на GPU медленно.

## GPU против CPU: принципиально разные конструкции

- CPU спроектирован для минимизации **задержки**: сокращения времени выполнения одной задачи. Он выделяет большую часть транзисторного бюджета на кеши, предсказатели переходов и внеочередное выполнение — все это приемы, позволяющие ускорить один поток.

- GPU спроектирован для максимизации **пропускной способности**: увеличения количества задач, выполняемых в секунду. Он выделяет большинство транзисторов на вычислительные блоки (АЛУ). Отдельные потоки работают медленно, но их тысячи.

| | CPU | GPU |
|--|-----|-----|
| Ядра | 4-128 (сложные, быстрые) | 1 000-20 000 (простые, медленные) |
| Тактовая частота | 3-5 ГГц | 1-2.5 ГГц |
| Кеш | Большой (32 МБ+ L3) | Маленький (разделяемая память на SM) |
| Предсказание переходов | Сложное | Отсутствует (все потоки следуют одним путем) |
| Лучшее применение | Низкая задержка, сложный поток управления | Высокая пропускная способность, параллельные данные |
| Типичные FLOPS (FP32) | 1-5 TFLOPS | 30-80 TFLOPS |
| Пропускная способность памяти | 50-100 ГБ/с | 1-3 ТБ/с |

- Преимущество GPU в пропускной способности памяти (в 10–30 раз) зачастую важнее, чем преимущество в вычислительной мощности. Многие операции машинного обучения ограничены скоростью памяти (поэлементные операции, нормализация, внимание), и пропускная способность GPU позволяет подавать данные на ядра достаточно быстро.

## Иерархия памяти GPU

- Понимание памяти GPU критически важно, поскольку **доступ к памяти является основным «узким местом»**, а не вычисления.

| Память | Размер | Задержка | Пропускная способность | Область видимости |
|--------|------|---------|-----------|-------|
| Регистры | ~256 КБ на SM | 0 циклов | Наивысшая | На поток |
| Разделяемая память | 48-228 КБ на SM | ~5 циклов | ~20 ТБ/с | На блок потоков |
| Кеш L1 | 128-256 КБ на SM | ~30 циклов | | На SM |
| Кеш L2 | 4-96 МБ | ~200 циклов | ~6 ТБ/с | Глобальная |
| Глобальная память (HBM) | 24-192 ГБ | ~400 циклов | 1-3.3 ТБ/с | Глобальная |

- **Регистры** — самые быстрые, но самые ограниченные. Каждый поток имеет собственный набор регистров (обычно максимум 255). Использование слишком большого количества регистров на поток снижает **заполняемость** (меньшее количество потоков может выполняться одновременно).

- **Разделяемая память** — это управляемый программистом кеш, общий для всех потоков в блоке. Это ключ к написанию быстрых ядер CUDA: загрузить фрагмент данных из медленной глобальной памяти в быструю разделяемую память, а затем выполнить вычисления. Это шаблон **тайлинга** (tiling), который доминирует в программировании для GPU.

- **Глобальная память (HBM)**: основная память GPU (VRAM). Большая, но медленная (задержка 400 циклов). Все данные начинаются и заканчиваются здесь. Цель оптимизации ядра — минимизировать обращения к глобальной памяти.

## Модель программирования CUDA

- CUDA (Compute Unified Device Architecture) — это модель программирования NVIDIA для GPU. Вы пишете **ядра**: функции, которые запускаются на GPU и выполняются тысячами потоков одновременно.

### Иерархия: сетки, блоки, потоки

```
Grid (the entire launch)
├── Block (0,0)
│   ├── Thread (0,0)
│   ├── Thread (1,0)
│   ├── Thread (2,0)
│   └── ... (up to 1024 threads per block)
├── Block (1,0)
│   ├── Thread (0,0)
│   └── ...
└── ... (millions of blocks possible)
```

- **Поток (thread)**: наименьшая единица. Каждый поток имеет уникальный идентификатор (`threadIdx.x`) внутри своего блока.
- **Блок (block)**: группа потоков, которые могут совместно использовать память и синхронизироваться. Идентификатор блока: `blockIdx.x`. Размер блока: `blockDim.x` (до 1024 потоков).
- **Сетка (grid)**: все блоки, запущенные одним ядром (kernel). Может быть одномерной, двумерной или трехмерной.

- Каждый поток вычисляет свой глобальный индекс: `int idx = blockIdx.x * blockDim.x + threadIdx.x;`

### Ваше первое CUDA-ядро (kernel)

```cpp
// vector_add.cu — CUDA source file (.cu extension)

#include <stdio.h>

// __global__ marks this as a GPU kernel (called from CPU, runs on GPU)
__global__ void vector_add(const float* a, const float* b, float* c, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {           // bounds check (grid may be larger than data)
        c[idx] = a[idx] + b[idx];
    }
}

int main() {
    int n = 1 << 20;  // ~1 million elements
    size_t bytes = n * sizeof(float);

    // Allocate host (CPU) memory
    float *h_a = new float[n];
    float *h_b = new float[n];
    float *h_c = new float[n];

    // Initialise
    for (int i = 0; i < n; i++) {
        h_a[i] = 1.0f;
        h_b[i] = 2.0f;
    }

    // Allocate device (GPU) memory
    float *d_a, *d_b, *d_c;
    cudaMalloc(&d_a, bytes);
    cudaMalloc(&d_b, bytes);
    cudaMalloc(&d_c, bytes);

    // Copy data from CPU to GPU
    cudaMemcpy(d_a, h_a, bytes, cudaMemcpyHostToDevice);
    cudaMemcpy(d_b, h_b, bytes, cudaMemcpyHostToDevice);

    // Launch kernel: 256 threads per block, enough blocks to cover n elements
    int block_size = 256;
    int grid_size = (n + block_size - 1) / block_size;  // ceiling division
    vector_add<<<grid_size, block_size>>>(d_a, d_b, d_c, n);

    // Copy result from GPU to CPU
    cudaMemcpy(h_c, d_a, bytes, cudaMemcpyDeviceToHost);

    // Verify
    printf("c[0] = %f (expected 3.0)\n", h_c[0]);

    // Free memory
    cudaFree(d_a); cudaFree(d_b); cudaFree(d_c);
    delete[] h_a; delete[] h_b; delete[] h_c;

    return 0;
}
```

```bash
# Compile with NVIDIA's compiler
nvcc -O3 -o vector_add vector_add.cu
./vector_add
```

- **Ключевые концепции C++ в CUDA**:
    - `__global__`: ключевое слово CUDA, помечающее функцию-ядро (kernel). Вызывается с CPU (`host`), выполняется на GPU (`device`).
    - `<<<grid_size, block_size>>>`: синтаксис запуска ядра. Определяет, сколько блоков и потоков использовать.
    - `cudaMalloc` / `cudaFree`: выделение/освобождение памяти GPU (аналогично `new`/`delete`, но для GPU).
    - `cudaMemcpy`: копирование данных между CPU и GPU. Это часто является самым узким местом (пропускная способность PCIe составляет ~32 ГБ/с, в то время как пропускная способность памяти GPU — ~3 ТБ/с).

### Варпы и SIMT

- GPU выполняет потоки группами по 32, называемыми **варпами** (warps). Все 32 потока в варпе выполняют **одну и ту же инструкцию** одновременно (Single Instruction, Multiple Threads — SIMT). Это аналог SIMD для GPU, но на уровне потоков.

- **Дивергенция варпа** (warp divergence) возникает, когда потоки внутри одного варпа переходят по разным веткам оператора `if`. GPU не может выполнять две разные инструкции одновременно в одном варпе, поэтому он выполняет обе ветки последовательно, маскируя потоки, которые не должны участвовать в вычислениях. Это снижает производительность вдвое (или более).

```cpp
// BAD: warp divergence (threads in same warp take different paths)
if (threadIdx.x % 2 == 0) {
    c[idx] = a[idx] + b[idx];    // even threads do this
} else {
    c[idx] = a[idx] - b[idx];    // odd threads do this (same warp, serialised)
}

// BETTER: branchless (no divergence)
float sign = (threadIdx.x % 2 == 0) ? 1.0f : -1.0f;
c[idx] = a[idx] + sign * b[idx];  // all threads execute the same instruction
```

### Объединение обращений к памяти (Memory Coalescing)

- **Объединенный доступ (coalesced access)**: когда последовательные потоки обращаются к последовательным адресам памяти, GPU объединяет их в одну транзакцию памяти. Это критически важно для производительности.

```cpp
// GOOD: coalesced — thread 0 reads a[0], thread 1 reads a[1], ...
c[idx] = a[idx] + b[idx];

// BAD: strided — thread 0 reads a[0], thread 1 reads a[stride], ...
c[idx] = a[idx * stride] + b[idx * stride];  // stride > 1 wastes bandwidth
```

- Для варпа из 32 потоков коалесцентный доступ считывает 128 байт (32 × 4 байта для `float32`) за одну транзакцию. Доступ с шагом требует нескольких транзакций, каждая из которых считывает 128 байт, но использует лишь их часть. Шаг, равный 32, является наихудшим случаем: каждая транзакция считывает 128 байт, но только один поток использует 4 байта (3% утилизации).

### Shared Memory and Tiling

- **Шаблон тайлинга (tiling pattern)** — это важнейший метод оптимизации GPU. Идея заключается в следующем: загрузить блок данных из медленной глобальной памяти в быструю общую память (shared memory), выполнить вычисления, а затем записать результаты обратно.

```cpp
// Matrix multiply with shared memory tiling (simplified)
__global__ void matmul_tiled(const float* A, const float* B, float* C,
                              int M, int N, int K) {
    // Shared memory for one tile of A and one tile of B
    __shared__ float tile_A[TILE_SIZE][TILE_SIZE];
    __shared__ float tile_B[TILE_SIZE][TILE_SIZE];

    int row = blockIdx.y * TILE_SIZE + threadIdx.y;
    int col = blockIdx.x * TILE_SIZE + threadIdx.x;
    float sum = 0.0f;

    // Loop over tiles
    for (int t = 0; t < (K + TILE_SIZE - 1) / TILE_SIZE; t++) {
        // Load one tile of A and B into shared memory
        if (row < M && t * TILE_SIZE + threadIdx.x < K)
            tile_A[threadIdx.y][threadIdx.x] = A[row * K + t * TILE_SIZE + threadIdx.x];
        else
            tile_A[threadIdx.y][threadIdx.x] = 0.0f;

        if (col < N && t * TILE_SIZE + threadIdx.y < K)
            tile_B[threadIdx.y][threadIdx.x] = B[(t * TILE_SIZE + threadIdx.y) * N + col];
        else
            tile_B[threadIdx.y][threadIdx.x] = 0.0f;

        __syncthreads();  // wait for all threads to finish loading

        // Compute partial dot product from this tile
        for (int k = 0; k < TILE_SIZE; k++) {
            sum += tile_A[threadIdx.y][k] * tile_B[k][threadIdx.x];
        }

        __syncthreads();  // wait before loading the next tile
    }

    if (row < M && col < N)
        C[row * N + col] = sum;
}
```

- **`__shared__`**: объявляет память, общую для всех потоков в блоке (быстрая, внутричиповая).
- **`__syncthreads()`**: барьер, который ожидает, пока все потоки в блоке не достигнут этой точки. Требуется между записью в общую память и чтением из неё (в противном случае некоторые потоки прочитают устаревшие данные).
- **Почему тайлинг работает**: без него каждый поток загружает данные из глобальной памяти для каждого умножения. При использовании тайлинга блок данных `TILE_SIZE × TILE_SIZE` загружается один раз в общую память и повторно используется всеми потоками в блоке. Коэффициент повторного использования равен `TILE_SIZE`, что сокращает трафик глобальной памяти во столько же раз.

## Потоки и конкурентность

- По умолчанию операции CUDA выполняются последовательно: CPU запускает ядро, ждёт его завершения, а затем запускает следующее. **Потоки** (streams) позволяют выполнять их параллельно:

```cpp
cudaStream_t stream1, stream2;
cudaStreamCreate(&stream1);
cudaStreamCreate(&stream2);

// These operations can overlap: different streams execute concurrently
cudaMemcpyAsync(d_a, h_a, bytes, cudaMemcpyHostToDevice, stream1);
cudaMemcpyAsync(d_b, h_b, bytes, cudaMemcpyHostToDevice, stream2);

kernel1<<<grid, block, 0, stream1>>>(d_a, d_c);
kernel2<<<grid, block, 0, stream2>>>(d_b, d_d);
```

- Потоки (streams) позволяют совмещать передачу данных с вычислениями: пока выполняется ядро одного потока, другой поток копирует данные. Это позволяет скрыть задержку передачи по шине PCIe и обеспечить постоянную загрузку GPU.

## Профилирование кода CUDA

```bash
# NVIDIA Nsight Compute: kernel-level profiling
ncu --set full ./my_program

# NVIDIA Nsight Systems: system-level timeline
nsys profile ./my_program

# Quick metrics
ncu --metrics sm__throughput,dram__throughput ./my_program
```

- **На что обратить внимание**:
    - **Заполненность (Occupancy)**: доля используемой емкости SM. Низкая заполненность (< 50%) означает, что потоков недостаточно для скрытия задержек памяти. Причины: слишком много регистров на поток, слишком большой объем разделяемой памяти на блок.
    - **Пропускная способность памяти**: сравните с пиковой пропускной способностью. Если вы достигаете < 50% от пика, значит, шаблоны доступа к памяти неэффективны (некоалесцированный доступ, банковские конфликты).
    - **Вычислительная пропускная способность**: сравните с пиковыми FLOPS. Если и пропускная способность памяти, и вычислительная производительность низки, значит, ядро ограничено задержками (недостаточный параллелизм).

## Продвинутые методы оптимизации

- Помимо базовых методов, таких как коалесценция и тайлинг в разделяемой памяти, высокопроизводительный код для GPU (и CPU) использует ряд продвинутых техник:

### Структура данных: AoS против SoA

- **Массив структур (Array of Structures, AoS)**: каждый элемент хранит все свои поля вместе. `[{x,y,z}, {x,y,z}, {x,y,z}]`.
- **Структура массивов (Structure of Arrays, SoA)**: каждое поле хранится в собственном непрерывном массиве. `{[x,x,x], [y,y,y], [z,z,z]}`.

```cpp
// AoS: BAD for SIMD/GPU (accessing all x values touches non-contiguous memory)
struct Particle { float x, y, z, mass; };
Particle particles[N];
// particles[0].x, particles[1].x are 16 bytes apart

// SoA: GOOD for SIMD/GPU (all x values are contiguous)
struct Particles {
    float x[N], y[N], z[N], mass[N];
};
// x[0], x[1] are 4 bytes apart — perfect for coalesced access and SIMD
```

- SoA почти всегда быстрее для задач с параллелизмом данных (SIMD, GPU). AoS лучше, когда вы всегда обращаетесь ко всем полям одного элемента вместе (редко встречается в численном коде). Тензоры PyTorch по своей природе являются SoA: каждый признак представляет собой непрерывное измерение.

### Программная предвыборка (Software Prefetching)

- Процессору можно дать команду начать загрузку данных до того, как они понадобятся, что позволяет скрыть задержку памяти:

```cpp
#include <xmmintrin.h>  // for _mm_prefetch

for (int i = 0; i < n; i += 4) {
    _mm_prefetch((char*)(a + i + 64), _MM_HINT_T0);  // prefetch 64 elements ahead
    // process a[i:i+4] with SIMD
    __m128 va = _mm_load_ps(a + i);
    // ...
}
```

- Инструкция `prefetch` является подсказкой: если данные уже находятся в кеше, она не выполняет никаких действий. Если нет, CPU начинает их подгрузку в фоновом режиме, продолжая выполнять другие инструкции. Дистанцию префетчинга (в данном примере 64 элемента вперед) следует настраивать в соответствии с задержкой памяти и временем итерации цикла.

### Слияние ядер (Kernel Fusion)

- **Слияние ядер (kernel fusion)** объединяет несколько операций в одно ядро, чтобы избежать записи промежуточных результатов в память. Это наиболее эффективная оптимизация GPU для машинного обучения:

```
// UNFUSED: 3 kernel launches, 3 global memory round-trips
y = matmul(x, W)     // write y to global memory
z = y + bias          // read y, write z
out = relu(z)         // read z, write out

// FUSED: 1 kernel launch, 1 global memory write
out = fused_matmul_bias_relu(x, W, bias)  // y and z never leave SRAM
```

- Для операций, ограниченных пропускной способностью памяти (bias add, ReLU, layer norm), объем передаваемых данных определяет время выполнения. Объединение (fusing) полностью устраняет этот трафик. `torch.compile` в PyTorch и Triton позволяют выполнять объединение автоматически или с минимальными усилиями.

### Ядра со смешанной точностью

- Использование пониженной точности (FP16, BF16, INT8) для вычислений и повышенной точности (FP32) для накопления результата позволяет получить преимущества обоих подходов:

```cpp
// Tensor Core: multiply FP16 matrices, accumulate in FP32
// Each Tensor Core instruction: D (FP32) = A (FP16) × B (FP16) + C (FP32)
nvcuda::wmma::mma_sync(c_frag, a_frag, b_frag, c_frag);
```

- FP16 в 2 раза меньше, чем FP32, поэтому он удваивает пропускную способность памяти (обычное «узкое место») и позволяет поместить в кеш в 2 раза больше данных. Тензорные ядра (Tensor Cores) обрабатывают FP16 в 8–16 раз быстрее, чем ядра CUDA в формате FP32. Именно поэтому обучение со смешанной точностью (глава 6) обеспечивает ускорение в 2–3 раза при минимальной потере точности.

### Пул аллокаторов памяти

- `cudaMalloc` работает медленно (~1 мс на вызов), поскольку синхронизируется с GPU. В цикле обучения, который выделяет временные буферы на каждой итерации, это приводит к значительным задержкам.

- **Пулы памяти** (кеширующий аллокатор PyTorch, пулы памяти CUDA) предварительно выделяют большой блок памяти GPU и выполняют подраспределение из него без системных вызовов:

```python
# PyTorch does this automatically — but understanding why matters
# Each torch.empty() reuses memory from the pool, no cudaMalloc
temp = torch.empty(1024, 1024, device='cuda')  # microseconds, not milliseconds
```

- Именно поэтому функции PyTorch `torch.cuda.memory_allocated()` и `torch.cuda.max_memory_allocated()` различаются: `allocated` — это то, что используется в данный момент, а `max` — пиковое значение (пул может содержать больше памяти, чем используется в текущий момент).

### Оптимизация на основе профилирования (Profile-Guided Optimisation)

- Не оптимизируйте вслепую. **Сначала профилируйте**, определите «узкое место», оптимизируйте его и профилируйте снова. Модель «крыши» (roofline model, файл 01) подскажет, является ли «узким местом» память или вычисления:

    - **Ограничено памятью (Memory-bound)** (низкая арифметическая интенсивность): оптимизируйте структуру данных (SoA), объединяйте ядра (fuse kernels), используйте пониженную точность, применяйте предвыборку (prefetch).
    - **Ограничено вычислениями (Compute-bound)** (высокая арифметическая интенсивность): используйте тензорные ядра (Tensor Cores), увеличивайте параллелизм, используйте более быстрые инструкции (FMA).
    - **Ограничено задержкой (Latency-bound)** (недостаточный параллелизм): увеличивайте заполняемость (occupancy), сокращайте использование регистров, запускайте больше потоков.

- Большинство рабочих нагрузок в машинном обучении **ограничены памятью**. Неожиданный вывод: более быстрый GPU (больше FLOPS) часто не помогает. Более быстрая память (HBM3 против HBM2e) помогает больше. Именно поэтому переход с A100 на H100 — это не только про FLOPS: у H100 также в 2 раза выше пропускная способность памяти.

## Поколения NVIDIA GPU

| Поколение | Год | Ключевая инновация | Значимость для ИИ |
|------------|------|----------------|--------------|
| Pascal (P100) | 2016 | HBM2, NVLink | Первый серьезный GPU для глубокого обучения |
| Volta (V100) | 2017 | **Tensor Cores** (матричное умножение смешанной точности) | Позволило обучение в FP16, 125 TFLOPS TF32 |
| Ampere (A100) | 2020 | TF32, разреженность, тензорные ядра 3-го поколения | 312 TFLOPS TF32, структурная разреженность 2:4 |
| Hopper (H100) | 2022 | **Transformer Engine** (FP8), HBM3 | 989 TFLOPS FP8, динамическое переключение точности |
| Blackwell (B200) | 2024 | Transformer Engine 2-го поколения, NVLink 5 | 2.5 PFLOPS FP4, многокристальный дизайн |

- **Tensor Cores** — это специализированные блоки для матричного умножения. Одна инструкция тензорного ядра вычисляет произведение матриц 4×4 (D = A×B + C) за один такт. Обычным ядрам CUDA потребовалось бы 64 операции FMA. Благодаря тензорным ядрам обучение со смешанной точностью (вычисления в float16, накопление в float32) происходит быстро.

- **Transformer Engine** (начиная с Hopper) динамически переключается между точностью FP8 и FP16 внутри одного слоя, выбирая более высокую точность только там, где это необходимо. Это максимизирует пропускную способность без ущерба для качества модели. Он специально разработан для архитектур трансформеров (внимание + MLP), которые доминируют в современном ИИ.

## Задачи по программированию (компилировать с помощью nvcc)

1. Напишите CUDA-ядро, которое применяет ReLU к массиву. Измерьте время, включая передачу памяти. Это упражнение обучает написанию ядер, использованию `cudaMalloc`/`cudaMemcpy` и пониманию «узкого места» при передаче данных между хостом и устройством (host↔device).
```cpp
// task1_relu.cu
// Compile: nvcc -O3 -o task1_relu task1_relu.cu

#include <stdio.h>
#include <stdlib.h>
#include <cuda_runtime.h>

__global__ void relu_kernel(const float* input, float* output, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        output[idx] = input[idx] > 0.0f ? input[idx] : 0.0f;
    }
}

int main() {
    const int N = 1 << 24;  // ~16M elements
    size_t bytes = N * sizeof(float);

    // Allocate host memory
    float* h_input  = (float*)malloc(bytes);
    float* h_output = (float*)malloc(bytes);
    for (int i = 0; i < N; i++) {
        h_input[i] = (float)(i % 100) - 50.0f;  // mix of positive and negative
    }

    // Allocate device memory
    float *d_input, *d_output;
    cudaMalloc(&d_input, bytes);
    cudaMalloc(&d_output, bytes);

    // Time the full pipeline: copy to GPU, compute, copy back
    cudaEvent_t start, stop;
    cudaEventCreate(&start);
    cudaEventCreate(&stop);

    cudaEventRecord(start);
    cudaMemcpy(d_input, h_input, bytes, cudaMemcpyHostToDevice);

    int block_size = 256;
    int grid_size = (N + block_size - 1) / block_size;
    relu_kernel<<<grid_size, block_size>>>(d_input, d_output, N);

    cudaMemcpy(h_output, d_output, bytes, cudaMemcpyDeviceToHost);
    cudaEventRecord(stop);
    cudaEventSynchronize(stop);

    float ms = 0;
    cudaEventElapsedTime(&ms, start, stop);

    // Verify
    int errors = 0;
    for (int i = 0; i < N; i++) {
        float expected = h_input[i] > 0.0f ? h_input[i] : 0.0f;
        if (h_output[i] != expected) errors++;
    }

    printf("Time (including transfers): %.2f ms\n", ms);
    printf("Bandwidth: %.1f GB/s\n", 2.0 * bytes / ms / 1e6);  // read + write
    printf("Errors: %d / %d\n", errors, N);

    cudaFree(d_input); cudaFree(d_output);
    free(h_input); free(h_output);
    return 0;
}
```

2. Напишите перемножение матриц с использованием тайлинга (tiled matrix multiply) в CUDA, задействовав общую память (shared memory). Сравните производительность с наивной (нетайлированной) версией. Это упражнение поможет понять работу общей памяти, `__syncthreads` и причины, по которым тайлинг важен.
```cpp
// task2_matmul.cu
// Compile: nvcc -O3 -o task2_matmul task2_matmul.cu

#include <stdio.h>
#include <cuda_runtime.h>

#define TILE 16

// Naive matmul: each thread computes one element of C
__global__ void matmul_naive(const float* A, const float* B, float* C, int N) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    if (row < N && col < N) {
        float sum = 0.0f;
        for (int k = 0; k < N; k++) {
            sum += A[row * N + k] * B[k * N + col];
        }
        C[row * N + col] = sum;
    }
}

// Tiled matmul: use shared memory to reduce global memory accesses
__global__ void matmul_tiled(const float* A, const float* B, float* C, int N) {
    __shared__ float sA[TILE][TILE];
    __shared__ float sB[TILE][TILE];

    int row = blockIdx.y * TILE + threadIdx.y;
    int col = blockIdx.x * TILE + threadIdx.x;
    float sum = 0.0f;

    for (int t = 0; t < (N + TILE - 1) / TILE; t++) {
        sA[threadIdx.y][threadIdx.x] = (row < N && t*TILE+threadIdx.x < N)
            ? A[row * N + t*TILE + threadIdx.x] : 0.0f;
        sB[threadIdx.y][threadIdx.x] = (t*TILE+threadIdx.y < N && col < N)
            ? B[(t*TILE + threadIdx.y) * N + col] : 0.0f;

        __syncthreads();
        for (int k = 0; k < TILE; k++)
            sum += sA[threadIdx.y][k] * sB[k][threadIdx.x];
        __syncthreads();
    }

    if (row < N && col < N)
        C[row * N + col] = sum;
}

int main() {
    const int N = 1024;
    size_t bytes = N * N * sizeof(float);

    float *d_A, *d_B, *d_C;
    cudaMalloc(&d_A, bytes); cudaMalloc(&d_B, bytes); cudaMalloc(&d_C, bytes);

    // Initialise with ones (easy to verify: C should be all N)
    float* h_A = new float[N*N];
    for (int i = 0; i < N*N; i++) h_A[i] = 1.0f;
    cudaMemcpy(d_A, h_A, bytes, cudaMemcpyHostToDevice);
    cudaMemcpy(d_B, h_A, bytes, cudaMemcpyHostToDevice);

    dim3 block(TILE, TILE);
    dim3 grid((N+TILE-1)/TILE, (N+TILE-1)/TILE);

    // Benchmark naive
    cudaEvent_t start, stop;
    cudaEventCreate(&start); cudaEventCreate(&stop);

    cudaEventRecord(start);
    for (int i = 0; i < 10; i++)
        matmul_naive<<<grid, block>>>(d_A, d_B, d_C, N);
    cudaEventRecord(stop);
    cudaEventSynchronize(stop);
    float naive_ms; cudaEventElapsedTime(&naive_ms, start, stop);

    // Benchmark tiled
    cudaEventRecord(start);
    for (int i = 0; i < 10; i++)
        matmul_tiled<<<grid, block>>>(d_A, d_B, d_C, N);
    cudaEventRecord(stop);
    cudaEventSynchronize(stop);
    float tiled_ms; cudaEventElapsedTime(&tiled_ms, start, stop);

    double gflops_naive = 2.0 * N * N * N * 10 / naive_ms / 1e6;
    double gflops_tiled = 2.0 * N * N * N * 10 / tiled_ms / 1e6;

    printf("Naive:  %.2f ms, %.1f GFLOPS\n", naive_ms/10, gflops_naive);
    printf("Tiled:  %.2f ms, %.1f GFLOPS\n", tiled_ms/10, gflops_tiled);
    printf("Speedup: %.1fx\n", naive_ms / tiled_ms);

    cudaFree(d_A); cudaFree(d_B); cudaFree(d_C);
    delete[] h_A;
    return 0;
}
```

3. Демонстрация дивергенции варпов (warp divergence). Напишите ядро, в котором потоки одного варпа выполняют разные ветви кода, и сравните его с версией без ветвлений.
```cpp
// task3_divergence.cu
// Compile: nvcc -O3 -o task3_diverge task3_divergence.cu

#include <stdio.h>
#include <cuda_runtime.h>

// BAD: warp divergence — even/odd threads take different paths
__global__ void divergent_kernel(float* data, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        if (idx % 2 == 0) {
            data[idx] = data[idx] * 2.0f + 1.0f;
        } else {
            data[idx] = data[idx] * 0.5f - 1.0f;
        }
    }
}

// GOOD: branchless — all threads execute the same instruction
__global__ void branchless_kernel(float* data, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        float scale = (idx % 2 == 0) ? 2.0f : 0.5f;
        float offset = (idx % 2 == 0) ? 1.0f : -1.0f;
        data[idx] = data[idx] * scale + offset;
    }
}

int main() {
    const int N = 1 << 24;
    float* d_data;
    cudaMalloc(&d_data, N * sizeof(float));
    cudaMemset(d_data, 0, N * sizeof(float));

    int block = 256, grid = (N + block - 1) / block;

    cudaEvent_t start, stop;
    cudaEventCreate(&start); cudaEventCreate(&stop);

    // Divergent
    cudaEventRecord(start);
    for (int i = 0; i < 100; i++)
        divergent_kernel<<<grid, block>>>(d_data, N);
    cudaEventRecord(stop);
    cudaEventSynchronize(stop);
    float div_ms; cudaEventElapsedTime(&div_ms, start, stop);

    // Branchless
    cudaEventRecord(start);
    for (int i = 0; i < 100; i++)
        branchless_kernel<<<grid, block>>>(d_data, N);
    cudaEventRecord(stop);
    cudaEventSynchronize(stop);
    float nodiv_ms; cudaEventElapsedTime(&nodiv_ms, start, stop);

    printf("Divergent:  %.2f ms\n", div_ms / 100);
    printf("Branchless: %.2f ms\n", nodiv_ms / 100);
    printf("Speedup:    %.2fx\n", div_ms / nodiv_ms);

    cudaFree(d_data);
    return 0;
}
```
