# ARM и NEON

*Процессоры ARM используются во всех смартфонах, большинстве планшетов, ноутбуках Apple и всё возрастающей доле серверов в дата-центрах. В этом файле рассматриваются архитектура ARM, программирование NEON SIMD с помощью C++ интринсиков, SVE/SVE2 для масштабируемой векторной обработки, особенности Apple Silicon и практические примеры векторизованных ядер.*

- Если у вас iPhone, MacBook или вы используете инстансы AWS Graviton, вы работаете на ARM. Энергоэффективность ARM делает эту архитектуру доминирующей в мобильных и встраиваемых системах, а также всё более конкурентоспособной в серверах и инференсе ML. Понимание ARM SIMD позволяет писать код, который работает быстро на оборудовании, используемом большинством людей.

- Реальный пример использования ядер ARM SIMD в продакшене — **Cactus**, низкозадержковый AI-движок для мобильных устройств и носимой электроники: [github.com/cactus-compute/cactus](https://github.com/cactus-compute/cactus). Cactus реализует кастомные ядра ARM NEON и NPU-ускоренные ядра для внимания (attention), квантования KV-кеша и chunked prefill, достигая самого быстрого инференса на CPU ARM с потреблением RAM в 10 раз меньшим, чем у других движков. Его трехуровневая архитектура (Engine → Graph → Kernels) — это конкретный пример того, как концепции SIMD из этого файла используются для построения инфраструктуры машинного обучения в продакшене.

## Основы архитектуры ARM

- ARM — это архитектура **RISC** (Reduced Instruction Set Computer) (глава 13). Ключевые характеристики:

    - **Архитектура load-store**: арифметические инструкции работают только с регистрами, никогда напрямую с памятью. Чтобы сложить два числа из памяти, вы должны: (1) загрузить их в регистры, (2) сложить регистры, (3) сохранить результат обратно в память. Это проще, чем в x86 (где можно сложить регистр и ячейку памяти за одну инструкцию), но позволяет организовать более чистый конвейер.

    - **Инструкции фиксированной ширины**: каждая инструкция ARMv8 (AArch64) имеет размер ровно 32 бита. Это делает декодирование быстрым и предсказуемым (в отличие от инструкций x86 переменной длины, которые могут составлять 1–15 байт).

    - **32 регистра общего назначения** (x0-x30, каждый по 64 бита), плюс указатель стека (sp) и нулевой регистр (xzr). Сравните с 16 регистрами общего назначения в x86. Больше регистров = меньше обращений к памяти = более быстрый код.

    - **32 SIMD/регистра с плавающей запятой** (v0-v31, каждый по 128 бит) для NEON и операций с плавающей запятой.

```cpp
// ARM assembly (just to see the flavour -- you will use intrinsics, not assembly)
// Add two registers
add x0, x1, x2    // x0 = x1 + x2

// Load from memory
ldr x0, [x1]      // x0 = *x1 (load 64 bits from address in x1)

// NEON: add four floats
fadd v0.4s, v1.4s, v2.4s  // v0 = v1 + v2 (four 32-bit floats)
```

- Вы не будете писать на ассемблере. Вы будете использовать **интринсики**: функции C/C++, которые отображаются 1:1 на конкретные инструкции. Компилятор берет на себя распределение регистров, планирование и другие низкоуровневые детали.

## NEON: 128-битный SIMD

- **NEON** — это SIMD-расширение ARM. Каждый регистр NEON имеет ширину 128 бит и может содержать:

| Тип данных | Элементов в регистре | Обозначение |
|-----------|-----------------------|----------|
| float32 | 4 | `float32x4_t` |
| float16 | 8 | `float16x8_t` |
| int32 | 4 | `int32x4_t` |
| int16 | 8 | `int16x8_t` |
| int8 | 16 | `int8x16_t` |

- 128 бит — это меньше, чем AVX (256 бит) или AVX-512 (512 бит) в x86. Но ARM компенсирует это превосходной энергоэффективностью и широкой доступностью.

### Интринсики NEON: основы

- Интринсики NEON следуют соглашению об именовании: `v[operation][qualifier]_[type]`

```cpp
#include <arm_neon.h>

// Load 4 floats from memory into a NEON register
float32x4_t a = vld1q_f32(ptr);        // vld1q = vector load 1, q = 128-bit (quad)

// Store 4 floats from a NEON register to memory
vst1q_f32(out_ptr, a);                   // vst1q = vector store 1, q = 128-bit

// Arithmetic
float32x4_t c = vaddq_f32(a, b);        // c = a + b (4 floats)
float32x4_t d = vmulq_f32(a, b);        // d = a * b (4 floats)
float32x4_t e = vfmaq_f32(c, a, b);     // e = c + a * b (fused multiply-add, 4 floats)

// Comparison (returns a mask: all 1s if true, all 0s if false)
uint32x4_t mask = vcgtq_f32(a, b);      // mask[i] = (a[i] > b[i]) ? 0xFFFFFFFF : 0

// Select elements based on mask (like numpy.where)
float32x4_t result = vbslq_f32(mask, a, b);  // result[i] = mask[i] ? a[i] : b[i]

// Reduce: sum all 4 elements to a scalar
float total = vaddvq_f32(a);             // total = a[0] + a[1] + a[2] + a[3]
```

- **`vfmaq_f32`** (fused multiply-add) — самая важная SIMD-инструкция для машинного обучения. Она вычисляет $c = c + a \times b$ за одну инструкцию с единственным этапом округления (это точнее, чем раздельное умножение и сложение). Скалярные произведения, матричные умножения и свёртки строятся на основе FMA.

### Практический пример: векторизованное скалярное произведение

- Скалярное произведение — это внутренний цикл матричного умножения. Давайте напишем его на обычном C++ и затем векторизуем с помощью NEON.

```cpp
#include <arm_neon.h>

// Scalar dot product
float dot_scalar(const float* a, const float* b, int n) {
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += a[i] * b[i];
    }
    return sum;
}

// NEON-vectorised dot product
float dot_neon(const float* a, const float* b, int n) {
    float32x4_t sum_vec = vdupq_n_f32(0.0f);  // initialise 4 accumulators to 0

    int i = 0;
    for (; i + 4 <= n; i += 4) {
        float32x4_t va = vld1q_f32(a + i);     // load 4 elements from a
        float32x4_t vb = vld1q_f32(b + i);     // load 4 elements from b
        sum_vec = vfmaq_f32(sum_vec, va, vb);   // sum_vec += va * vb
    }

    // Reduce the 4 accumulators to a single scalar
    float sum = vaddvq_f32(sum_vec);

    // Handle remaining elements (if n is not a multiple of 4)
    for (; i < n; i++) {
        sum += a[i] * b[i];
    }

    return sum;
}
```

- **Ключевые концепции C++**:
    - `const float*`: указатель на данные типа float, доступные только для чтения. `const` гарантирует, что мы не будем изменять данные через этот указатель.
    - `a + i`: адресная арифметика. `a + i` указывает на $i$-й элемент массива (то же самое, что `&a[i]`).
    - «Очищающий цикл» в конце обрабатывает случай, когда $n$ не кратно 4. Это универсальный паттерн в SIMD-коде: обработать основной объем данных векторизованными чанками, а затем обработать остаток с помощью обычного скалярного кода.

- **Почему в `sum_vec` 4 аккумулятора**: вместо одного скалярного аккумулятора мы используем 4 независимых аккумулятора (по одному на каждую SIMD-линию). Это позволяет избежать зависимости по данным: FMA на каждой итерации зависит от `sum_vec`, но при наличии 4 независимых линий процессор может конвейеризировать выполнение FMA. В конце мы сводим 4 частичные суммы в одну.

### Практический пример: векторизованная ReLU

```cpp
#include <arm_neon.h>

void relu_neon(const float* input, float* output, int n) {
    float32x4_t zero = vdupq_n_f32(0.0f);

    int i = 0;
    for (; i + 4 <= n; i += 4) {
        float32x4_t x = vld1q_f32(input + i);
        float32x4_t result = vmaxq_f32(x, zero);  // max(x, 0) = ReLU
        vst1q_f32(output + i, result);
    }

    // Scalar cleanup
    for (; i < n; i++) {
        output[i] = input[i] > 0 ? input[i] : 0;
    }
}
```

- `vmaxq_f32` вычисляет поэлементный максимум двух векторов. Поскольку один из векторов состоит из нулей, это в точности соответствует ReLU. Никаких ветвлений, никаких сравнений — только одна инструкция.

## I8MM: умножение целочисленных матриц

- **I8MM** (Int8 Matrix Multiply) — это расширение ARMv8.6, добавляющее специализированные инструкции для умножения матриц INT8 с накоплением в INT32, что именно и требуется для квантованного инференса в машинном обучении.

- Ключевой инструкцией является **`SMMLA`** (Signed Matrix Multiply-Accumulate): она берет два блока 8×2 значений INT8 и накапливает результат в блок 2×2 значений INT32:

```cpp
#include <arm_neon.h>

// I8MM: multiply two 8-element INT8 vectors, accumulate into 4 INT32 results
// This computes a 2x2 tile of the output matrix from 2x8 x 8x2 input tiles
void matmul_i8mm_tile(const int8_t* A, const int8_t* B, int32_t* C) {
    // Load 8 bytes from A (2 rows of 4 elements, packed)
    int8x16_t va = vld1q_s8(A);   // 16 bytes = 2 rows × 8 elements
    int8x16_t vb = vld1q_s8(B);   // 16 bytes = 2 rows × 8 elements

    // Load existing accumulator (2x2 = 4 int32 values)
    int32x4_t acc = vld1q_s32(C);

    // I8MM instruction: acc += A_tile × B_tile^T
    // Computes 2×2 output from 2×8 × 8×2 inputs
    acc = vmmlaq_s32(acc, va, vb);  // THE I8MM instruction

    vst1q_s32(C, acc);
}
```

- **Почему I8MM важен**: без I8MM умножение матриц INT8 на NEON требует расширяющих умножений (`vmull`) с последующим попарным сложением — это несколько инструкций на один выходной элемент. С I8MM оборудование выполняет скалярное произведение 8 элементов (2×8 × 8×2 = 2×2) за одну инструкцию. Для задач инференса INT8 это в 4–8 раз быстрее, чем обычный NEON.

- **Доступность**: Apple M1+ (весь Apple Silicon), ARM Cortex-A510/A710/X2+ (ARMv9), AWS Graviton3+. Проверяется через `#ifdef __ARM_FEATURE_MATMUL_INT8`.

- Для инференса в машинном обучении: квантованные модели INT8 (глава 18), работающие на серверах ARM (Graviton) или Apple Silicon, получают огромный прирост от I8MM. Такие фреймворки, как ONNX Runtime и llama.cpp, автоматически определяют наличие I8MM во время выполнения и используют оптимизированные ядра.

## SME и SME2: Scalable Matrix Extension

- **SME** (Scalable Matrix Extension) — это ответ ARM на Intel AMX и тензорные ядра NVIDIA: специализированное оборудование для матричных операций. SME2 (ARMv9.2) расширяет эти возможности.

- SME вводит **регистры тайлов ZA**: 2D-матрицы, хранящиеся в оборудовании, размером до SVL×SVL байт (где SVL — длина потокового вектора, обычно 128–512 бит на измерение). В отличие от NEON (1D-векторы) или даже SVE (масштабируемые 1D-векторы), SME работает с **2D-тайлами** нативно.

- Модель программирования имеет два режима:
    - **Нормальный режим**: стандартное выполнение ARM (NEON, SVE работают как обычно).
    - **Потоковый режим SVE (Streaming SVE mode)**: активируется через `smstart`, включает инструкции SME. Инструкции SVE также работают в этом режиме, но могут использовать другую ширину регистров.

```cpp
#include <arm_sme.h>

// SME2: outer product accumulation for matrix multiply
// Accumulates A_col × B_row into the ZA tile register
void sme2_matmul_outer(const float* A_col, const float* B_row, int K) {
    // Enter streaming mode
    // smstart;  // (done via compiler intrinsic or inline asm)

    // Zero the ZA tile accumulator
    svzero_za();

    for (int k = 0; k < K; k++) {
        // Load a column of A and a row of B into SVE registers
        svfloat32_t a = svld1_f32(svptrue_b32(), &A_col[k * SVL]);
        svfloat32_t b = svld1_f32(svptrue_b32(), &B_row[k * SVL]);

        // Outer product: ZA += a × b^T
        // This accumulates an SVL×SVL tile in one instruction
        svmopa_za32_f32_m(0, svptrue_b32(), svptrue_b32(), a, b);
    }

    // Store the ZA tile to memory
    // svst1_za(...);

    // Exit streaming mode
    // smstop;
}
```

- **Ключевые концепции**:
    - **`svmopa`** (накопление внешнего произведения): основная инструкция SME. Она вычисляет полное внешнее произведение двух векторов и накапливает его в тайл ZA. Для SVL=512 бит (16 чисел с плавающей запятой) это внешнее произведение 16×16 — 256 операций FMA за одну инструкцию.
    - **Тайл ZA**: сохраняет состояние между инструкциями в потоковом режиме. Вы накапливаете несколько внешних произведений (по одному на итерацию K) в один и тот же тайл, формируя полный тайл умножения матриц.
    - **Потоковый режим**: инструкции SME работают только в потоковом режиме. Издержки на вход/выход из потокового режима означают, что SME лучше всего подходит для длительных матричных вычислений, а не для коротких всплесков.

- **Дополнения SME2**: векторные операции (обработка 2 или 4 векторов SVE одновременно), дополнительные операции с тайлами и улучшенная интеграция с нормальным режимом.

- **Доступность**: ARM Neoverse V2 (AWS Graviton4), некоторые будущие мобильные чипы. Пока отсутствует в Apple Silicon (по состоянию на 2026 год). SME все еще находится на ранней стадии — большинство фреймворков машинного обучения пока не имеют ядер, оптимизированных под SME.

- **Эволюция**: NEON (128-битные векторы, поэлементные операции) → I8MM (матричные тайлы INT8) → SVE (масштабируемые векторы) → SME (масштабируемые 2D-матричные тайлы). Каждое поколение приближается к нативным матричным операциям на уровне оборудования.

## SVE и SVE2: Scalable Vector Extensions

- NEON имеет фиксированную ширину 128 бит. **SVE** (Scalable Vector Extension) внедряет программирование, **не зависящее от длины вектора (VLA)**: вы пишете код один раз, и он работает на оборудовании с любой шириной вектора (от 128 до 2048 бит). Оборудование определяет ширину во время выполнения.

```cpp
#include <arm_sve.h>

void add_sve(const float* a, const float* b, float* c, int n) {
    int i = 0;
    svbool_t pred = svwhilelt_b32(i, n);  // predicate: which lanes are active

    while (svptest_any(svptrue_b32(), pred)) {
        svfloat32_t va = svld1(pred, a + i);
        svfloat32_t vb = svld1(pred, b + i);
        svst1(pred, c + i, svadd_x(pred, va, vb));

        i += svcntw();  // advance by the hardware vector width (in 32-bit elements)
        pred = svwhilelt_b32(i, n);
    }
}
```

- **Предикатные регистры** (`svbool_t`) заменяют скалярный цикл очистки хвоста. Каждый канал имеет бит предиката: активные каналы участвуют в вычислениях, неактивные — маскируются. Инструкция `svwhilelt_b32(i, n)` создает предикат, в котором активны каналы, соответствующие `i, i+1, ..., n-1`. Это автоматически обрабатывает остаток массива.

- **`svcntw()`** возвращает количество 32-битных элементов в векторном регистре во время выполнения. На CPU с 256-битным SVE это значение равно 8. На 512-битном SVE — 16. Ваш код адаптируется автоматически.

- SVE доступен на ARM Neoverse V1/V2 (AWS Graviton3/4, некоторые серверные чипы). Он пока не поддерживается в Apple Silicon.

## Особенности Apple Silicon

- Чипы Apple M-серии (M1, M2, M3, M4) основаны на архитектуре ARM с кастомной микроархитектурой:

- **Производительные и энергоэффективные ядра**: P-ядра (Firestorm/Avalanche и т.д.) для тяжелых вычислений, E-ядра (Icestorm/Blizzard и т.д.) для фоновых задач. Планировщик распределяет потоки по соответствующим типам ядер.

- **AMX** (Apple Matrix eXtensions): выделенные блоки для матричного умножения, отдельные от NEON. AMX не задокументирован (Apple не публикует ISA), но фреймворк Accelerate использует его внутренне для операций BLAS. Когда вы вызываете `np.dot` на Mac, вызов проходит через Accelerate, который задействует AMX. Вы не можете программировать AMX напрямую (без обратной разработки).

- **Объединенная память (Unified memory)**: CPU и GPU используют одну и ту же физическую RAM. В других системах данные должны копироваться из памяти CPU в память GPU (через PCIe, ~32 ГБ/с). В Apple Silicon копирования нет — GPU читает ту же память, в которую записал CPU. Это устраняет критическое «бутылочное горлышко» для задач машинного обучения.

- **Neural Engine**: 16-ядерный специализированный ускоритель для машинного обучения. Выполняет ~30 TOPS (триллионов операций в секунду) для инференса в INT8. Используется Core ML для инференса на устройстве.

- **Для машинного обучения на Apple Silicon**: используйте MLX (фреймворк Apple для машинного обучения), который разработан специально для архитектуры с объединенной памятью. PyTorch также имеет поддержку бэкенда MPS (Metal Performance Shaders), хотя он менее зрелый, чем CUDA.

## Автовекторизация

- Написание SIMD-интринсиков — утомительная задача. Может ли **компилятор** векторизовать ваш код автоматически?

- Да, с оговорками. Современные компиляторы (GCC, Clang) могут автоматически векторизовать простые циклы:

```cpp
// The compiler CAN auto-vectorise this (with -O3 -march=native)
void add_auto(const float* a, const float* b, float* c, int n) {
    for (int i = 0; i < n; i++) {
        c[i] = a[i] + b[i];
    }
}
```

- **Шаблоны, способствующие автовекторизации**:
    - Простые циклы с известным числом итераций.
    - Отсутствие зависимостей по данным между итерациями (`c[i]` не зависит от `c[i-1]`).
    - Последовательный доступ к памяти (без scatter/gather).
    - Использование указателей `const` и `restrict` (сообщает компилятору, что массивы не перекрываются).

```cpp
// restrict tells the compiler: a, b, c point to non-overlapping memory
void add_restrict(const float* __restrict__ a,
                  const float* __restrict__ b,
                  float* __restrict__ c, int n) {
    for (int i = 0; i < n; i++) {
        c[i] = a[i] + b[i];
    }
}
```

- Без `restrict` компилятор должен предполагать, что `c` может перекрываться с `a` или `b` (запись в `c[i]` может изменить `a[i+1]`), что предотвращает векторизацию.

- **Шаблоны, препятствующие автовекторизации**:
    - Зависимости по данным: `a[i] = a[i-1] + b[i]` (каждая итерация зависит от предыдущей).
    - Сложная логика управления: операторы `if` внутри цикла (если только компилятор не может преобразовать их в предикацию).
    - Вызовы функций внутри цикла (если только функция не встраивается/inline).
    - Алиасинг указателей (массивы могут перекрываться, если не указан `restrict`).

- **Проверка автовекторизации**: используйте флаги компилятора, чтобы увидеть, что было векторизовано:

```bash
# GCC: show vectorisation decisions
g++ -O3 -march=native -fopt-info-vec-optimized code.cpp

# Clang: show vectorisation report
clang++ -O3 -march=native -Rpass=loop-vectorize code.cpp
```

- **Когда использовать интринсики, а когда автовекторизацию**: начните с чистого C++ и оптимизаций компилятора. Если компилятор векторизует ваш цикл — отлично. Если производительности все еще недостаточно, изучите отчет компилятора по векторизации, чтобы понять причину, и только после этого пишите интринсики для критически важного внутреннего цикла. Преждевременное использование интринсиков делает код нечитаемым без гарантированного выигрыша.

## Задачи по программированию (компилируйте с помощью g++ или clang++ на ARM — Mac M-серии или Linux aarch64)

1. Напишите скалярное произведение и NEON-векторизованное скалярное произведение. Проведите бенчмарк обоих вариантов и измерьте ускорение.

```cpp
// task1_neon_dot.cpp
// Compile (Mac/ARM Linux): clang++ -O3 -o task1 task1_neon_dot.cpp
// Note: NEON is enabled by default on AArch64, no special flags needed

#include <iostream>
#include <chrono>
#include <vector>
#include <arm_neon.h>

float dot_scalar(const float* a, const float* b, int n) {
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += a[i] * b[i];
    }
    return sum;
}

float dot_neon(const float* a, const float* b, int n) {
    float32x4_t sum_vec = vdupq_n_f32(0.0f);
    int i = 0;
    for (; i + 4 <= n; i += 4) {
        float32x4_t va = vld1q_f32(a + i);
        float32x4_t vb = vld1q_f32(b + i);
        sum_vec = vfmaq_f32(sum_vec, va, vb);
    }
    float sum = vaddvq_f32(sum_vec);
    for (; i < n; i++) sum += a[i] * b[i];
    return sum;
}

int main() {
    const int N = 10'000'000;
    std::vector<float> a(N, 1.0f), b(N, 2.0f);

    // Warm up
    volatile float s1 = dot_scalar(a.data(), b.data(), N);
    volatile float s2 = dot_neon(a.data(), b.data(), N);

    // Benchmark scalar
    auto start = std::chrono::high_resolution_clock::now();
    for (int t = 0; t < 100; t++) {
        s1 = dot_scalar(a.data(), b.data(), N);
    }
    auto end = std::chrono::high_resolution_clock::now();
    double scalar_ms = std::chrono::duration<double, std::milli>(end - start).count() / 100;

    // Benchmark NEON
    start = std::chrono::high_resolution_clock::now();
    for (int t = 0; t < 100; t++) {
        s2 = dot_neon(a.data(), b.data(), N);
    }
    end = std::chrono::high_resolution_clock::now();
    double neon_ms = std::chrono::duration<double, std::milli>(end - start).count() / 100;

    std::cout << "Scalar: " << scalar_ms << " ms (result: " << s1 << ")\n";
    std::cout << "NEON:   " << neon_ms << " ms (result: " << s2 << ")\n";
    std::cout << "Speedup: " << scalar_ms / neon_ms << "x\n";
    return 0;
}
```

2. Реализуйте ReLU и поиск максимума (для softmax) с помощью NEON. Попрактикуйтесь в использовании паттерна «загрузка→вычисление→сохранение» (load→compute→store) с различными операциями.

```cpp
// task2_neon_ops.cpp
// Compile: clang++ -O3 -o task2 task2_neon_ops.cpp

#include <iostream>
#include <vector>
#include <cmath>
#include <arm_neon.h>

void relu_neon(const float* in, float* out, int n) {
    float32x4_t zero = vdupq_n_f32(0.0f);
    int i = 0;
    for (; i + 4 <= n; i += 4) {
        float32x4_t x = vld1q_f32(in + i);
        vst1q_f32(out + i, vmaxq_f32(x, zero));
    }
    for (; i < n; i++) out[i] = in[i] > 0 ? in[i] : 0;
}

float max_neon(const float* data, int n) {
    float32x4_t max_vec = vdupq_n_f32(-INFINITY);
    int i = 0;
    for (; i + 4 <= n; i += 4) {
        max_vec = vmaxq_f32(max_vec, vld1q_f32(data + i));
    }
    float result = vmaxvq_f32(max_vec);
    for (; i < n; i++) result = result > data[i] ? result : data[i];
    return result;
}

int main() {
    std::vector<float> data = {-3, 1, -1, 4, 2, -5, 0, 7, -2, 3};
    std::vector<float> out(data.size());

    relu_neon(data.data(), out.data(), data.size());
    std::cout << "ReLU: ";
    for (float x : out) std::cout << x << " ";
    std::cout << "\n";

    float mx = max_neon(data.data(), data.size());
    std::cout << "Max: " << mx << " (expected: 7)\n";
    return 0;
}
```

3. Сравните автовекторизованный код с кодом, написанным вручную на интринсиках NEON. Выполните компиляцию с флагами `-fopt-info-vec` (GCC) или `-Rpass=loop-vectorize` (Clang), чтобы увидеть, что делает компилятор.

```cpp
// task3_auto_vs_manual.cpp
// Compile: clang++ -O3 -Rpass=loop-vectorize -o task3 task3_auto_vs_manual.cpp
//    (or): g++ -O3 -fopt-info-vec-optimized -o task3 task3_auto_vs_manual.cpp

#include <iostream>
#include <chrono>
#include <vector>
#include <arm_neon.h>

// Let the compiler auto-vectorise
void add_auto(const float* __restrict__ a, const float* __restrict__ b,
              float* __restrict__ c, int n) {
    for (int i = 0; i < n; i++) {
        c[i] = a[i] + b[i];
    }
}

// Hand-written NEON
void add_neon(const float* a, const float* b, float* c, int n) {
    int i = 0;
    for (; i + 4 <= n; i += 4) {
        vst1q_f32(c + i, vaddq_f32(vld1q_f32(a + i), vld1q_f32(b + i)));
    }
    for (; i < n; i++) c[i] = a[i] + b[i];
}

int main() {
    const int N = 10'000'000;
    std::vector<float> a(N, 1.0f), b(N, 2.0f), c(N);

    auto bench = [&](auto fn, const char* name) {
        fn(a.data(), b.data(), c.data(), N);  // warm up
        auto start = std::chrono::high_resolution_clock::now();
        for (int t = 0; t < 100; t++) fn(a.data(), b.data(), c.data(), N);
        auto end = std::chrono::high_resolution_clock::now();
        double ms = std::chrono::duration<double, std::milli>(end - start).count() / 100;
        std::cout << name << ": " << ms << " ms\n";
    };

    bench(add_auto, "Auto-vectorised");
    bench(add_neon, "Hand-written NEON");
    // They should be very close — the compiler auto-vectorises this simple loop well
    return 0;
}
```
