# Почему C++ и как работают ML-фреймворки

*Каждый вызов `jnp.matmul`, `torch.nn.Linear` и `np.dot` в этой книге на самом деле исполняет код на C++ и CUDA. Эта глава приоткрывает завесу: почему ML-фреймворки устроены именно так, краткие основы C++ для Python-разработчиков, когда стоит писать собственные C++ ядра и как подключать их к Python — это мост между кодом, который вы пишете, и оборудованием, на котором он выполняется.*

- Вы потратили 15 глав на написание кода на Python. Вы импортировали JAX, вызывали `jax.grad`, запускали циклы обучения и создавали модели. Всё это ощущалось как Python. Но вот правда: **почти никакие реальные вычисления не происходили в Python.**

- Когда вы пишете `output = model(input)` в PyTorch или `output = jnp.matmul(W, x)` в JAX, Python почти ничего не делает. Он строит описание вычислений (граф операций), а затем передает его бэкенду на C++/CUDA, который и выполняет основную работу. Python — это руль; C++ — это двигатель.

## Почему Python-фронтенд, C++ бэкенд

- Такая архитектура на двух языках существует потому, что Python и C++ хороши в противоположных вещах:

| | Python | C++ |
|--|--------|-----|
| Скорость разработки | Высокая (динамическая типизация, REPL, без компиляции) | Низкая (статическая типизация, заголовочные файлы, время компиляции) |
| Скорость выполнения | ~100x медленнее C (интерпретируемый, GIL) | Скорость, близкая к аппаратному уровню (компилируемый, без накладных расходов) |
| Управление памятью | Автоматическое (GC), нет контроля над размещением | Ручное, точный контроль над каждым байтом |
| Доступ к оборудованию | Нет (нет SIMD, нет GPU, нет кастомной памяти) | Полный (интринсики, CUDA, встроенный ассемблер) |
| Экосистема | Богатая для ML (ноутбуки, визуализация, данные) | Богатая для системного программирования (ОС, драйверы, движки) |

- Идея проста: **используйте каждый язык для того, в чем он хорош**. Python берет на себя части, где важна продуктивность человека (проектирование экспериментов, подбор гиперпараметров, исследование данных). C++ берет на себя части, где важна производительность машины (матричное умножение, свёртка, ядра внимания).

- Одно матричное умножение `jnp.matmul(A, B)`, где $A$ имеет размер $4096 \times 4096$, выполняет ~137 миллиардов операций с плавающей запятой. На чистом Python (вложенные циклы) это занимает ~30 минут. В оптимизированном C++ с использованием AVX-512 SIMD и многопоточности это занимает ~10 миллисекунд. Это разница в **180 000 раз**. Никакая изобретательность в Python не сократит этот разрыв.

## Как устроены ML-фреймворки

- Каждый крупный ML-фреймворк следует одной и той же архитектуре:

```
User code (Python)
    ↓
Python API layer (torch.nn, jax.numpy, numpy)
    ↓
Dispatch / JIT compiler (torch.compile, XLA, NumPy dispatch)
    ↓
C++ kernel library (ATen/PyTorch, XLA, BLAS/LAPACK)
    ↓
Hardware-specific backends (CUDA, cuDNN, MKL, oneDNN, Metal)
    ↓
Hardware (CPU SIMD units, GPU cores, TPU MXUs)
```

### NumPy

- Ядро NumPy написано на C. Когда вы вызываете `np.dot(A, B)`, Python вызывает функцию на C, которая обращается к BLAS (Basic Linear Algebra Subprograms), обычно это Intel MKL или OpenBLAS. BLAS — это оптимизированный вручную код на C и Fortran, который использует SIMD-инструкции, паттерны доступа к памяти с учетом кеша и многопоточность. Десятилетия оптимизаций были потрачены на то, чтобы сделать матричное умножение быстрым.

- NumPy работает только на CPU. Он не использует GPU. Но на CPU он чрезвычайно быстр, потому что делегирует вычисления лучшей доступной реализации BLAS.

### PyTorch

- Вычислительный движок PyTorch — это **ATen** (A Tensor Library), написанная на C++. ATen реализует ~2000 тензорных операций (add, matmul, conv2d, softmax, ...), каждая из которых имеет бэкенды для CPU и CUDA.

- Когда вы вызываете `torch.matmul(A, B)`:
    1. Python выполняет диспетчеризацию к функции ATen на C++.
    2. ATen проверяет устройство (CPU или CUDA) и тип данных (dtype).
    3. На CPU: вызывает MKL/OpenBLAS. На GPU: вызывает cuBLAS (оптимизированный для GPU BLAS от NVIDIA).
    4. Результат оборачивается в объект тензора Python и возвращается.

- **torch.compile** (PyTorch 2.0+) идет дальше: он трассирует ваш Python-код, строит граф вычислений и компилирует его с помощью **Triton** (для GPU) или **C++/OpenMP** (для CPU). Скомпилированный код объединяет операции (fusing), устраняет накладные расходы Python и может быть в 2-5 раз быстрее, чем в режиме eager.

### JAX

- JAX компилирует функции Python в **XLA** (Accelerated Linear Algebra), компилятор Google для ML-нагрузок. Когда вы применяете `jax.jit` к функции:
    1. JAX трассирует функцию, захватывая операции в виде графа вычислений XLA (HLO — High Level Operations).
    2. XLA оптимизирует граф: объединяет операции, устраняет избыточные вычисления, оптимизирует размещение в памяти.
    3. XLA компилирует код для целевого бэкенда: CPU (через LLVM), GPU (через CUDA/PTX) или TPU (через специфичные для TPU инструкции).
    4. Скомпилированный код выполняется непосредственно на оборудовании без участия Python.

- Вот почему `jax.jit` так важен: без него каждая операция — это отдельный цикл Python→C++. С ним вся функция превращается в единое скомпилированное ядро.

## Краткие основы C++ для Python-разработчиков

- Вам не нужно становиться экспертом в C++. Вам нужно понимать достаточно, чтобы читать код ядер, писать простые расширения и разбираться в дискуссиях о производительности. Вот самое основное.

### Типы и переменные

```cpp
// C++ requires explicit types (unlike Python)
int count = 0;           // 32-bit integer
float loss = 0.5f;       // 32-bit float
double lr = 3e-4;        // 64-bit float
bool training = true;    // boolean

// Arrays (fixed size, stack-allocated)
float weights[1024];     // 1024 floats, contiguous in memory

// Pointers: a variable that holds a memory address
float* ptr = weights;    // ptr points to the first element of weights
float val = ptr[42];     // access element 42 via pointer arithmetic
// ptr[42] is equivalent to *(ptr + 42)
```

- **Указатели** — это главное концептуальное отличие от Python. В Python всё является ссылкой, и вы никогда не задумываетесь об адресах памяти. В C++ указатели дают прямой доступ к памяти — это мощно, но опасно (висячие указатели, переполнение буфера).

### Функции

```cpp
// Function declaration: return_type name(param_type param_name)
float relu(float x) {
    return x > 0.0f ? x : 0.0f;
}

// Passing by reference (avoids copying large objects)
void scale_vector(std::vector<float>& vec, float factor) {
    for (size_t i = 0; i < vec.size(); i++) {
        vec[i] *= factor;
    }
}

// const reference: read-only, no copy
float sum(const std::vector<float>& vec) {
    float total = 0.0f;
    for (float x : vec) {  // range-based for loop (like Python's for x in vec)
        total += x;
    }
    return total;
}
```

### Память: стек против кучи

```cpp
// Stack allocation: fast, automatic lifetime (freed when function returns)
float buffer[256];   // 256 floats on the stack

// Heap allocation: manual, survives beyond the function
float* data = new float[n];   // allocate n floats on the heap
// ... use data ...
delete[] data;                 // YOU must free it (no garbage collector)

// Modern C++: smart pointers (automatic cleanup, like Python references)
#include <memory>
auto data = std::make_unique<float[]>(n);  // freed automatically when out of scope
```

- **Ключевое правило**: стек работает быстро, но его объем ограничен (обычно 1–8 МБ). Большие массивы (тензоры, карты признаков) должны размещаться в куче. В Python всё находится в куче, а очисткой памяти занимается сборщик мусора (GC). В C++ вы управляете этим самостоятельно (или используете умные указатели).

### Шаблоны (Generics)

```cpp
// A function that works with any numeric type
template <typename T>
T add(T a, T b) {
    return a + b;
}

add<float>(1.5f, 2.5f);   // returns 4.0f
add<int>(3, 4);             // returns 7
```

- Шаблоны — это то, как библиотеки C++ (например, ATen) пишут код, который работает с float16, float32, float64 и т. д., не дублируя реализацию.

### Основы стандартной библиотеки

```cpp
#include <vector>      // dynamic array (like Python list)
#include <string>      // string type

#include <algorithm>   // sort, find, transform, etc.

```

## Когда писать собственные ядра на C++

- Большинству инженеров по машинному обучению никогда не нужно писать на C++. Встроенные операции фреймворка покрывают 99% случаев использования. Вам стоит рассматривать написание собственного кода на C++ только если:

1. **Ваша операция отсутствует во фреймворке**: новая функция активации, нестандартный механизм внимания (attention), специализированная функция потерь, которую нельзя выразить как композицию существующих операций.

2. **Объединение (fusing) операций для производительности**: ваша модель выполняет `relu(layernorm(matmul(x, W) + b))`. Каждая операция запускает отдельное ядро, считывает и записывает данные в память и синхронизируется. Объединенное ядро выполняет всё за один проход, избегая лишних обращений к памяти. Это может быть в 2–5 раз быстрее.

3. **Снижение использования памяти**: собственное ядро может вычислять градиенты, не сохраняя все промежуточные активации (gradient checkpointing на уровне ядра).

4. **Работа с новым оборудованием**: новый ускоритель (например, Cerebras, Groq) может не иметь поддержки во фреймворке. Вы пишете ядра напрямую.

- Для случаев 1–2 часто достаточно **Triton** (глава 16, файл 05), и это гораздо проще, чем писать на CUDA C напрямую. Переходите на CUDA C только тогда, когда Triton не может выразить то, что вам нужно.

## Как связать C++ с Python

- Написание кода на C++ — это только половина дела. Вам также нужно вызывать его из Python.

### pybind11 (общего назначения)

- pybind11 создает Python-биндинги для функций C++ с минимальным количеством шаблонного кода:

```cpp
// my_ops.cpp
#include <pybind11/pybind11.h>
#include <pybind11/numpy.h>
namespace py = pybind11;

// A simple custom operation
py::array_t<float> custom_relu(py::array_t<float> input) {
    auto buf = input.request();
    float* ptr = static_cast<float*>(buf.ptr);
    size_t n = buf.size;

    auto result = py::array_t<float>(n);
    float* out = static_cast<float*>(result.request().ptr);

    for (size_t i = 0; i < n; i++) {
        out[i] = ptr[i] > 0 ? ptr[i] : 0;
    }
    return result;
}

PYBIND11_MODULE(my_ops, m) {
    m.def("custom_relu", &custom_relu, "Custom ReLU operation");
}
```

```bash
# Compile
pip install pybind11
c++ -O3 -shared -std=c++17 -fPIC $(python3 -m pybind11 --includes) my_ops.cpp -o my_ops$(python3-config --extension-suffix)
```

```python
# Use from Python
import my_ops
import numpy as np

x = np.array([-1.0, 2.0, -3.0, 4.0], dtype=np.float32)
y = my_ops.custom_relu(x)
print(y)  # [0. 2. 0. 4.]
```

### C++ расширения PyTorch

- PyTorch предоставляет упрощенный способ добавления пользовательских операций:

```cpp
// custom_op.cpp
#include <torch/extension.h>

torch::Tensor custom_gelu(torch::Tensor x) {
    return x * 0.5 * (1.0 + torch::erf(x / std::sqrt(2.0)));
}

PYBIND11_MODULE(TORCH_EXTENSION_NAME, m) {
    m.def("custom_gelu", &custom_gelu, "Custom GELU activation");
}
```

```python
# Load and compile on-the-fly
from torch.utils.cpp_extension import load

custom_ops = load(
    name="custom_ops",
    sources=["custom_op.cpp"],
    extra_cflags=["-O3"],
)

x = torch.randn(1000)
y = custom_ops.custom_gelu(x)
```

- `torch.utils.cpp_extension.load` компилирует код на C++, создает разделяемую библиотеку и загружает её как модуль Python за один вызов. Это самый простой способ экспериментировать с пользовательскими операциями на C++ в PyTorch.

### Пользовательские вызовы JAX

- JAX использует пользовательские вызовы XLA. Этот процесс более сложный (вы регистрируете функцию C в XLA), но концепция та же: напишите на C/C++, создайте биндинги и вызывайте из Python.

- Для большинства пользователей JAX лучшим выбором будет **Pallas** (рассматривается в файле 05): он позволяет писать GPU-ядра на синтаксисе, похожем на Python, который компилирует XLA, не покидая экосистему JAX.

## Общая картина

- В этом файле объясняется слой между Python и оборудованием. Оставшиеся файлы в этой главе углубляются в детали:
    - **Файл 01**: само оборудование (архитектура CPU, архитектура GPU, системы памяти)
    - **Файлы 02–03**: SIMD-программирование на CPU (ARM NEON, x86 AVX) — где вы пишете на C++, используя векторные блоки процессора
    - **Файл 04**: программирование GPU с помощью CUDA — где вы пишете на C++, который выполняется на тысячах ядер GPU
    - **Файл 05**: Triton, Pallas и высокоуровневое программирование GPU — где вы пишете на Python, который компилируется в GPU-ядра

- Эта последовательность отражает лестницу абстракций: интринсики C++ (низший уровень, максимальный контроль) → CUDA (специфично для GPU) → Triton/Pallas (Python-стиль, компилируемый) → JAX/PyTorch (высший уровень, автоматизация). Каждый уровень жертвует контролем ради удобства. Понимание нижних уровней делает вас более эффективным пользователем верхних.

## Задачи по программированию (компилируйте с помощью g++ или clang++)

1. Напишите свою первую программу на C++. Выделите память под массив, заполните его, вычислите сумму и измерьте время выполнения. Это упражнение познакомит вас с компиляцией, массивами, указателями и замером времени.
```cpp
// task1_basics.cpp
// Compile: g++ -O3 -o task1 task1_basics.cpp
// Run: ./task1

#include <iostream>
#include <chrono>
#include <vector>

int main() {
    const int N = 10'000'000;  // C++ allows ' as digit separator
    std::vector<float> data(N);

    // Fill the array
    for (int i = 0; i < N; i++) {
        data[i] = static_cast<float>(i) * 0.001f;
    }

    // Compute sum
    auto start = std::chrono::high_resolution_clock::now();
    float sum = 0.0f;
    for (int i = 0; i < N; i++) {
        sum += data[i];
    }
    auto end = std::chrono::high_resolution_clock::now();
    double elapsed = std::chrono::duration<double, std::milli>(end - start).count();

    std::cout << "Sum: " << sum << std::endl;
    std::cout << "Time: " << elapsed << " ms" << std::endl;
    std::cout << "Elements: " << N << std::endl;
    std::cout << "Throughput: " << (N * sizeof(float)) / elapsed / 1e6 << " GB/s" << std::endl;

    return 0;
}
```

2. Напишите функцию на C++, которая вычисляет ReLU для массива, а затем создайте Python-биндинг с помощью pybind11. Вызовите её из Python и сравните скорость работы с NumPy.
```cpp
// task2_relu.cpp
// Compile: c++ -O3 -shared -std=c++17 -fPIC $(python3 -m pybind11 --includes) \
//          task2_relu.cpp -o my_relu$(python3-config --extension-suffix)

#include <pybind11/pybind11.h>
#include <pybind11/numpy.h>
namespace py = pybind11;

py::array_t<float> cpp_relu(py::array_t<float> input) {
    auto buf = input.request();
    float* ptr = static_cast<float*>(buf.ptr);
    int n = buf.size;

    auto result = py::array_t<float>(n);
    float* out = static_cast<float*>(result.request().ptr);

    for (int i = 0; i < n; i++) {
        out[i] = ptr[i] > 0.0f ? ptr[i] : 0.0f;
    }
    return result;
}

PYBIND11_MODULE(my_relu, m) {
    m.def("relu", &cpp_relu, "C++ ReLU");
}
```
```python
# test_relu.py — run after compiling the C++ module above
import numpy as np
import time
import my_relu  # the compiled C++ module

x = np.random.randn(10_000_000).astype(np.float32)

# C++ ReLU
start = time.time()
for _ in range(100):
    y_cpp = my_relu.relu(x)
cpp_time = (time.time() - start) / 100

# NumPy ReLU
start = time.time()
for _ in range(100):
    y_np = np.maximum(x, 0)

print(f"C++ ReLU:   {cpp_time*1000:.2f} ms")

```

3. Напишите программу на C++, которая демонстрирует, почему важно расположение данных в памяти. Сравните шаблоны доступа к памяти по строкам (row-major) и по столбцам (column-major) и измерьте разницу в производительности.
```cpp
// task3_layout.cpp
// Compile: g++ -O3 -o task3 task3_layout.cpp

#include <iostream>
#include <chrono>
#include <vector>

int main() {
    const int N = 4096;
    std::vector<float> matrix(N * N, 1.0f);

    // Row-major access: sequential memory addresses (cache-friendly)
    auto start = std::chrono::high_resolution_clock::now();
    float sum_row = 0.0f;
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < N; j++) {
            sum_row += matrix[i * N + j];  // stride-1 access
        }
    }
    auto end = std::chrono::high_resolution_clock::now();
    double row_ms = std::chrono::duration<double, std::milli>(end - start).count();

    // Column-major access: stride-N access (cache-unfriendly)
    start = std::chrono::high_resolution_clock::now();
    float sum_col = 0.0f;
    for (int j = 0; j < N; j++) {
        for (int i = 0; i < N; i++) {
            sum_col += matrix[i * N + j];  // stride-N access (cache misses!)
        }
    }
    end = std::chrono::high_resolution_clock::now();
    double col_ms = std::chrono::duration<double, std::milli>(end - start).count();

    std::cout << "Row-major (cache-friendly): " << row_ms << " ms" << std::endl;
    std::cout << "Col-major (cache-hostile):  " << col_ms << " ms" << std::endl;
    std::cout << "Slowdown: " << col_ms / row_ms << "x" << std::endl;
    std::cout << "(Both sums: " << sum_row << ", " << sum_col << ")" << std::endl;

    return 0;
}
```
