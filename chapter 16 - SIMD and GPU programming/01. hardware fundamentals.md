# Основы аппаратного обеспечения

*Перед написанием кода для SIMD или GPU необходимо понимать аппаратное обеспечение, для которого вы программируете. В этом файле рассматривается, почему параллелизм заменил тактовую частоту, как современные процессоры выполняют инструкции, что такое SIMD, модель «roofline» для анализа производительности и ландшафт архитектур чипов.*

- На протяжении десятилетий программное обеспечение ускорялось «бесплатно»: вы покупали новый процессор с более высокой тактовой частотой, и ваша программа работала быстрее, не меняя ни строчки кода. Эта эра закончилась примерно в 2005 году. Понимание того, почему это произошло и что пришло на смену, необходимо каждому, кто хочет писать быстрый код.

## Конец эпохи бесплатной производительности

- **Закон Мура** (1965) гласит, что количество транзисторов на чипе удваивается примерно каждые два года. Это правило соблюдалось 60 лет. Больше транзисторов означало их меньший размер, что приводило к более высоким тактовым частотам, а значит — к более быстрым программам.

- Но около 2005 года тактовые частоты уперлись в потолок на уровне ~4 ГГц. Проблема заключается в **энергопотреблении**. Мощность, потребляемая чипом, приблизительно равна:

$$P \propto C \cdot V^2 \cdot f$$

- где $C$ — емкость (пропорциональна количеству транзисторов), $V$ — напряжение, а $f$ — тактовая частота. Чтобы увеличить частоту, необходимо увеличить напряжение (чтобы транзисторы переключались быстрее). Но мощность масштабируется как $V^2 \cdot f$, поэтому небольшое увеличение частоты вызывает значительный рост энергопотребления (и выделения тепла). На частоте 4 ГГц чипы уже потребляли более 100 Вт. Переход на 8 ГГц потребовал бы непрактичного охлаждения.

- Решение: вместо того чтобы делать одно ядро быстрее, разместить **несколько ядер** на одном чипе. 4-ядерный чип на частоте 3 ГГц потребляет примерно столько же энергии, сколько одно ядро на частоте 4,5 ГГц, но может выполнять в 4 раза больше параллельной работы. Именно поэтому каждый современный процессор имеет несколько ядер, и именно поэтому параллелизм (SIMD, многопоточность, вычисления на GPU) — единственный путь к повышению производительности.

- **Значение для машинного обучения**: шаг обучения, который занимает 10 минут на одном ядре, нельзя ускорить покупкой более быстрого процессора. Его можно ускорить только за счет использования большего количества ядер (параллелизм данных, глава 6), более широких SIMD-блоков (эта глава) или GPU (тысячи ядер).

## Как современные процессоры выполняют инструкции

- Современное ядро процессора гораздо сложнее, чем простая модель «выборка-декодирование-выполнение» из главы 13. Оно использует несколько приемов для выполнения большего количества инструкций за такт:

- **Суперскалярное выполнение**: процессор имеет несколько исполнительных блоков (АЛУ, блоки операций с плавающей запятой, блоки загрузки/сохранения) и может выполнять несколько независимых инструкций одновременно. Современное ядро может выполнять 4-6 инструкций за такт, если они не зависят друг от друга.

- **Внеочередное выполнение (OoO)**: процессор не выполняет инструкции в порядке их следования в программе. Он просматривает поток инструкций вперед, находит те, чьи входные данные готовы, и выполняет их немедленно, независимо от их позиции. Это скрывает задержки: пока одна инструкция ожидает данные из памяти (100+ тактов), процессор выполняет другие готовые инструкции.

- **Предсказание переходов**: условные переходы (операторы `if`, условия циклов) создают неопределенность: процессор не знает, по какому пути пойти, пока условие не будет вычислено. Вместо простоя процессор **предсказывает** результат и спекулятивно выполняет инструкции по предсказанному пути. Если предсказание верно (>95% случаев с современными предсказателями), время не теряется. Если неверно, спекулятивная работа отбрасывается и выполняется правильный путь (штраф около 15 тактов).

- **Спекулятивное выполнение**: расширение предсказания переходов. Процессор выполняет инструкции, которые *могут* не понадобиться, делая ставку на то, что они будут нужны. Это заполняет конвейер и поддерживает исполнительные блоки в рабочем состоянии.

- Все это происходит автоматически — процессор делает это без какого-либо вмешательства программиста. Но это помогает только с **параллелизмом на уровне инструкций (ILP)**: независимыми инструкциями внутри одного потока. Для **параллелизма на уровне данных** (одна и та же операция над множеством элементов данных) нам нужен SIMD.

## SIMD: одна инструкция, множество данных

- **SIMD** — это идея применения одной инструкции к нескольким элементам данных одновременно. Вместо сложения двух чисел складывайте два вектора из 4 (или 8, или 16) чисел за одну инструкцию.

- Без SIMD (скалярно):

```cpp
// Add two arrays element by element: 4 add instructions
for (int i = 0; i < 4; i++) {
    c[i] = a[i] + b[i];  // one add per iteration
}
```

- С SIMD (векторизованно):

```cpp
// Add two arrays: 1 SIMD instruction does all 4 adds
#include <immintrin.h>  // x86 SIMD intrinsics

__m128 va = _mm_load_ps(a);    // load 4 floats into a 128-bit register
__m128 vb = _mm_load_ps(b);    // load 4 floats into another register
__m128 vc = _mm_add_ps(va, vb); // add all 4 pairs simultaneously
_mm_store_ps(c, vc);            // store 4 results
```

- Версия с SIMD выполняет ту же работу за 1/4 количества инструкций. Это теоретическое ускорение в 4 раза, достигаемое за счет обработки 4 чисел float за инструкцию вместо 1.

### Векторные регистры

- SIMD-инструкции работают с **векторными регистрами**: широкими регистрами, которые хранят несколько элементов данных.

| Ширина регистра | Числа float (32-бит) | Числа double (64-бит) | Название |
|----------------|-----------------|-------------------|------|
| 128-бит | 4 | 2 | SSE (x86), NEON (ARM) |
| 256-бит | 8 | 4 | AVX/AVX2 (x86) |
| 512-бит | 16 | 8 | AVX-512 (x86) |
| Переменная (128-2048) | варьируется | варьируется | SVE/SVE2 (ARM) |

- Более широкие регистры = больше параллелизма. Инструкция AVX-512 шириной 512 бит обрабатывает 16 чисел float одновременно, что дает теоретическое ускорение в 16 раз по сравнению со скалярным кодом. На практике ускорение ниже из-за ограничений пропускной способности памяти (вы можете вычислять быстрее, чем успеваете подавать данные в процессор).

- Для машинного обучения: матричное умножение значений float32 дает огромный выигрыш от SIMD. Внутренний цикл (скалярное произведение двух векторов) напрямую отображается на SIMD-инструкции умножения-сложения. Именно поэтому библиотеки BLAS (которые вызывают NumPy и PyTorch) так сильно оптимизированы с помощью SIMD.

## Модель «roofline»

- Как узнать, является ли ваш код быстрым? Модель **«roofline»** предоставляет структуру для характеристики производительности с точки зрения двух аппаратных ограничений:

1. **Пиковая вычислительная мощность** (FLOPS): максимальное количество операций с плавающей запятой в секунду. Для процессора 4 ГГц с 256-битным AVX (8 чисел float за инструкцию) и 2 блоками FMA: $4 \times 10^9 \times 8 \times 2 = 64$ GFLOPS.

2. **Пиковая пропускная способность памяти** (байт/секунда): скорость, с которой данные могут передаваться из памяти в CPU. Современный CPU может обладать пропускной способностью памяти в 50 ГБ/с.

- **Арифметическая интенсивность** вашего кода — это отношение объема вычислений к объему обращений к памяти:

$$\text{Arithmetic Intensity} = \frac{\text{FLOPS}}{\text{Bytes transferred}}$$

- Если арифметическая интенсивность низка (мало операций на загруженный байт), ваш код является **ограниченным памятью** (memory-bound): он проводит большую часть времени в ожидании данных. Ускорение вычислений (более широкий SIMD, более высокая тактовая частота) не поможет.

- Если арифметическая интенсивность высока (много операций на байт), ваш код является **ограниченным вычислениями** (compute-bound): он проводит большую часть времени за вычислениями. Более быстрая память не поможет.

- Модель «крыши» (roofline):

$$\text{Achievable FLOPS} = \min\left(\text{Peak FLOPS}, \; \text{Bandwidth} \times \text{Arithmetic Intensity}\right)$$

- **Матричное умножение** обладает высокой арифметической интенсивностью: $O(n^3)$ операций на $O(n^2)$ данных, поэтому интенсивность $\approx O(n)$. Для больших матриц оно ограничено вычислениями. Именно поэтому GPU (высокая вычислительная мощность) доминируют в задачах машинного обучения с интенсивными матричными вычислениями.

- **Поэлементные операции** (ReLU, сложение, умножение) имеют низкую арифметическую интенсивность: 1 операция на загруженный элемент. Они ограничены памятью. Ускорение GPU не поможет; вам нужна более быстрая память (или объединение этих операций с вычислительно емкими, чтобы избежать лишних циклов обращения к памяти).

- Модель «крыши» объясняет, почему **слияние ядер** (kernel fusion) так важно: объединение матричного умножения с прибавлением смещения и ReLU в одно ядро позволяет избежать записи промежуточных результатов в память и их последующего чтения, превращая три операции, ограниченные памятью, в одну операцию, ограниченную вычислениями.

## Задержка против пропускной способности

- **Задержка** (latency) — это время выполнения одной операции. **Пропускная способность** (throughput) — это количество операций, выполненных за единицу времени.

- Аналогия: автобус имеет высокую задержку (останавливается на каждой остановке), но высокую пропускную способность (перевозит 50 человек сразу). Такси имеет низкую задержку (едет прямо к месту назначения), но низкую пропускную способность (перевозит 1–4 человек).

- GPU — это автобусы: высокая задержка на операцию (каждая инструкция требует много циклов для завершения), но огромная пропускная способность (тысячи ядер, работающих одновременно). CPU — это такси: низкая задержка (исполнение с изменением порядка, предсказание переходов, кэши большого объема минимизируют задержки), но ограниченная пропускная способность (4–64 ядра).

- Именно поэтому GPU лучше подходят для обучения моделей машинного обучения (важна пропускная способность: обработка миллионов примеров), а CPU — для задач операционной системы (важна задержка: мгновенный отклик на нажатие клавиши).

- **Конвейеризация** (pipelining) преобразует задержку в пропускную способность. Если инструкция выполняется 5 циклов, но конвейер запускает новую инструкцию каждый цикл, пропускная способность составляет 1 инструкцию за цикл (даже если на выполнение каждой уходит 5 циклов). Это тот же принцип, что и конвейер CPU из главы 13, но он применяется на всех уровнях: блоки SIMD, контроллеры памяти и ядра GPU — все они конвейеризированы.

## Ландшафт архитектур чипов

- Аппаратное обеспечение, для которого вы пишете код, определяет доступные SIMD-инструкции:

### x86 (Intel, AMD)

- Доминируют на настольных компьютерах, ноутбуках и серверных CPU. SIMD: SSE (128 бит), AVX/AVX2 (256 бит), AVX-512 (512 бит). Intel AMX предоставляет специализированные блоки матричного умножения для задач ИИ.

- **Сильные стороны**: высочайшая производительность на одно ядро, самый широкий SIMD, зрелая экосистема ПО (MKL, oneDNN).
- **Слабые стороны**: высокое энергопотребление, сложный набор инструкций, высокая стоимость.

### ARM

- Доминирует в мобильном сегменте (каждый смартфон), растет в серверах (AWS Graviton, Ampere Altra) и ноутбуках (Apple M-series). SIMD: NEON (128 бит), SVE/SVE2 (масштабируемый, 128–2048 бит).

- **Сильные стороны**: отличная энергоэффективность (производительность на ватт), кастомные ядра (Apple M4 соперничает с Intel по производительности на одно ядро при значительно меньшем энергопотреблении).
- **Слабые стороны**: более узкий SIMD (NEON только 128-битный, хотя SVE может быть шире), меньшая экосистема ПО для высокопроизводительных вычислений (HPC).

### Apple Silicon (M1/M2/M3/M4)

- Основаны на ARM с собственными дополнениями. Включают **AMX** (Apple Matrix eXtensions) — недокументированные блоки матричного умножения, которые фреймворк Accelerate использует для операций BLAS. Архитектура унифицированной памяти: CPU и GPU используют одну и ту же физическую память, устраняя «бутылочное горлышко» копирования данных между CPU и GPU.

- **Для машинного обучения**: Apple Neural Engine (16-ядерный специализированный ускоритель ML) и унифицированная память делают чипы серии M на удивление эффективными для локального инференса и обучения моделей небольшого масштаба. Однако CUDA отсутствует: необходимо использовать Metal (графический API Apple) или MLX (фреймворк машинного обучения Apple).

### RISC-V

- ISA с открытым исходным кодом. Отсутствие лицензионных отчислений (в отличие от ARM). Растет в сегментах встраиваемых систем, IoT и исследований. SIMD: векторное расширение «V» обеспечивает масштабируемую векторную обработку, аналогичную ARM SVE.

- **Для машинного обучения**: пока не конкурирует с x86/ARM в задачах ML, но стоит следить за развитием. Ряд стартапов по созданию ИИ-ускорителей используют ядра RISC-V.

### GPU (NVIDIA, AMD, Intel)

- Подробно рассмотрены в файлах 04–05. Тысячи простых ядер, оптимизированных для пропускной способности. NVIDIA доминирует в ML с CUDA; AMD конкурирует с ROCm; Intel выходит на рынок с GPU Arc и ускорителями Gaudi.

### TPU (Google)

- Кастомный ASIC, разработанный специально для машинного обучения. Систолические массивы, оптимизированные для матричного умножения. Рассмотрены в файле 05.

## Тепловые и энергетические ограничения

- Производительность в конечном итоге ограничивается питанием и охлаждением:

- **TDP** (Thermal Design Power): максимальная мощность, которую чип может потреблять при длительной работе. У CPU ноутбука TDP может составлять 15 Вт; у серверного CPU — 250 Вт; у GPU для дата-центров — 700 Вт (NVIDIA B200).

- **«Темный кремний»** (dark silicon): в любой момент времени значительная часть транзисторов должна быть отключена, чтобы не выйти за рамки теплового бюджета. Теоретически чип мог бы использовать все свои транзисторы одновременно, но он бы расплавился.

- **Энергоэффективность** (FLOPS/ватт) становится все более важным показателем, чем «сырые» FLOPS. Именно поэтому:
    - ARM захватывает дата-центры (лучше FLOPS/ватт, чем у x86).
    - TPU конкурируют с GPU, несмотря на более низкие пиковые FLOPS (гораздо лучше FLOPS/ватт для задач ML).
    - Квантование (INT8, FP8) важно не только для памяти: оно также снижает энергопотребление на операцию.

- Для машинного обучения в больших масштабах: обучение передовой LLM потребляет мегаватты энергии в течение месяцев. Стоимость электроэнергии может превышать стоимость оборудования. Энергоэффективность напрямую влияет на экономику исследований в области ИИ.

## Практика: измерение производительности в C++

- Чтобы рассуждать о производительности, её нужно измерить. Вот минимальный шаблон для бенчмаркинга на C++:

```cpp
#include <iostream>
#include <chrono>
#include <vector>

// Scalar addition
void add_scalar(const float* a, const float* b, float* c, int n) {
    for (int i = 0; i < n; i++) {
        c[i] = a[i] + b[i];
    }
}

int main() {
    const int N = 1 << 24;  // ~16 million elements
    std::vector<float> a(N, 1.0f), b(N, 2.0f), c(N);

    // Warm up (fill caches, trigger frequency scaling)
    add_scalar(a.data(), b.data(), c.data(), N);

    // Benchmark
    auto start = std::chrono::high_resolution_clock::now();

    for (int trial = 0; trial < 100; trial++) {
        add_scalar(a.data(), b.data(), c.data(), N);
    }

    auto end = std::chrono::high_resolution_clock::now();
    double elapsed = std::chrono::duration<double>(end - start).count();

    double total_bytes = 3.0 * N * sizeof(float) * 100;  // read a, read b, write c
    double bandwidth = total_bytes / elapsed / 1e9;        // GB/s

    std::cout << "Time: " << elapsed << " s\n";
    std::cout << "Bandwidth: " << bandwidth << " GB/s\n";

    return 0;
}
```

```bash
# Compile with optimisations
g++ -O3 -march=native -o bench bench.cpp
./bench
```

- **Ключевые концепции C++ в этом коде**:
    - `#include <vector>`: динамический массив (`std::vector<float>`) — как `list` в Python, но типизированный и непрерывный в памяти.
    - `a.data()`: возвращает «сырой» указатель (`float*`) на лежащий в основе массив — необходим для SIMD-интринсиков.
    - `std::chrono`: высокоточный таймер для бенчмаркинга.
    - `-O3`: максимальный уровень оптимизации компилятора. Компилятор может автоматически векторизовать ваши циклы (использовать SIMD автоматически). `-march=native` включает все SIMD-инструкции, поддерживаемые вашим процессором.

- **Зачем нужен прогрев (warm up)**: первый прогон заполняет кеши и может спровоцировать изменение тактовой частоты процессора (turbo boost). Последующие прогоны являются более репрезентативными.

- **Зачем измерять пропускную способность (bandwidth)**: для операций, ограниченных памятью (memory-bound, например, поэлементное сложение), значимой метрикой является пропускная способность (ГБ/с), а не FLOPS. Если измеренная пропускная способность близка к аппаратному пределу (~50 ГБ/с для DDR5), значит, вы ограничены памятью, и SIMD не сильно поможет (узким местом является память, а не вычисления).

def arithmetic_intensity(flops, bytes_transferred):

2. Продемонстрируйте, почему важен параллелизм. Сравните последовательное и параллельное (NumPy) выполнение по мере роста размера данных.

import numpy as np
import time

for n in [1000, 10000, 100000, 1000000, 10000000]:
    a = np.random.randn(n).astype(np.float32)
    b = np.random.randn(n).astype(np.float32)

    # "Sequential" (Python loop)
    start = time.time()
    c = [a[i] * b[i] for i in range(min(n, 100000))]  # cap at 100K for sanity
    seq_time = time.time() - start
    if n > 100000:
        seq_time *= n / 100000  # extrapolate

    # "Parallel" (NumPy, uses SIMD + multithreading internally)
    start = time.time()
    c = a * b
    par_time = time.time() - start

    print(f"n={n:>10,}  sequential={seq_time:.4f}s  parallel={par_time:.6f}s  "
