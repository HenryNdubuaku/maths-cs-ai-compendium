# 操作系统

*操作系统是介于硬件与应用程序之间的软件层，负责管理资源、提供抽象并实施隔离。本文件涵盖 OS 的职责、process、thread、CPU 调度、内存管理、file system 与系统调用*

- 一台没有操作系统的计算机就像一间没有厨师的厨房：食材（硬件）都在，但没有人协调谁用炉子、把盘子放哪儿，或如何防止两个人抢同一把刀。**OS** 就是那位协调者。

- 对 ML 从业者来说，OS 概念解释了：为什么 `nvidia-smi` 按 process 显示 GPU 内存用量，为什么训练会以“out of memory”崩溃，为什么 `fork()` 会复制你的 Python 进程，以及为什么 Docker 容器能提供隔离环境。

## 操作系统做什么

- OS 有三项核心职责：

    - **抽象**：用干净的接口隐藏硬件复杂性。程序读写“文件”而无需知道底层存储是 SSD、HDD 还是网络驱动。它们分配“内存”而无需管理物理 RAM 芯片。它们在“CPU”上运行而无需操心中断和 cache 一致性。

    - **资源管理**：多个程序共享 CPU、内存、磁盘和网络。OS 决定谁在何时获得什么、获得多久。公平而高效的分配策略让系统保持响应性。

    - **隔离与保护**：程序之间不得互相干扰。你浏览器里的 bug 不应让 kernel 崩溃。恶意程序不应读取另一个程序的密码。OS 借助硬件支持（特权级、虚拟内存）来强制边界。

## 进程

- **process（进程）**是一个正在运行的程序。它是 OS 工作的基本单位。每个 process 拥有：

    - **代码**（程序指令，只读）。
    - **数据**（全局变量、heap 分配）。
    - **栈**（函数调用帧、局部变量）。
    - **状态**（寄存器值、程序计数器、打开的文件等）。

- **进程控制块（PCB）**是 OS 用来跟踪一个 process 的数据结构。它存储进程 ID（PID）、状态、程序计数器、寄存器内容、内存映射、打开的文件描述符和 scheduling 优先级。当 OS 从一个 process 切换到另一个时，它把当前 process 的状态保存到其 PCB，并装载下一个 process 的状态。这就是一次**上下文切换**。

- 上下文切换代价高昂：保存和恢复寄存器、冲刷 cache、令 TLB 表项失效需要微秒级时间。在运行数千个 process 的系统上，这一开销可能很可观。这就是为什么“每请求一个 process”的服务器架构（如老式 Apache）被基于 thread 或事件驱动的架构取代。

- Unix 中的**进程创建**使用 `fork()` 和 `exec()`：

    - `fork()` 创建当前 process 的一个**副本**。子 process 获得父 process 内存、文件描述符和状态的复制品。两个 process 都从同一点继续执行，但 `fork()` 在子中返回 0，在父中返回子的 PID。

    - `exec()` 用一个新程序替换当前 process 的代码。在 `fork()` 之后，子 process 通常调用 `exec()` 来运行一个不同的程序。

    - 这种先 fork 后 exec 的模型很优雅：创建新 process（fork）与装载新程序（exec）是分离的操作，可以各自独立定制。在 fork 和 exec 之间，子 process 可以重定向 I/O、改变环境变量或放弃特权。

![进程状态转换：new → ready → running → blocked/terminated，带抢占和 I/O 等待](../images/process_states.svg)

- **进程状态**：一个 process 处于以下状态之一：
    - **运行（Running）**：正在某个 CPU 核上执行。
    - **就绪（Ready）**：在等待某个 CPU 核（可运行但尚未被调度）。
    - **阻塞（Blocked）**（等待）：在某些事件发生前无法继续（I/O 完成、锁获取、定时器到期）。
    - **终止（Terminated）**：执行完毕，等待父 process 收取其退出状态。

## 线程

- **thread（线程）**是 process 内的轻量执行单元。一个 process 中的所有 thread 共享相同的代码、数据和 heap，但每个 thread 有自己的栈和寄存器状态。

- 相对于多 process 的优势：thread 共享内存，所以它们之间的通信很快（直接读写共享变量即可）。process 则需要进程间通信（管道、套接字、共享内存映射），更慢也更复杂。

- 缺点：共享内存是危险的。两个 thread 同时写入同一个变量会造成**竞态条件**（结果取决于哪个 thread 先运行）。这引出文件 4 中讨论的同步问题。

- **内核线程**由 OS 调度器管理。每个 thread 独立调度到 CPU 核上。创建和切换内核线程涉及 syscall，其开销与 process 上下文切换相近（但更小）。

- **用户线程**（绿色线程）由用户空间的运行时库管理，对 OS 不可见。它们的创建和切换更便宜（无需 syscall），但一个用户 thread 的阻塞操作会阻塞该 process 中的所有 thread（因为 OS 只看到一个内核 thread）。

- 现代系统使用**混合模型**：许多用户线程映射到较少数量的内核 thread（M:N 线程）。Go 的 goroutine 和 Erlang 的 process 都是由语言运行时调度到 OS thread 上的用户级 thread。

- **线程池**预先创建固定数量的 thread 等待任务。当任务到来时，把它分配给一个空闲 thread。这避免了为每个任务创建和销毁 thread 的开销。Web 服务器、数据库引擎和 ML 推理服务器都使用线程池。

## CPU 调度

- **调度器**决定在每个时刻哪个 process/thread 在哪个 CPU 核上运行。目标是：最大化 CPU 利用率、最小化响应时间（对交互式任务）、最大化吞吐率（对批处理任务），并保证公平性。

- **先来先服务（FCFS）**：process 按到达顺序运行。简单但受**护航效应**困扰：一个长时间运行的 process 会阻塞其后所有更短的 process。

- **最短作业优先（SJF）**：先运行最短的 process。可证明能最小化平均等待时间，但需要事先知道作业长度（一般不可能）。其抢占式版本**最短剩余时间优先（SRTF）**：如果有更短的作业到达，会中断正在运行的作业。

- **轮转（RR）**：每个 process 获得一个固定的**时间片**（例如 10 ms），然后被抢占并移到队列末尾。公平且响应快，但时间片大小很关键：太小意味着过多的上下文切换，太大则退化为 FCFS。

- **优先级调度**：每个 process 有一个优先级。高优先级 process 先运行。危险在于**饥饿**：如果高优先级 process 不断到来，低优先级 process 可能永远得不到运行。**老化**可解决：一个 process 等得越久，其优先级就越高。

- **多级反馈队列（MLFQ）**：多个具有不同优先级和时间片的队列。新 process 从最高优先级队列（短时间片）开始。如果一个 process 用完了完整时间片（CPU 密集型），它被降级到更低优先级队列（更长的时间片）。交互式 process 自然留在高优先级队列（它们在用完时间片前就为 I/O 阻塞）。这无需事先知道作业类型就能适应工作负载。

- **完全公平调度器（CFS）**：Linux 的调度器。它维护一棵红黑树（一种平衡二叉搜索树），按“虚拟运行时间”——即它们消耗了多少 CPU 时间——排序 process。虚拟运行时间最小的 process 下一个运行。这保证了随着时间推移，每个 process 都获得其公平份额。CFS 每个调度决策在 $O(\log n)$ 内完成。

## 内存管理

- OS 管理物理 RAM，把它分配给 process 并在不再需要时回收。

- **分页**（来自文件 2）把虚拟内存划分为定长的页，把物理内存划分为帧。页表把页映射到帧。分页消除了外部碎片（分配之间浪费的空间），因为所有页大小相同。

- **按需分页**只在页首次被访问时（而非 process 启动时）才把页装入 RAM。这节省了内存：一个有 1 GB 代码的程序在典型运行中可能只用到 50 MB。其余永远不会被装入。

- 当 RAM 已满且需要新页时，OS 必须**驱逐**一个已有页。**页面置换**算法（LRU、FIFO、时钟，来自文件 2）决定驱逐哪一页。好的置换最小化缺页；差的置换导致颠簸。

- **分段**把内存划分为变长的段（代码、数据、栈、heap），各有自己的基址和长度。段提供逻辑组织，而分页提供物理管理。现代系统很少使用分段（主要用于保护），而依赖分页做内存管理。

- **heap** 是动态分配内存所在之处（C 中的 `malloc`/`free`，Java 中的 `new`，Python 中是隐式的）。OS 向 process 提供大块内存，由**内存分配器**（例如 `glibc malloc`、`jemalloc`、`tcmalloc`）把这些大块细分为更小的分配。分配器设计影响性能：碎片浪费空间，thread 间的争用浪费时间。

## 文件系统

- **file system（文件系统）**把持久存储（SSD、HDD）上的数据组织为命名文件和目录的层级。

- **inode**（索引节点）存储文件的元数据：大小、所有权、权限、时间戳，以及指向磁盘上数据块的指针。文件名存储在目录中，目录把名字映射到 inode 编号。这种分离意味着一个文件可以有多个名字（**硬链接**）指向同一个 inode。

- **FAT**（文件分配表）：一种简单的 file system，用于 U 盘和 SD 卡。一张表把每个簇（块）映射到文件中的下一个簇，形成一条链表。简单，但对权限、日志和大文件支持不佳。

- **ext4**：Linux 默认的 file system。使用带直接、间接、二级间接和三级间接块指针的 inode，处理任意大小的文件。支持**区段**（连续的块范围）以提升对大文件的效率。最大文件大小：16 TB，最大分区：1 EB。

- **日志**防止崩溃导致的损坏。在修改 file system 结构之前，把变更写入一个**日志**（log）。如果系统在中途崩溃，重启时会重放日志以完成或撤销该操作。没有日志，写入期间的崩溃可能让 file system 处于不一致状态（文件的数据块已更新而 inode 未更新，或反之）。

- **基于 B 树的 file system**（Btrfs、ZFS）使用 B 树（平衡搜索树）组织数据和元数据，支持高效搜索、写时复制快照和内置校验和以保证数据完整性。这些与数据库索引中使用的 B 树相同。

## 系统调用与内核模式

- **系统调用（system call）**是用户程序与 OS kernel 之间的接口。当程序需要做某些特权操作（读文件、分配内存、创建 process、发送网络包）时，它发起一次 syscall。

- CPU 在两种模式下运行：
    - **用户模式**：受限。程序可以执行自己的代码、访问自己的内存，但不能直接访问硬件、其他 process 的内存或 OS 数据结构。
    - **内核模式**：不受限。OS kernel 可以访问所有硬件和内存。syscall 是从用户模式到内核模式的受控通道。

- 当程序调用 `read()` 时，会发生以下过程：
    1. 程序把参数放入寄存器，并触发一次**陷入**（软件中断）。
    2. CPU 切换到 kernel 模式，并跳转到 syscall 处理程序。
    3. kernel 校验参数，执行 I/O 操作，并把数据复制到用户缓冲区。
    4. kernel 切换回用户模式并返回结果。

- 常见的 syscall：`open`、`read`、`write`、`close`（文件），`fork`、`exec`、`wait`、`exit`（进程），`mmap`、`brk`（内存），`socket`、`bind`、`listen`、`accept`（网络）。

- **中断**是硬件信号，强制 CPU 临时停止当前工作并运行一个中断处理程序（在 kernel 中）。一次按键、一个网络包到来或一次定时器滴答都会产生中断。定时器中断尤为重要：正是它让 OS 能抢占正在运行的 process 并切换到另一个（抢占式多任务）。

## 网络基础

- 网络协议栈是 OS 的一个子系统，使机器之间能够通信。理解它就能解释分布式训练如何同步梯度、模型服务如何处理请求，以及为什么延迟很重要。

![TCP/IP 协议栈：应用层、传输层、网络层、链路层，每层添加各自的首部](../images/tcp_ip_layers.svg)

- **TCP/IP 模型**把网络组织为层级，每层向上层提供抽象：

    - **链路层**：处理单条物理链路上的通信（以太网、Wi-Fi）。处理 MAC 地址和帧。
    - **网络层（IP）**：跨多个网络从源到目的路由包。每台机器有一个 **IP 地址**（例如 IPv4 的 192.168.1.1，或 128 位的 IPv6 地址）。路由器根据目的 IP 逐跳转发包。
    - **传输层（TCP/UDP）**：提供应用程序之间的端到端通信。
    - **应用层**：应用程序直接使用的协议，如 HTTP、DNS、gRPC。

- **TCP**（传输控制协议）提供可靠、按序的交付。它建立连接（三次握手：SYN、SYN-ACK、ACK），保证所有数据按序到达（使用序列号和确认），重传丢失的包，并控制发送速率以避免压垮网络（**拥塞控制**）。代价是延迟：握手增加一个往返，重传增加延迟。

- **UDP**（用户数据报协议）提供不可靠、不按序的交付。无握手、无重传、无顺序保证。延迟比 TCP 低得多。用于速度比可靠性更重要的场景：视频流、在线游戏、DNS 查询。在 ML 中，一些梯度同步协议使用基于 UDP 的 RDMA 以获得更低延迟。

- **套接字**是 OS 网络通信的 API。一个**套接字**是由（IP 地址，端口号）标识的端点。服务器创建一个套接字，绑定到某个端口（例如 HTTP 用 80），监听连接并接受它们。客户端创建一个套接字并连接到服务器的地址:端口。然后通过套接字读写数据，就像对文件一样。

- **DNS**（域名系统）把人类可读的名字（google.com）翻译成 IP 地址（142.250.80.46）。它是一个分布式、层次化的数据库：你的机器询问本地解析器，本地解析器询问根服务器，根服务器再委托给每个域的权威服务器。

- **HTTP**（超文本传输协议）是 Web 的请求-响应协议。客户端发送一个请求（方法 + URL + 首部 + 可选的正文），服务器返回一个响应（状态码 + 首部 + 正文）。ML 模型服务（例如 TensorFlow Serving、Triton）把模型暴露为 HTTP 或 gRPC 端点。

- **延迟 vs 带宽**：延迟是一个包从源到目的所用的时间（由物理距离和网络跳数决定）。带宽是数据速率（每秒多少字节）。高带宽、高延迟的连接（卫星互联网）可以传输大量数据，但每个字节都要花很长时间才能到达。对分布式训练来说，**延迟**对同步屏障很重要（所有 GPU 必须等最慢的那个），而**带宽**对传输大梯度张量很重要（第 6 章）。

## 虚拟化与容器

- **虚拟化**在单台物理机上运行多个操作系统。**hypervisor**（VMware、KVM、Xen）创建**虚拟机（VM）**，每个有自己的虚拟 CPU、内存、磁盘和网络接口。每个 VM 运行一个完整的 OS（客户 OS），它以为自己拥有专用硬件。

- VM 提供强隔离（一个 VM 崩溃不影响其他）和灵活性（在同一台机器上运行 Linux 和 Windows，在物理主机间迁移 VM）。代价是开销：每个 VM 运行一个完整的 OS kernel，用于 OS 操作的内存和 CPU 与主机 OS 冗余。

![VM 在虚拟硬件上运行独立的客户 OS；容器共享主机 kernel，更轻量](../images/container_vs_vm.svg)

- **容器**（Docker、Podman）提供更轻的替代方案。容器不虚拟化整套硬件，而是共享主机 OS kernel，并使用 kernel 特性来隔离 process：

    - **命名空间**隔离一个 process 能看到什么：每个容器有自己的进程树视图（PID 命名空间）、网络接口（网络命名空间）、文件系统挂载点（挂载命名空间）和主机名（UTS 命名空间）。容器内的 process 看不到其他容器中的 process。

    - **cgroup**（控制组）限制一个 process 能使用什么：CPU 时间、内存、磁盘 I/O、网络带宽。一个容器不能消耗超过其 cgroup 允许的资源，防止一个容器饿死其他容器。

- 容器在毫秒内启动（无需 OS 引导），开销极小（共享 kernel），并由一个 **Dockerfile** 定义，该文件指定基础镜像、依赖和命令。这使它们可复现：`docker build` 在任何地方都产生相同的环境。

- 对 ML 而言，容器解决了“在我机器上能跑”的问题。一个带有特定版本 CUDA、cuDNN、PyTorch 和 Python 的训练环境被封装为容器镜像。任何人都可以在任何机器上复现完全相同的环境。云训练平台（AWS SageMaker、GCP Vertex AI）在容器中运行训练作业。

- **Kubernetes**（K8s）大规模编排容器：它把容器调度到一组机器上，重启失败的容器，按负载伸缩，并管理容器之间的网络。大规模 ML 服务（数千个模型副本处理数百万请求）就运行在 Kubernetes 上。

## 安全基础

- OS 通过多种机制实施安全：

- **权限**：每个文件有一个所有者、一个组，以及权限位（对所有者、组、其他人的读/写/执行）。一个 process 以启动它的用户的身份（UID）运行，只能访问权限位允许的文件。**root** 用户（UID 0）绕过所有权限检查，这就是为什么以 root 运行很危险。

- **特权分离**：process 以所需的最小特权运行。Web 服务器不需要 root 访问权；它应以一个受限用户运行，只能读取 Web 文件并绑定到端口 80。如果服务器被攻陷，攻击者的访问被限制在该受限用户能做的事情。

- **沙箱**：在文件权限之外限制一个 process 能做什么。**seccomp**（Linux）限制一个 process 能发起哪些 syscall。**AppArmor** 和 **SELinux** 定义强制访问控制策略。容器结合命名空间、cgroup 和 seccomp 实现多层隔离。

- **地址空间布局随机化（ASLR）**：每次程序运行时随机化栈、heap 和库的内存位置。这使得攻击者更难利用内存损坏 bug（缓冲区溢出），因为他们无法预测代码或数据在内存中的位置。

- 安全是系统全局性的关注点：一条链条的强度取决于最薄弱的环节。一个模型服务系统需要安全的网络通信（TLS/HTTPS）、经过认证的 API 访问（API 密钥、OAuth）、输入校验（防止对抗性输入）以及隔离的执行（具有最小特权的容器）。

1. 探索进程创建。使用 Python 的 `os.fork()`（仅 Unix）创建一个子 process，观察父 process 与子 process 都从同一点继续。
```python
import os

pid = os.fork()

if pid == 0:
    # Child process
    print(f"Child: my PID is {os.getpid()}, parent PID is {os.getppid()}")
else:
    # Parent process
    print(f"Parent: my PID is {os.getpid()}, child PID is {pid}")
    os.wait()  # wait for child to finish
```

```python
def round_robin(processes, quantum=3):
    """Simulate round-robin scheduling.
    processes: list of (name, burst_time) tuples.
    """
    queue = [(name, burst, 0) for name, burst in processes]  # (name, remaining, wait)
    time = 0
    log = []

    while queue:
        name, remaining, waited = queue.pop(0)
        waited += (time - waited - (processes[[p[0] for p in processes].index(name)][1] - remaining))
        run_time = min(quantum, remaining)
        log.append(f"  t={time:3d}: {name} runs for {run_time} (remaining: {remaining - run_time})")
        time += run_time
        remaining -= run_time

        if remaining > 0:
            queue.append((name, remaining, time))
        else:
            log.append(f"  t={time:3d}: {name} DONE (turnaround: {time})")

    for line in log:
        print(line)

print("Round Robin (quantum=3):")
round_robin([("P1", 10), ("P2", 4), ("P3", 6)], quantum=3)
```

3. 用 LRU 模拟页面置换。给定一串页面访问和固定数量的帧，统计缺页数。
```python
def lru_page_replacement(pages, n_frames):
    """Simulate LRU page replacement."""
    frames = []
    faults = 0

    for page in pages:
        if page in frames:
            frames.remove(page)
            frames.append(page)  # move to most recently used
            status = "HIT "
        else:
            faults += 1
            if len(frames) >= n_frames:
                evicted = frames.pop(0)  # remove least recently used
                status = f"MISS (evict {evicted})"
            else:
                status = "MISS (cold)"
            frames.append(page)
        print(f"  Page {page}: {status}  frames={frames}")

    print(f"\nTotal faults: {faults}/{len(pages)} ({faults/len(pages):.0%})")

print("LRU with 3 frames:")
lru_page_replacement([1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5], n_frames=3)
```
