# 计算机体系结构

*计算机体系结构是关于如何构建执行指令的机器的学问。本文件涵盖数制、逻辑门、CPU 设计、指令集架构、流水线、存储层级和虚拟内存，是每个程序、框架和 AI 模型最终运行其上的硬件基础。*

- 每个神经网络、每个训练循环、每次推理调用最终都变成流经晶体管的一串电信号。对于严谨的 ML 从业者来说，理解硬件并非可有可无：它解释了为什么矩阵乘法很快、为什么 memory 是瓶颈、为什么 GPU 在 AI 训练中占主导，以及为什么对 cache 友好的代码可能比朴素代码快 100 倍。

## 数制

- 计算机把一切都表示为**二进制**（基 2）：0 和 1 的序列。每个数字是一个**比特**。一组 8 个比特是一个**字节**。一个二进制数 $b_{n-1} b_{n-2} \ldots b_1 b_0$ 的值为 $\sum_{i=0}^{n-1} b_i \cdot 2^i$。

- 例如，$1011_2 = 1 \cdot 8 + 0 \cdot 4 + 1 \cdot 2 + 1 \cdot 1 = 11_{10}$。

- **十六进制**（基 16）是二进制的一种紧凑记法。每个十六进制数字代表 4 个比特：$0\text{-}9$ 映射到 $0000\text{-}1001$，$A\text{-}F$ 映射到 $1010\text{-}1111$。所以 $\text{0xFF} = 1111\,1111_2 = 255_{10}$。内存地址和颜色代码通常用十六进制书写。

- **二进制补码**表示带符号整数。对于 $n$ 位数字，最高位的权重是 $-2^{n-1}$ 而不是 $+2^{n-1}$。8 位补码的范围从 $-128$ 到 $+127$。取负的方法：翻转所有位再加 1。这种表示使加法和减法使用同一套硬件电路，因此被普遍采用。

- **IEEE 754 浮点数**把实数表示为 $(-1)^s \times 1.m \times 2^{e-\text{bias}}$，其中 $s$ 是符号位，$m$ 是尾数（小数部分），$e$ 是经偏移的指数。

![IEEE 754 float32 布局：1 位符号位，8 位指数，23 位尾数](../images/ieee754_float.svg)

    - **float32**（单精度）：1 符号 + 8 指数 + 23 尾数 = 32 位。范围：$\approx \pm 3.4 \times 10^{38}$，精度：约 7 位十进制数字。
    - **float64**（双精度）：1 符号 + 11 指数 + 52 尾数 = 64 位。范围：$\approx \pm 1.8 \times 10^{308}$，精度：约 15 位十进制数字。
    - **float16**（半精度）：1 + 5 + 10 = 16 位。范围和精度有限，但占用一半的 memory 和带宽。广泛用于 ML 训练（混合精度，第 6 章）。
    - **bfloat16**：1 + 8 + 7 = 16 位。与 float32 相同的指数范围，但精度更低。由 Google 专为 ML 设计：完整的指数范围防止训练期间溢出，而降低的精度对梯度更新是可接受的。

- 浮点运算**并不精确**。在 float64 中 $0.1 + 0.2 \neq 0.3$（它等于 $0.30000000000000004$）。这是因为 $0.1$ 没有精确的二进制表示，就像 $1/3$ 没有精确的十进制表示。在数百万次运算中累积这些误差（如梯度下降）会导致数值不稳定，因此出现了损失缩放（第 6 章）和 Kahan 求和这类技术。

## 逻辑门

- 所有计算都归约为**逻辑门**：实现布尔运算（文件 1 中的命题逻辑）的物理电路。

- 基本门：
    - **AND**：仅当两个输入都为 1 时输出为 1。
    - **OR**：至少一个输入为 1 时输出为 1。
    - **NOT**（反相器）：翻转输入。
    - **NAND**（NOT-AND）：通用门。任何其他门都可以仅用 NAND 门构建。这就是 NAND 成为数字电路基础构件的原因。
    - **XOR**（异或）：当输入不同时输出为 1。对加法（二进制加法的和位是 XOR）和密码学至关重要。

- **半加器**用 XOR（和）和 AND（进位）把两个单比特相加。**全加器**把两个比特加上进位输入链接起来，构成一个 $n$ 位加法器。CPU 就是这样执行整数加法的：一串简单逻辑门的级联。

- **多路选择器**（MUX）根据控制信号从多个输入中选择一个。用 $n$ 个控制位，它可以从 $2^n$ 个输入中选择。多路选择器是 if-else 链的硬件等价物，在 CPU 数据通路中广泛用于路由数据。

- 现代处理器包含数十亿晶体管，每个都充当一个微小开关。一个晶体管要么导通（导电，表示 1）要么截止（不导电，表示 0）。门由晶体管构成，加法器由门构成，ALU 由加法器构成，CPU 由 ALU 构成。整个计算层级都建立在这个基础之上。

## CPU 架构

- **中央处理器（CPU）**执行指令。它的核心组件：

    - **ALU**（算术逻辑单元）：执行整数算术（加、减、乘）和逻辑运算（AND、OR、XOR、移位）。这是实际计算发生的地方，由上述逻辑门构成。

    - **寄存器**：CPU 内部微小、超快的存储位置。现代 CPU 有数十个通用寄存器，每个保存一个字（在 64 位 CPU 上是 64 位）。寄存器是系统中最快的 memory：访问约需 0.3 纳秒。

    - **程序计数器（PC）**：保存下一条要执行指令的内存地址。

    - **控制单元**：解码指令并编排数据通路，告诉 ALU 执行什么运算以及使用哪些寄存器。

- **指令周期**（取指-译码-执行）每秒重复数十亿次：

    1. **取指**：从 PC 中的地址处读取内存中的指令。
    2. **译码**：确定指令做什么（加法？从内存装载？分支？）以及使用哪些操作数。
    3. **执行**：执行运算（ALU 计算、内存访问或分支）。
    4. 递增 PC（除非该指令是分支/跳转）。

- 运行在 4 GHz 的 CPU 每秒执行 40 亿个周期。每个周期耗时 0.25 纳秒。在这段时间里，光大约行进 7.5 厘米，这就是为什么芯片的物理尺寸很重要：信号无法在一个周期内跨越一块大芯片。

## 指令集架构

- **指令集架构（ISA）**是硬件与软件之间的契约：它定义了 CPU 能理解的指令、寄存器组、内存模型和编码格式。

- **CISC**（复杂指令集计算机）：指令可以很复杂、变长，并可以直接访问内存。一条指令可能将两个内存中的值相乘并存储结果。**x86**（Intel/AMD）是主导的 CISC ISA，驱动着大多数台式机和服务器。它的向后兼容性（现代 x86 CPU 仍能运行 1980 年代的代码）既是它的优势，也是它的负担。

- **RISC**（精简指令集计算机）：指令简单、定长，只在寄存器上运算。访问内存需要单独的 load/store 指令。更简单的指令能实现更快的时钟速度和更简便的流水线。

    - **ARM**：移动设备以及越来越多服务器和笔记本（Apple M 系列芯片是 ARM）的主导 RISC ISA。ARM 的能效使其成为电池供电和热受限设备的理想选择。
    - **RISC-V**：一种开源 RISC ISA。任何人都可以设计 RISC-V 芯片而无需支付授权费。在嵌入式系统、研究和 AI 加速器中采用率不断增长。

- CISC 与 RISC 的界限已变得模糊：现代 x86 CPU 在内部把复杂的 CISC 指令解码为更简单的微操作（本质上是内部 RISC），兼得两者之长。

## 流水线

- 没有流水线时，CPU 在开始下一条指令前要完整地完成当前指令。这浪费了硬件：当 ALU 在执行时，取指和译码单元却闲置着。

![CPU 流水线：指令在取指、译码、执行、内存、写回等阶段间重叠](../images/cpu_pipeline.svg)

- **流水线**让指令的执行重叠，就像流水线一样。当指令 1 在执行时，指令 2 在译码，指令 3 在取指。一个 5 级流水线（取指、译码、执行、内存访问、写回）可以同时有 5 条指令在飞行中。

- 吞吐率接近每周期一条指令（尽管每条指令需 5 个周期才能完成）。这与 ML 中流水线的原理相同：数据并行性让计算与通信重叠（第 6 章）。

- **冒险**是流水线失效的情形：

    - **数据冒险**：指令 2 需要一个尚未由指令 1 生成的结果。“Add R1, R2, R3”后接“Sub R4, R1, R5”——第二条指令需要 R1，而第一条仍在计算它。**前递**（旁路）通过把结果从一个流水级直接路由到另一个、无需等待写回阶段来解决此问题。

    - **控制冒险**：一条分支指令（if-else）意味着 CPU 在分支解析之前不知道接下来取哪条指令。**分支预测**猜测分支走向，并沿预测路径投机地取指。现代预测器准确率超过 95%，使用历史表和类似神经网络的模式匹配。一次误预测代价约 15 个周期（必须冲刷并重启流水线）。

    - **结构冒险**：两条指令同时需要同一硬件资源（例如都需要内存端口）。通过复制资源或插入停顿来解决。

## 存储层级

- 计算机内存中的根本矛盾：快的 memory 贵且小，便宜的 memory 慢且大。**存储层级**通过利用**局部性**来弥合这一差距：程序倾向于反复访问同一数据（时间局部性）并访问邻近数据（空间局部性）。

![存储层级金字塔：顶端是寄存器（快、小），底端是 HDD（慢、大）](../images/memory_hierarchy.svg)

- 该层级从快到慢依次为：

    - **寄存器**：约 0.3 ns 访问，总量约 KB 级。位于 CPU 内部。
    - **L1 cache**：约 1 ns，每核 32-64 KB。分为指令 cache 和数据 cache。
    - **L2 cache**：约 4 ns，每核 256 KB-1 MB。
    - **L3 cache**：约 10 ns，跨核共享 8-64 MB。
    - **RAM（DRAM）**：约 50-100 ns，8-512 GB。主存。
    - **SSD**：约 10-100 μs，256 GB-8 TB。持久存储。
    - **HDD**：约 5-10 ms，1-20 TB。机械式，随机访问非常慢。

- 寄存器与 RAM 之间的速度差距约为 300 倍。寄存器与磁盘之间约为 30,000,000 倍。cache 层级隐藏了这一差距：如果 CPU 需要的数据在 L1 cache 中（一次 **cache 命中**），访问很快。否则（一次 **cache 缺失**），CPU 停顿，从更慢的层级取数据。

- **cache 相联度**决定一个内存地址可以存放在 cache 中的何处：
    - **直接映射**：每个地址恰好映射到一条 cache 行。简单但易产生冲突。
    - **全相联**：任何地址可以放在任何位置。灵活但搜索代价高。
    - **组相联**（$k$ 路）：每个地址映射到 $k$ 个位置组成的一组。真实 CPU 中使用的实用折中（通常是 4 路或 8 路）。

- **cache 一致性**确保所有 CPU 核看到一致的内存视图。当核 1 写入核 2 已 cache 的地址时，一致性协议（例如 MESI）使核 2 的副本失效或更新它。这对并发编程（文件 4）至关重要，也是共享内存并行难以实现的原因之一。

- 对 ML 从业者来说，存储层级解释了为什么：
    - 矩阵运算应当顺序地访问内存（行优先 vs 列优先布局很重要）。
    - 批量大小影响性能：更大的批量摊薄了内存延迟。
    - 混合精度（float16/bfloat16）使有效内存带宽加倍，而带宽往往是瓶颈。

## 虚拟内存

- **虚拟内存**给每个 process 赋予一种拥有自己大而连续的内存空间的错觉，尽管物理 RAM 有限且为多个 process 共享。

- 地址空间被划分为定长的**页**（通常 4 KB）。**页表**把虚拟页号映射到物理帧号。当程序访问虚拟地址 0x1234 时，CPU 通过查页表把它翻译为物理地址。

- **转换后备缓冲器（TLB）**是页表项的 cache。由于页表位于 RAM 中（慢），TLB 把最近使用的翻译存放在快速硬件中。一次 TLB 缺失需要在内存中遍历页表，耗费数百个周期。

- 当程序访问一个不在物理 RAM 中的页时发生**缺页**。OS 从磁盘加载该页（交换），耗时数百万个周期。过多的缺页（**颠簸**）会严重拖垮性能。这就是为什么 ML 训练需要足够的 RAM 来容纳模型、优化器状态和合理批量的数据。

- **页面置换**算法决定 RAM 满时驱逐哪一页：
    - **LRU**（最近最少使用）：驱逐最长时间未被访问的页。对大多数工作负载在实践中最优。在硬件中用**时钟算法**（一个带引用位的循环列表）来近似。
    - **FIFO**：驱逐最旧的页。简单但可能驱逐频繁使用的页。
    - **最优**（Bélády 算法）：驱逐将来最长时间不会被使用的页。无法实现（需要未来知识），但作为理论基准很有用。

- 虚拟内存还提供**隔离**：每个 process 有自己的虚拟地址空间。一个 process 中的 bug 不能破坏另一个 process 的内存，因为它们的虚拟地址映射到不同的物理帧。这是 OS 安全与稳定性的基础。

## I/O、中断与 DMA

- CPU 需要与外部世界通信：磁盘、网卡、键盘、GPU。这就是 **I/O 子系统**。

- **程序 I/O**（轮询）：CPU 在循环中反复检查设备的状态寄存器，等待数据就绪。简单但浪费 CPU 周期在空转上，而非做有用的工作。

- **中断驱动 I/O**：当数据就绪时，设备发送一个硬件**中断**。CPU 继续正常执行，直到中断到来，然后运行一个**中断处理程序**（一个 kernel 函数）来处理数据。这比轮询高效得多，因为 CPU 在等待时不会闲置。

- 中断机制：
    1. 设备通过硬件线路发出中断信号。
    2. CPU 完成当前指令，把当前状态（寄存器、程序计数器）保存到栈上。
    3. CPU 在**中断向量表**（一个函数指针表，每种中断类型一个）中查找中断处理程序地址。
    4. 处理程序在 kernel 模式下运行，处理 I/O，然后返回。
    5. CPU 恢复保存的状态并恢复被中断的程序。

- 这与上下文切换（文件 3）是相同的保存/恢复模式，但由硬件而非定时器触发。

- **DMA**（直接内存访问）：对于大数据传输（磁盘读取、网络包、GPU 内存拷贝），让 CPU 逐字节复制数据很浪费。**DMA 控制器**在设备和 RAM 之间直接传输数据而无需 CPU 介入。CPU 设置好传输（源、目的、大小），DMA 控制器处理它，完成时 CPU 收到一个中断。

- DMA 对 ML 至关重要：当你调用 `model.to('cuda')` 时，数据经 PCIe 总线通过 DMA 从系统 RAM 传到 GPU 内存。在训练期间，跨 GPU 的梯度同步使用基于 DMA 的 RDMA（远程 DMA）以获得高带宽、低延迟的传输（第 6 章）。

- **总线**把 CPU 连接到内存和 I/O 设备。现代系统使用 **PCIe**（外围组件互连高速）连接高速设备（GPU、NVMe SSD、网卡）。PCIe 4.0 每个 x16 插槽提供约 32 GB/s；PCIe 5.0 翻倍。总线带宽常常是 GPU 训练的瓶颈：GPU 的计算速度可以超过数据供给它的速度。

- **MMIO**（内存映射 I/O）：设备寄存器被映射到内存地址。CPU 使用普通的 load/store 指令读写这些地址，硬件把访问路由到设备而非 RAM。这把内存访问和 I/O 访问统一为单一机制，简化了硬件和软件。

1. 探索 IEEE 754 浮点表示。把一个 float 转换为它的二进制表示，观察符号、指数和尾数字段。
```python
import struct

def float_to_bits(f):
    """Show the IEEE 754 binary representation of a float32."""
    packed = struct.pack('>f', f)
    bits = ''.join(f'{byte:08b}' for byte in packed)
    sign = bits[0]
    exponent = bits[1:9]
    mantissa = bits[9:]
    return sign, exponent, mantissa

for val in [1.0, -1.0, 0.1, 0.5, 3.14, float('inf'), float('nan')]:
    s, e, m = float_to_bits(val)
    print(f"{val:>10}  sign={s}  exp={e} ({int(e, 2) - 127:>4d})  mantissa={m[:10]}...")
```

2. 模拟一个直接映射 cache。对一系列内存访问跟踪命中和缺失。
```python
def simulate_cache(accesses, cache_size=8, block_size=1):
    """Simulate a direct-mapped cache."""
    cache = [None] * cache_size
    hits, misses = 0, 0

    for addr in accesses:
        cache_line = addr % cache_size
        if cache[cache_line] == addr:
            hits += 1
            status = "HIT "
        else:
            misses += 1
            cache[cache_line] = addr
            status = "MISS"
        print(f"  Access {addr:3d} → line {cache_line}: {status}")

    print(f"\nHits: {hits}, Misses: {misses}, Hit rate: {hits/(hits+misses):.1%}")

# Sequential access (good locality)
print("Sequential access:")
simulate_cache([0, 1, 2, 3, 4, 5, 6, 7, 0, 1, 2, 3])

# Strided access (conflict misses)
print("\nStrided access (stride = cache size):")
simulate_cache([0, 8, 0, 8, 0, 8])
```
