# Языки программирования

*Языки программирования — это интерфейс между человеческим намерением и машинным исполнением. В этом файле рассматриваются парадигмы языков, системы типов, стратегии управления памятью, конвейер компиляции, интерпретация и JIT-компиляция, ключевые особенности языков, предметно-ориентированные языки и компромиссы при проектировании.*

- Любое программное обеспечение, любая модель машинного обучения, любая операционная система написаны на языке программирования. Но существуют сотни языков, и у каждого свои сильные стороны. Почему? Потому что проектирование языка подразумевает фундаментальные компромиссы: производительность против безопасности, выразительность против простоты, контроль против абстракции. Понимание этих компромиссов помогает выбрать правильный инструмент для задачи и осознать ограничения, в рамках которых вы работаете.

## Парадигмы программирования

- **Парадигма** — это стиль программирования: набор принципов, которые определяют, как вы структурируете код и мыслите при решении задач.

- **Императивное** программирование описывает вычисления как последовательность команд, изменяющих состояние. «Присвоить x значение 5. Прибавить к x число 3. Если x > 7, вывести его». C, Python и Java по своей сути являются императивными. Ментальная модель здесь — это машина с памятью, которую вы модифицируете шаг за шагом.

- **Объектно-ориентированное (ООП)** программирование организует код вокруг **объектов**: наборов данных (атрибутов) и поведения (методов). Объекты взаимодействуют, посылая друг другу сообщения. Ключевые идеи — это **инкапсуляция** (скрытие внутреннего состояния за публичным интерфейсом), **наследование** (создание новых классов путем расширения существующих) и **полиморфизм** (единообразная обработка различных типов через общий интерфейс). Java, C++ и Python поддерживают ООП.

- **Функциональное программирование (ФП)** рассматривает вычисления как вычисление математических функций. Основные принципы: **неизменяемость** (данные не меняются после создания), **чистые функции** (результат зависит только от входных данных, нет побочных эффектов) и **функции первого класса** (функции являются значениями, которые можно передавать в качестве аргументов, возвращать из других функций и сохранять в переменных). Haskell является чисто функциональным языком. Python, JavaScript и Scala поддерживают функциональный стиль.

- Чистые функции легко анализировать, тестировать и распараллеливать (отсутствие общего изменяемого состояния означает отсутствие состояний гонки). Именно поэтому функциональные идеи всё чаще используются в распределенных системах и пайплайнах обработки данных. JAX (используемый в этой книге) является функциональным: `jax.grad` работает, потому что функции JAX чистые.

- **Логическое программирование** описывает, *что* должно быть истинным, а не *как* это вычислить. Вы задаете факты и правила, а среда выполнения находит решения. Prolog — классический пример: при заданных «Сократ — человек» и «все люди смертны» движок выводит «Сократ смертен». Логическое программирование используется в базах знаний ИИ и проверке типов.

- Большинство современных языков являются **мультипарадигмальными**: Python поддерживает императивный, объектно-ориентированный и функциональный стили. Rust поддерживает императивный и функциональный. Парадигма — это инструмент, а не религия.

## Системы типов

- **Тип** классифицирует значения и определяет, какие операции допустимы. Целое число 3 и строка "3" — это разные типы: вы можете складывать целые числа, но не строки (вернее, вы можете конкатенировать строки, но это другая операция).

- **Статическая типизация**: типы проверяются на этапе **компиляции**, до запуска программы. Ошибки типов обнаруживаются рано. C, Java, Rust и Go являются статически типизированными. Вы должны объявлять типы (или компилятор выводит их сам):

```rust
let x: i32 = 5;     // Rust: x is a 32-bit integer
let y: f64 = 3.14;  // y is a 64-bit float
// let z = x + y;    // compile error: cannot add i32 and f64
```

- **Динамическая типизация**: типы проверяются во время **выполнения**, когда операции фактически исполняются. Это более гибко, но ошибки типов обнаруживаются только при запуске кода. Python, JavaScript и Ruby являются динамически типизированными:

```python
x = 5       # x is an int (for now)
x = "hello" # now x is a string -- no error
```

- **Строгая типизация**: язык предотвращает неявное приведение типов. Python строго типизирован: `"3" + 5` вызывает TypeError. **Слабая типизация**: язык неявно преобразует типы. JavaScript слабо типизирован: `"3" + 5` дает `"35"` (число приводится к строке). C слабо типизирован: вы можете привести указатель к целому числу.

- **Вывод типов** позволяет компилятору определять типы без явных аннотаций:

```rust
let x = 5;        // compiler infers: i32
let y = x + 3.0;  // compile error: mixed types, even with inference
```

- **Дженерики** (параметрический полиморфизм) позволяют писать код, который работает с любым типом:

```rust
fn largest<T: PartialOrd>(list: &[T]) -> &T {
    let mut max = &list[0];
    for item in &list[1..] {
        if item > max { max = item; }
    }
    max
}
// Works for integers, floats, strings -- any type that supports comparison
```

- Для машинного обучения: динамическая типизация Python ускоряет эксперименты, но скрывает ошибки. Системы машинного обучения в продакшене всё чаще используют подсказки типов (`def train(model: nn.Module, lr: float) -> float`) и инструменты статического анализа (mypy) для обнаружения ошибок до развёртывания. PyTorch и JAX используют Python для гибкости; TensorRT и ONNX Runtime используют C++ для производительности.

## Управление памятью

- Каждая программа выделяет и освобождает память. То, как это реализовано, является одним из самых важных решений при проектировании языка.

![Схема памяти: стек растет вниз от старших адресов, куча растет вверх, код и данные находятся внизу](../images/stack_vs_heap.svg)

- **Стек** хранит локальные переменные и кадры вызова функций. Выделение памяти тривиально (сдвиг указателя стека), а освобождение происходит автоматически (удаление кадра при возврате из функции). Доступ к стеку быстрый, так как он всегда находится в кеше. Однако стек имеет фиксированный размер (обычно 1-8 МБ) и поддерживает только LIFO-выделение (последним пришел — первым ушел).

- **Куча** хранит динамически выделяемые данные (объекты, массивы, строки, размер которых неизвестен на этапе компиляции). Выделение памяти в куче медленнее (нужно найти свободный блок) и требует явного или автоматического освобождения. Куча может расти, заполняя доступную память.

- **Ручное управление памятью** (C, C++): программист явно выделяет (`malloc`) и освобождает (`free`) память в куче. Это дает максимальный контроль и производительность, но крайне подвержено ошибкам:
    - **Use-after-free**: обращение к памяти, которая уже была освобождена. Приводит к сбоям или уязвимостям безопасности.
    - **Double free**: повторное освобождение одной и той же памяти. Повреждает внутренние структуры данных аллокатора.
    - **Утечка памяти**: выделение памяти без последующего освобождения. Программа постепенно поглощает всю доступную оперативную память.

- **Сборка мусора (GC)**: среда выполнения автоматически обнаруживает и освобождает память, которая больше не является доступной. Программист никогда не вызывает `free`.

    - **Трассирующая сборка мусора (Tracing GC)** (Java, Go, сборщик циклических ссылок в Python): периодически обходит все доступные объекты, начиная с «корней» (переменные стека, глобальные переменные), и освобождает недоступные объекты. Это просто, но вызывает **паузы GC**: программа останавливается на время работы сборщика. Современные сборщики (конкурентный GC в Go, ZGC в Java) минимизируют время пауз до долей миллисекунды.

    - **Подсчет ссылок (Reference counting)** (основной механизм Python, Swift, Objective-C): каждый объект отслеживает количество ссылок, указывающих на него. Когда счетчик падает до 0, объект немедленно освобождается. Пауз нет, но этот метод не справляется с **циклами** (A ссылается на B, B ссылается на A, у обоих счетчик > 0, но ни один из них не является доступным). Python использует отдельный детектор циклов для решения этой проблемы.

- **Владение (Ownership)** (Rust): компилятор обеспечивает правила безопасности памяти на этапе компиляции без каких-либо накладных расходов во время выполнения.

    - У каждого значения есть ровно один **владелец**. Когда владелец выходит из области видимости, значение удаляется (освобождается).
    - Значения можно **заимствовать** (получать ссылки), но компилятор накладывает ограничение: либо одна изменяемая ссылка, ЛИБО любое количество неизменяемых ссылок, но никогда и то, и другое одновременно.
    - Это предотвращает использование после освобождения (use-after-free), двойное освобождение, гонки данных и «висячие» указатели — всё на этапе компиляции. Никакого GC, никаких затрат во время выполнения.

- **Borrow checker** (проверщик заимствований) — это главная особенность Rust и самый сложный этап в его освоении. Он гарантирует безопасность памяти и потокобезопасность без сборки мусора, поэтому Rust всё чаще используется для критически важных систем (ядра ОС, игровые движки, среды выполнения для инференса ML, такие как Candle и Burn).

## Конвейер компиляции

- **Компилятор** переводит исходный код в машинный код (или другой целевой язык) до запуска программы. Конвейер состоит из нескольких этапов:

![Конвейер компиляции: исходный код → лексер → парсер → семантический анализ → оптимизатор → генерация кода → машинный код](../images/compilation_pipeline.svg)

1. **Лексический анализ (Lexing)** (токенизация): преобразование исходного текста в поток токенов. `x = 3 + y` превращается в `[IDENT("x"), EQUALS, INT(3), PLUS, IDENT("y")]`. Лексер удаляет пробелы и комментарии.

2. **Синтаксический анализ (Parsing)**: построение **абстрактного синтаксического дерева (AST)** из потока токенов. AST представляет иерархическую структуру программы. `3 + y * 2` разбирается как `Add(3, Mul(y, 2))` (умножение имеет более высокий приоритет). Парсер проверяет синтаксис: здесь обнаруживаются несоответствующие скобки и пропущенные точки с запятой.

3. **Семантический анализ**: проверка типов, разрешение имен переменных, проверка того, что функции вызываются с правильными аргументами. Именно здесь происходит статическая проверка типов. Результатом является типизированное, аннотированное AST.

4. **Оптимизация**: преобразование программы для более быстрого выполнения без изменения её поведения. Распространенные оптимизации:
    - **Свертка констант (Constant folding)**: вычисление `3 + 5` на этапе компиляции с заменой на `8`.
    - **Удаление мертвого кода (Dead code elimination)**: удаление кода, который никогда не будет выполнен.
    - **Развертывание циклов (Loop unrolling)**: замена цикла на повторяющийся встроенный код для уменьшения накладных расходов на переходы.
    - **Инлайнинг (Inlining)**: замена вызова функции на её тело, что устраняет накладные расходы на вызов.

5. **Генерация кода**: перевод оптимизированного представления в целевой машинный код (x86, ARM) или промежуточное представление.

- **LLVM** — доминирующая инфраструктура компиляторов. Она предоставляет общее промежуточное представление (LLVM IR), в которое компилируются многие языки. Оптимизатор LLVM работает с этим IR, а его бэкенды генерируют машинный код для множества целей. Clang (C/C++), Rust, Swift, Julia и многие другие языки используют LLVM. Это означает, что улучшения оптимизатора LLVM приносят пользу всем этим языкам одновременно.

## Интерпретация и JIT-компиляция

- **Интерпретатор** выполняет программу построчно (или пооператорно) без создания машинного кода. Это ускоряет запуск и делает разработку интерактивной, но выполнение происходит медленнее (каждая строка анализируется заново при каждом запуске).

- Большинство интерпретируемых языков на самом деле компилируются в **байт-код**: промежуточное представление, которое проще исходного кода, но не привязано к конкретной машине. Байт-код выполняется на **виртуальной машине (VM)**.

    - **CPython** (стандартная реализация Python) компилирует исходный код Python в байт-код (файлы `.pyc`), который выполняется виртуальной машиной CPython. VM интерпретирует байт-код по одной инструкции за раз. Именно поэтому Python примерно в 100 раз медленнее C для вычислительно сложных задач.

    - **JVM** (Java Virtual Machine): Java компилируется в байт-код JVM (файлы `.class`). JVM сначала интерпретирует байт-код, а затем **JIT-компилирует** часто выполняемые участки кода («горячие точки») в нативный машинный код. Вот почему Java запускается медленнее, чем C (накладные расходы на интерпретацию), но приближается к скорости C для долго работающих программ (благодаря JIT-оптимизированным «горячим точкам»).

- **JIT-компиляция (Just-In-Time)** компилирует код в машинный код во время выполнения, используя информацию, доступную только в процессе исполнения. JIT может оптимизировать код на основе реальных данных: если функция всегда вызывается с целочисленными аргументами, JIT генерирует специализированный машинный код только для целых чисел, пропуская проверки типов.

- **PyPy** — альтернативная реализация Python с JIT-компилятором. Она выполняет большинство программ на Python в 5–10 раз быстрее, чем CPython, за счет JIT-компиляции «горячих» циклов в машинный код. Однако у неё ограниченная совместимость с модулями расширения на C (NumPy, PyTorch), что ограничивает её использование в ML.

- Спектр от интерпретации до компиляции не является бинарным:
    - Чистая интерпретация: Bash-скрипты.
    - Интерпретация байт-кода: CPython.
    - Байт-код + JIT: JVM, .NET CLR, LuaJIT, PyPy.
    - Компиляция «заранее» (AOT): C, C++, Rust, Go.
    - AOT + генерация кода во время выполнения: `jax.jit` в JAX компилирует функции Python в оптимизированный код XLA при первом вызове, а затем кэширует скомпилированную версию.

## Ключевые возможности языков

- **Замыкания (Closures)**: функции, которые захватывают переменные из своей объемлющей области видимости. Функция «замыкает» окружение, в котором она была определена:

```python
def make_adder(n):
    def add(x):
        return x + n  # n is captured from the enclosing scope
    return add

add5 = make_adder(5)
print(add5(3))  # 8
```

- Замыкания (closures) — это механизм, лежащий в основе колбэков, декораторов и частичного применения функций. Они фундаментальны для функционального программирования.

- **Сопоставление с образцом (pattern matching)**: мощный механизм управления потоком выполнения, который деструктурирует данные и выполняет ветвление на основе их структуры:

```rust
match value {
    Some(x) if x > 0 => println!("Positive: {}", x),
    Some(0)           => println!("Zero"),
    Some(x)           => println!("Negative: {}", x),
    None              => println!("Nothing"),
}
```

- Сопоставление с образцом более выразительно, чем цепочки if-else: оно проверяет структуру данных (является ли она `Some` или `None`? содержит ли она значение, соответствующее условию?), а не просто равенство. В Python структурное сопоставление с образцом появилось в версии 3.10 (`match`/`case`).

- **Алгебраические типы данных (ADTs)**: типы, которые могут принимать один из нескольких вариантов, каждый из которых содержит различные данные. Тип `Result` — это либо `Ok(value)`, либо `Err(error)`. `Tree` — это либо `Leaf(value)`, либо `Node(left, right)`. Алгебраические типы данных в сочетании с сопоставлением с образцом позволяют исчерпывающе обрабатывать все случаи, устраняя целые классы ошибок (исключения нулевого указателя, необработанные коды ошибок).

- **Трейты и интерфейсы**: определяют набор методов, которые тип должен реализовать, не указывая, как именно. Это обеспечивает полиморфизм: функция, принимающая «всё, что реализует трейт Display», работает с целыми числами, строками и пользовательскими типами. В Rust используются трейты, в Java — интерфейсы, в Go — неявные интерфейсы, в Python — утиная типизация («если оно ходит как утка...»).

## Предметно-ориентированные языки (DSL)

- **Предметно-ориентированный язык (DSL)** разработан для конкретной предметной области, жертвуя общностью ради выразительности в рамках этой области.

- **SQL**: язык реляционных баз данных. Запрос `SELECT name FROM users WHERE age > 30` гораздо более читаем и оптимизируем, чем эквивалентный императивный цикл. Движок базы данных оптимизирует план выполнения запроса, автоматически выбирая стратегии соединения и использование индексов.

- **Регулярные выражения**: мини-язык для сопоставления с образцом в тексте. `\d{3}-\d{4}` соответствует номерам телефонов вида «555-1234». Движки регулярных выражений компилируют шаблоны в конечные автоматы для эффективного поиска.

- **Языки шейдеров** (GLSL, HLSL, Metal Shading Language): программы, которые выполняются на ядрах GPU для вычисления цветов пикселей, положений вершин или выполнения вычислительных операций. Шейдеры обладают массовым параллелизмом: каждое выполнение обрабатывает один пиксель или один элемент независимо. Это та же модель выполнения, которую использует CUDA для вычислений в машинном обучении.

- В машинном обучении такие фреймворки, как PyTorch и JAX, по сути являются DSL для тензорных вычислений, встроенными в Python. Они предоставляют предметно-ориентированные абстракции (тензоры, автоматическое дифференцирование, размещение на устройстве), используя при этом экосистему Python.

## Компромиссы при проектировании языков

- Нет языка, который был бы лучшим во всём. Проектирование — это выбор того, на какие компромиссы пойти:

- **Производительность против безопасности**: C дает «сырую» скорость и контроль над оборудованием, но позволяет повредить память. Rust обеспечивает сопоставимую скорость при безопасности памяти на этапе компиляции. Java обеспечивает безопасность памяти ценой накладных расходов на сборку мусора. Python обеспечивает максимальную безопасность и выразительность при выполнении, которое в 100 раз медленнее.

- **Выразительность против простоты**: система типов Haskell позволяет выражать очень точные ограничения, но имеет крутую кривую обучения. Go намеренно опускает дженерики (до недавнего времени), наследование и исключения ради простоты. Философия Python «должен существовать один очевидный способ сделать это» позволяет легко изучать язык.

- **Контроль против абстракции**: C/C++ дают контроль над структурой памяти, поведением кеша и взаимодействием с оборудованием. Python скрывает всё это. Для обучения моделей (где доминируют вычисления на GPU) накладные расходы Python незначительны. Для инференса (где важна каждая микросекунда) могут потребоваться C++ или Rust.

- **Скорость компиляции против скорости выполнения**: Go компилируется за секунды (простая система типов, минимальная оптимизация). Rust компилируется за минуты (сложная система типов, агрессивная оптимизация). Компромисс заключается в скорости итерации разработчика против производительности развернутого приложения.

- Экосистема машинного обучения отражает эти компромиссы: Python для экспериментов и обучения (побеждает выразительность), C++/CUDA для ядер и инференса (побеждает производительность), Rust для инфраструктуры и критически важных систем (побеждает безопасность).

```python
def make_multiplier(factor):
    """Returns a function that multiplies by factor."""
    def multiply(x):
        return x * factor
    return multiply

double = make_multiplier(2)
triple = make_multiplier(3)

# Closures capture by reference, not by value
def make_counter():
    count = [0]  # mutable container to allow modification
    def increment():
        count[0] += 1
        return count[0]
    return increment

counter = make_counter()
print(f"counter() = {counter()}")  # 1
print(f"counter() = {counter()}")  # 2
print(f"counter() = {counter()}")  # 3
```

2. Сравните поведение динамической и статической типизации. Покажите, как динамическая типизация Python обеспечивает гибкость, но может скрывать ошибки.
```python
def add(a, b):
    return a + b

# Works with different types -- flexible!
print(add(3, 5))           # 8 (int + int)
print(add("hello ", "world"))  # "hello world" (str + str)
print(add([1, 2], [3, 4]))    # [1, 2, 3, 4] (list + list)

# But type errors only surface at runtime:
try:
    print(add("hello", 5))  # TypeError! str + int
except TypeError as e:
    print(f"Runtime error: {e}")

```

3. Измерьте разницу в производительности между интерпретируемым Python и скомпилированными/JIT-подходами для вычислительно сложной задачи.
```python
import time
import jax
import jax.numpy as jnp

n = 1_000_000

# Pure Python loop (interpreted)
start = time.time()
total = 0.0
for i in range(n):
    total += i * i
python_time = time.time() - start

# JAX (compiled via XLA)
@jax.jit
def sum_squares_jax(n):
    return jnp.sum(jnp.arange(n, dtype=jnp.float32) ** 2)

_ = sum_squares_jax(10)  # warm up JIT
start = time.time()
result = sum_squares_jax(n)
jax_time = time.time() - start

print(f"Python loop: {python_time:.4f}s")
print(f"JAX (JIT):   {jax_time:.6f}s")
print(f"Speedup:     {python_time / jax_time:.0f}x")
```
