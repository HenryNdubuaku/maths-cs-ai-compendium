# Конкурентность и параллелизм

*Конкурентность и параллелизм — это способы, которыми программы выполняют несколько действий одновременно. В этом файле рассматриваются различие между конкурентностью и параллелизмом, примитивы синхронизации, классические проблемы конкурентности, взаимная блокировка (deadlock), неблокирующие структуры данных, модели параллельного программирования, асинхронное программирование и законы масштабирования — концепции, лежащие в основе многопоточных серверов, распределенного обучения и любого современного приложения.*

- Одно ядро CPU выполняет одну инструкцию за раз. Но современные системы имеют 8, 64 или даже тысячи ядер (GPU). И даже на одном ядре мы хотим обрабатывать несколько задач: скачивать файл, отрисовывать интерфейс и обрабатывать ввод пользователя. **Конкурентность** (concurrency) и **параллелизм** (parallelism) — это две стратегии управления множеством действий.

## Конкурентность vs Параллелизм

![Конкурентность чередует задачи на одном ядре; параллелизм выполняет задачи одновременно на нескольких ядрах](../images/concurrency_vs_parallelism.svg)

- **Конкурентность** — это *управление* множеством задач. Задачи продвигаются за счет чередования: задача A выполняется некоторое время, затем задача B, затем снова A. На одном ядре конкурентность создает иллюзию одновременного выполнения. Задачи не являются по-настоящему одновременными; они работают по очереди.

- **Параллелизм** — это *выполнение* множества задач одновременно. При наличии $n$ ядер $n$ задач могут действительно выполняться в одно и то же время. Параллелизм требует наличия нескольких аппаратных исполнительных устройств.

- Аналогия: конкурентность — это один повар, который чередует нарезку овощей и помешивание супа. Параллелизм — это два повара, каждый из которых выполняет одну задачу одновременно. Система может быть конкурентной, но не параллельной (одно ядро, чередующиеся задачи), параллельной, но не конкурентной (несколько ядер, выполняющих независимые программы без взаимодействия), или и той, и другой (несколько ядер, выполняющих чередующиеся задачи, которые взаимодействуют друг с другом).

- В машинном обучении конкурентность проявляется при загрузке данных (совмещение предобработки данных с вычислениями на GPU), тогда как параллелизм проявляется в распределенном обучении (несколько GPU одновременно вычисляют градиенты, глава 6).

## Примитивы синхронизации

- Когда несколько потоков совместно используют данные, **синхронизация** предотвращает состояние гонки (race condition). Состояние гонки возникает, когда результат зависит от непредсказуемого порядка выполнения потоков.

- Рассмотрим два потока, которые одновременно увеличивают общий счетчик: `counter += 1`. На самом деле это три операции: (1) чтение счетчика, (2) прибавление 1, (3) запись счетчика. Если оба потока прочитают одно и то же значение (скажем, 5), оба прибавят 1 и оба запишут 6, счетчик в итоге окажется равен 6 вместо правильного 7. Одно приращение будет потеряно.

- **Мьютекс** (mutex, взаимная блокировка) гарантирует, что только один поток обращается к критической секции в данный момент времени. Поток **захватывает** (acquires) блокировку перед входом в критическую секцию и **освобождает** (releases) её после. Любой другой поток, пытающийся захватить уже занятую блокировку, блокируется до тех пор, пока она не будет освобождена.

```
lock.acquire()
counter += 1      # only one thread at a time here
lock.release()
```

- Мьютексы корректны, но создают **конкуренцию за ресурсы** (contention): если многие потоки соперничают за одну и ту же блокировку, они тратят время на ожидание вместо вычислений. Это ограничивает масштабируемость. В крайнем случае, когда все потоки хотят получить одну и ту же блокировку, происходит сериализация всей программы.

- **Семафор** (semaphore) обобщает понятие мьютекса. Счётный семафор поддерживает счетчик: `wait()` уменьшает счетчик (блокируя поток, если значение станет отрицательным), а `signal()` увеличивает его. Семафор, инициализированный значением 1, ведет себя как мьютекс. Семафор, инициализированный значением $n$, позволяет одновременно находиться в критической секции до $n$ потоков (полезно для пулов ресурсов, таких как соединения с базой данных).

- **Переменная условия** (condition variable) позволяет потоку ждать, пока не будет выполнено определенное условие. Поток освобождает блокировку, ждет на переменной условия и «просыпается», когда другой поток сигнализирует о выполнении условия. Это позволяет избежать активного ожидания (busy-waiting) — многократной проверки условия в цикле, что тратит ресурсы CPU.

- **Монитор** (monitor) объединяет мьютекс с переменными условия и общими данными в единую абстракцию. Ключевое слово `synchronized` в Java и `threading.Condition` в Python реализуют семантику, близкую к мониторам.

- **Блокировки чтения-записи** (read-write locks) различают читателей (которые могут иметь общий доступ, так как чтение не изменяет данные) и писателей (которым нужен исключительный доступ). Несколько читателей могут удерживать блокировку одновременно, но писатель блокирует всех читателей и других писателей. Это оптимально, когда количество операций чтения значительно превышает количество операций записи (например, кешированная модель, выдающая предсказания).

## Классические проблемы конкурентности

- **Производитель-потребитель** (Producer-Consumer, ограниченный буфер): производители создают элементы и помещают их в буфер фиксированного размера; потребители извлекают элементы. Проблемы: производители должны ждать, когда буфер полон, потребители должны ждать, когда он пуст, и обе стороны должны избегать повреждения данных в буфере.

- Решение использует два семафора (один считает пустые слоты, другой — заполненные) плюс мьютекс для самого буфера. Это паттерн, лежащий в основе большинства очередей сообщений, систем логирования и пайплайнов обработки данных.

- **Читатели-писатели** (Readers-Writers): несколько читателей могут читать одновременно, но писателям нужен исключительный доступ. Проблема заключается в справедливости: если читатели продолжают поступать, писатель может «голодать» (никогда не получить доступ). Решения либо отдают приоритет читателям или писателям, либо чередуют их справедливо.

- **Обедающие философы** (Dining Philosophers): пять философов сидят за столом, между ними пять вилок. Каждому нужно две вилки, чтобы поесть. Если все пятеро одновременно возьмут левую вилку, никто не сможет взять правую, и все останутся голодными (взаимная блокировка). Решения включают: брать обе вилки атомарно, ввести асимметрию (один философ берет правую вилку первым) или использовать официанта (семафор, ограничивающий количество обедающих до 4).

## Взаимная блокировка

- **Взаимная блокировка** (deadlock) возникает, когда набор потоков ожидает ресурс, удерживаемый другим потоком из этого же набора, создавая цикл зависимостей. Никто не может продолжить работу.

![Взаимная блокировка: Поток A удерживает Блокировку 1 и хочет Блокировку 2, Поток B удерживает Блокировку 2 и хочет Блокировку 1 — циклическое ожидание](../images/deadlock_cycle.svg)

- Четыре **необходимых условия** для возникновения взаимной блокировки (все должны выполняться одновременно):

    1. **Взаимное исключение**: ресурс может удерживаться только одним потоком.
    2. **Удержание и ожидание**: поток удерживает один ресурс, ожидая другой.
    3. **Отсутствие вытеснения**: ресурсы не могут быть принудительно отобраны у потока.
    4. **Циклическое ожидание**: в графе ожидания существует цикл.

- **Предотвращение взаимных блокировок (deadlock prevention)** нарушает одно из четырех условий:
    - Устранение циклического ожидания: введение полного упорядочивания ресурсов. Все потоки захватывают ресурсы в одном и том же порядке. Если каждый поток всегда захватывает блокировку A перед блокировкой B, возникновение цикла невозможно.
    - Устранение удержания и ожидания: требование к потокам запрашивать все ресурсы одновременно (атомарно).

- **Избежание взаимных блокировок (deadlock avoidance)** динамически решает, может ли удовлетворение запроса на ресурс привести к блокировке. **Алгоритм банкира** отслеживает максимально возможную потребность каждого потока и удовлетворяет только те запросы, которые оставляют систему в «безопасном состоянии» (состоянии, при котором все потоки в конечном итоге могут завершиться). Алгоритм имеет сложность $O(n^2 m)$ на запрос ($n$ потоков, $m$ типов ресурсов), что слишком затратно для большинства реальных систем.

- **Обнаружение взаимных блокировок (deadlock detection)** допускает возникновение блокировок, а затем обнаруживает их (путем поиска циклов в графе ожидания) и восстанавливает систему (путем завершения потока или отката транзакции).

- На практике большинство систем используют предотвращение (упорядочивание ресурсов) для типичных случаев и обнаружение для редких. Базы данных — классический пример: они обнаруживают блокировки между транзакциями и прерывают одну из них, чтобы разорвать цикл.

## Неблокирующие (lock-free) и не требующие ожидания (wait-free) структуры данных

- Блокировки вызывают конкуренцию, инверсию приоритетов и риск взаимной блокировки. **Неблокирующие (lock-free)** структуры данных полностью избегают блокировок, используя **атомарные операции**, предоставляемые аппаратным обеспечением.

- Ключевой атомарной операцией является **Compare-And-Swap (CAS)**: атомарная проверка того, содержит ли ячейка памяти ожидаемое значение, и если да, то замена его на новое. В псевдокоде:

```
CAS(address, expected, new_value):
    if *address == expected:
        *address = new_value
        return true
    else:
        return false
```

- CAS реализована как единая аппаратная инструкция, поэтому она атомарна даже без блокировок. Неблокирующие алгоритмы используют CAS в цикле повторных попыток: чтение текущего значения, вычисление нового, попытка CAS. Если другой поток тем временем изменил значение, CAS завершается неудачей, и поток повторяет попытку.

- **Неблокирующий (lock-free)**: по крайней мере один поток продвигается вперед за конечное число шагов (взаимная блокировка невозможна, но отдельные потоки могут бесконечно повторять попытки при высокой конкуренции).

- **Не требующий ожидания (wait-free)**: каждый поток продвигается вперед за ограниченное число шагов (сильнейшая гарантия, но наиболее труднодостижимая).

- Неблокирующие стеки, очереди и хеш-таблицы широко используются в высокопроизводительных системах. `ConcurrentHashMap` в Java и атомарные операции в Go построены на основе CAS.

## Модели параллельного программирования

- Параллелизм с **разделяемой памятью (shared memory)**: все потоки имеют доступ к одному и тому же пространству памяти. Синхронизация — ответственность программиста. **OpenMP** предоставляет директивы компилятора для распараллеливания циклов:

```c
#pragma omp parallel for
for (int i = 0; i < n; i++) {
    result[i] = compute(data[i]);
}
```

- Компилятор распределяет итерации цикла между доступными ядрами. OpenMP эффективен для задач с параллелизмом по данным (одна и та же операция над множеством точек данных) и широко используется в научных вычислениях.

- Параллелизм с **передачей сообщений (message passing)**: каждый процесс имеет свою собственную память. Обмен данными происходит путем отправки и получения сообщений. **MPI** (Message Passing Interface) является стандартом для распределенных вычислений между узлами:

```c
MPI_Send(data, count, MPI_FLOAT, dest, tag, MPI_COMM_WORLD);
MPI_Recv(data, count, MPI_FLOAT, src, tag, MPI_COMM_WORLD, &status);
```

- MPI масштабируется до тысяч узлов, так как нет разделяемого состояния, требующего синхронизации. Распределенное глубокое обучение (глава 6) использует коллективные операции, такие как `MPI_AllReduce` (кольцевой all-reduce), для синхронизации градиентов между GPU.

- **Параллелизм GPU** следует модели **SIMT** (Single Instruction, Multiple Threads): тысячи потоков выполняют одну и ту же инструкцию над разными данными. Это идеально подходит для матричных операций (глава 2), где одно и то же умножение-сложение применяется к каждому элементу. Мы подробно рассмотрим программирование GPU в одной из следующих глав.

## Асинхронное и событийно-ориентированное программирование

- Не всякая конкурентность требует потоков. **Асинхронное** программирование обрабатывает множество задач, ограниченных операциями ввода-вывода (I/O-bound), в одном потоке с использованием **цикла событий (event loop)**.

- Цикл событий поддерживает очередь задач. Когда задаче нужно дождаться ввода-вывода (сетевой ответ, чтение файла), она регистрирует обратный вызов (callback) и освобождает управление. Цикл событий выбирает следующую готовую задачу. Когда ввод-вывод завершается, обратный вызов ставится в очередь и в конечном итоге выполняется. Ни один поток не блокируется во время ожидания.

- **Корутины (coroutines)** — это функции, которые могут приостанавливать и возобновлять свое выполнение. Синтаксис `async/await` (Python, JavaScript, Rust) делает корутины похожими на обычный последовательный код:

```python
async def fetch_data(url):
    response = await http_get(url)  # suspends here, event loop runs other tasks
    return process(response)         # resumes when response arrives
```

- Ключевое слово `await` приостанавливает корутину и возвращает управление циклу событий. Когда ожидаемая операция завершается, корутина возобновляется с того места, где была прервана. Это кооперативная многозадачность: корутины добровольно освобождают управление, в отличие от вытесняющей многозадачности, где ОС принудительно переключает потоки.

- Асинхронность идеально подходит для задач, **ограниченных вводом-выводом (I/O-bound)**, с большим количеством одновременных соединений (веб-серверы, обслуживающие тысячи клиентов). Она не подходит для задач, **ограниченных процессором (CPU-bound)** (однопоточный цикл событий не может использовать несколько ядер). Для задач, ограниченных процессором, используйте потоки или процессы.

- **Глобальная блокировка интерпретатора (GIL)** в Python предотвращает истинный параллелизм с помощью потоков: только один поток может выполнять байт-код Python одновременно. Именно поэтому Python использует мультипроцессинг (отдельные процессы, каждый со своим интерпретатором) для параллелизма на уровне CPU и асинхронность для конкурентности ввода-вывода. GIL удаляется в Python 3.13+ (free-threaded Python), что обеспечит настоящий многопоточный параллелизм.

## Законы масштабируемости

- **Закон Амдала** описывает теоретическое ускорение при распараллеливании программы. Если доля $p$ программы поддается распараллеливанию, а остальная часть $1 - p$ является последовательной:

$$\text{Speedup}(n) = \frac{1}{(1-p) + \frac{p}{n}}$$

![Amdahl's law: the serial fraction limits maximum speedup — 10% serial means 10x max, no matter how many cores](../images/amdahl_serial_bottleneck.svg)

- где $n$ — количество процессоров. При $n \to \infty$ максимальное ускорение стремится к $\frac{1}{1-p}$. Если 95% программы распараллелено, максимальное ускорение составит $\frac{1}{0.05} = 20\times$, независимо от того, сколько ядер вы добавите. Последовательная часть является «узким местом».

- Это имеет глубокие последствия для машинного обучения: если загрузка данных занимает 10% времени обучения и выполняется последовательно, добавление большего количества GPU может ускорить обучение максимум в 10 раз. 10-процентное последовательное «узкое место» ограничивает всё (вот почему эффективные пайплайны данных и совмещение вычислений с вводом-выводом так важны, глава 6).

- **Закон Густафсона** предлагает более оптимистичный взгляд. Вместо фиксации размера задачи и добавления процессоров, он фиксирует общее время и задается вопросом, какой объем работы можно выполнить дополнительно. Если параллельная часть масштабируется вместе с размером задачи:

$$\text{Speedup}(n) = 1 - p + p \cdot n$$

- Это линейная зависимость от $n$. Аргумент следующий: с большим количеством процессоров мы решаем более крупные задачи, а не ту же самую задачу быстрее. В машинном обучении это соответствует увеличению размера батча при использовании большего количества GPU (слабое масштабирование), а не сохранению размера батча неизменным (сильное масштабирование).

## Задания по программированию (используйте CoLab или ноутбук)

1. Продемонстрируйте состояние гонки (race condition). Два потока увеличивают значение общего счетчика без синхронизации, что приводит к потере обновлений.
```python
import threading

counter = 0

def increment(n):
    global counter
    for _ in range(n):
        counter += 1  # NOT atomic: read, add, write

threads = [threading.Thread(target=increment, args=(100000,)) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()

print(f"Expected: {4 * 100000}")
print(f"Actual:   {counter}")
print(f"Lost updates: {4 * 100000 - counter}")
```

2. Исправьте состояние гонки с помощью блокировки (lock) и измерьте накладные расходы.
```python
import threading
import time

lock = threading.Lock()
counter = 0

def increment_locked(n):
    global counter
    for _ in range(n):
        with lock:
            counter += 1

start = time.time()
threads = [threading.Thread(target=increment_locked, args=(100000,)) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
elapsed = time.time() - start

print(f"Counter: {counter} (correct: {4 * 100000})")
print(f"Time with lock: {elapsed:.3f}s")
```

3. Визуализируйте закон Амдала. Постройте график зависимости ускорения от количества процессоров для различных долей параллельных вычислений.

import jax.numpy as jnp
import matplotlib.pyplot as plt

n_procs = jnp.arange(1, 65)

for p, color in [(0.5, "#e74c3c"), (0.9, "#f39c12"), (0.95, "#27ae60"), (0.99, "#3498db")]:
    speedup = 1 / ((1 - p) + p / n_procs)
    plt.plot(n_procs, speedup, color=color, linewidth=2, label=f"p={p}")
    # Max speedup line
    plt.axhline(1 / (1 - p), color=color, linestyle="--", alpha=0.3)

plt.xlabel("Number of processors")
plt.ylabel("Speedup")
plt.title("Amdahl's Law: Serial Fraction Limits Speedup")
plt.legend()
plt.grid(True)
plt.show()
