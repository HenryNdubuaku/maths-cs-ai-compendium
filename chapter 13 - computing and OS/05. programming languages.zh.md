# 编程语言

*编程语言是人类意图与机器执行之间的接口。本文件涵盖语言范式、类型系统、内存管理策略、编译流水线、解释与 JIT 编译、关键语言特性、领域专用语言以及设计权衡*

- 每一款软件、每一个 ML 模型、每一个操作系统都是用某种编程语言写成的。但有数百种语言，各有不同的强项。为什么？因为语言设计涉及根本的权衡：性能 vs 安全、表达力 vs 简单、控制 vs 抽象。理解这些权衡有助于你为任务选择合适工具，并理解你所处约束。

## 语言范式

- **范式**是一种编程风格：一组指导你如何组织代码、如何思考问题的原则。

- **命令式**编程把计算描述为一串改变状态的命令。“把 x 设为 5。给 x 加 3。如果 x > 7，就打印它。”C、Python 和 Java 在核心上都是命令式的。其心智模型是一台有内存、你一步步修改的机器。

- **面向对象（OOP）**编程围绕**对象**组织代码：把数据（属性）和行为（方法）捆绑在一起。对象之间通过互相发送消息来交互。关键思想是**封装**（在公共接口背后隐藏内部状态）、**继承**（通过扩展现有类来创建新类）和**多态**（通过共享接口统一对待不同类型）。Java、C++ 和 Python 都支持 OOP。

- **函数式编程（FP）**把计算视为数学函数的求值。核心原则：**不可变性**（数据一旦创建就不再改变）、**纯函数**（输出只依赖输入，无副作用）和**一等函数**（函数是值，可以作为参数传入、从其他函数返回、存入变量）。Haskell 是纯函数式的。Python、JavaScript 和 Scala 支持函数式风格。

- 纯函数易于推理、测试和并行化（没有共享可变状态意味着没有竞态条件）。这就是为什么函数式思想越来越多地用于分布式系统和数据流水线。本书通篇使用的 JAX 是函数式的：`jax.grad` 之所以能工作，正是因为 JAX 的函数是纯函数。

- **逻辑式编程**描述*什么*应当为真，而不是*如何*计算。你陈述事实和规则，运行时找出解。Prolog 是经典例子：给定“苏格拉底是人”和“所有人都会死”，引擎推导出“苏格拉底会死”。逻辑编程用于 AI 知识库和类型检查。

- 大多数现代语言是**多范式**的：Python 支持命令式、OOP 和函数式风格。Rust 支持命令式和函数式。范式是工具，不是信仰。

## 类型系统

- **类型**对值进行分类，并决定哪些运算合法。整数 3 和字符串 "3" 是不同类型：你可以把整数相加，但不能把字符串相加（好吧，你可以拼接字符串，但那是一种不同的运算）。

- **静态类型**：类型在程序运行之前的**编译期**检查。类型错误能被及早发现。C、Java、Rust 和 Go 是静态类型的。你必须声明类型（或由编译器推断）：

```rust
let x: i32 = 5;     // Rust: x is a 32-bit integer
let y: f64 = 3.14;  // y is a 64-bit float
// let z = x + y;    // compile error: cannot add i32 and f64
```

- **动态类型**：类型在运算实际执行时的**运行期**检查。更灵活，但类型错误只在代码运行时才浮现。Python、JavaScript 和 Ruby 是动态类型的：

```python
x = 5       # x is an int (for now)
x = "hello" # now x is a string -- no error
```

- **强类型**：语言阻止隐式类型转换。Python 是强类型的：`"3" + 5` 抛出 TypeError。**弱类型**：语言悄悄转换类型。JavaScript 是弱类型的：`"3" + 5` 得到 `"35"`（数字被转换为字符串）。C 是弱类型的：你可以把指针转换为整数。

- **类型推断**让编译器无需显式注解就能推导类型：

```rust
let x = 5;        // compiler infers: i32
let y = x + 3.0;  // compile error: mixed types, even with inference
```

- **泛型**（参数化多态）让你编写适用于任意类型的代码：

```rust
fn largest<T: PartialOrd>(list: &[T]) -> &T {
    let mut max = &list[0];
    for item in &list[1..] {
        if item > max { max = item; }
    }
    max
}
// Works for integers, floats, strings -- any type that supports comparison
```

- 对 ML 而言：Python 的动态类型让实验快速，但隐藏了 bug。生产 ML 系统越来越多地使用类型提示（`def train(model: nn.Module, lr: float) -> float`）和静态分析工具（mypy）在部署前捕捉错误。PyTorch 和 JAX 出于灵活性使用 Python；TensorRT 和 ONNX Runtime 出于性能使用 C++。

## 内存管理

- 每个程序都要分配和释放内存。这如何管理是最具后果的语言设计决策之一。

![内存布局：栈从高地址向下增长，heap 向上增长，代码和数据位于底部](../images/stack_vs_heap.svg)

- **栈**存储局部变量和函数调用帧。分配很简单（移动栈指针），释放是自动的（函数返回时弹出该帧）。栈访问很快，因为它总在 cache 中。但栈大小固定（通常 1-8 MB），且只支持 LIFO（后进先出）分配。

- **heap** 存储动态分配的数据（对象、数组、大小在编译期未知的字符串）。heap 分配更慢（需要找一个空闲块），并需要显式或自动的释放。heap 可以增长以填满可用内存。

- **手动内存管理**（C、C++）：程序员显式分配（`malloc`）和释放（`free`）heap 内存。控制力和性能最大，但极易出错：
    - **释放后使用**：访问已被释放的内存。导致崩溃或安全漏洞。
    - **双重释放**：把同一内存释放两次。破坏分配器的内部数据结构。
    - **内存泄漏**：分配内存但从不释放。程序慢慢耗尽所有可用 RAM。

- **垃圾回收（GC）**：运行时自动检测并释放不再可达的内存。程序员从不调用 `free`。

    - **追踪式 GC**（Java、Go、Python 的循环收集器）：周期性地从“根”（栈变量、全局变量）出发遍历所有可达对象，释放不可达对象。简单但会造成 **GC 暂停**：收集器运行时程序停下。现代收集器（Go 的并发 GC、Java 的 ZGC）把暂停时间最小化到亚毫秒级。

    - **引用计数**（Python 的主要机制、Swift、Objective-C）：每个对象跟踪有多少引用指向它。当计数降为 0 时，对象立即被释放。无暂停，但无法处理**循环**（A 引用 B，B 引用 A，两者计数都大于 0 但都不可达）。Python 用一个独立的循环检测器来处理这种情况。

- **所有权**（Rust）：编译器在编译期强制内存安全规则，零运行时开销。

    - 每个值有且只有一个**所有者**。当所有者离开作用域时，该值被丢弃（释放）。
    - 值可以被**借用**（引用），但编译器强制：要么一个可变引用，要么任意数量的不可变引用，二者不能同时存在。
    - 这在编译期就防止了释放后使用、双重释放、数据竞争和悬垂指针。无需 GC，无运行时成本。

- **借用检查器**是 Rust 的杀手锏，也是它最陡的学习曲线。它在没有垃圾回收的前提下保证内存安全和线程安全，这就是为什么 Rust 越来越多地用于性能关键系统（OS kernel、游戏引擎、像 Candle 和 Burn 这样的 ML 推理运行时）。

## 编译流水线

- **编译器**在程序运行之前把源代码翻译成机器代码（或另一种目标语言）。流水线有几个阶段：

![编译流水线：源码 → 词法分析器 → 语法分析器 → 语义分析 → 优化器 → 代码生成 → 机器代码](../images/compilation_pipeline.svg)

1. **词法分析**（分词）：把源文本转换为一串 token。`x = 3 + y` 变成 `[IDENT("x"), EQUALS, INT(3), PLUS, IDENT("y")]`。词法分析器剥离空白和注释。

2. **语法分析**：从 token 流构建一棵**抽象语法树（AST）**。AST 表示程序的层次结构。`3 + y * 2` 被解析为 `Add(3, Mul(y, 2))`（乘法优先级更高）。解析器检查语法：括号不匹配和缺失分号在此被捕捉。

3. **语义分析**：检查类型、解析变量名、验证函数被以正确参数调用。静态类型检查就在这里发生。输出是一棵带类型注解的 AST。

4. **优化**：在不改变行为的前提下变换程序使其运行更快。常见优化：
    - **常量折叠**：在编译期计算 `3 + 5`，替换为 `8`。
    - **死代码消除**：移除永远不会执行的代码。
    - **循环展开**：用重复的内联代码替换循环，以减少分支开销。
    - **内联**：把函数调用替换为函数体，消除调用开销。

5. **代码生成**：把优化后的表示翻译成目标机器代码（x86、ARM）或某种中间表示。

- **LLVM** 是主导的编译器基础设施。它提供一个公共的中间表示（LLVM IR），许多语言都编译到它。LLVM 的优化器作用于此 IR，其后端为许多目标生成机器代码。Clang（C/C++）、Rust、Swift、Julia 和许多其他语言都使用 LLVM。这意味着对 LLVM 优化器的改进同时让所有这些语言受益。

## 解释与 JIT 编译

- **解释器**逐行（或逐语句）执行程序，而不产生机器代码。这使启动快、开发具交互性，但执行更慢（每行每次运行时都被重新分析）。

- 大多数被“解释”的语言其实编译为**字节码**：一种比源代码简单但非机器特定的中间表示。字节码在**虚拟机（VM）**上运行。

    - **CPython**（标准 Python 实现）把 Python 源码编译为字节码（`.pyc` 文件），由 CPython VM 执行。VM 逐条解释字节码。这就是为什么对于计算密集的代码，Python 比 C 慢约 100 倍。

    - **JVM**（Java 虚拟机）：Java 编译为 JVM 字节码（`.class` 文件）。JVM 先解释字节码，然后对频繁执行的代码路径（“热点”）**JIT 编译**为本地机器代码。这就是为什么 Java 启动比 C 慢（解释开销），但对长时间运行的程序接近 C 的速度（JIT 优化的热路径）。

- **JIT（即时）编译**在运行期把代码编译为机器代码，利用只在执行期间可用的信息。JIT 可以基于实际运行数据优化：如果一个函数总是以整数参数被调用，JIT 生成专门的仅整数机器代码，跳过类型检查。

- **PyPy** 是一个带 JIT 编译器的替代 Python 实现。它通过把热循环 JIT 编译为机器代码，使大多数 Python 代码比 CPython 快 5-10 倍。然而，它与 C 扩展模块（NumPy、PyTorch）兼容性有限，这限制了它在 ML 中的使用。

- 从解释到编译的谱系不是非此即彼的：
    - 纯解释：Bash shell 脚本。
    - 字节码解释：CPython。
    - 字节码 + JIT：JVM、.NET CLR、LuaJIT、PyPy。
    - 预先（AOT）编译：C、C++、Rust、Go。
    - AOT + 运行时代码生成：JAX 的 `jax.jit` 在首次调用时把 Python 函数编译为优化的 XLA 代码，然后缓存编译后的版本。

## 关键语言特性

- **闭包**：一个从其外层作用域捕获变量的函数。该函数“闭合”了它定义时所处的环境：

```python
def make_adder(n):
    def add(x):
        return x + n  # n is captured from the enclosing scope
    return add

add5 = make_adder(5)
print(add5(3))  # 8
```

- 闭包是回调、装饰器和偏应用背后的机制。它是函数式编程的基础。

- **模式匹配**：一种强大的控制流机制，对数据进行解构并依据其形状分支：

```rust
match value {
    Some(x) if x > 0 => println!("Positive: {}", x),
    Some(0)           => println!("Zero"),
    Some(x)           => println!("Negative: {}", x),
    None              => println!("Nothing"),
}
```

- 模式匹配比 if-else 链更具表达力：它检查数据的结构（是 Some 还是 None？是否包含满足某条件的值？），而不仅是相等性。Python 在 3.10 中加入了结构化模式匹配（`match`/`case`）。

- **代数数据类型（ADT）**：可以是若干变体之一、每种携带不同数据的类型。`Result` 类型要么是 `Ok(value)` 要么是 `Err(error)`。一棵 `Tree` 要么是 `Leaf(value)` 要么是 `Node(left, right)`。ADT 与模式匹配结合，能穷尽地处理所有情形，消除整类 bug（空指针异常、未处理的错误码）。

- **trait 与接口**：定义一组类型必须实现的方法，但不指明如何实现。这实现了多态：一个接受“任何实现了 Display trait 的东西”的函数可以与整数、字符串和自定义类型一起工作。Rust 用 trait，Java 用接口，Go 用隐式接口，Python 用鸭子类型（“如果它走起路来像鸭子……”）。

## 领域专用语言

- **领域专用语言（DSL）**是为特定问题领域设计的，在该领域内以通用性换表达力。

- **SQL**：关系数据库的语言。`SELECT name FROM users WHERE age > 30` 比等价的命令式循环可读得多，也更可优化。数据库引擎优化查询执行计划，自动选择连接策略和索引使用。

- **正则表达式**：用于文本模式匹配的迷你语言。`\d{3}-\d{4}` 匹配形如 "555-1234" 的电话号码。正则引擎把模式编译为有限自动机以高效匹配。

- **着色器语言**（GLSL、HLSL、Metal Shading Language）：运行在 GPU 核上、用于计算像素颜色、顶点位置或计算操作的程序。着色器是大规模并行的：每次调用独立处理一个像素或一个元素。这与 CUDA 用于 ML 计算的执行模型相同。

- 在 ML 中，像 PyTorch 和 JAX 这样的框架本质上是嵌入 Python 中的张量计算 DSL。它们提供领域专用抽象（张量、自动微分、设备放置），同时利用 Python 的生态。

## 语言设计权衡

- 没有哪种语言在所有方面都最好。设计就是选择做哪些权衡：

- **性能 vs 安全**：C 给你原始速度和硬件控制，但允许你损坏内存。Rust 给出可比的速度，并带来编译期内存安全。Java 借助垃圾回收给出内存安全，但有开销。Python 给出最大的安全性和表达力，但执行慢 100 倍。

- **表达力 vs 简单**：Haskell 的类型系统能表达非常精确的约束，但学习曲线陡峭。Go 为简单起见刻意省略了泛型（直到最近）、继承和异常。Python 的“做一件事应该有一种显然的方式”哲学让语言保持易学。

- **控制 vs 抽象**：C/C++ 让你控制内存布局、cache 行为和硬件交互。Python 把这一切都隐藏了。对 ML 训练（由 GPU 计算主导），Python 的开销可忽略。对 ML 推理（每微秒都算数），可能需要 C++ 或 Rust。

- **编译速度 vs 运行速度**：Go 在数秒内编译（简单的类型系统、最小的优化）。Rust 在数分钟内编译（复杂的类型系统、激进的优化）。权衡的是开发者迭代速度 vs 部署性能。

- ML 生态反映了这些权衡：用 Python 做实验和训练（表达力胜出），用 C++/CUDA 做内核和推理（性能胜出），用 Rust 做基础设施和安全关键系统（安全胜出）。

## 编程任务（使用 CoLab 或 notebook）

1. 探索闭包与高阶函数。实现一个简单的函数工厂，并验证闭包捕获了其环境。
```python
def make_multiplier(factor):
    """Returns a function that multiplies by factor."""
    def multiply(x):
        return x * factor
    return multiply

double = make_multiplier(2)
triple = make_multiplier(3)

print(f"double(5) = {double(5)}")  # 10
print(f"triple(5) = {triple(5)}")  # 15

# Closures capture by reference, not by value
def make_counter():
    count = [0]  # mutable container to allow modification
    def increment():
        count[0] += 1
        return count[0]
    return increment

counter = make_counter()
print(f"counter() = {counter()}")  # 1
print(f"counter() = {counter()}")  # 2
print(f"counter() = {counter()}")  # 3
```

2. 比较动态与静态类型的行为。展示 Python 的动态类型如何带来灵活性，但可能隐藏 bug。
```python
def add(a, b):
    return a + b

# Works with different types -- flexible!
print(add(3, 5))           # 8 (int + int)
print(add("hello ", "world"))  # "hello world" (str + str)
print(add([1, 2], [3, 4]))    # [1, 2, 3, 4] (list + list)

# But type errors only surface at runtime:
try:
    print(add("hello", 5))  # TypeError! str + int
except TypeError as e:
    print(f"Runtime error: {e}")
    print("A static type checker would catch this before running")
```

3. 对一个计算密集任务，测量解释执行 Python 与编译/JIT 方法的性能差异。
```python
import time
import jax
import jax.numpy as jnp

n = 1_000_000

# Pure Python loop (interpreted)
start = time.time()
total = 0.0
for i in range(n):
    total += i * i
python_time = time.time() - start

# JAX (compiled via XLA)
@jax.jit
def sum_squares_jax(n):
    return jnp.sum(jnp.arange(n, dtype=jnp.float32) ** 2)

_ = sum_squares_jax(10)  # warm up JIT
start = time.time()
result = sum_squares_jax(n)
jax_time = time.time() - start

print(f"Python loop: {python_time:.4f}s")
print(f"JAX (JIT):   {jax_time:.6f}s")
print(f"Speedup:     {python_time / jax_time:.0f}x")
```
