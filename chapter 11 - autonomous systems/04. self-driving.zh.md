# 自动驾驶汽车

*自动驾驶汽车是商业上最成熟的 autonomous system，把 perception、prediction、planning 和 control 集成到一辆车中。本文件涵盖自动驾驶栈、HD 地图、运动预测、planning、端到端驾驶、仿真、安全标准和自动化等级。*

- 自动驾驶汽车可以说是大规模尝试的最困难机器人问题。与在受控环境中运行的工厂机器人不同，自动驾驶汽车必须应对开放世界：不可预测的人类驾驶员、乱穿马路的行人、一夜之间出现的施工区，以及随时变化的天气。

- 代价也独特地高。自动驾驶汽车在高速公路速度下行驶于弱势道路使用者之间。对安全关键失效而言，容错率几乎为零。

## 自动驾驶栈

- 经典的自动驾驶架构是**模块化流程**，包含四个阶段，每个都馈入下一个：

$$\text{Perception} \to \text{Prediction} \to \text{Planning} \to \text{Control}$$

![自动驾驶栈：sensor 馈入 perception，再馈入 prediction、planning，最后是 control](../images/autonomous_driving_stack.svg)

- **Perception**（本章文件 1 所述）把原始 sensor 数据处理为结构化的场景表示：检测到的物体及其 3D 位置、速度和类别标签；车道标线；交通灯；可驾驶表面边界。

- **Prediction** 预测其他 agent（车辆、行人、骑行者）未来如何运动。给定当前场景状态，prediction 模块在时间域内为每个 agent 输出轨迹（通常预测未来 3-8 秒）。

- **Planning** 决定自车应该做什么：跟随哪条路径、何时变道、何时让行、何时加速或刹车。它接收预测后的场景，为自车生成安全、舒适且向目的地推进的轨迹。

- **Control** 把规划轨迹转换为执行器命令：方向盘转角、油门和刹车。这是最低层，把抽象轨迹翻译为物理运动。

- 模块化设计有明显的工程优势：每个模块可以独立开发、测试和改进。但它也有弱点：误差向下游传播（漏检对 planner 不可见），且在每个接口处信息丢失（planner 看到的是边界框，而非产生它们的丰富 sensor 数据）。

## HD 地图

- **High-Definition (HD) map（高精地图）**是详细的、厘米级精度的数字地图，编码道路结构：车道边界、车道连通性（在交叉口哪条车道连接到哪条）、交通标志位置、限速、人行横道位置和路面高程。

- HD 地图为驾驶任务提供强先验。Perception 模块不必每帧从零发现车道边界，只需在地图中定位车辆并验证现实与存储结构匹配即可。这极大地简化了 planning。

- 构建 HD 地图需要配备高端 LiDAR、camera 和 RTK-GPS 的专用测量车辆。地图必须随道路变化进行维护和更新。这很昂贵，且不易扩展到地球上的每条道路。

- **Mapless driving（无图驾驶）**（也称"在线建图"）旨在消除对预建 HD 地图的依赖。车辆转而从其 sensor 实时构建局部地图。**MapTR** 和 **MapTRv2** 等模型使用 transformer 架构直接从 camera 图像预测矢量化地图元素（车道中心线、道路边界、人行横道），输出以有序点序列表示的多段线。

- 无图方法以地图精度换取可扩展性：汽车能行驶的任何道路，它都能建图。但它要求 perception 系统足够鲁棒，能在复杂交叉口、高速公路匝道和施工区中实时检测所有相关道路结构。

- 实践中，许多系统采用混合方法：使用带有粗略道路拓扑的轻量地图（来自现有地图提供商），并由车辆 sensor 实时丰富。

## 运动预测

- 预测其他道路使用者会去哪里是自动驾驶中最困难的子问题之一。人类难以预测、意图隐藏、可能未来的空间迅速分叉。

- 预测模型的输入是**场景上下文**：所有检测到的 agent 在最近过去（通常 1-2 秒历史）的位置和速度，加上静态上下文（车道几何、交通信号、道路边界）。

- 输出是每个 agent 的一组**预测轨迹**，通常覆盖未来 3-8 秒。由于未来不确定，好的预测模型输出多个带相关概率的可能轨迹，而非单点估计。

- **Trajectory forecasting（轨迹预测）**作为回归问题：预测每个 agent 在离散未来时间步的 $(x, y)$ 坐标。损失通常是 $K$ 条预测轨迹上的最小平均位移误差（minADE）：

$$\text{minADE}_K = \min_{k \in \{1, \ldots, K\}} \frac{1}{T} \sum_{t=1}^{T} \| \hat{\mathbf{p}}_t^{(k)} - \mathbf{p}_t \|_2$$

- 这是一个"$K$ 中取最佳"的指标：只要模型 $K$ 个预测中的任何一个接近真值，模型就得分。这鼓励多样化、多模态的预测。

- **Social force（社会力）**模型把行人行为建模为动力系统，每个人受到吸引力（朝向目标）和排斥力（远离其他行人和障碍物）。人 $i$ 的加速度为：

$$\mathbf{a}_i = \frac{\mathbf{v}_i^{\text{desired}} - \mathbf{v}_i}{\tau} + \sum_{j \neq i} \mathbf{f}_{ij}^{\text{repulsive}} + \sum_{\text{walls}} \mathbf{f}_{\text{wall}}$$

- 这是一个类似于本章文件 2 机器人动力学方程的微分方程组。该模型优雅但依赖手工调参的力参数，且在复杂多 agent 交互中表现不佳。

- 用于预测的 **Graph Neural Network (GNN)** 把场景建模为 graph：每个 agent 是一个 node，edge 表示空间关系（邻近、共享车道）。node 之间的 message passing 捕获交互："这辆车正在让行那个行人"或"这两辆车正并入同一车道"。

- 现代预测架构（如 **MTR**、**QCNet**）使用基于 transformer 的模型，联合推理 agent 历史、地图上下文和 agent-agent 交互。Agent 通过 cross-attention 关注相关地图特征（当前车道、即将到来的交叉口）和其他 agent（前车、人行横道上的行人）。输出是通过自回归或混合模型生成的一组轨迹假设。

- **Goal-conditioned prediction（目标条件预测）**先预测 agent 可能去哪里（一组候选目标点，如车道端点或交叉口出口），然后预测到达每个目标的轨迹。这把问题分解为"去哪里"（离散、可管理）和"如何去"（给定目标的连续路径），使多模态预测问题更可处理。

## Planning

- 给定预测后的场景，planner 必须为自车生成轨迹。这是一个约束优化问题：找到安全、舒适、高效且合法的轨迹。

- **Rule-based planner（基于规则的规划器）**把驾驶行为编码为一组 if-then 规则："如果人行横道上有行人，让行"、"如果与前车间距小于 2 秒，不变道"、"如果接近红灯，减速停在停止线"。这些规则可解释、可审计，但在复杂场景下变得臃肿（数千条规则、许多边缘情况、规则间交互）。

- **Optimisation-based planner（基于优化的规划器）**把驾驶构造为轨迹优化。自车轨迹被参数化（如未来时间步的 $(x, y, \theta, v)$ 状态序列），并最小化目标函数：

$$\min_{\boldsymbol{\xi}} \underbrace{w_1 \cdot J_{\text{progress}}(\boldsymbol{\xi})}_{\text{get to destination}} + \underbrace{w_2 \cdot J_{\text{comfort}}(\boldsymbol{\xi})}_{\text{smooth ride}} + \underbrace{w_3 \cdot J_{\text{safety}}(\boldsymbol{\xi})}_{\text{avoid collisions}}$$

$$\text{subject to: } \text{kinematic constraints, speed limits, lane boundaries}$$

- 进展项惩罚偏离期望路线。舒适项惩罚高横向加速度、jerk（加速度的导数）和急转向，因为乘客会感受到这些。安全项惩罚靠近其他 agent，使用预测轨迹评估碰撞风险。

- 这是约束优化（第 3 章）：在等式约束下最小化代价函数。权重 $w_1, w_2, w_3$ 在竞争目标间权衡（激进驾驶更快但更不舒适、更不安全）。

- **Learning-based planner（基于学习的规划器）**使用在人类驾驶数据上训练的神经网络生成轨迹。模型观察场景并直接输出规划轨迹，从专家人类驾驶示例中隐式学习复杂的权衡。

- 优势在于人类驾驶行为被整体捕获，包括那些微妙、难以形式化的方面：并线有多激进、在交叉口何时缓缓向前、给骑行者多少空间。劣势则与 imitation learning（文件 2）相同的分布偏移问题：模型在训练数据中代表性不足的情况下可能行为不可预测。

## 端到端驾驶

- **End-to-end driving（端到端驾驶）**完全消除模块边界。单个神经网络接收原始 sensor 输入（camera 图像、LiDAR 点云）并直接输出驾驶命令（转向、油门、刹车）或规划轨迹。没有独立的 perception、prediction 或 planning 模块。

- 其吸引力在于整个系统为最终任务（安全驾驶）联合优化，因此在模块边界处没有信息丢失。Perception 模块学会提取 planner 所需的确切特征，而非可能不捕获任务相关细节的通用物体检测。

- **UniAD**（Unified Autonomous Driving，统一自动驾驶）是一个里程碑式的端到端架构。它通过 BEV 编码器处理多 camera 图像，然后应用一系列基于 transformer 的模块级联：跟踪、在线建图、运动预测、occupancy 预测和 planning。虽然它有内部模块，但都是可微的并端到端联合训练，planning 损失通过整个网络反向传播。

- UniAD 的 planning 模块通过关注预测的 BEV 特征、预测的 agent 轨迹和预测的 occupancy 来生成未来自车路径点。这是多元链式法则（第 3 章）在行动：梯度从 planning 损失一路回流到图像编码器，告诉 perception 特征如何对 planning 更有用。

- 更新的端到端方法使用 VLA 风格架构（本章文件 3）。**DriveVLM** 等模型接收 camera 图像和导航指令（或路线），使用 VLM 主干产生驾驶动作。这把大规模预训练的好处（视觉理解、推理）直接带入驾驶栈。

- 端到端驾驶的张力在于**可解释性**。模块化系统可以报告"我在 (x, y) 检测到行人并预测他们会穿过"——失效模式可诊断。端到端系统是一个产生方向盘转角的黑盒。当它失效时，诊断原因很困难，这对安全认证是严重关切。

## 驾驶的世界模型

- **World model** 学习在给定当前状态和自车动作的情况下预测驾驶场景的未来状态：$p(s_{t+1} \mid s_t, a_t)$（如第 10 章所介绍）。在驾驶中，这意味着生成逼真的未来帧或 BEV 布局："如果我加速并向左转，3 秒后场景会是这样。"

- 世界模型为自动驾驶提供两种强大能力：

    - **基于想象的 planning**：planner 不必承诺一个动作然后看会发生什么，而是通过在世界模型中 rollout 多条候选轨迹来"想象"它们，评估每条的安全性和舒适性，并选最好的。这是 model-based RL（本章文件 2 所述）应用于驾驶。

    - **学习型仿真**：在真实驾驶数据上训练的世界模型实际上是一个数据驱动的仿真器。它无需手工构建仿真器的力气即可生成逼真场景（包括罕见的边缘情况）。关键地，它捕获真实驾驶的统计模式：其他驾驶员实际如何行为、光照如何变化、雨如何影响能见度。

- **GAIA-1**（Wayve）是一个生成式驾驶世界模型。给定过去 camera 帧序列和自车动作，它自回归地生成未来视频帧。它使用以动作输入为条件的视频 diffusion 架构。模型学会生成合理的未来：遵守交通规则的车辆、在人行道上行走的行人、正确切换的交通灯，这些都从训练数据中涌现而非编程规则。

- **DriveDreamer** 和 **GenAD** 采取类似方法但在 BEV 空间而非像素空间操作。预测未来 BEV 布局比生成完整视频帧更紧凑（类似于机器人学中的 DreamerV3 在潜空间而非像素空间预测，如文件 2 所述）。BEV 世界模型预测所有 agent 在哪里、道路结构会是什么样、哪里有自由空间，planner 直接使用这些。

- **神经闭环仿真**使用世界模型替代手工仿真器进行测试。给定一段真实驾驶日志作为起点，世界模型生成如果自车采取不同动作会发生什么。这实现了反事实评估："如果我晚刹车 0.5 秒会怎样？"而无需在物理上重建场景。

- 与 **JEPA** 框架（第 10 章）的联系在这里很自然。驾驶世界模型不需要预测像素完美的未来（每个像素的精确 RGB 值）。它们需要预测对 planning 有意义的方面：agent 在哪里、移动多快、自由空间在哪里。Embedding 空间预测（JEPA 风格）捕获这些语义有意义的属性，而不把容量浪费在无关视觉细节如精确云纹理上。

- 主要挑战是**长时域保真度**。世界模型随时间累积误差：第 2 帧的小错误会偏移所有后续帧。对驾驶而言，3 秒预测时域对战术决策（现在并线吗？）有用，但 30 秒时域（路线规划等战略决策所需）仍不可靠。当前工作通过重新锚定（周期性地用真实观测重置模型）和不确定性估计（标记预测何时不可靠）来缓解。

## 仿真

- 通过在真实道路上驾驶来测试自动驾驶汽车是必要但不充分的。危险场景（近乎碰撞、边缘情况）罕见，所以按行驶里程测试效率低。一辆车需要行驶数亿英里才能在统计上证明安全，这不可行。

- **Simulation（仿真）**提供无限、可控、安全的测试。现实世界中罕见的场景（孩子跑入道路、轮胎爆胎、突发障碍物）可以在仿真中测试数百万次。

- **CARLA** 是基于 Unreal Engine 构建的开源驾驶仿真器。它提供逼真的城市环境、动态天气、交通 agent 和 sensor 仿真（camera、LiDAR、radar）。研究人员使用 CARLA 训练基于 RL 的驾驶 agent 并评估 perception 算法。

- **nuPlan**（Motional）是一个闭环 planning 基准。不同于开环评估（回放日志数据并比较 planner 输出与人类驾驶员实际轨迹），闭环评估让 planner 的决策影响仿真：如果 planner 决定变道，仿真相应演化。这测试反应行为，而不仅仅是轨迹相似度。

![开环回放日志无交互；闭环让模型的动作改变仿真状态](../images/open_vs_closed_loop.svg)

- **开环**和**闭环**评估的区别至关重要：

    - 开环：回放录制场景，计算模型输出与人类驾驶员动作的相似度。这易于设置但有误导性：一个总是预测"直行"的模型在高速公路上误差可能很低，但在第一个弯道就会撞车。

    - 闭环：模型的动作改变仿真状态，仿真据此演化。这测试模型从自身错误中恢复和对动态情况反应的能力。它代价更高但有意义得多。

- **Scenario generation（场景生成）**创建给系统施压的测试用例。对抗性场景（车辆突然刹车、藏在停放车后的行人）通过优化自动驾驶系统表现最差的情况来生成。这与 ML 中的对抗训练（第 6 章）相关：寻找最大化损失的输入。

## 安全

- 自动驾驶中的安全由工程标准而非仅仅是 ML 指标来治理。

- **ISO 26262**（Functional Safety，功能安全）是安全关键电子系统的汽车标准。它定义了从 A（最低）到 D（最高）的 **Automotive Safety Integrity Level (ASIL)**，基于潜在危害的严重性、暴露率和可控性。自动驾驶系统的 perception 和 planning 组件通常是 ASIL-D，最高等级，需要广泛的验证、冗余和失效安全设计。

- **SOTIF**（Safety of the Intended Functionality，预期功能安全，ISO 21448）针对另一类危害：不是硬件失效（由 ISO 26262 覆盖），而是系统按设计工作但仍然产生不安全结果的情况。把白色卡车误分类为天空的 perception 模型（一个真实事件）是 SOTIF 问题：硬件工作正常，但算法的局限导致危害。

- **Operational Design Domain (ODD)（运行设计域）**定义自动驾驶系统设计运行的条件：特定地理区域、道路类型（仅高速公路、城市、两者）、天气条件（无大雪）、速度范围和一天中的时间。超出 ODD 运行是不允许的：如果系统不能应对雪，就不能在雪中行驶。

- **Fail-safe** 与 **fail-operational** 设计：
    - Fail-safe（失效安全）：检测到故障时，系统转移到安全状态（如靠边停车）。这是最低要求。
    - Fail-operational（失效可运行）：尽管故障，系统仍安全运行，使用冗余组件。具有冗余转向、制动和计算的自动驾驶车可以在单个组件失效后仍驾驶到安全位置。

- **Redundancy（冗余）**是基础。关键 perception sensor 是重复的：多个 camera 覆盖重叠视场、LiDAR 和 radar 都提供独立的深度测量、双计算平台运行相同软件。如果任何单一组件失效，其他组件仍提供足够信息安全驾驶。

## 自动化等级

![SAE 自动化等级从 L0（无自动化）到 L5（完全自动化），显示责任如何从人转移到系统](../images/sae_autonomy_levels.svg)

- **SAE J3016** 标准定义了六个驾驶自动化等级，从 0（无自动化）到 5（完全自动化）：

    - **Level 0（无自动化）**：人做一切。系统可能提供警告（车道偏离警报）但不控制车辆。

    - **Level 1（驾驶员辅助）**：系统控制转向或速度之一，但不同时控制两者。自适应巡航控制（维持速度和跟车距离）或车道保持辅助（使车辆保持在车道中央）属于 Level 1。

    - **Level 2（部分自动化）**：系统同时控制转向和速度，但人必须始终监控并准备接管。Tesla Autopilot、GM Super Cruise 以及大多数当前"自动驾驶"功能属于 Level 2。人仍是责任驾驶员。

    - **Level 3（有条件自动化）**：系统驾驶并监控环境，但仅在特定条件（ODD）下。人可以脱手但必须准备在系统请求时接管（有时间缓冲，通常 10+ 秒）。Mercedes Drive Pilot（特定高速公路、低于 60 km/h）是首个认证的 Level 3 系统。

    - **Level 4（高度自动化）**：系统在其 ODD 内驾驶并处理所有情况，无需人干预。如果遇到 ODD 外的情况，它可以安全地自行停车。Waymo 的 robotaxi 服务在特定地理区域以 Level 4 运行。

    - **Level 5（完全自动化）**：系统在人能去的所有地方、所有条件下驾驶。无需方向盘或踏板。这尚不存在。

- 关键区别是**谁对安全负责**。Level 0-2 由人负责。Level 3-5 由系统负责（在其 ODD 内）。这具有深远的法律、保险和伦理含义。

- 当前行业状态是 Level 2（广泛部署）、Level 3（开始部署）和 Level 4（有限地理部署）的混合。Level 5 仍是长期研究目标。

import jax
import jax.numpy as jnp
import matplotlib.pyplot as plt

# Trajectory: N waypoints, each (x, y)
N = 20
start = jnp.array([0.0, 0.0])
goal = jnp.array([10.0, 0.0])
obstacle = jnp.array([5.0, 0.0])
obs_radius = 1.5

# Initialise: straight line from start to goal
waypoints_init = jnp.linspace(start, goal, N)

def cost(waypoints):
    wp = jnp.concatenate([start[None], waypoints, goal[None]], axis=0)

    # Smoothness: penalise acceleration (second differences)
    accel = wp[2:] - 2 * wp[1:-1] + wp[:-2]
    smooth_cost = jnp.sum(accel ** 2)

    # Obstacle avoidance: penalise proximity
    dists = jnp.linalg.norm(wp - obstacle, axis=1)
    collision_cost = jnp.sum(jnp.maximum(0, obs_radius + 0.5 - dists) ** 2)

    return 10 * smooth_cost + 100 * collision_cost

grad_cost = jax.grad(cost)

# Optimise the interior waypoints
waypoints = waypoints_init[1:-1]
lr = 0.01
for _ in range(500):
    g = grad_cost(waypoints)
    waypoints = waypoints - lr * g

# Plot
full_path = jnp.concatenate([start[None], waypoints, goal[None]], axis=0)
theta = jnp.linspace(0, 2 * jnp.pi, 100)

import jax.numpy as jnp
import matplotlib.pyplot as plt

# Ground truth: vehicle turning right
dt = 0.1
T = 40  # 4 seconds
v = 10.0  # m/s
omega = 0.3  # rad/s (turning rate)

# True trajectory (constant turn rate)
t = jnp.arange(T) * dt
theta = omega * t
gt_x = (v / omega) * jnp.sin(theta)
gt_y = (v / omega) * (1 - jnp.cos(theta))

# Constant velocity prediction from t=0
# Assumes the car continues straight at its current heading
obs_steps = 10  # observe first 1 second
vx0 = v * jnp.cos(theta[obs_steps - 1])
vy0 = v * jnp.sin(theta[obs_steps - 1])
pred_t = jnp.arange(T - obs_steps) * dt
pred_x = gt_x[obs_steps - 1] + vx0 * pred_t
pred_y = gt_y[obs_steps - 1] + vy0 * pred_t

plt.figure(figsize=(8, 6))
plt.plot(gt_x[:obs_steps], gt_y[:obs_steps], "ko-", label="Observed")
plt.plot(gt_x[obs_steps:], gt_y[obs_steps:], "g-", linewidth=2, label="True future")

3. 实现一个简单的基于规则的 planner，根据检测到的障碍物在车道保持和停车之间决策。
```python
import jax.numpy as jnp

def rule_based_planner(ego_speed, obstacles, speed_limit=13.9):
    """
    Simple rule-based planner.
    ego_speed: current speed (m/s)
    obstacles: list of (distance, speed) tuples for vehicles ahead
    speed_limit: max allowed speed (m/s), default ~50 km/h

    Returns: (target_speed, action_label)
    """
    min_following_distance = 2.0 * ego_speed  # 2-second rule
    emergency_distance = 5.0  # metres

    if not obstacles:
        return speed_limit, "cruise"

    # Find closest obstacle ahead
    closest_dist, closest_speed = min(obstacles, key=lambda o: o[0])

    if closest_dist < emergency_distance:
        return 0.0, "EMERGENCY STOP"
    elif closest_dist < min_following_distance:
        # Match speed of vehicle ahead
        target = min(closest_speed, speed_limit)
        return target, "following"
    else:
        return speed_limit, "cruise"

# Test scenarios
scenarios = [
    (13.9, [], "Empty road"),
    (13.9, [(30.0, 10.0)], "Slower car ahead"),
    (13.9, [(3.0, 0.0)], "Stopped car very close"),
    (13.9, [(50.0, 13.9)], "Car ahead at same speed"),
]

for speed, obs, desc in scenarios:
    target, action = rule_based_planner(speed, obs)
    print(f"{desc:30s}  →  {action:15s} target_speed={target:.1f} m/s ({target*3.6:.0f} km/h)")
```
