# Массивы и хеширование

*Массивы и хеш-таблицы — две самые фундаментальные структуры данных в программировании. В этом файле рассматривается их внутреннее устройство, а затем на основе постепенно усложняющихся задач выстраиваются ключевые паттерны решения: два указателя, скользящее окно, префиксные суммы и поиск по хеш-таблице, с разбором типичных ошибок на каждом этапе.*

- Если вы глубоко понимаете массивы и хеш-карты, вы сможете решить ~40% всех задач на собеседованиях по программированию. Эти две структуры встречаются повсеместно, так как они предоставляют две вещи, наиболее необходимые алгоритмам: **быстрый доступ по индексу** (массивы) и **быстрый поиск по ключу** (хеш-карты).

- Этот файл обучает паттернам, а не готовым решениям. Цель состоит в том, чтобы при встрече с новой задачей вы понимали, какой паттерн применим и почему, вместо того чтобы пытаться вспомнить заученное решение.

## Массивы

- **Массив** — это непрерывный блок памяти, где элементы хранятся по фиксированным смещениям. Доступ к элементу $i$ стоит $O(1)$, так как адрес вычисляется просто как `base + i * element_size`. Это максимально быстрый доступ к данным, и именно поэтому массивы являются выбором по умолчанию.

- **Динамические массивы** (Python `list`, Java `ArrayList`, C++ `vector`) автоматически увеличиваются при заполнении. Стратегия заключается в **амортизированном удвоении**: когда массив заполнен, выделяется новый массив удвоенного размера, и все данные копируются в него. Копирование стоит $O(n)$, но это происходит настолько редко (каждые $n$ вставок), что амортизированная стоимость одной вставки составляет $O(1)$.

- **Локальность кеша** — причина, по которой массивы быстры на практике, а не только в теории. Поскольку элементы хранятся непрерывно, доступ к одному элементу загружает соседние элементы в кеш процессора (глава 13). Итерация по массиву благоприятна для кеша; следование по указателям в связном списке — нет. Эта разница в константный множитель на практике может составлять 10–100 раз.

| Операция | Массив | Динамический массив |
|-----------|-------|---------------|
| Доступ по индексу | $O(1)$ | $O(1)$ |
| Добавление в конец | н/д | $O(1)$ амортизированное |
| Вставка в позицию $i$ | $O(n)$ | $O(n)$ |
| Удаление в позиции $i$ | $O(n)$ | $O(n)$ |
| Поиск (неотсортированный) | $O(n)$ | $O(n)$ |

- **Подводный камень**: вставка или удаление в середине массива требует $O(n)$, так как все последующие элементы должны быть сдвинуты. Если вам часто требуется вставка в середину, рассмотрите связный список или принципиально иной подход.

## Строки

- **Строка** — это массив символов. В Python строки неизменяемы (immutable): каждая конкатенация создает новую строку. Построение строки символ за символом в цикле имеет сложность $O(n^2)$, так как каждая конкатенация копирует всю строку, созданную к этому моменту.

```python
# BAD: O(n^2) string concatenation
s = ""
for c in characters:
    s += c  # copies entire string each time

# GOOD: O(n) using a list then join
parts = []
for c in characters:
    parts.append(c)
s = "".join(parts)
```

- **Подводный камень**: в Python `s += c` внутри цикла — одна из самых частых ошибок производительности. Всегда собирайте символы в список и используйте `.join()`.

- **Кодировка**: ASCII использует 7 бит (128 символов). **UTF-8** имеет переменную длину: символы ASCII занимают 1 байт, символы с диакритическими знаками — 2, китайские/японские иероглифы — 3, эмодзи — 4. Когда в задаче говорится «строчные английские буквы», размер алфавита равен 26, что означает, что можно использовать массив фиксированного размера вместо хеш-карты.

## Хеш-таблицы

- **Хеш-таблица** отображает ключи в значения с $O(1)$ в среднем случае для поиска, вставки и удаления. Она работает за счет вычисления **хеш-функции** $h(key)$, которая преобразует ключ в индекс массива.

- Хеш-функция должна быть: **детерминированной** (один и тот же ключ всегда дает один и тот же хеш), **равномерной** (распределяет ключи по бакетам равномерно) и **быстрой** в вычислении.

- **Коллизии** возникают, когда два разных ключа имеют одинаковый хеш-индекс. Две основные стратегии:

    - **Цепочки (Chaining)**: каждый бакет хранит связный список пар «ключ-значение». При коллизии элемент добавляется в список. Худший случай (все ключи попадают в один бакет): $O(n)$. Средний случай с хорошей хеш-функцией: $O(1)$.

    - **Открытая адресация (Open addressing)**: при коллизии выполняется поиск следующего свободного слота. **Линейное пробирование** проверяет следующий слот, затем следующий и т. д. Это благоприятно для кеша, но страдает от **кластеризации** (длинные последовательности занятых слотов). **Хеширование Робин Гуда** снижает дисперсию за счет вытеснения записей, которые находятся «ближе к дому».

- **Коэффициент заполнения** $\alpha = n / m$ (элементы / бакеты) определяет производительность. Когда $\alpha$ превышает порог (обычно 0.75), таблица выполняет **рехеширование**: выделяется таблица большего размера и все элементы вставляются заново. Это стоит $O(n)$, но происходит редко.

- **Хеш-карты** (`dict` в Python, `HashMap` в Java) хранят пары «ключ-значение». **Хеш-множества** (`set` в Python, `HashSet` в Java) хранят только ключи (используются для быстрой проверки на наличие элемента).

| Операция | В среднем | Худший случай |
|-----------|---------|------------|
| Поиск | $O(1)$ | $O(n)$ |
| Вставка | $O(1)$ | $O(n)$ |
| Удаление | $O(1)$ | $O(n)$ |

- **Фильтры Блума** — это вероятностные множества, эффективные по памяти. Они могут сказать «точно нет в множестве» или «возможно, есть в множестве» (с настраиваемой частотой ложноположительных срабатываний). Они используют $k$ хеш-функций и битовый массив. Используются в базах данных (избегают чтения с диска для отсутствующих ключей), веб-кешах и проверке орфографии.

- **Когда использовать хеш-карту**: всякий раз, когда вам нужно ответить на вопрос «видел ли я это раньше?» или «какое значение/индекс/счетчик связаны с этим ключом?» за $O(1)$. Если вы выполняете повторяющиеся линейные сканирования в поисках чего-либо, хеш-карта почти наверняка ускорит процесс.

## Паттерн: Поиск по хеш-карте

- Самый базовый паттерн: используйте хеш-карту, чтобы заменить сканирование $O(n)$ на поиск $O(1)$.

### Легкий уровень: Two Sum

- **Задача**: дан массив целых чисел и целевое значение (target), верните индексы двух чисел, которые в сумме дают target.

- **Метод грубой силы** $O(n^2)$: проверить каждую пару.

- **Идея паттерна**: для каждого числа `num` вам нужно, чтобы `target - num` существовало где-то в массиве. Вместо того чтобы сканировать массив в его поиске, сохраняйте ранее встреченные числа в хеш-карте.

```python
def two_sum(nums, target):
    seen = {}  # value -> index
    for i, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i
```

- **Почему это работает**: один проход по массиву. Для каждого элемента поиск в хеш-таблице занимает $O(1)$. Итого: $O(n)$ по времени, $O(n)$ по памяти.

- **Подводный камень**: не добавляйте текущее число в хеш-таблицу до проверки дополнения, иначе вы можете сопоставить элемент с самим собой. Порядок в коде выше верный: сначала проверка, затем вставка.

### Medium: Группировка анаграмм

- **Задача**: дан список строк, сгруппируйте анаграммы вместе. ("eat", "tea", "ate") — это одна группа.

- **Идея паттерна**: анаграммы содержат одни и те же символы в разном порядке. Если отсортировать каждую строку, анаграммы дадут одинаковый отсортированный ключ. Используйте этот ключ в качестве ключа хеш-таблицы.

```python
from collections import defaultdict

def group_anagrams(strs):
    groups = defaultdict(list)
    for s in strs:
        key = tuple(sorted(s))  # or use character count tuple
        groups[key].append(s)
    return list(groups.values())
```

- **Оптимизация**: сортировка каждой строки занимает $O(k \log k)$, где $k$ — длина строки. Для более быстрого получения ключа подсчитайте частоту символов и используйте кортеж с этими значениями в качестве ключа:

```python
def group_anagrams_fast(strs):
    groups = defaultdict(list)
    for s in strs:
        count = [0] * 26
        for c in s:
            count[ord(c) - ord('a')] += 1
        groups[tuple(count)].append(s)
    return list(groups.values())
```

- Это $O(k)$ для каждой строки вместо $O(k \log k)$. Кортеж с количеством символов является **канонической формой**: представлением, которое одинаково для всех членов группы.

- **Подводный камень**: в Python списки не являются хешируемыми (не могут быть ключами словаря). Вы должны преобразовать их в кортеж. Это часто вызывает ошибки, когда пытаются использовать `groups[count].append(s)`.

### Hard: Самая длинная последовательность

- **Задача**: дан несортированный массив, найдите длину самой длинной последовательности идущих подряд чисел (например, [100, 4, 200, 1, 3, 2] → 4, так как [1, 2, 3, 4]).

- **Грубая сила** $O(n \log n)$: отсортируйте массив, затем просканируйте его на наличие последовательных участков.

- **Идея паттерна**: поместите все числа в хеш-множество для поиска за $O(1)$. Для каждого числа проверьте, является ли оно **началом** последовательности (т.е. `num - 1` отсутствует в множестве). Если да, посчитайте, насколько далеко простирается последовательность.

```python
def longest_consecutive(nums):
    num_set = set(nums)
    best = 0

    for num in num_set:
        # only start counting from the beginning of a sequence
        if num - 1 not in num_set:
            length = 1
            while num + length in num_set:
                length += 1
            best = max(best, length)

    return best
```

- **Почему $O(n)$**: внутренний цикл `while` выполняется в сумме не более $n$ раз для всех итераций (каждое число посещается не более двух раз: один раз во внешнем цикле и один раз при расширении в `while`). Условие `if num - 1 not in num_set` гарантирует, что мы начинаем отсчет только с начала последовательности.

- **Подводный камень**: без проверки `if num - 1 not in num_set` вы бы начинали отсчет с каждого элемента, что привело бы к $O(n^2)$ в худшем случае (например, [1, 2, 3, ..., n] сканировало бы всю последовательность из каждой начальной точки).

## Паттерн: Два указателя

- Паттерн **два указателя** использует два индекса, которые перемещаются по массиву, обычно с противоположных концов или с одного конца с разной скоростью. Он работает, когда массив отсортирован или когда нужно сравнивать пары.

- **Когда использовать**: задача включает пары, подмассивы или разбиение, а массив отсортирован (или может быть отсортирован без потери необходимой информации).

### Easy: Валидный палиндром

- **Задача**: определите, является ли строка палиндромом, учитывая только буквенно-цифровые символы и игнорируя регистр.

- **Паттерн**: один указатель в начале, другой в конце. Двигайте их навстречу друг другу, сравнивая символы.

```python
def is_palindrome(s):
    left, right = 0, len(s) - 1

    while left < right:
        # skip non-alphanumeric characters
        while left < right and not s[left].isalnum():
            left += 1
        while left < right and not s[right].isalnum():
            right -= 1

        if s[left].lower() != s[right].lower():
            return False

        left += 1
        right -= 1

    return True
```

- **Подводный камень**: забыть проверку `left < right` внутри внутренних циклов while. Без нее указатель может выйти за границы массива на строках вроде "!!!" (состоящих только из не буквенно-цифровых символов).

### Medium: Сумма трех чисел

- **Задача**: найдите все уникальные тройки в массиве, которые в сумме дают ноль.

- **Паттерн**: отсортируйте массив. Зафиксируйте один элемент, затем используйте два указателя на оставшейся части, чтобы найти пары, сумма которых равна отрицательному значению зафиксированного элемента.

```python
def three_sum(nums):
    nums.sort()
    result = []

    for i in range(len(nums) - 2):
        # skip duplicate fixed elements
        if i > 0 and nums[i] == nums[i - 1]:
            continue

        left, right = i + 1, len(nums) - 1
        target = -nums[i]

        while left < right:
            total = nums[left] + nums[right]
            if total < target:
                left += 1
            elif total > target:
                right -= 1
            else:
                result.append([nums[i], nums[left], nums[right]])
                # skip duplicates
                while left < right and nums[left] == nums[left + 1]:
                    left += 1
                while left < right and nums[right] == nums[right - 1]:
                    right -= 1
                left += 1
                right -= 1

    return result
```

- **Почему это работает**: сортировка занимает $O(n \log n)$. Для каждого зафиксированного элемента сканирование двумя указателями занимает $O(n)$. Итого: $O(n^2)$, что является оптимальным для этой задачи (необходимо рассмотреть все пары).

- **Подводный камень**: обработка дубликатов — самая сложная часть. Без логики пропуска дубликатов (как для зафиксированного элемента, так и для результатов работы двух указателей) вы получите дублирующиеся тройки. Строка `if i > 0 and nums[i] == nums[i-1]: continue` критически важна.

### Hard: Ловушка для дождевой воды

- **Задача**: дана карта высот (массив неотрицательных целых чисел), вычислите, сколько воды может удержаться после дождя.

- **Идея паттерна**: для каждой позиции уровень воды определяется минимумом из максимальной высоты слева и максимальной высоты справа минус текущая высота. Два указателя с обоих концов отслеживают эти текущие максимумы.

```python
def trap(height):
    left, right = 0, len(height) - 1
    left_max, right_max = 0, 0
    water = 0

    while left < right:
        if height[left] < height[right]:
            if height[left] >= left_max:
                left_max = height[left]
            else:
                water += left_max - height[left]
            left += 1
        else:
            if height[right] >= right_max:
                right_max = height[right]
            else:
                water += right_max - height[right]
            right -= 1

    return water
```

- **Почему это работает**: ключевая идея заключается в том, что если `height[left] < height[right]`, то объем воды в позиции `left` ограничен значением `left_max` (мы знаем, что справа есть более высокий столбик, поэтому правая сторона не может быть «узким местом»). Мы обрабатываем более короткую сторону, будучи уверенными, что с другой стороны есть более высокий столбик.

- **Подводный камень**: многие пытаются сначала предварительно вычислить массивы `left_max[i]` и `right_max[i]` (это работает, но требует $O(n)$ памяти). Подход с двумя указателями позволяет достичь $O(1)$ памяти. Кроме того, путаница между `>=` и `>` при обновлении максимумов может привести к ошибкам на единицу при расчете объема воды.

## Шаблон: Скользящее окно (Sliding Window)

- Шаблон **скользящего окна** поддерживает окно (непрерывный подмассив), которое расширяется и сужается по мере итерации. Он подходит для задач, связанных с подмассивами или подстроками, удовлетворяющими некоторому условию.

- **Когда использовать**: если задача требует найти самый длинный/короткий подмассив или подстроку, удовлетворяющие ограничению, и расширение/сужение окна является монотонным (добавление элементов может только усложнить или упростить выполнение условия, но не то и другое одновременно).

- **Шаблон**:

```python
def sliding_window(arr):
    left = 0
    state = ...  # window state (counts, sum, etc.)
    best = ...

    for right in range(len(arr)):
        # expand: add arr[right] to the window state
        update_state(state, arr[right])

        # contract: shrink from the left while constraint is violated
        while constraint_violated(state):
            remove_from_state(state, arr[left])
            left += 1

        # update answer
        best = max(best, right - left + 1)  # or min, depending on problem

    return best
```

### Легкий уровень: Лучшее время для покупки и продажи акций

- **Задача**: даны ежедневные цены, найти максимальную прибыль от одной покупки и одной продажи (покупка перед продажей).

- **Шаблон**: отслеживайте минимальную цену, встреченную на данный момент (левая граница окна), и вычисляйте прибыль для каждого дня.

```python
def max_profit(prices):
    min_price = float('inf')
    max_profit = 0

    for price in prices:
        min_price = min(min_price, price)
        max_profit = max(max_profit, price - min_price)

    return max_profit
```

- Это вырожденный случай скользящего окна: левый указатель (минимальная цена) сдвигается вперед только тогда, когда найден новый минимум. Временная сложность $O(n)$, пространственная сложность $O(1)$.

### Средний уровень: Самая длинная подстрока без повторяющихся символов

- **Задача**: найти длину самой длинной подстроки без повторяющихся символов.

- **Шаблон**: расширяйте окно, перемещая `right`. Когда найден дубликат, сужайте окно слева, пока дубликат не будет удален.

```python
def length_of_longest_substring(s):
    char_index = {}  # character -> its most recent index
    left = 0
    best = 0

    for right, char in enumerate(s):
        if char in char_index and char_index[char] >= left:
            left = char_index[char] + 1  # jump past the duplicate

        char_index[char] = right
        best = max(best, right - left + 1)

    return best
```

- **Почему `char_index[char] >= left`**: символ может находиться в словаре еще с тех пор, как текущее окно не началось. Без этой проверки вы бы ошибочно сузили окно для символа, которого на самом деле нет в текущем окне.

- **Подводный камень**: использование множества и удаление символов по одному слева — это корректно, но медленнее. Подход с хеш-картой позволяет сразу перейти к нужной позиции справа.

### Сложный уровень: Минимальное окно подстроки

- **Задача**: даны строки `s` и `t`, найти минимальное окно в `s`, которое содержит все символы из `t`.

- **Шаблон**: расширяйте окно, чтобы включить все необходимые символы, затем сужайте его слева, чтобы найти минимальное валидное окно.

```python
from collections import Counter

def min_window(s, t):
    if not t or not s:
        return ""

    need = Counter(t)       # characters we need and their counts
    have = 0                # how many unique characters we have in sufficient quantity
    required = len(need)    # how many unique characters we need

    left = 0
    best = (float('inf'), 0, 0)  # (length, left, right)

    window_counts = {}

    for right in range(len(s)):
        char = s[right]
        window_counts[char] = window_counts.get(char, 0) + 1

        # check if this character's count now meets the requirement
        if char in need and window_counts[char] == need[char]:
            have += 1

        # contract from the left while the window is valid
        while have == required:
            # update best
            if (right - left + 1) < best[0]:
                best = (right - left + 1, left, right)

            # remove leftmost character
            left_char = s[left]
            window_counts[left_char] -= 1
            if left_char in need and window_counts[left_char] < need[left_char]:
                have -= 1
            left += 1

    length, start, end = best
    return s[start:end + 1] if length != float('inf') else ""
```

- **Подводный камень**: счетчик `have` — это критически важная оптимизация. Без него вам пришлось бы сравнивать весь словарь `window_counts` с `need` на каждом шаге, что дает $O(|\text{unique chars}|)$ на шаг. Счетчик `have` делает проверку валидности $O(1)$.

- **Подводный камень**: проверка `window_counts[char] == need[char]` (а не `>=`) гарантирует, что мы увеличиваем `have` ровно один раз для каждого символа. Если бы мы использовали `>=`, мы бы пересчитали количество.

## Шаблон: Префиксные суммы

- Массив **префиксных сумм** хранит накопленные суммы: `prefix[i] = sum(arr[0:i])`. После построения за $O(n)$ сумму любого подмассива можно вычислить за $O(1)$: `sum(arr[l:r]) = prefix[r] - prefix[l]`.

```python
def build_prefix(arr):
    prefix = [0] * (len(arr) + 1)
    for i in range(len(arr)):
        prefix[i + 1] = prefix[i] + arr[i]
    return prefix

# sum of arr[l:r] (inclusive l, exclusive r)
def range_sum(prefix, l, r):
    return prefix[r] - prefix[l]
```

- **Когда использовать**: задача включает множественные запросы суммы подмассива или поиск подмассивов с заданной суммой.

### Легкий уровень: Запрос суммы на отрезке (Range Sum Query)

- **Задача**: дан массив, нужно ответить на несколько запросов вида «чему равна сумма от индекса $l$ до $r$?»

- Без префиксных сумм: каждый запрос выполняется за $O(n)$. С префиксными суммами: $O(n)$ на предварительные вычисления, затем $O(1)$ на каждый запрос.

### Средний уровень: Сумма подмассива равна K

- **Задача**: подсчитать количество непрерывных подмассивов, сумма которых равна $k$.

- **Идея паттерна**: сумма подмассива от индекса $l$ до $r$ равна `prefix[r+1] - prefix[l]`. Мы хотим, чтобы это было равно $k$, следовательно, `prefix[l] = prefix[r+1] - k`. Для каждой позиции подсчитайте, сколько предыдущих префиксных сумм равны `current_prefix - k`, используя хеш-таблицу.

```python
def subarray_sum(nums, k):
    count = 0
    prefix = 0
    prefix_counts = {0: 1}  # empty prefix sum

    for num in nums:
        prefix += num
        # how many earlier prefix sums equal prefix - k?
        count += prefix_counts.get(prefix - k, 0)
        prefix_counts[prefix] = prefix_counts.get(prefix, 0) + 1

    return count
```

- Этот метод объединяет префиксные суммы с поиском по хеш-таблице: время $O(n)$, память $O(n)$.

- **Ловушка**: забыть инициализировать `prefix_counts = {0: 1}`. Пустой префикс (перед любым элементом) имеет сумму 0. Без этого вы пропустите подмассивы, начинающиеся с индекса 0.

### Сложный уровень: Произведение массива кроме самого себя

- **Задача**: дан массив, верните массив, где каждый элемент является произведением всех остальных элементов. Нельзя использовать деление.

- **Паттерн**: постройте префиксные произведения слева и суффиксные произведения справа. Ответ для каждой позиции — `left_product * right_product`.

```python
def product_except_self(nums):
    n = len(nums)
    result = [1] * n

    # left pass: result[i] = product of nums[0..i-1]
    prefix = 1
    for i in range(n):
        result[i] = prefix
        prefix *= nums[i]

    # right pass: multiply by product of nums[i+1..n-1]
    suffix = 1
    for i in range(n - 1, -1, -1):
        result[i] *= suffix
        suffix *= nums[i]

    return result
```

- Время $O(n)$, дополнительная память $O(1)$ (результирующий массив не учитывается). Здесь используется сам результирующий массив для хранения промежуточных префиксных произведений, а затем во втором проходе умножаются суффиксные произведения.

- **Ловушка**: если массив содержит ноль, подходы с делением не работают. Этот подход с префиксами/суффиксами корректно обрабатывает нули, так как деление не используется.

## Резюме типичных ловушек

| Ловушка | Пример | Решение |
|---------|---------|-----|
| Ошибка на единицу в размере окна | `right - left` против `right - left + 1` | Нарисуйте пример из 2 элементов |
| Изменяемое значение по умолчанию в Python | `def f(seen={})` сохраняет состояние между вызовами | Используйте `def f(seen=None)` |
| Конкатенация строк в цикле | `s += c` имеет сложность $O(n^2)$ в Python | Используйте `list.append` + `"".join"` |
| Забыли `{0: 1}` в префиксной сумме | Пропуск подмассивов, начинающихся с индекса 0 | Всегда инициализируйте пустым префиксом |
| Хеш-таблица перед проверкой | Two Sum: добавление `num` до проверки дополнения | Сначала проверьте, затем вставьте |
| Необработка дубликатов | Three Sum возвращает дублирующиеся тройки | Пропускайте последовательные равные значения |
| Переполнение целых чисел | Сумма больших массивов в C++/Java | Используйте `long` или проверяйте границы |
