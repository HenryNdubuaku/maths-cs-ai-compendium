## Почему паттерны, а не заучивание

- На LeetCode, NeetCode и HackerRank тысячи задач по программированию. Никто не может запомнить их все, и попытка сделать это — проигрышная стратегия. Интервьюеры не выбирают задачи из фиксированного списка: они видоизменяют, комбинируют и маскируют их. Заученное решение задачи «Two Sum» не поможет вам, когда интервьюер попросит решить вариант, который вы никогда не видели.

- Хорошая новость: существует всего около **15–20 основных паттернов** (два указателя, скользящее окно, BFS/DFS, динамическое программирование, поиск с возвратом и т. д.). Любая задача, как бы необычно она ни выглядела на первый взгляд, сводится к одному из этих паттернов или их комбинации. Интервью проверяет не то, видели ли вы эту конкретную задачу раньше. Оно проверяет, способны ли вы **отбросить контекст**, историю, специфические типы данных, граничные случаи и распознать лежащий в основе паттерн.

- Рассмотрим три задачи:
    - «Найти в массиве два числа, сумма которых равна целевому значению».
    - «Найти две молекулы, сумма энергий связи которых равна пороговому значению».
    - «Дан список балансов счетов, найти два счета, суммарная стоимость которых равна долгу».

- Они выглядят по-разному. Но это одна и та же задача: **Two Sum**. Контекст (числа, молекулы, счета) не имеет значения. Структура такова: поиск дополнения в коллекции → поиск в хеш-таблице.

- Именно поэтому в этой главе мы обучаем **паттернам через интуицию**, а не решениям через повторение. Для каждого паттерна мы объясняем:
    - **Какое структурное свойство** задачи указывает на этот паттерн (отсортированный ввод → два указателя; ограничение на подмассив → скользящее окно; оптимальная подструктура + перекрывающиеся подзадачи → динамическое программирование).
    - **Почему паттерн работает** — математическое или логическое обоснование, а не просто «это дает правильный ответ».
    - **Как его адаптировать** — показывая простые, средние и сложные варианты, где одна и та же базовая идея применяется в разных контекстах.

- Когда вы глубоко понимаете, *почему* работает скользящее окно (монотонность ограничения означает, что расширения/сужения достаточно), вы можете применить его к любой задаче с такой структурой, даже если никогда ее не видели. Если же вы просто зазубрили код для задачи «Longest Substring Without Repeating Characters», вы окажетесь в тупике, как только задача изменится.

- Практическая стратегия:
    1. **Изучите паттерн** (эта глава).
    2. **Учитесь распознавать его** в замаскированных задачах (домашние задания NeetCode в конце каждого файла).
    3. **Практикуйтесь в реализации** в условиях нехватки времени.
    4. На интервью: прочитайте задачу → отбросьте контекст → определите паттерн → реализуйте.

## Нотация Big O

- Когда мы говорим, что алгоритм «быстрый» или «медленный», нам нужен точный способ измерения. **Нотация Big O** описывает, как время выполнения алгоритма (или использование памяти) растет по мере увеличения размера входных данных $n$, игнорируя константные множители и члены низших порядков.

- Формальное определение: $f(n) = O(g(n))$ означает, что существуют константы $c > 0$ и $n_0$ такие, что $f(n) \leq c \cdot g(n)$ для всех $n \geq n_0$. Простыми словами: $f$ растет не быстрее, чем $g$ для больших входных данных.

- Почему мы игнорируем константы? Потому что алгоритм $2n$ и алгоритм $5n$ — оба являются $O(n)$: они масштабируются одинаково. На более быстром компьютере константы изменятся, но масштабирование — нет. Big O фиксирует **внутреннюю** сложность задачи, независимо от оборудования.

### Иерархия скорости роста

- От самого быстрого к самому медленному:

| Big O | Название | Пример | $n = 10^6$ операций |
|-------|----------|----------|----------------------|
| $O(1)$ | Константная | Доступ к массиву, хеш-поиск | 1 |
| $O(\log n)$ | Логарифмическая | Бинарный поиск | 20 |
| $O(n)$ | Линейная | Линейный проход, один цикл | $10^6$ |
| $O(n \log n)$ | Линейно-логарифмическая | Сортировка слиянием, эффективная сортировка | $2 \times 10^7$ |
| $O(n^2)$ | Квадратичная | Вложенные циклы, перебор пар | $10^{12}$ (слишком медленно) |
| $O(n^3)$ | Кубическая | Три вложенных цикла, умножение матриц | $10^{18}$ (очень медленно) |
| $O(2^n)$ | Экспоненциальная | Все подмножества, перебор с возвратом | $10^{301030}$ (невозможно) |
| $O(n!)$ | Факториальная | Все перестановки | абсурдно |

- **Эмпирическое правило**: современные компьютеры выполняют примерно $10^8$–$10^9$ простых операций в секунду. Для ограничения по времени в 1 секунду:
    - $O(n)$ работает при $n \leq 10^8$
    - $O(n \log n)$ работает при $n \leq 10^7$
    - $O(n^2)$ работает при $n \leq 10^4$
    - $O(2^n)$ работает при $n \leq 25$

- Эта таблица сразу подсказывает, достаточно ли быстр ваш подход. Если $n = 10^5$, а ваше решение $O(n^2)$, это будет $10^{10}$ операций — слишком медленно. Вам нужен алгоритм получше.

### Как анализировать Big O

- **Один цикл** по $n$ элементам: $O(n)$.

```python
total = 0
for x in arr:   # n iterations
    total += x   # O(1) per iteration
# Total: O(n)
```

- **Вложенные циклы**: перемножайте количество итераций.

```python
for i in range(n):       # n iterations
    for j in range(n):   # n iterations each
        process(i, j)    # O(1)
# Total: O(n^2)
```

- **Цикл с делением пополам**: $O(\log n)$. Каждая итерация уменьшает размер задачи вдвое, поэтому требуется $\log_2 n$ итераций.

```python
i = n
while i > 0:
    process(i)
    i //= 2
# Total: O(log n)
```

- **Вложенный цикл, где внутренний зависит от внешнего**:

```python
for i in range(n):
    for j in range(i):   # j goes from 0 to i-1
        process(i, j)
# Total: 0 + 1 + 2 + ... + (n-1) = n(n-1)/2 = O(n^2)
```

- **Рекурсия**: запишите рекуррентное соотношение и решите его (в главе 13 рассматривалась основная теорема). Например, сортировка слиянием: $T(n) = 2T(n/2) + O(n) = O(n \log n)$.

### Common Traps

- **Hidden loops**: `x in list` is $O(n)$ in Python (linear scan), but `x in set` is $O(1)$. Using `in` on a list inside a loop gives $O(n^2)$, not $O(n)$.

```python
# BAD: O(n^2) — "in" on a list is O(n)
for x in arr:
    if x in another_list:
        process(x)

# GOOD: O(n) — convert to set first
another_set = set(another_list)
for x in arr:
    if x in another_set:
        process(x)
```

- **String concatenation**: `s += c` in Python copies the entire string each time. Inside a loop of $n$ iterations: $O(1 + 2 + \cdots + n) = O(n^2)$.

- **Sorting dominance**: if your algorithm sorts ($O(n \log n)$) and then does a linear scan ($O(n)$), the total is $O(n \log n)$ — the sort dominates.

- **Amortised complexity**: some operations are expensive occasionally but cheap on average. Dynamic array append is $O(1)$ amortised because the rare $O(n)$ resize is spread across $n$ cheap appends. Do not confuse amortised $O(1)$ with worst-case $O(1)$.

### Space Complexity

- Space complexity follows the same Big O rules, applied to memory usage instead of time.

- **In-place** algorithms use $O(1)$ extra space (not counting the input). Quicksort is $O(\log n)$ space (recursion stack depth). Merge sort is $O(n)$ (temporary arrays for merging).

- **Recursion stack**: every recursive call uses stack space. A recursion $n$ levels deep uses $O(n)$ space, even if each call does not allocate additional memory. This is why recursive DFS on a graph with $n$ nodes uses $O(n)$ space.

- For interviews, always state both time and space complexity. An $O(n)$ time, $O(n)$ space solution is often acceptable, but an $O(n)$ time, $O(1)$ space solution is better. The interviewer may ask you to optimise one or the other.

## Recursion

- **Recursion** is when a function calls itself to solve a smaller instance of the same problem. It is the most natural approach for problems with recursive structure: trees, nested structures, divide-and-conquer, and mathematical sequences.

- Every recursive function has two parts:
    1. **Base case**: the smallest instance that can be solved directly (without recursion). This is what stops the recursion.
    2. **Recursive case**: break the problem into smaller subproblems, solve them recursively, and combine the results.

### Example: Factorial

```python
def factorial(n):
    if n <= 1:        # base case
        return 1
    return n * factorial(n - 1)  # recursive case
```

- How it executes for `factorial(4)`:
    - `factorial(4)` calls `factorial(3)`
    - `factorial(3)` calls `factorial(2)`
    - `factorial(2)` calls `factorial(1)`
    - `factorial(1)` returns `1` (base case)
    - `factorial(2)` returns `2 * 1 = 2`
    - `factorial(3)` returns `3 * 2 = 6`
    - `factorial(4)` returns `4 * 6 = 24`

- Each call is pushed onto the **call stack**. The stack grows until the base case is reached, then unwinds as each call returns. If the recursion is too deep (e.g., `factorial(1000000)` in Python), the stack overflows (`RecursionError`). Python's default recursion limit is 1000.

### How to Think Recursively

- The key mental shift: **trust the recursion**. When writing a recursive function, assume that the recursive call returns the correct answer for the smaller subproblem. Your job is only to:
    1. Handle the base case.
    2. Break the problem into smaller pieces.
    3. Combine the results.

- You do not need to trace through every recursive call in your head. That is like trying to understand a loop by mentally executing every iteration. Instead, verify: "if the recursive call gives me the right answer for the smaller input, does my combination step give the right answer for the full input?"

### Example: Recursion on a Linked List

- Reverse a linked list recursively:

```python
def reverse(head):
    if not head or not head.next:   # base case: 0 or 1 nodes
        return head

    new_head = reverse(head.next)   # reverse the rest
    head.next.next = head           # point the next node back to me
    head.next = None                # I am now the tail
    return new_head
```

- **Trust the recursion**: `reverse(head.next)` correctly reverses the rest of the list and returns the new head. We just need to attach the current node at the end.

### Example: Recursion on a Tree

- Compute the height of a binary tree:

```python
def height(root):
    if not root:           # base case: empty tree has height 0
        return 0
    left_h = height(root.left)    # height of left subtree
    right_h = height(root.right)  # height of right subtree
    return 1 + max(left_h, right_h)  # this node adds 1 level
```

- This pattern — "recurse on left, recurse on right, combine" — solves the vast majority of tree problems (see file 03).

### Recursion vs Iteration

- Every recursive algorithm can be converted to an iterative one (using an explicit stack or loop). Iteration avoids the call stack overhead and the risk of stack overflow.

- **When to prefer recursion**: the problem has natural recursive structure (trees, nested data, divide-and-conquer). The recursive solution is cleaner and easier to reason about.

- **When to prefer iteration**: the recursion depth could be very large (e.g., processing a linked list of $10^6$ nodes). The iterative solution avoids stack overflow.

- **Tail recursion**: a recursive call is "tail recursive" if it is the last operation in the function (no work is done after the recursive call returns). Some languages (Scheme, Scala) optimise tail calls to use constant stack space. Python does **not** optimise tail calls, so tail recursion in Python still uses $O(n)$ stack space.

### Common Pitfalls

| Pitfall | Example | Fix |
|---------|---------|-----|
| Missing base case | Infinite recursion → stack overflow | Always define when to stop |
| Wrong base case | Off-by-one in recursive decomposition | Test with the smallest inputs (0, 1, 2) |
| Not reducing the problem | `f(n)` calls `f(n)` instead of `f(n-1)` | Ensure subproblem is strictly smaller |
| Redundant computation | Fibonacci: `f(n) = f(n-1) + f(n-2)` recomputes exponentially | Use memoisation (→ DP) |
| Python recursion limit | `factorial(10000)` crashes | Use `sys.setrecursionlimit` or convert to iteration |

## Backtracking

- **Backtracking** (поиск с возвратом) — это систематический способ исследования всех возможных решений путем их постепенного построения и отказа от частичных решений, которые заведомо не могут привести к корректному ответу.

- Представьте это как прохождение лабиринта. На каждой развилке вы выбираете путь. Если вы зашли в тупик, вы возвращаетесь к последней развилке и пробуете другой путь. Вы не начинаете всё сначала — вы **возвращаетесь** (backtrack) к самой последней точке принятия решения.

### Три шага

Каждый алгоритм поиска с возвратом следует одной и той же схеме:

1. **Выбор (Choose)**: выбор кандидата для расширения текущего частичного решения.
2. **Исследование (Explore)**: рекурсивная попытка построить полное решение из этого кандидата.
3. **Отмена выбора (Unchoose)**: отмена сделанного выбора (возврат) и попытка следующего кандидата.

```python
def backtrack(state, choices, result):
    if is_complete(state):
        result.append(state.copy())
        return

    for choice in choices:
        if is_valid(choice, state):
            state.add(choice)           # 1. choose
            backtrack(state, choices, result)  # 2. explore
            state.remove(choice)        # 3. unchoose (backtrack)
```

- Шаг **отмены выбора** — это то, что отличает поиск с возвратом от обычной рекурсии. Без него состояние накапливает все сделанные выборы, и вы не можете исследовать альтернативные пути.

### Когда использовать поиск с возвратом

- Задача требует **перечислить все допустимые конфигурации**: все перестановки, все подмножества, все допустимые расстановки (например, задача о N ферзях).
- Задача требует **найти любую допустимую конфигурацию**: решение судоку, поиск пути в лабиринте.
- Пространство поиска велико, но его можно **отсечь (prune)**: большинство частичных решений можно отбросить на раннем этапе, не исследуя их полностью.

### Как отсечение ускоряет работу

- Без отсечения поиск с возвратом исследует каждую возможную комбинацию — это экспоненциальное время. **Отсечение** обрезает ветви на раннем этапе:

```python
for choice in choices:
    if not is_valid(choice, state):
        continue  # PRUNE: skip this entire subtree

    state.add(choice)
    backtrack(state, choices, result)
    state.remove(choice)
```

- В задаче о N ферзях (файл 05) проверка конфликтов по столбцам и диагоналям перед установкой ферзя сокращает дерево поиска с $n^n$ примерно до $n!$ кандидатов. Для $n = 8$ это означает переход от 16 миллионов к 40 000 операций. Хорошее отсечение делает экспоненциальные алгоритмы пригодными для использования при умеренных $n$.

### Генерация всех подмножеств (простейший поиск с возвратом)

```python
def subsets(nums):
    result = []

    def backtrack(start, path):
        result.append(path[:])  # every partial solution is a valid subset

        for i in range(start, len(nums)):
            path.append(nums[i])        # choose
            backtrack(i + 1, path)       # explore (i+1: no reuse)
            path.pop()                   # unchoose

    backtrack(0, [])
    return result
```

- Для `[1, 2, 3]` дерево рекурсии выглядит так:
    - `[]` → `[1]` → `[1,2]` → `[1,2,3]` (backtrack) → `[1,3]` (backtrack) → `[2]` → `[2,3]` (backtrack) → `[3]`

- Каждый узел в дереве — это один вызов `backtrack`. Каждый лист (и промежуточный узел) создает подмножество. Общее количество подмножеств: $2^n$.

### Генерация всех перестановок

```python
def permutations(nums):
    result = []

    def backtrack(path, remaining):
        if not remaining:
            result.append(path[:])
            return

        for i in range(len(remaining)):
            path.append(remaining[i])                    # choose
            backtrack(path, remaining[:i] + remaining[i+1:])  # explore
            path.pop()                                   # unchoose

    backtrack([], nums)
    return result
```

- Общее количество перестановок: $n!$. Каждая требует $O(n)$ операций для создания `remaining`, поэтому общая сложность составляет $O(n \cdot n!)$.

### Типичные ошибки

| Ошибка | Пример | Исправление |
|---------|---------|-----|
| Забыли скопировать путь | `result.append(path)` — все записи ссылаются на один и тот же список | `result.append(path[:])` или `path.copy()` |
| Отсутствие возврата (отмены выбора) | Состояние продолжает расти, последующие кандидаты видят устаревшее состояние | Всегда используйте `path.pop()` или `state.remove()` после рекурсивного вызова |
| Неверный старт цикла | Подмножества с дубликатами или перестановки с нежелательным повторным использованием | Используйте параметр `start`, чтобы избежать повторного посещения предыдущих индексов |
| Пропуск отсечения | Исследование заведомо неверных ветвей | Добавьте `if not is_valid: continue` перед рекурсивным вызовом |

## Динамическое программирование

- **Динамическое программирование (DP)** — это метод оптимизации для задач, в которых одни и те же подзадачи решаются многократно. Вместо повторных вычислений DP решает каждую подзадачу один раз и сохраняет результат.

- DP применяется, когда задача обладает двумя свойствами:
    1. **Оптимальная подструктура**: оптимальное решение можно построить из оптимальных решений подзадач.
    2. **Перекрывающиеся подзадачи**: одни и те же подзадачи возникают несколько раз в процессе рекурсии.

### Мотивация на примере чисел Фибоначчи

- Наивная рекурсивная реализация Фибоначчи:

```python
def fib(n):
    if n <= 1:
        return n
    return fib(n - 1) + fib(n - 2)
```

- Для `fib(5)` дерево рекурсии выглядит так:
    - `fib(5)` вызывает `fib(4)` и `fib(3)`
    - `fib(4)` вызывает `fib(3)` и `fib(2)`
    - `fib(3)` вычисляется **дважды**, `fib(2)` вычисляется **трижды**

- Сложность составляет $O(2^n)$, так как дерево ветвится на каждом уровне, и большинство ветвей пересчитывают одни и те же значения. Для `fib(50)` потребуется более $10^{15}$ операций — это неосуществимо.

- С использованием **мемоизации** (DP сверху вниз):

```python
def fib_memo(n, memo={}):
    if n in memo:
        return memo[n]
    if n <= 1:
        return n
    memo[n] = fib_memo(n - 1, memo) + fib_memo(n - 2, memo)
    return memo[n]
```

- Теперь `fib(3)` вычисляется один раз, сохраняется и извлекается при последующих вызовах. Итого: $O(n)$ по времени, $O(n)$ по памяти.

- С использованием **табуляции** (DP снизу вверх):

```python
def fib_tab(n):
    if n <= 1:
        return n
    dp = [0] * (n + 1)
    dp[1] = 1
    for i in range(2, n + 1):
        dp[i] = dp[i - 1] + dp[i - 2]
    return dp[n]
```

- Та же сложность $O(n)$ по времени, но решение строится снизу вверх без рекурсии. Может быть дополнительно оптимизировано до $O(1)$ по памяти, так как каждое значение зависит только от двух предыдущих.

### Рецепт динамического программирования (DP)

Для любой задачи DP следуйте этим шагам:

1. **Определите состояние**: что представляет собой `dp[i]` (или `dp[i][j]`)? Это самый сложный шаг. Состояние должно содержать достаточно информации для принятия оптимальных решений.

2. **Запишите рекуррентное соотношение**: как `dp[i]` связано с меньшими подзадачами? Это формула перехода.

3. **Определите базовый случай**: какие подзадачи являются самыми маленькими и могут быть решены напрямую?

4. **Определите порядок итерации**: какие подзадачи должны быть решены раньше других? Снизу вверх (bottom-up): итерируйтесь в порядке, гарантирующем разрешение зависимостей. Сверху вниз (top-down): рекурсия обрабатывает это автоматически.

5. **Оптимизируйте память** (опционально): если `dp[i]` зависит только от предыдущей строки или нескольких предыдущих элементов, вам не нужна вся таблица целиком.

### Пример: процесс мышления

**Задача**: дан массив положительных целых чисел, найти максимальную сумму не соседних элементов (House Robber).

**Шаг 1 — Определите состояние**: `dp[i]` = максимальная сумма с учетом элементов `nums[0..i]`.

**Шаг 2 — Запишите рекуррентное соотношение**: для элемента $i$ мы либо:
- Пропускаем его: `dp[i] = dp[i-1]` (лучшая сумма без элемента $i$).
- Берем его: `dp[i] = dp[i-2] + nums[i]` (нужно пропустить элемент $i-1$, затем добавить элемент $i$).

Итого: `dp[i] = max(dp[i-1], dp[i-2] + nums[i])`.

**Шаг 3 — Базовые случаи**: `dp[0] = nums[0]`, `dp[1] = max(nums[0], nums[1])`.

**Шаг 4 — Порядок итерации**: слева направо (каждое состояние зависит от двух предыдущих).

**Шаг 5 — Оптимизация памяти**: нужны только два последних значения.

```python
def rob(nums):
    if len(nums) == 1:
        return nums[0]

    prev2, prev1 = nums[0], max(nums[0], nums[1])

    for i in range(2, len(nums)):
        curr = max(prev1, prev2 + nums[i])
        prev2, prev1 = prev1, curr

    return prev1
```

### Как распознать задачи DP

- Задача требует найти **оптимум** (минимальную стоимость, максимальную прибыль, самую длинную последовательность) или **количество** (число способов).
- В задаче есть **выбор на каждом шаге** (взять/пропустить, идти влево/вправо, использовать эту монету или нет), и лучший общий ответ зависит от лучших ответов на подзадачи.
- Построение дерева рекурсии выявляет **повторяющиеся подзадачи**.
- Метод грубой силы (brute force) имеет экспоненциальную сложность, но существует гораздо меньше **различных состояний**, чем рекурсивных вызовов.

### Категории DP

- **1D DP**: состояние зависит от одного индекса. Примеры: подъем по лестнице, House Robber, максимальный подмассив.

- **2D DP**: состояние зависит от двух индексов. Примеры: наибольшая общая подпоследовательность (`dp[i][j]` для первых $i$ символов строки 1 и первых $j$ символов строки 2), расстояние редактирования, задачи на пути в сетке.

- **Интервальное DP**: состояние — это диапазон `dp[i][j]`, представляющий подзадачу на `arr[i..j]`. Примеры: умножение цепочки матриц, лопающиеся шарики.

- **DP на рюкзаке (Knapsack DP)**: состояние — это индекс предмета и вместимость. Примеры: рюкзак 0/1, размен монет, сумма подмножества.

- **DP с битовой маской (Bitmask DP)**: состояние включает битовую маску, представляющую, какие элементы были использованы. Примеры: задача коммивояжера (TSP), задача о назначениях. Пространство состояний составляет $O(2^n \cdot n)$, что приемлемо для $n \leq 20$.

### Сверху вниз vs Снизу вверх

| | Сверху вниз (Мемоизация) | Снизу вверх (Табуляция) |
|--|---|---|
| Реализация | Рекурсия + кеш | Итерация + таблица |
| Вычисления | Только те подзадачи, которые действительно нужны | Все подзадачи до целевой |
| Риск переполнения стека | Да (глубокая рекурсия) | Нет |
| Оптимизация памяти | Сложнее | Проще (использование скользящего массива) |
| Простота написания | Часто естественнее (написать рекурсию, добавить кеш) | Требует продумывания порядка итерации |

- На собеседованиях код сверху вниз обычно пишется быстрее. В продакшене подход снизу вверх обычно предпочтительнее из-за производительности (нет накладных расходов на рекурсию, лучше работа с кешем).

### Типичные ошибки

| Ошибка | Пример | Исправление |
|---------|---------|-----|
| Неверное определение состояния | `dp[i]` не содержит достаточно информации для принятия решений | Добавьте размерности (например, `dp[i][j]` вместо `dp[i]`) |
| Отсутствие базового случая | `dp[0]` неверно → все последующие значения неверны | Проверьте базовый случай вручную |
| Неверный порядок итерации | Вычисление `dp[i]` до того, как вычислены его зависимости | Нарисуйте стрелки зависимостей и итерируйтесь в соответствии с ними |
| Неверная инициализация `dp` | Использование 0 там, где должна быть бесконечность (для задач на минимум) | `float('inf')` для минимизации, `float('-inf')` для максимизации |
| Забыли рассмотреть вариант "пропустить" | Всегда берется текущий элемент | Рекуррентное соотношение обычно содержит `max(take, skip)` |
| Изменяемый аргумент по умолчанию | `def f(memo={})` использует общий кеш для всех вызовов | `def f(memo=None): if memo is None: memo = {}` |
| Ошибка на единицу в 2D DP | Индексация `text1[i]`, когда `dp` имеет индексацию с 1 | `dp` имеет размер `(m+1) x (n+1)`, обращайтесь к `text1[i-1]` |

## Итоги

- Эти четыре концепции образуют последовательность:
    1. **Big O** говорит вам, достаточно ли быстр подход.
    2. **Рекурсия** разбивает задачи на подзадачи.
    3. **Поиск с возвратом (Backtracking)** — это рекурсия + выбор + отмена действия для исчерпывающего поиска.
    4. **DP** — это рекурсия + кеширование для оптимизации перекрывающихся подзадач.

- Когда вы видите новую задачу:
    - Оцените размер входных данных $n$. Какая сложность Big O приемлема?
    - Если метод грубой силы имеет экспоненциальную сложность, а задача требует перечисления/поиска конфигураций: **backtracking** (с прунингом для практической применимости).
    - Если метод грубой силы имеет экспоненциальную сложность, а задача требует поиска оптимума или количества, и вы видите перекрывающиеся подзадачи: **DP**.
    - Если задача имеет структуру, которая делит пространство поиска пополам: **бинарный поиск** или **разделяй и властвуй**.
    - Если задача связана с последовательностями с ограничением на подмассивы: **скользящее окно** или **два указателя**.
    - Если задаче нужен быстрый поиск: **хеш-мапа**.
