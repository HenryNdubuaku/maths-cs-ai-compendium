# Scaling 与 Deployment

*把大模型 serving 给数百万用户，需要把推理分布到多 GPU、在需要前就预测 token、缓存共享上下文，并选择合适的框架。本文件涵盖推理时的并行、speculative decoding、前缀缓存、推理框架、成本优化与监控*

- 单张 H100 GPU 以交互式 latency serving 70B 模型约可处理 100 并发用户。要服务 1000 万用户需要 100,000 张 GPU —— 云计算成本约每年 30 亿美元。每提高一个百分点的效率都省下数千万美元。这也是为什么推理优化不是纸上谈兵：它直接决定 AI 产品的经济性。

## 推理时的模型并行

- 当模型单卡放不下时，必须切分到多张 GPU。训练时的并行策略（第 6 章）在推理时同样适用，但权衡不同。

### Tensor Parallelism

- **Tensor parallelism**（Megatron 风格，第 6 章）把单个权重矩阵切分到多 GPU。对线性层 $Y = XW$，权重矩阵 $W$ 按列切分到 $N$ 张 GPU。每张 GPU 计算部分结果，然后 all-reduce 聚合：

$$W = [W_1 | W_2 | \cdots | W_N], \quad Y_i = X W_i, \quad Y = \text{concat}(Y_1, \ldots, Y_N)$$

- 推理时，对单卡放不下的模型，tensor parallelism 是默认选择。70B 模型 FP16 需 140 GB —— 用 tensor parallelism 切到 2 张 80 GB GPU 上。

- **Latency 影响**：tensor parallelism 每层多一次 all-reduce 通信。在 NVLink（900 GB/s）上每层约增加 0.1 ms；在 PCIe（32 GB/s）上每层约增加 3 ms。70B 模型 80 层、2 张 GPU 时：NVLink 总共约增加 8 ms，PCIe 约 240 ms。这也是为什么 NVLink 对多 GPU 推理至关重要。

### Pipeline Parallelism

- **Pipeline parallelism** 把不同层分给不同 GPU。GPU 1 处理 0-39 层，GPU 2 处理 40-79 层。token 顺序流过流水线。

- 推理时，pipeline parallelism 的 latency 比 tensor parallelism 更高（每个 token 都要遍历整条流水线），但通信开销更低（GPU 之间只传激活，无 all-reduce）。当 GPU 之间用慢速互联（不同节点、无 NVLink）时，它更受青睐。

### Sequence Parallelism

- 对超长序列，即便模型能放下，KV-cache 本身也可能单卡放不下。**Sequence parallelism** 把 KV-cache 分片到多 GPU：每张 GPU 存储序列缓存 key 和 value 的一部分。

- attention 时，每张 GPU 对自己缓存的片段计算部分 attention 分数，再通过 reduction 合并结果。用于长上下文推理（128K+ token），此时 KV-cache 超出单卡显存。

## Speculative Decoding

- **Speculative decoding** 是最具影响力的 LLM 推理优化之一。洞察：decode 慢是因为它一次只生成一个 token，而每个 token 都要对大模型做一次完整前向传播。但小模型可以快得多地生成候选 token，而大模型可以并行**验证**多个候选。

![Speculative decoding：快速的 draft model 生成 5 个候选 token，target model 一次验证全部，接受的 token 保留、拒绝的重新采样](../images/speculative_decoding.svg)

- **算法**：
    1. 一个 **draft model**（小而快，如 1B 参数）自回归地生成 $k$ 个候选 token。
    2. **target model**（大而准，如 70B）对整个 draft 序列做一次前向传播，计算每个候选 token 的概率。
    3. 若 target model 认可（其对该 token 的概率足够高），则候选被**接受**。被拒的候选从 target model 的分布重新采样。
    4. 平均而言，每次验证步接受多个 token，加速比与接受率成正比。

$$\text{Speedup} \approx \frac{k \times \text{acceptance\_rate}}{\text{cost\_ratio}} \approx 2\text{-}3\times$$

- **为何无质量损失**：拒绝采样方案保证输出分布与 target model 完全一致。Speculative decoding 是无损的 —— 输出在统计上与单独跑 target model 完全相同，只是更快。

- **变体**：
    - **Medusa**（Cai et al., 2024）：不用单独的 draft model，而是在 target model 上加多个轻量 "head" 同时预测多个未来 token。无需单独模型。
    - **EAGLE**（Li et al., 2024）：训练一个轻量 draft head，用 target model 的 hidden state 预测未来 token。接受率比独立 draft model 更高。
    - **自 speculative decoding**：target model 自身用提前退出产生 draft（只跑前几层做 draft，再用完整模型验证）。
    - **并行解码**：并行生成多个续写（候选树）并一次性验证整棵树。throughput 更高，但分支 KV-cache 占用更多显存。

## 前缀缓存

- 许多请求共享公共前缀：系统 prompt、few-shot 示例或常见查询模式。**前缀缓存** 为这些前缀存储 KV-cache 并跨请求复用。

- **系统 prompt 缓存**：若每个请求都以相同的 2000 token 系统 prompt 开头，那 2000 token 的 KV-cache 只算一次就跨所有请求共享。对 70B、80 层模型，每请求省下约 200 MB。

- **基数树缓存**（SGLang）：把缓存的前缀组织成基数树（trie）。新请求到来时，找最长缓存前缀匹配并从那里开始生成，跳过匹配前缀的计算。

- **影响**：对有长共享前缀的应用（带系统 prompt 的聊天机器人、带常见检索段落的 RAG），前缀缓存把 TTFT 降低 50-90%，并节省相应比例的 GPU 算力。

## KV-Cache Eviction

- 除了量化 KV-cache（文件 01）和用 GQA/MLA 缩减其大小（文件 02），**KV-cache eviction** 策略有选择地移除未来不太可能被 attend 的缓存 token。

- **H2O**（Heavy-Hitter Oracle，Zhang et al., 2023）观察到 attention 分数服从幂律：少数 token（"重击者"）获得大部分 attention，大多数 token 获得可忽略的 attention。H2O 保留：

    1. **近期 token**（最近 $w$ 个 token 的滑动窗口，类似 StreamingLLM）。
    2. **重击 token**（按所有过去 decode 步累计 attention 分数取 top-$k$）。

- 既非近期又非重击的 token 被驱逐。这维持固定大小的 KV-cache，同时保留真正影响生成的 token。H2O 仅用 20% 的显存即可达到接近完整 KV-cache 的质量。

- **Scissorhands**（Liu et al., 2023）思路类似，但用更精细的重要性度量：保留**当前**步获得高 attention 的 token，驱逐已 $T$ 步未被 attend 的 token。这能适应生成过程中 attention 模式的变化。

- **动态驱逐 + StreamingLLM**：把 attention sink（永久保留前几个 token）与动态驱逐（保留近期 + 重击 token）结合。这是超长生成下最省显存的方法，可在有界质量损失下实现无限长度生成。

- 所有 eviction 方法的共同关键洞察：LLM 的 attention 在**实践中是稀疏的** —— 尽管架构对所有缓存 token 计算 attention，实际 attention 权重集中在小子集上。驱逐其余的对输出质量影响极小。

## 推理框架

- LLM serving 生态已收敛到几个主流框架：

| 框架 | 优势 | 最适合 |
|-----------|-----------|----------|
| **vLLM** | PagedAttention、continuous batching、高 throughput | 通用 LLM serving、最高 throughput |
| **TensorRT-LLM** | NVIDIA 优化 kernel、FP8、in-flight batching | 在 NVIDIA GPU 上极致性能 |
| **SGLang** | 前缀缓存（RadixAttention）、快速结构化生成 | 共享前缀、约束输出的应用 |
| **llama.cpp** | CPU/Metal/CUDA/Vulkan、GGUF 量化、可移植 | 消费级硬件、设备端推理 |
| **TGI**（HuggingFace） | API 简单、易部署、model hub 集成 | 快速部署、HuggingFace 生态 |
| **Ollama** | 一条命令下载并 serving 模型 | 个人使用、本地开发 |
| **ExLlamaV2** | 极限量优化（EXL2 格式） | 显存受限的 GPU 推理 |

- **vLLM** 是生产 LLM serving 的默认选择。它支持 continuous batching、PagedAttention、tensor parallelism、speculative decoding、LoRA serving 以及大多数开源模型。

- **TensorRT-LLM** 在 NVIDIA 硬件上达到最高原始性能（同 GPU 上比 vLLM 快 10-30%），但灵活性较低、定制较难。

- **SGLang** 在应用有结构化输出（JSON、特定格式代码）或共享前缀时表现出色，这得益于其基数 attention 缓存和约束解码引擎。

## 成本优化

- 在规模上，推理成本主导 ML 预算。降低成本的策略：

- **合理选卡**：并非每个模型都需要 H100。量化的 7B 模型在 A10G（约 $1/hr）上跑得很好，而非 H100（约 $8/hr）。让 GPU 匹配工作负载。

- **Spot 实例**：云厂商以 60-90% 折扣提供闲置 GPU 算力（AWS Spot、GCP Preemptible）。Spot 实例可能被中断，适合批处理推理而非 latency 敏感的 serving。配合抢占处理（保存状态、在新实例上恢复），spot 实例也能服务交互流量。

- **Autoscaling**：按流量扩缩 GPU 数量。高峰期扩容，夜间缩容。Kubernetes HPA（Horizontal Pod Autoscaler）或云原生 autoscaling（AWS SageMaker、GCP Vertex AI）负责处理。

- **Batching + 利用率**：30% 与 90% 的 GPU 利用率之间，每 token 成本相差 3 倍。Continuous batching、智能调度和 PagedAttention 都提升利用率。

- **量化**：INT4 vs FP16 少 4 倍显存 → 可放在更小的 GPU → 成本低 2-4 倍。此外，更多请求能塞进同一 batch → throughput 更高 → 每 token 成本更低。

- **每 token 成本基准**（约 2026 年）：

| 配置 | 每 1M token 成本 |
|-------|-------------------|
| GPT-4o API | $2.50 |
| Claude 3.5 Sonnet API | $3.00 |
| Llama-70B on H100 (vLLM, FP16) | $0.50 |
| Llama-70B on H100 (TRT-LLM, INT8) | $0.25 |
| Llama-8B on A10G (vLLM, INT4) | $0.05 |
| Llama-3B on-device (llama.cpp) | $0（硬件摊销） |

## 监控

- 生产推理需要持续监控，在用户受影响前抓住退化：

- **Latency 监控**：跟踪 p50、p95、p99 的 TTFT 和 TPOT。为 p99 超 SLO 设告警。p99 飙升常意味着：KV-cache 显存压力（thrashing）、长请求独占 batch、或 GPU 热降频。

- **Throughput 监控**：跟踪每 GPU 每秒 token 数。下降意味着：batch 效率下降（许多短请求 → batch 利用率低）、序列长度增加（每请求 KV-cache 显存更多）、或硬件问题（GPU 进入 ECC 纠错模式，变慢）。

- **GPU 利用率**：跟踪 SM 占用率、显存利用率、显存带宽。低 SM 占用 + 高显存利用 = 内存受限（需更多带宽或量化）。高 SM 占用 + 低显存利用 = 计算受限（需更多 FLOPS 或更小模型）。

- **模型质量监控**：跟踪逐请求指标（响应长度、held-out 集困惑度、用户反馈信号）。模型质量可能因以下原因退化：数据漂移（入站请求分布变化）、KV-cache 量化误差在长对话中累积、或 serving 流水线的 bug。

- **成本监控**：跟踪每模型每 GPU 类型的每 token 成本。若成本上升而 throughput 未升，调查效率回归（新版本模型显存占用更高、batch 配置不佳、或 GPU 未充分利用）。

- **工具**：Prometheus + Grafana（第 15 章）用于基础设施指标，vLLM/TRT-LLM 内置 metrics 端点，自定义日志用于模型级指标。

## 编程练习（使用 CoLab 或 notebook）

1. 模拟 speculative decoding。用一个快 "draft" 函数和一个慢 "target" 函数，测量一次生成并验证多个 token 的加速。
```python
import random
import time

def target_model(tokens):
    """Slow but accurate model. Returns probability of each candidate token."""
    time.sleep(0.01)  # simulate 10ms per forward pass
    # For simulation: accept tokens that are even numbers
    return [0.9 if t % 2 == 0 else 0.1 for t in tokens]

def draft_model():
    """Fast but approximate model. Generates one candidate token."""
    time.sleep(0.001)  # simulate 1ms per token
    return random.randint(0, 9)

def standard_decoding(n_tokens):
    """Generate one token at a time with the target model."""
    tokens = []
    for _ in range(n_tokens):
        time.sleep(0.01)  # target model generates 1 token
        tokens.append(random.randint(0, 9))
    return tokens

def speculative_decoding(n_tokens, k=4):
    """Generate k draft tokens, verify with target, accept/reject."""
    tokens = []
    total_target_calls = 0

    while len(tokens) < n_tokens:
        # Draft: generate k candidates quickly
        candidates = [draft_model() for _ in range(k)]

        # Verify: one target model call for all k candidates
        probs = target_model(candidates)
        total_target_calls += 1

        # Accept tokens until one is rejected
        for i, (tok, prob) in enumerate(zip(candidates, probs)):
            if random.random() < prob:
                tokens.append(tok)
                if len(tokens) >= n_tokens:
                    break
            else:
                # Resample from target distribution
                tokens.append(tok + 1)  # simplified resampling
                break

    return tokens, total_target_calls

n = 50

start = time.time()
_ = standard_decoding(n)
standard_time = time.time() - start

start = time.time()
_, target_calls = speculative_decoding(n, k=5)
spec_time = time.time() - start

print(f"Standard:    {standard_time:.2f}s ({n} target calls)")
print(f"Speculative: {spec_time:.2f}s ({target_calls} target calls)")
print(f"Speedup:     {standard_time / spec_time:.1f}x")
```

2. 估计不同优化策略给 LLM serving 部署带来的成本节省。
```python
def serving_cost_analysis(
    model_name, params_B, precision_bits,
    gpu_name, gpu_mem_gb, gpu_cost_per_hr,
    target_throughput_tps,
):
    """Estimate serving cost for an LLM deployment."""
    model_size_gb = params_B * 1e9 * precision_bits / 8 / 1e9
    gpus_for_model = max(1, int((model_size_gb * 1.2) / gpu_mem_gb + 0.99))  # 1.2x for KV-cache

    # Rough throughput estimate (memory-bandwidth limited)
    tokens_per_gpu = 500 / (params_B * precision_bits / 16)  # normalised to 500 tok/s for 7B FP16
    total_throughput = tokens_per_gpu * gpus_for_model

    replicas = max(1, int(target_throughput_tps / total_throughput + 0.99))
    total_gpus = gpus_for_model * replicas
    cost_per_hr = total_gpus * gpu_cost_per_hr
    cost_per_1M_tokens = cost_per_hr / (total_throughput * replicas * 3600 / 1e6)

    print(f"{model_name} @ {precision_bits}-bit on {gpu_name}:")
    print(f"  Model size: {model_size_gb:.0f} GB → {gpus_for_model} GPU(s)/replica")
    print(f"  Throughput: {total_throughput:.0f} tok/s/replica")
    print(f"  Replicas for {target_throughput_tps} tok/s: {replicas}")
    print(f"  Total GPUs: {total_gpus}")
    print(f"  Cost: ${cost_per_hr:.0f}/hr, ${cost_per_1M_tokens:.2f}/1M tokens")
    print()

print("=== Cost Comparison ===\n")

# Baseline: FP16 on H100
serving_cost_analysis("Llama-70B", 70, 16, "H100", 80, 8.0, 1000)

# Quantised: INT8 on H100
serving_cost_analysis("Llama-70B", 70, 8, "H100", 80, 8.0, 1000)

# Quantised: INT4 on A100
serving_cost_analysis("Llama-70B", 70, 4, "A100", 80, 4.0, 1000)

# Smaller model: 8B on A10G
serving_cost_analysis("Llama-8B", 8, 4, "A10G", 24, 1.0, 1000)
```
