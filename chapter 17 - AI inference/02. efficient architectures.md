## StreamingLLM: генерация бесконечной длины

- Стандартные трансформеры хранят все предыдущие токены в KV-кеше, который растет линейно с длиной последовательности. В какой-то момент кеш превышает объем памяти GPU, и генерация завершается с ошибкой. **StreamingLLM** (Xiao et al., 2023) решает эту проблему с помощью **rolling KV-cache** (скользящего KV-кеша) фиксированного размера.

- Ключевое наблюдение: первые несколько токенов в последовательности получают непропорционально высокие оценки внимания независимо от их содержания. Они называются **attention sinks** (стоками внимания). Если удалить их из кеша, распределение внимания разрушается, а качество генерации катастрофически падает.

- Решение StreamingLLM: постоянно хранить в кеше небольшое количество **sink tokens** (первых 1–4 токенов), а также **rolling window** (скользящее окно) из последних $w$ токенов. Общий размер кеша составляет $\text{sink} + w$, что остается неизменным независимо от того, сколько токенов было сгенерировано.

$$\text{Cache} = [\text{token}_0, \text{token}_1, \text{token}_{t-w+1}, \ldots, \text{token}_t]$$

- Стоки внимания закрепляют распределение softmax, а скользящее окно обеспечивает недавний контекст. Это позволяет осуществлять **генерацию бесконечной длины** с постоянным объемом памяти ценой потери доступа к контексту в середине последовательности.

- StreamingLLM работает без какого-либо дообучения для моделей, которые естественным образом формируют стоки внимания (большинство предобученных LLM делают это). Для моделей, у которых этого не происходит, добавление одного обучаемого sink-токена во время обучения решает проблему.

## Разреженное внимание

- Полное самовнимание имеет сложность $O(n^2)$ относительно длины последовательности $n$, так как каждый токен учитывает все остальные токены. Для $n = 128K$ матрица внимания содержит $128K^2 = 16$ миллиардов элементов. Паттерны **разреженного внимания** (sparse attention) уменьшают это число, ограничивая токены, которые могут взаимодействовать друг с другом.

![Attention sparsity patterns: full attention is O(n²), sliding window is O(n·w), local+global adds long-range tokens](../images/attention_sparsity_patterns.svg)

- **Sliding window attention** (скользящее окно внимания) (Mistral, Gemma): каждый токен учитывает только предыдущие $w$ токенов (например, $w = 4096$). Внимание имеет сложность $O(n \cdot w)$ вместо $O(n^2)$. Информация распространяется за пределы окна через несколько слоев: после $L$ слоев эффективный контекст составляет $L \times w$.

- **Local + global attention** (локальное + глобальное внимание) (Longformer, BigBird): большинство токенов используют внимание со скользящим окном (локальное), но несколько выделенных токенов (например, [CLS], каждый 512-й токен) учитывают все токены (глобальное). Это позволяет захватывать как локальные паттерны, так и долгосрочные зависимости.

- **Dilated attention** (расширенное внимание): учет каждого $k$-го токена внутри окна, что создает разреженный паттерн, охватывающий больший диапазон при том же количестве оценок внимания. Увеличение расширения (dilation) по слоям создает иерархический паттерн, похожий на расширенные свёртки (глава 8).

- Практическим победителем для современных LLM является **чередование скользящего окна и полного внимания**: некоторые слои используют скользящее окно (дешево, обрабатывает локальный контекст), некоторые слои используют полное внимание (дорого, захватывает долгосрочные зависимости). Mistral/Mixtral используют этот паттерн.

## Линейное внимание и модели пространства состояний

- Можно ли полностью заменить внимание $O(n^2)$? **Линейное внимание** и **модели пространства состояний (SSM)** обрабатывают последовательности за время $O(n)$, избегая явного вычисления матрицы внимания.

- **Линейное внимание** заменяет softmax-внимание аппроксимацией ядра:

$$\text{Standard: } O = \text{softmax}(QK^T / \sqrt{d}) V$$
$$\text{Linear: } O = \phi(Q) (\phi(K)^T V)$$

- Благодаря ассоциативности произведения $K^T V$ (которое имеет размер $d \times d$ и не зависит от длины последовательности), вычисления становятся $O(n \cdot d^2)$ вместо $O(n^2 \cdot d)$. Для длинных последовательностей, где $n \gg d$, это дает колоссальную экономию.

- **RWKV** объединяет идеи из RNN и трансформеров. Он использует рекуррентную формулировку, которая обрабатывает токены последовательно (как RNN), но может быть распараллелена во время обучения (как трансформер). Инференс составляет $O(1)$ на токен (постоянная память, отсутствие роста KV-кеша).

- **Mamba** (Gu & Dao, 2023) — это селективная модель пространства состояний. Она обрабатывает последовательности через обучаемые переходы состояний:

$$h_t = \bar{A} h_{t-1} + \bar{B} x_t, \quad y_t = C h_t$$

- где $\bar{A}$ и $\bar{B}$ зависят от входных данных (селективные), что позволяет Mamba динамически фокусироваться на частях входных данных или игнорировать их. В отличие от фиксированных SSM, селективность делает Mamba конкурентоспособной по сравнению с трансформерами в языковых задачах, сохраняя при этом масштабируемость $O(n)$.

- **Компромисс**: линейное внимание и SSM быстрее для длинных последовательностей, но, как правило, менее эффективны, чем полное внимание, в задачах, требующих точного извлечения информации на больших расстояниях. Гибридные архитектуры (некоторые слои трансформера + некоторые слои Mamba) часто дают лучшее из обоих миров.

## Multi-Query и Grouped-Query внимание

- Стандартное многоголовое внимание (MHA, глава 7) использует отдельные проекции $K, V$ для каждой головы. Для $h$ голов это означает $h$ отдельных тензоров ключей и значений в KV-кеше. **Multi-Query Attention (MQA)** и **Grouped-Query Attention (GQA)** уменьшают этот объем.

- **MQA** (Shazeer, 2019): все головы разделяют один набор проекций $K, V$. Каждая голова по-прежнему имеет свою собственную проекцию $Q$. KV-кеш уменьшается в $h$ раз (например, в 32 раза для 32 голов).

- **GQA** (Ainslie et al., 2023): золотая середина. Головы группируются, и каждая группа разделяет один набор проекций $K, V$. При $h = 32$ головах и $g = 8$ группах каждая группа из 4 голов разделяет K/V. KV-кеш уменьшается в $h/g = 4$ раза.

$$\text{MHA: } h \text{ heads, } h \text{ K/V sets} \quad \to \quad \text{GQA: } h \text{ heads, } g \text{ K/V sets} \quad \to \quad \text{MQA: } h \text{ heads, } 1 \text{ K/V set}$$

![MHA против GQA против MQA: MHA предоставляет каждой «голове» собственный KV, GQA распределяет KV между группами, MQA использует один KV для всех «голов» — это значительно уменьшает размер KV-кеша](../images/mha_gqa_mqa.svg)

- Большинство современных LLM используют GQA (Llama 2/3, Gemma, Mistral). Это позволяет уменьшить объем памяти для KV-кеша и задержку инференса при пренебрежимо малой потере качества по сравнению с MHA.

### Multi-head Latent Attention (MLA)

- **MLA** (DeepSeek-V2, 2024) идет дальше, чем GQA, сжимая KV-кеш в **низкоранговое латентное пространство**. Вместо кеширования полных векторов ключей (key) и значений (value), MLA кеширует сжатый латентный вектор $\mathbf{c}_t$ для каждого токена и восстанавливает K/V «на лету» во время работы механизма внимания:

$$\mathbf{c}_t = W_{\text{compress}} \cdot [\mathbf{k}_t; \mathbf{v}_t], \quad \mathbf{k}_t = W_K^{\text{up}} \cdot \mathbf{c}_t, \quad \mathbf{v}_t = W_V^{\text{up}} \cdot \mathbf{c}_t$$

- Сжатый вектор $\mathbf{c}_t$ намного меньше, чем исходные K и V вместе взятые. DeepSeek-V2 достигает **93,3% сокращения** размера KV-кеша по сравнению с MHA, превосходя даже MQA, при этом сохраняя качество уровня MHA.

- Компромисс: восстановление K/V из латентного представления добавляет небольшие вычислительные затраты на каждую операцию внимания. Но поскольку декодирование в LLM ограничено пропускной способностью памяти (а не вычислительной мощностью), это дает чистый выигрыш: меньше памяти для загрузки > чуть больше вычислений на токен.

### Flash Attention

- **Flash Attention** (Dao et al., 2022, подробно рассматривается в главе 16, файле 05) — это не изменение архитектуры, а оптимизация реализации, которую стоит упомянуть в любом обсуждении эффективного внимания. Она вычисляет точное стандартное внимание с помощью:

    - **O(n) памяти** вместо O(n²) (матрица внимания никогда не материализуется в HBM).
    - **2–4-кратного ускорения** по сравнению со стандартным вниманием (за счет удержания данных в SRAM с помощью тайлинга и онлайн-softmax).
    - **Отсутствия потери качества** — выходные данные математически идентичны стандартному вниманию.

- Flash Attention теперь является реализацией внимания по умолчанию в PyTorch (`torch.nn.functional.scaled_dot_product_attention`), JAX и всех основных фреймворках для инференса. Если вы используете внимание в 2024 году и позже, вы почти наверняка используете Flash Attention.

### Ring Attention

- **Ring Attention** (Liu et al., 2023) распределяет вычисления внимания между несколькими устройствами для последовательностей, которые слишком длинны, чтобы поместиться в память одного GPU, даже с использованием Flash Attention.

- Идея: разбить последовательность между $N$ устройствами. Каждое устройство хранит Q, K, V для $n/N$ токенов. Устройства расположены в кольцо. На каждом шаге:
    1. Каждое устройство вычисляет локальное внимание (свои Q против своих локальных K/V).
    2. Каждое устройство отправляет свой блок K/V следующему устройству в кольце.
    3. Каждое устройство получает K/V от предыдущего устройства и вычисляет внимание по отношению к ним.
    4. После $N$ шагов каждое устройство обработало каждый блок K/V.

- Передача данных **совмещается** с вычислениями: пока вычисляется внимание для текущего блока K/V, следующий блок уже передается. Это практически полностью скрывает задержку передачи данных.

- Ring Attention позволяет создавать **контекстные окна на миллион токенов**, распределяя KV-кеш по кольцу GPU. Память на устройство составляет O(n/N), что делает возможными произвольно длинные последовательности (ограниченные только количеством устройств).

## Mixture of Experts при инференсе

- Модели MoE (глава 7) активируют лишь часть своих параметров для каждого токена (обычно 2 из 8 экспертов). При инференсе уникальной задачей является **кеширование экспертов**: все эксперты должны находиться в памяти (потому что любой токен может быть направлен к любому эксперту), но активны только 2 из них на токен.

- Для модели Mixtral 8x7B: общее количество параметров = 47B (8 экспертов по 7B, но с общими компонентами). Активных параметров на токен ≈ 13B (2 эксперта + общие слои). Модель обладает качеством уровня LLM-70B при стоимости инференса уровня LLM-13B, но требует 47B параметров в памяти.

- **Выгрузка экспертов (expert offloading)**: для развертывания в условиях ограниченной памяти GPU неактивные эксперты хранятся в CPU или на SSD и загружаются по требованию. Это работает, потому что маршрутизация токенов достаточно предсказуема, чтобы заранее подгружать нужных экспертов.

- **Кеширование экспертов**: поддержание LRU-кеша недавно использованных экспертов в памяти GPU. Если одни и те же эксперты активируются повторно (что часто бывает для данных внутри домена), частота попаданий в кеш будет высокой.

## Дистилляция знаний

- **Дистилляция** (глава 6) обучает небольшую «студенческую» модель имитировать большую «учительскую». Студент учится на мягких предсказаниях учителя (распределениях вероятностей по классам), которые содержат больше информации, чем просто жесткие метки.

$$\mathcal{L} = \alpha \cdot \text{KL}(p_{\text{teacher}}^{T} \| p_{\text{student}}^{T}) + (1 - \alpha) \cdot \mathcal{L}_{\text{CE}}(y, p_{\text{student}})$$

- где $T$ — температура (более высокое значение $T$ сглаживает распределения, выявляя неуверенность учителя), а $\alpha$ балансирует функцию потерь дистилляции с обычной функцией потерь кросс-энтропии.

- **Для LLM**: дистилляция используется для создания небольших и быстрых моделей на основе больших и мощных. GPT-4 → студент 7B, который перенимает большую часть поведения GPT-4 для конкретной задачи. Обслуживание студента может быть в 10–100 раз дешевле.

- **Дистилляция под конкретную задачу**: дистилляция только на данных, относящихся к вашей задаче развертывания. Модель 7B, дистиллированная на медицинских вопросах и ответах от учителя 70B, может превзойти модель 70B в этой конкретной задаче (потому что ограниченная емкость студента полностью сфокусирована на целевом домене).

## Прунинг

- **Прунинг** удаляет ненужные веса (обнуляя их), уменьшая размер модели и объем вычислений.

- **Неструктурированный прунинг** (на основе величины): удаление отдельных весов с наименьшими абсолютными значениями. Это создает разреженную матрицу весов. Метод прост и эффективен для сжатия, но современное оборудование (GPU) не может эффективно ускорять разреженные операции, если разреженность не следует определенным паттернам.

- **Структурированный прунинг**: удаление целых единиц — «голов» внимания, нейронов MLP или слоев. Это позволяет получить меньшую плотную модель, которую легко ускорить на стандартном оборудовании. Компромисс заключается в более грубой гранулярности (удаление целой «головы» может привести к удалению как полезных, так и бесполезных весов одновременно).

- **Разреженность 2:4** (NVIDIA Ampere+): аппаратно поддерживаемый паттерн разреженности, при котором 2 из каждых 4 весов равны нулю. Разреженные тензорные ядра (Tensor Cores) GPU пропускают умножения на ноль, достигая ускорения примерно в 2 раза. На сегодняшний день это единственный паттерн разреженности с практическим аппаратным ускорением.

- **Гипотеза лотерейного билета** (Frankle & Carlin, 2019): внутри случайно инициализированной нейронной сети существует подсеть («выигрышный билет»), которую можно обучить изолированно, чтобы она достигла производительности полной сети. Поиск таких подсетей (путем обучения, прунинга и «отмотки» весов) является затратным, но эта идея стимулирует исследования в области прунинга.

## Поиск нейронных архитектур (NAS)

- **NAS** автоматизирует проектирование архитектур, выполняя поиск в пространстве возможных архитектур для нахождения той, которая максимизирует точность с учетом аппаратных ограничений (задержка, память, энергопотребление).

- **EfficientNet** (глава 8) была найдена с помощью NAS: правило составного масштабирования (баланс глубины, ширины и разрешения) возникло в результате поиска, а не благодаря человеческой интуиции.

- Для эффективности инференса NAS может находить архитектуры, оптимизированные под конкретные аппаратные цели: «найти модель с задержкой <5 мс на iPhone Neural Engine и точностью >80% на ImageNet». Пространство поиска включает типы слоев, ширину, функции активации и паттерны внимания.

- **Сети «один для всех» (Once-for-all networks)** обучают одну перепараметризованную сеть и извлекают из нее подсети для различных целей развертывания. Один цикл обучения позволяет получить модели для облачных GPU, мобильных GPU и CPU, каждая из которых оптимизирована под свою целевую платформу.

## Задачи по программированию (используйте CoLab или ноутбук)

1. Реализуйте внимание со скользящим окном и сравните использование памяти с полным вниманием.
```python
import jax
import jax.numpy as jnp

def full_attention(Q, K, V):
    """Standard O(n^2) attention."""
    scores = Q @ K.T / jnp.sqrt(Q.shape[-1])
    weights = jax.nn.softmax(scores, axis=-1)
    return weights @ V

def sliding_window_attention(Q, K, V, window_size=128):
    """Sliding window attention: each token attends to window_size previous tokens."""
    n = Q.shape[0]
    d = Q.shape[-1]
    output = jnp.zeros_like(Q)

    for i in range(n):
        start = max(0, i - window_size + 1)
        k_window = K[start:i+1]
        v_window = V[start:i+1]
        scores = Q[i] @ k_window.T / jnp.sqrt(d)
        weights = jax.nn.softmax(scores)
        output = output.at[i].set(weights @ v_window)

    return output

n, d = 512, 64
key = jax.random.PRNGKey(0)
Q = jax.random.normal(key, (n, d))
K = jax.random.normal(jax.random.PRNGKey(1), (n, d))
V = jax.random.normal(jax.random.PRNGKey(2), (n, d))

print(f"Full attention memory:    O(n^2) = {n*n} entries")
print(f"Window (w=128) memory:   O(n*w) = {n*128} entries")
print(f"Reduction: {n*n / (n*128):.1f}x")
```

3. Смоделируйте структурный прунинг, удалив наименее важные головы внимания из случайного слоя внимания и измерив изменение выходных данных.
```python
import jax
import jax.numpy as jnp

key = jax.random.PRNGKey(0)
n_heads, seq_len, d_head = 8, 64, 32

# Random multi-head attention output (one per head)
head_outputs = jax.random.normal(key, (n_heads, seq_len, d_head))

# Full output: concatenate all heads
full_output = head_outputs.reshape(seq_len, n_heads * d_head)

# Importance: measure each head's contribution by its norm
head_norms = jnp.linalg.norm(head_outputs, axis=(1, 2))
print("Head importance (by norm):", jnp.round(head_norms, 2))

# Prune least important heads
for n_keep in [8, 6, 4, 2]:
    top_heads = jnp.argsort(head_norms)[-n_keep:]
    pruned = head_outputs[top_heads].reshape(seq_len, n_keep * d_head)

    # Pad to original size for comparison (zero out pruned heads)
    full_pruned = jnp.zeros_like(head_outputs)
    full_pruned = full_pruned.at[top_heads].set(head_outputs[top_heads])
    full_pruned = full_pruned.reshape(seq_len, n_heads * d_head)

    error = jnp.linalg.norm(full_output - full_pruned) / jnp.linalg.norm(full_output)
    print(f"Keep {n_keep}/{n_heads} heads: relative error = {error:.4f}, "
          f"memory = {n_keep/n_heads:.0%}")
```
