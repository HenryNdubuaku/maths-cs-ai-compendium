# Инференс и батчинг

*Для предоставления доступа к LLM тысячам одновременных пользователей недостаточно просто загрузить модель и запустить инференс. В этом файле рассматриваются разделение на префилл и декодирование, непрерывный батчинг, PagedAttention и vLLM, стратегии планирования, дезагрегированный инференс, обслуживание нескольких моделей и LoRA, а также важные метрики.*

- Один запрос к LLM прост: подать токены на вход, сгенерировать выходные токены. Но обслуживание LLM для 10 000 одновременных пользователей с низкой задержкой и высокой пропускной способностью — это системная задача. Наивный подход (обработка по одному запросу за раз) тратит впустую более 90% ресурсов GPU. Умный батчинг и планирование могут увеличить пропускную способность в 10–50 раз без добавления оборудования.

## Префилл и декодирование: две принципиально разные фазы

- Инференс LLM состоит из двух отдельных фаз с фундаментально разными вычислительными характеристиками:

- **Префилл** (обработка промпта): обработка всех входных токенов одновременно. Это одно большое матричное умножение: $O(\text{prompt\_length} \times d_{\text{model}}^2)$. Промпт можно обрабатывать параллельно (все токены известны). Префилл **ограничен вычислительной мощностью** (compute-bound): узким местом являются ALU графического процессора.

- **Декодирование** (генерация токенов): генерация по одному токену за раз, авторегрессионно. Каждый новый токен требует применения механизма внимания ко всем предыдущим токенам через KV-кеш. Декодирование **ограничено пропускной способностью памяти** (memory-bandwidth-bound): GPU тратит большую часть времени на загрузку весов модели и KV-кеша из памяти, а не на вычисления. Каждый шаг декодирования создает только один токен, но требует загрузки всей модели (~140 ГБ для модели 70B в формате FP16).

- Выводы:

| | Префилл | Декодирование |
|--|---------|--------|
| Обработка токенов | Все сразу (параллельно) | По одному (последовательно) |
| Узкое место | Вычисления (FLOPS) | Пропускная способность памяти |
| Арифметическая интенсивность | Высокая | Очень низкая |
| Использование GPU | Высокое (50–80%) | Низкое (1–10%) без батчинга |
| Метрика задержки | **Время до первого токена (TTFT)** | **Время на выходной токен (TPOT)** |

- TTFT важен для пользовательского опыта (сколько времени пройдет до начала потоковой передачи ответа). TPOT определяет воспринимаемую скорость генерации. Пользователи терпимы к более высокому TTFT (1–5 секунд), но ожидают быстрого TPOT (30–100 мс на токен для диалоговых приложений).

## Статический батчинг (наивный)

- Простейший батчинг: собрать $B$ запросов, дополнить их до одинаковой длины (padding) и обработать как единый батч.

- **Проблема 1**: запросы имеют разную длину промптов и генерируют разное количество выходных токенов. Короткие запросы завершаются раньше, но должны ждать самый длинный запрос в батче, прежде чем начнется следующий батч. GPU простаивает, пока генерирует ответ для одного оставшегося длинного запроса.

- **Проблема 2**: дополнение (padding) тратит вычислительные ресурсы впустую. Если самый длинный промпт составляет 2000 токенов, а самый короткий — 50, батч дополняется до 2000. GPU обрабатывает 1950 токенов дополнения для короткого запроса — это чистая потеря.

![Статический батчинг тратит слоты GPU впустую, ожидая самый длинный запрос; непрерывный батчинг немедленно заполняет освободившиеся слоты](../images/static_vs_continuous_batching.svg)

## Непрерывный батчинг

- **Непрерывный батчинг** (также называемый батчингом на уровне итераций) решает обе проблемы, работая на уровне отдельных шагов декодирования, а не целых запросов.

- На каждом шаге декодирования:
    1. Все активные запросы генерируют по одному токену параллельно (как батч).
    2. Запросы, которые завершились (сгенерировали токен EOS), немедленно **удаляются** из батча.
    3. Новые запросы из очереди немедленно **вставляются** в освободившиеся слоты.

- Размер батча динамически меняется на каждом шаге. GPU никогда не простаивает в ожидании отстающих, и нет бесполезного дополнения (каждый запрос использует только те слоты, которые ему нужны).

- **Результат**: непрерывный батчинг обычно увеличивает пропускную способность в 2–10 раз по сравнению со статическим батчингом без изменения качества модели или значительного увеличения задержки.

## PagedAttention и vLLM

- KV-кеш создает проблему управления памятью. У каждого запроса есть KV-кеш, который растет с каждым сгенерированным токеном. Разные запросы находятся на разных стадиях (разные размеры кеша). Выделение непрерывной памяти для каждого запроса тратит место впустую (вы должны выделить память под максимально возможную длину, даже если запрос генерирует всего несколько токенов).

![PagedAttention отображает виртуальные страницы KV-кеша на несмежную физическую память GPU, устраняя фрагментацию и обеспечивая выделение по запросу](../images/paged_attention.svg)

- **PagedAttention** (Kwon et al., 2023) применяет концепцию виртуальной памяти из ОС (глава 13) к KV-кешу. Кеш делится на **страницы** фиксированного размера (блоки позиций токенов). Страницы выделяются по запросу и могут быть несмежными в физической памяти GPU.

- Преимущества:
    - **Отсутствие фрагментации**: страницы имеют одинаковый размер, поэтому между запросами нет «дыр» впустую потраченной памяти.
    - **Ленивое выделение**: память выделяется только тогда, когда токены действительно генерируются, а не резервируется заранее под максимальную длину.
    - **Copy-on-write**: запросы, имеющие общий префикс (например, системные промпты), используют одни и те же страницы KV-кеша. Страницы копируются только тогда, когда запросы расходятся.

- **vLLM** — это движок инференса, построенный на базе PagedAttention. Он достигает в 2–4 раза более высокой пропускной способности, чем обслуживание со статическим выделением памяти (например, text-generation-inference от HuggingFace без PagedAttention), практически полностью устраняя потери памяти KV-кеша.

## Стратегии планирования

- Когда ожидается несколько запросов, а GPU может обработать только ограниченный батч, **планирование** решает, какие запросы обслуживать:

- **Первым пришел — первым обслужен (FCFS)**: обработка запросов в порядке их поступления. Просто, но несправедливо: пользователь, отправивший запрос на генерацию 10 тыс. токенов, блокирует всех пользователей за собой.

- **Сначала короткие задания (SJF)**: обработка запроса, который завершится раньше всех. Минимизирует среднюю задержку, но наказывает долго выполняющиеся запросы (они могут «голодать»). На практике ожидаемая длина вывода неизвестна, поэтому SJF использует эвристики (длина промпта, история пользователя).

- **Вытеснение (Preemption)**: если поступает высокоприоритетный запрос, приостановить выполнение низкоприоритетного запроса (переместить его KV-кеш в память CPU или на SSD), обслужить высокоприоритетный запрос, а затем возобновить приостановленный. vLLM поддерживает эту функцию.

- **На основе приоритетов**: назначение приоритетов пользователям или типам запросов. Интерактивные запросы в реальном времени получают более высокий приоритет, чем пакетные задания. В сочетании с вытеснением это гарантирует соблюдение SLO по задержке для высокоприоритетного трафика.

- **Токеновый бюджет**: ограничение общего количества токенов в активном батче. Это предотвращает ситуацию, когда несколько длинных запросов монополизируют память GPU и лишают ресурсов новые запросы.

## Дезагрегированный инференс

- Фазы префилла (prefill) и декодирования (decode) имеют противоположные вычислительные профили. Выполнение обеих на одном и том же GPU означает, что GPU постоянно переключается между состояниями, ограниченными вычислениями (префилл) и пропускной способностью памяти (декодирование), не используя полностью ни один из ресурсов.

- **Дезагрегированный инференс** разделяет их:
    - **Узлы префилла**: GPU, оптимизированные для вычислений (высокие FLOPS, возможно, с меньшим объемом памяти). Обрабатывают все входящие промпты.
    - **Узлы декодирования**: GPU, оптимизированные для пропускной способности памяти (большой объем KV-кеша, высокая пропускная способность памяти). Обрабатывают генерацию всех токенов.

- Узел префилла вычисляет начальный KV-кеш и отправляет его на узел декодирования (через NVLink или сеть). Узел декодирования генерирует токены, используя полученный кеш.

- Это архитектура **Mooncake** (Moonshot AI), которую исследуют многие команды, занимающиеся обслуживанием LLM. Преимущество: каждый тип GPU соответствует характеристикам своей рабочей нагрузки, что повышает общую утилизацию.

## Обслуживание нескольких моделей и LoRA

- В продакшене часто приходится обслуживать несколько моделей (разных размеров для разных уровней нагрузки, разные дообученные варианты для разных задач).

- **Мультиплексирование моделей**: загрузка нескольких моделей на один GPU и маршрутизация запросов к соответствующей модели. Память GPU используется совместно: GPU на 40 ГБ может одновременно содержать модель 13B (26 ГБ) и модель 7B (14 ГБ).

- **Обслуживание LoRA**: вместо развертывания отдельных дообученных моделей развертывается одна базовая модель с несколькими **LoRA-адаптерами** (глава 6). Каждый адаптер добавляет <1% параметров. Запросы направляются к соответствующему адаптеру во время инференса.

- **S-LoRA** (Sheng et al., 2023): обслуживает тысячи LoRA-адаптеров на основе одной базовой модели. Адаптеры хранятся в CPU и подгружаются в память GPU по мере необходимости. KV-кеш и веса базовой модели являются общими; для каждого запроса различаются только небольшие матрицы LoRA.

- **Punica** (Chen et al., 2023): выполняет батчинг запросов для разных LoRA-адаптеров с помощью специализированного CUDA-ядра, которое применяет разные матрицы LoRA к разным запросам внутри одного батча. Это позволяет избежать накладных расходов на переключение адаптеров для каждого запроса.

## Ограниченная и управляемая генерация

- Многим приложениям требуется, чтобы LLM выдавала результат в определенном формате: валидный JSON, SQL-запросы, код на конкретном языке или ответы, соответствующие схеме. **Ограниченная генерация** гарантирует, что вывод соответствует грамматике или схеме.

- **Декодирование с грамматическими ограничениями**: на каждом шаге декодирования маскируются токены, которые нарушают грамматику. Если текущий вывод выглядит как `{"name": "Alice", "age":`, а грамматика требует далее целое число, маскируются все токены, кроме цифр. Распределение вероятностей LLM перенормируется по допустимым токенам.

- **Outlines** (Willard & Louf, 2023): компилирует JSON-схему или регулярное выражение в конечный автомат (FSM). На каждом шаге декодирования FSM определяет, какие токены являются допустимыми продолжениями. Недопустимые токены получают вероятность 0. Это гарантирует 100% соответствие схеме без повторных попыток.

- **SGLang** интегрирует ограниченную генерацию нативно: вы задаете структуру вывода на Python, а движок эффективно управляет маскированием токенов и кешированием. Это сочетается с RadixAttention (кеширование префиксов), благодаря чему структурированные выводы повторно используют кешированные префиксы.

- **Почему это важно**: без ограниченной генерации вы генерируете текст свободно, а затем парсите его, повторяя попытки при ошибках. Для сложных JSON-схем частота повторных попыток составляет 10-30%, что приводит к пустой трате вычислительных ресурсов. Ограниченная генерация полностью исключает необходимость повторных попыток.

## Маршрутизация запросов

- Не каждому запросу нужна самая большая модель. **Маршрутизация запросов** направляет запросы к разным моделям в зависимости от их предполагаемой сложности:

- **Каскадирование**: сначала пробуется небольшая модель. Если уверенность небольшой модели ниже порога (например, softmax-вероятность топового токена < 0.8), запрос передается более крупной модели. Простые запросы (80%+ трафика) дешево обрабатываются маленькой моделью; только сложные запросы используют дорогую модель.

- **Обучаемая маршрутизация**: обучение легковесного классификатора (или использование перплексии небольшой модели) для предсказания того, какой уровень модели нужен запросу. Маршрутизация запроса "Сколько будет 2+2?" к модели 3B, а запроса "Объясни математические основы квантовой запутанности" — к модели 70B.

- **Влияние**: если 80% запросов могут быть обработаны моделью, которая стоит в 10 раз дешевле, средняя стоимость одного запроса снижается примерно на 70%. Это одна из наиболее эффективных оптимизаций затрат для развертываний с несколькими моделями.

- **Гибридная маршрутизация (на устройстве + облако)**: **Cactus** ([github.com/cactus-compute/cactus](https://github.com/cactus-compute/cactus)) реализует маршрутизацию запросов на уровне устройства. Она запускает небольшую модель локально (на телефоне, ноутбуке, носимом устройстве) с помощью специализированных ARM SIMD-ядер и автоматически перенаправляет запрос в облачную модель, если уверенность локальной модели низка или запрос выходит за рамки возможностей устройства. Приложение использует API, совместимый с OpenAI, для обоих путей — маршрутизация происходит прозрачно. Это каскадирование на уровне инфраструктуры: первый уровень бесплатен (на устройстве), второй уровень платный (облачный API). Для приложений, где большинство запросов просты (ответы помощника, автодополнение, транскрибация), локальная обработка покрывает 70-90% трафика с нулевыми предельными затратами.

## Метрики инференса

- Выбор правильных метрик зависит от варианта использования:

| Метрика | Что измеряет | Цель (диалог) | Цель (батч) |
|--------|-----------------|------------------------|-----------------|
| **TTFT** | Время до первого токена | <1 с | менее важно |
| **TPOT** | Время на один выходной токен | <100 мс | менее важно |
| **Пропускная способность** | Токенов/секунду (всего) | менее важно | максимизировать |
| **p99 Задержка** | Худшие 1% запросов | <5 с | <30 с |
| **Стоимость токена** | $/1M токенов | минимизировать | минимизировать |
| **Соблюдение SLO** | % запросов, уложившихся в задержку | >99% | >95% |

- **Компромисс между TTFT и TPOT**: агрессивный батчинг увеличивает пропускную способность (больше токенов/с в сумме), но увеличивает TPOT (каждый токен генерируется дольше, так как GPU обрабатывает больше запросов одновременно). Стратегия планирования должна находить баланс между пропускной способностью (выручкой) и задержкой (пользовательским опытом).

- **Стоимость одного токена** — это главный показатель для продакшена. Он объединяет стоимость оборудования (аренда GPU), пропускную способность (токенов/с) и утилизацию. Система, работающая при 50% утилизации GPU, стоит в 2 раза дороже в пересчете на токен, чем система при 100%. Именно поэтому батчинг, планирование и PagedAttention так важны — они повышают утилизацию.

## Задачи по программированию (используйте CoLab или ноутбук)

1. Смоделируйте непрерывный (continuous) и статический батчинг и измерьте разницу в пропускной способности.
```python
import random
import time

def simulate_static_batching(requests, batch_size=8):
    """Process requests in fixed batches. Wait for all to finish."""
    total_tokens = 0
    total_time = 0

    for i in range(0, len(requests), batch_size):
        batch = requests[i:i + batch_size]
        max_len = max(r['output_len'] for r in batch)
        # All requests in the batch take as long as the longest
        batch_time = max_len * 0.01  # 10ms per token
        total_time += batch_time
        total_tokens += sum(r['output_len'] for r in batch)

    return total_tokens / total_time  # tokens per second

def simulate_continuous_batching(requests, max_batch=8):
    """Process with continuous batching. Remove finished, add new."""
    total_tokens = 0
    total_time = 0
    active = []
    queue = list(requests)

    while active or queue:
        # Fill batch
        while len(active) < max_batch and queue:
            active.append({'remaining': queue.pop(0)['output_len']})

        if not active:
            break

        # One decode step: all active requests generate 1 token
        for req in active:
            req['remaining'] -= 1
        total_tokens += len(active)
        total_time += 0.01  # 10ms per step

        # Remove finished requests
        active = [r for r in active if r['remaining'] > 0]

    return total_tokens / total_time

# Generate requests with varied output lengths
random.seed(42)
requests = [{'output_len': random.randint(10, 500)} for _ in range(100)]

static_tps = simulate_static_batching(requests)
continuous_tps = simulate_continuous_batching(requests)

print(f"Static batching:     {static_tps:.0f} tokens/s")
print(f"Continuous batching: {continuous_tps:.0f} tokens/s")
print(f"Speedup: {continuous_tps / static_tps:.1f}x")
```

    # Paged: allocate only what is used (with page granularity)
    import math
    avg_pages = math.ceil(avg_seq_len / page_size)
    paged_gb = n_requests * avg_pages * page_size * kv_per_token_bytes / 1e9

    waste_preallocated = (max_seq_len - avg_seq_len) / max_seq_len
    waste_paged = (avg_pages * page_size - avg_seq_len) / (avg_pages * page_size)

    print(f"Requests: {n_requests}, Max seq: {max_seq_len}, Avg seq: {avg_seq_len}")
    print(f"  Preallocated: {preallocated_gb:.1f} GB (waste: {waste_preallocated:.0%})")
    print(f"  Paged:        {paged_gb:.1f} GB (waste: {waste_paged:.0%})")
    print(f"  Savings:      {preallocated_gb - paged_gb:.1f} GB ({preallocated_gb/paged_gb:.1f}x)")
    print()

# Llama-70B: ~1.3 KB per token per layer, 80 layers = ~100 KB per token total
kv_bytes = 100_000

# Scenario 1: short requests, large max
paged_vs_preallocated(256, max_seq_len=4096, avg_seq_len=256, page_size=16, kv_per_token_bytes=kv_bytes)

# Scenario 2: varied lengths
paged_vs_preallocated(256, max_seq_len=8192, avg_seq_len=1024, page_size=16, kv_per_token_bytes=kv_bytes)

# Scenario 3: long context
paged_vs_preallocated(64, max_seq_len=131072, avg_seq_len=16000, page_size=16, kv_per_token_bytes=kv_bytes)
