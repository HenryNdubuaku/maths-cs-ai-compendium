# Масштабирование и развёртывание

*Обслуживание больших моделей для миллионов пользователей требует распределения инференса между несколькими GPU, предсказания токенов до того, как они потребуются, кеширования общего контекста и выбора подходящего фреймворка. В этом файле рассматриваются параллелизм при инференсе, спекулятивное декодирование, кеширование префиксов, фреймворки для инференса, оптимизация затрат и мониторинг.*

- Один GPU H100, обслуживающий модель 70B, может обрабатывать около 100 одновременных пользователей с интерактивной задержкой. Обслуживание 10 миллионов пользователей требует 100 000 GPU, что обходится примерно в 3 миллиарда долларов в год в облачных вычислениях. Каждый процент повышения эффективности экономит десятки миллионов долларов. Именно поэтому оптимизация инференса — это не академическая задача: она напрямую определяет экономику продуктов на базе ИИ.

## Параллелизм моделей при инференсе

- Когда модель слишком велика для одного GPU, её необходимо разделить между несколькими GPU. Стратегии параллелизма из обучения (глава 6) применимы и при инференсе, но с другими компромиссами.

### Тензорный параллелизм

- **Тензорный параллелизм** (в стиле Megatron, глава 6) разделяет отдельные весовые матрицы между GPU. Для линейного слоя $Y = XW$ весовая матрица $W$ разбивается по столбцам между $N$ GPU. Каждый GPU вычисляет частичный результат, а затем операция all-reduce выполняет агрегацию:

$$W = [W_1 | W_2 | \cdots | W_N], \quad Y_i = X W_i, \quad Y = \text{concat}(Y_1, \ldots, Y_N)$$

- При инференсе тензорный параллелизм является стандартным решением для моделей, которые не помещаются на один GPU. Модель 70B в формате FP16 требует 140 ГБ — она распределяется между 2 × 80 ГБ GPU с помощью тензорного параллелизма.

- **Влияние на задержку**: тензорный параллелизм добавляет этап коммуникации all-reduce на каждый слой. При использовании NVLink (900 ГБ/с) это добавляет около 0,1 мс на слой. При использовании PCIe (32 ГБ/с) это добавляет около 3 мс на слой. Для модели 70B с 80 слоями на 2 GPU: NVLink добавляет в сумме около 8 мс, PCIe — около 240 мс. Именно поэтому NVLink критически важен для многопроцессорного инференса.

### Конвейерный параллелизм

- **Конвейерный параллелизм** назначает разные слои разным GPU. GPU 1 обрабатывает слои 0-39, GPU 2 — слои 40-79. Токены проходят через конвейер последовательно.

- При инференсе конвейерный параллелизм имеет бо́льшую задержку, чем тензорный (весь конвейер должен быть пройден для каждого токена), но меньшие накладные расходы на коммуникацию (между GPU передаются только активации, без all-reduce). Он предпочтителен, когда GPU соединены медленными интерфейсами (разные узлы, отсутствие NVLink).

### Параллелизм по последовательности

- Для очень длинных последовательностей сам KV-кеш может не поместиться на один GPU, даже если модель помещается. **Параллелизм по последовательности** шардирует KV-кеш между GPU: каждый GPU хранит часть кешированных ключей и значений последовательности.

- Во время работы механизма внимания каждый GPU вычисляет частичные оценки внимания для своего кешированного сегмента, а затем операция редукции объединяет результаты. Это используется при инференсе с длинным контекстом (128K+ токенов), когда KV-кеш превышает память одного GPU.

## Спекулятивное декодирование

- **Спекулятивное декодирование** — одна из наиболее эффективных оптимизаций инференса LLM. Идея заключается в следующем: декодирование происходит медленно, так как токены генерируются по одному, и каждый токен требует полного прохода большой модели. Однако маленькая модель может генерировать токены-кандидаты гораздо быстрее, а большая модель может **верифицировать** несколько кандидатов параллельно.

![Спекулятивное декодирование: быстрая черновая модель генерирует 5 токенов-кандидатов, целевая модель верифицирует их все за один проход, принятые токены сохраняются, а отвергнутые пересемплируются](../images/speculative_decoding.svg)

- **Алгоритм**:
    1. **Черновая модель** (маленькая, быстрая — например, 1B параметров) авторегрессионно генерирует $k$ токенов-кандидатов.
    2. **Целевая модель** (большая, точная — например, 70B) выполняет один прямой проход по всей черновой последовательности, вычисляя вероятность каждого токена-кандидата.
    3. Каждый кандидат **принимается**, если целевая модель «согласна» (её вероятность для этого токена достаточно высока). Отвергнутые кандидаты пересемплируются из распределения целевой модели.
    4. В среднем за один шаг верификации принимается несколько токенов, что дает ускорение, пропорциональное коэффициенту принятия.

$$\text{Speedup} \approx \frac{k \times \text{acceptance\_rate}}{\text{cost\_ratio}} \approx 2\text{-}3\times$$

- **Почему это работает без потери качества**: схема сэмплирования с отклонением гарантирует, что выходное распределение в точности соответствует целевой модели. Спекулятивное декодирование не приводит к потерям — результат статистически идентичен результату работы только целевой модели, но достигается быстрее.

- **Варианты**:
    - **Medusa** (Cai et al., 2024): вместо отдельной черновой модели добавляются несколько легковесных «голов» к целевой модели, которые предсказывают несколько будущих токенов одновременно. Отдельная модель не требуется.
    - **EAGLE** (Li et al., 2024): обучает легковесную черновую голову, которая использует скрытые состояния целевой модели для предсказания будущих токенов. Более высокий коэффициент принятия, чем у независимых черновых моделей.
    - **Самоспекулятивное декодирование**: целевая модель сама генерирует черновики с помощью раннего выхода (для черновика выполняются только первые несколько слоев, затем верификация проводится полной моделью).
    - **Параллельное декодирование**: генерация нескольких продолжений параллельно (дерево кандидатов) и верификация всего дерева целиком. Более высокая пропускная способность, но требует больше памяти для ветвящихся KV-кешей.

## Кеширование префиксов

- Многие запросы имеют общие префиксы: системные промпты, примеры few-shot или типичные шаблоны запросов. **Кеширование префиксов** сохраняет KV-кеш для этих префиксов и повторно использует его в разных запросах.

- **Кеширование системного промпта**: если каждый запрос начинается с одного и того же системного промпта из 2000 токенов, KV-кеш для этих 2000 токенов вычисляется один раз и используется совместно всеми запросами. Для модели 70B с 80 слоями это экономит около 200 МБ на запрос.

- **Кеширование на основе radix-дерева** (SGLang): организация кешированных префиксов в виде radix-дерева (префиксного дерева). Когда поступает новый запрос, находится самое длинное совпадение с кешированным префиксом, и генерация начинается с этого места, пропуская вычисления для совпадающего префикса.

- **Результат**: для приложений с длинными общими префиксами (чат-боты с системными промптами, RAG с часто извлекаемыми фрагментами) кеширование префиксов сокращает TTFT на 50-90% и пропорционально экономит вычислительные ресурсы GPU.

## KV-Cache Eviction

- Помимо квантования KV-кеша (файл 01) и использования GQA/MLA для уменьшения его размера (файл 02), политики **KV-cache eviction** (вытеснения из KV-кеша) выборочно удаляют закешированные токены, на которые с малой вероятностью будет обращено внимание в будущем.

- **H2O** (Heavy-Hitter Oracle, Zhang et al., 2023) отмечает, что веса внимания подчиняются степенному закону: небольшая доля токенов («тяжеловесы» или heavy hitters) получает большую часть внимания, в то время как на большинство токенов внимание практически не обращается. H2O сохраняет:

    1. **Недавние токены** (скользящее окно из последних $w$ токенов, как в StreamingLLM).
    2. **Токены-тяжеловесы** (топ-$k$ токенов по совокупному весу внимания за все прошлые шаги декодирования).

- Токены, которые не являются ни недавними, ни «тяжеловесами», вытесняются. Это позволяет поддерживать фиксированный размер KV-кеша, сохраняя при этом токены, которые действительно влияют на генерацию. H2O достигает качества, близкого к полному KV-кешу, используя лишь 20% памяти.

- **Scissorhands** (Liu et al., 2023) использует похожий подход, но применяет более сложную метрику важности: токены, получающие высокое внимание на **текущем** шаге, сохраняются, а токены, на которые не обращалось внимание в течение $T$ шагов, вытесняются. Это позволяет адаптироваться к изменяющимся паттернам внимания во время генерации.

- **Динамическое вытеснение + StreamingLLM**: объединение «стоков внимания» (постоянное сохранение первых нескольких токенов) с динамическим вытеснением (сохранение недавних токенов + токенов-тяжеловесов). Это наиболее эффективный с точки зрения памяти подход для очень длинной генерации, позволяющий генерировать текст бесконечной длины с ограниченным снижением качества.

- Ключевой вывод для всех методов вытеснения: внимание в LLM **на практике разреженно** — несмотря на то, что архитектура вычисляет внимание по всем закешированным токенам, фактические веса внимания концентрируются на небольшой подвыборке. Вытеснение остальных токенов оказывает минимальное влияние на качество вывода.

## Инфраструктура инференса

- Экосистема обслуживания LLM пришла к нескольким доминирующим фреймворкам:

| Фреймворк | Преимущества | Лучшее применение |
|-----------|-----------|----------|
| **vLLM** | PagedAttention, непрерывный батчинг, высокая пропускная способность | Общее обслуживание LLM, максимальная пропускная способность |
| **TensorRT-LLM** | Оптимизированные ядра NVIDIA, FP8, in-flight batching | Максимальная производительность на GPU NVIDIA |
| **SGLang** | Кеширование префиксов (RadixAttention), быстрая структурированная генерация | Приложения с общими префиксами, ограниченный вывод |
| **llama.cpp** | CPU/Metal/CUDA/Vulkan, квантование GGUF, портативность | Потребительское оборудование, инференс на устройстве |
| **TGI** (HuggingFace) | Простой API, легкость развертывания, интеграция с модельным хабом | Быстрое развертывание, экосистема HuggingFace |
| **Ollama** | Загрузка и запуск модели одной командой | Личное использование, локальная разработка |
| **ExLlamaV2** | Экстремальная оптимизация квантования (формат EXL2) | Инференс на GPU с ограниченной памятью |

- **vLLM** — это выбор по умолчанию для продакшн-обслуживания LLM. Он поддерживает непрерывный батчинг, PagedAttention, тензорный параллелизм, спекулятивное декодирование, обслуживание LoRA и большинство моделей с открытым исходным кодом.

- **TensorRT-LLM** обеспечивает высочайшую «сырую» производительность на оборудовании NVIDIA (на 10-30% быстрее, чем vLLM на том же GPU), но менее гибок и сложнее в настройке.

- **SGLang** превосходит аналоги, когда приложения требуют структурированного вывода (JSON, код в определенном формате) или имеют общие префиксы, благодаря кешу radix attention и движку ограниченного декодирования.

## Оптимизация затрат

- При масштабировании стоимость инференса становится основной статьей бюджета на ML. Стратегии снижения затрат:

- **Правильный выбор GPU**: не каждой модели нужен H100. Квантованная модель 7B хорошо работает на A10G (~$1/hr) вместо H100 (~$8/hr). Подбирайте GPU под рабочую нагрузку.

- **Spot-инстансы**: облачные провайдеры предлагают неиспользуемые мощности GPU со скидкой 60-90% (AWS Spot, GCP Preemptible). Spot-инстансы могут быть прерваны, поэтому они подходят для пакетного инференса, но не для сервисов, критичных к задержке. В сочетании с обработкой прерываний (сохранение состояния, возобновление на новом инстансе) spot-инстансы могут обслуживать и интерактивный трафик.

- **Автомасштабирование**: масштабируйте количество GPU в зависимости от трафика. Увеличивайте количество в часы пик, уменьшайте ночью. Это решается с помощью Kubernetes HPA (Horizontal Pod Autoscaler) или облачных средств автомасштабирования (AWS SageMaker, GCP Vertex AI).

- **Батчинг + утилизация**: разница между 30% и 90% утилизации GPU — это трехкратная разница в стоимости за токен. Непрерывный батчинг, умное планирование и PagedAttention повышают утилизацию.

- **Квантование**: INT4 против FP16 — это в 4 раза меньше памяти → помещается на меньший GPU → в 2-4 раза ниже стоимость. Кроме того, в один батч помещается больше запросов → выше пропускная способность → ниже стоимость за токен.

- **Бенчмарки стоимости за токен** (приблизительно, 2026):

## Мониторинг

- Продакшн-инференс требует непрерывного мониторинга, чтобы выявлять деградацию до того, как она затронет пользователей:

- **Мониторинг задержки**: отслеживайте TTFT и TPOT на уровне p50, p95 и p99. Установите алерты, если p99 превышает SLO. Скачок p99 часто указывает на: нехватку памяти KV-кеша (thrashing), «тяжелый» запрос, монополизирующий батч, или тепловой троттлинг GPU.

- **Мониторинг пропускной способности**: отслеживайте количество токенов в секунду на GPU. Падение указывает на: снижение эффективности батчинга (много коротких запросов → низкая утилизация батча), увеличение длины последовательностей (больше памяти KV-кеша на запрос) или проблемы с оборудованием (GPU в режиме коррекции ошибок ECC, работает медленнее).

- **Утилизация GPU**: отслеживайте загрузку SM, использование памяти и пропускную способность памяти. Низкая загрузка SM + высокая утилизация памяти = ограничение по памяти (нужна большая пропускная способность или квантование). Высокая загрузка SM + низкая утилизация памяти = ограничение по вычислениям (нужно больше FLOPS или модель поменьше).

- **Мониторинг качества модели**: отслеживайте метрики для каждого запроса (длина ответа, перплексия на отложенной выборке, сигналы обратной связи от пользователей). Качество модели может снижаться из-за: дрейфа данных (изменилось распределение входящих запросов), накопления ошибки квантования KV-кеша при длинных диалогах или ошибок в пайплайне обслуживания.

- **Мониторинг затрат**: отслеживайте стоимость в расчете на токен, на модель и на тип GPU. Если затраты растут без увеличения пропускной способности, исследуйте регрессии эффективности (новая версия модели с более высоким потреблением памяти, неоптимальная конфигурация батчей или недоиспользование GPU).

- **Инструменты**: Prometheus + Grafana (глава 15) для метрик инфраструктуры, встроенные эндпоинты метрик vLLM/TRT-LLM и пользовательское логирование для метрик уровня модели.

## Задачи по программированию (используйте CoLab или ноутбук)

1. Смоделируйте спекулятивное декодирование (speculative decoding). Используйте быструю «черновую» (draft) функцию и медленную «целевую» (target) функцию, а также измерьте ускорение от генерации и проверки нескольких токенов одновременно.
```python
import random
import time

def target_model(tokens):
    """Slow but accurate model. Returns probability of each candidate token."""
    time.sleep(0.01)  # simulate 10ms per forward pass
    # For simulation: accept tokens that are even numbers
    return [0.9 if t % 2 == 0 else 0.1 for t in tokens]

def draft_model():
    """Fast but approximate model. Generates one candidate token."""
    time.sleep(0.001)  # simulate 1ms per token
    return random.randint(0, 9)

def standard_decoding(n_tokens):
    """Generate one token at a time with the target model."""
    tokens = []
    for _ in range(n_tokens):
        time.sleep(0.01)  # target model generates 1 token
        tokens.append(random.randint(0, 9))
    return tokens

def speculative_decoding(n_tokens, k=4):
    """Generate k draft tokens, verify with target, accept/reject."""
    tokens = []
    total_target_calls = 0

    while len(tokens) < n_tokens:
        # Draft: generate k candidates quickly
        candidates = [draft_model() for _ in range(k)]

        # Verify: one target model call for all k candidates
        probs = target_model(candidates)
        total_target_calls += 1

        # Accept tokens until one is rejected
        for i, (tok, prob) in enumerate(zip(candidates, probs)):
            if random.random() < prob:
                tokens.append(tok)
                if len(tokens) >= n_tokens:
                    break
            else:
                # Resample from target distribution
                tokens.append(tok + 1)  # simplified resampling
                break

    return tokens, total_target_calls

n = 50

start = time.time()
_ = standard_decoding(n)
standard_time = time.time() - start

start = time.time()
_, target_calls = speculative_decoding(n, k=5)
spec_time = time.time() - start

print(f"Standard:    {standard_time:.2f}s ({n} target calls)")
print(f"Speculative: {spec_time:.2f}s ({target_calls} target calls)")
print(f"Speedup:     {standard_time / spec_time:.1f}x")
```

2. Оцените экономию затрат при использовании различных стратегий оптимизации, примененных к развертыванию LLM для инференса.
```python
def serving_cost_analysis(
    model_name, params_B, precision_bits,
    gpu_name, gpu_mem_gb, gpu_cost_per_hr,
    target_throughput_tps,
):
    """Estimate serving cost for an LLM deployment."""
    model_size_gb = params_B * 1e9 * precision_bits / 8 / 1e9
    gpus_for_model = max(1, int((model_size_gb * 1.2) / gpu_mem_gb + 0.99))  # 1.2x for KV-cache

    # Rough throughput estimate (memory-bandwidth limited)
    tokens_per_gpu = 500 / (params_B * precision_bits / 16)  # normalised to 500 tok/s for 7B FP16
    total_throughput = tokens_per_gpu * gpus_for_model

    replicas = max(1, int(target_throughput_tps / total_throughput + 0.99))
    total_gpus = gpus_for_model * replicas
    cost_per_hr = total_gpus * gpu_cost_per_hr
    cost_per_1M_tokens = cost_per_hr / (total_throughput * replicas * 3600 / 1e6)

    print(f"{model_name} @ {precision_bits}-bit on {gpu_name}:")
    print(f"  Model size: {model_size_gb:.0f} GB → {gpus_for_model} GPU(s)/replica")
    print(f"  Throughput: {total_throughput:.0f} tok/s/replica")
    print(f"  Replicas for {target_throughput_tps} tok/s: {replicas}")
    print(f"  Total GPUs: {total_gpus}")
    print(f"  Cost: ${cost_per_hr:.0f}/hr, ${cost_per_1M_tokens:.2f}/1M tokens")
    print()

print("=== Cost Comparison ===\n")

# Baseline: FP16 on H100
serving_cost_analysis("Llama-70B", 70, 16, "H100", 80, 8.0, 1000)

# Quantised: INT8 on H100
serving_cost_analysis("Llama-70B", 70, 8, "H100", 80, 8.0, 1000)

# Quantised: INT4 on A100
serving_cost_analysis("Llama-70B", 70, 4, "A100", 80, 4.0, 1000)

# Smaller model: 8B on A10G
serving_cost_analysis("Llama-8B", 8, 4, "A10G", 24, 1.0, 1000)
```
