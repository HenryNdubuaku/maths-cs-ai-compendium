# Edge Inference

*Edge inference 在用户设备（手机、笔记本、IoT 传感器）上运行模型，不把数据发到云端。本文件涵盖边缘约束、模型压缩流水线、设备端 runtime、编译器栈、硬件目标（NPU、Neural Engine）、设备端 LLM、联邦学习与 latency 优化*

- 云端推理需要网络连接、增加 latency（50-200 ms 往返）、每次请求都要花钱，并把用户数据发到第三方服务器。**Edge inference** 一举消除这四点：模型在本地运行、即时响应、每次推理零成本、数据保持私密。

- 权衡：边缘设备的算力和显存比数据中心 GPU 少 100-1000 倍。要让模型在这些约束下运行，需要在每一层做激进优化。

- **Cactus**（[github.com/cactus-compute/cactus](https://github.com/cactus-compute/cactus)）是专为移动与可穿戴设备打造的低 latency AI 引擎。它在生产中演示了本文件涵盖的诸多技术：用于 attention 和矩阵运算的自定义 ARM SIMD kernel（第 16 章）、KV-cache 量化（第 17 章文件 01）、分块 prefill、在 Apple 和 Qualcomm 芯片上的 NPU 加速推理、零拷贝内存映射（RAM 占用降低 10 倍），以及在端侧算力不足时自动回退到云端。Cactus 在 iOS、Android、macOS 与嵌入式 Linux 上支持多模态推理（LLM、视觉、语音），并提供 Swift、Kotlin、Python、Flutter、React Native 与 Rust 的 SDK。其基准显示在 M4 Pro 上对 1.2B 模型 INT4 达到 100 tokens/s decode，在 iPhone 17 Pro 上达到 48 tokens/s，这是优化过的 edge inference 的具体范例。

## 边缘约束

| 资源 | 云端 GPU (H100) | 笔记本 (M4) | 手机 (Snapdragon 8 Gen 3) | IoT (ESP32) |
|----------|-----------------|-------------|---------------------------|-------------|
| RAM | 80 GB HBM3 | 16-36 GB 统一内存 | 8-12 GB LPDDR5 | 520 KB |
| 算力 | 989 TFLOPS (FP8) | 38 TOPS (Neural Engine) | 45 TOPS (NPU) | 0.001 TOPS |
| 功耗 | 700 W | 15-30 W | 5-10 W | 0.1 W |
| 存储 | TB | 256 GB-2 TB | 128-512 GB | 4 MB |

- 云端 GPU 与手机 NPU 的算力差距约 20 倍。GPU 与微控制器之间约 1,000,000 倍。不同设备需要不同程度的压缩和不同的模型架构。

## 模型压缩流水线

- 对边缘部署，压缩不是单一技术 —— 它是顺序应用多种互补技术的**流水线**：

```
Full model (FP32, 70B params)
    ↓ Knowledge distillation → smaller model (7B params)
    ↓ Structured pruning → remove redundant heads/layers (4B effective)
    ↓ Quantisation (INT4) → 4x smaller (2 GB)
    ↓ Compiler optimisation → fused kernels, optimised memory layout
    ↓ Runtime → on-device execution
```

- 每一步都减小体积和 latency。顺序很重要：先 distill（缩减架构），再 prune（移除结构），再 quantise（降低精度），最后编译（针对目标硬件优化）。量化之后再 distill 等于压缩已经损失过的模型。

## 设备端 Runtime

- **runtime** 加载模型、分配显存并在目标硬件上执行推理。每个平台都有其偏好的 runtime：

- **ONNX Runtime**：跨平台（Windows、Linux、macOS、iOS、Android）。支持 CPU、GPU（CUDA、DirectML、CoreML、NNAPI）以及众多加速器后端。可移植性最强。模型从 PyTorch/TensorFlow 导出为 ONNX 格式。

- **TensorFlow Lite (TFLite)**：Google 的边缘 runtime。针对 ARM CPU 和 Android NPU 优化。二进制极小（约 1 MB）。支持 INT8 和 float16。Android 部署的标准选择。

- **Core ML**：Apple 面向 iOS/macOS 的 runtime。根据模型特征自动使用 Neural Engine、GPU 或 CPU。模型用 `coremltools` 从 PyTorch/TensorFlow 转换。与 Apple 硬件紧密集成（统一内存、Neural Engine）。

- **ExecuTorch**：Meta 新的设备端 PyTorch runtime。面向边缘部署，支持提前编译和算子级硬件加速器委派。是 PyTorch Mobile 的继任者。

- **TensorRT**：NVIDIA 面向 GPU 推理优化的 runtime（第 15 章）。融合层、选择最优 kernel、自动量化。在 NVIDIA GPU 上比 PyTorch eager 模式快 2-5 倍。

- **llama.cpp**：单文件 C++ LLM 推理引擎。支持 GGUF 量化（Q4、Q5、Q8）、CPU（AVX/NEON）、Metal（Apple GPU）、CUDA 和 Vulkan。在消费级硬件上跑 LLM 的首选。

## 编译器栈

- 高层模型（PyTorch 图）与硬件（NPU 指令）之间是**编译器栈**，它针对具体目标优化模型：

```
PyTorch model
    ↓ Export (torch.export, ONNX, TorchScript)
Graph IR (intermediate representation)
    ↓ Graph optimisations
        - Constant folding (compute constant expressions at compile time)
        - Dead code elimination (remove unused operations)
        - Operator fusion (conv + bn + relu → single fused op)
        - Layout transformation (NCHW → NHWC for ARM, channels-last)
    ↓ Lowering
Hardware-specific IR
    ↓ Backend optimisations
        - Tiling and loop ordering (cache-friendly access patterns)
        - Vectorisation (SIMD, chapter 16)
        - Memory planning (reuse buffers to minimise peak memory)
        - Kernel selection (choose the best implementation per op)
    ↓ Code generation
Machine code / NPU instructions
```

- **算子融合** 是影响最大的优化。一个 transformer 块约有 20 个运算（matmul、add、layernorm、softmax 等）。不融合时，每个运算把输出写入显存、下一个再读回。融合后，多个运算合并为单个 kernel，数据留在寄存器/cache 中。这能快 2-5 倍（第 16 章 roofline 模型）。

- **显存规划**：编译器分析模型图，确定哪些张量在生命周期上重叠、可共享同一显存缓冲区。一个含 100 个中间张量的模型可能只需 10 份显存，因为大多数在被其他张量创建之前就已被消费释放。这在 RAM 受限的设备上至关重要。

## 硬件目标

### 移动 GPU

- **Qualcomm Adreno**（Android）：支持 OpenCL、Vulkan compute（第 16 章）以及 Qualcomm 自家的 SNPE（Snapdragon Neural Processing Engine）。Adreno GPU 有 256-1024 个 ALU，支持 FP16 和 INT8。

- **ARM Mali**（Android）：支持 OpenCL 和 Vulkan。Mali GPU 采用 tile 架构（不同于桌面 GPU），影响最优显存访问模式。

- **Apple GPU**（iOS/macOS）：通过 Metal（Apple 的 GPU API）访问。统一内存架构意味着无 CPU↔GPU 拷贝开销。Metal Performance Shaders (MPS) 提供优化的 ML 原语。

### 神经处理单元（NPU）

- NPU 是专为 ML inference 设计的固定功能加速器。对标准 ML 运算（matmul、conv、激活），它比 GPU 能效高得多。

- **Apple Neural Engine**：16 核，约 38 TOPS（INT8）。通过 Core ML 访问。擅长视觉模型和设备端 diffusion。不能跑任意代码 —— 只支持 Core ML 支持的运算。

- **Qualcomm Hexagon NPU**：集成在 Snapdragon SoC 中。支持 INT8 和 INT4 推理。通过 SNPE 或带 QNN 后端的 ONNX Runtime 访问。驱动设备端功能如背景虚化、语音识别、实时翻译。

- **Google Edge TPU**：云端 TPU 的小型低功耗版本。4 TOPS、2W。用于 Coral 设备的端侧推理。仅支持 INT8 量化的 TFLite 模型。

- **委派模式**：runtime 把模型图在 NPU（支持的运算）和 CPU（不支持的运算）之间切分。最大化在 NPU 上运行的比例是性能与能效的关键。

## 设备端 LLM

- 借助小模型和激进量化，在手机和笔记本上跑 LLM 已可行：

| 模型 | 参数 | 量化后大小 | 目标设备 | 性能 |
|-------|--------|---------------|---------------|-------------|
| Phi-3 Mini | 3.8B | ~2 GB (Q4) | 手机/笔记本 | 在 iPhone 15 上约 15 tokens/s |
| Gemma 2B | 2B | ~1.5 GB (Q4) | 手机 | 在 Pixel 8 上约 20 tokens/s |
| Llama 3.2 1B | 1B | ~700 MB (Q4) | 手机 | 约 30 tokens/s |
| Llama 3.2 3B | 3B | ~2 GB (Q4) | 手机/笔记本 | 约 15 tokens/s |
| Llama 3.1 8B | 8B | ~4.5 GB (Q4) | 笔记本 | 在 M2 上约 20 tokens/s |

- **挑战**：
    - **显存**：3B Q4 模型塞进 2 GB，但长对话的 KV-cache 显著增加。手机上上下文长度通常限制在 2-4K token。
    - **热降频**：持续推理会让手机发热。连续生成 30 秒后，SoC 会降频以防止过热，性能下降 30-50%。
    - **电池**：以 15 tokens/s 跑 3B 模型约耗 3-5W。30 分钟对话耗掉典型手机电池约 5%。偶尔使用可接受，常驻应用则成问题。

- **llama.cpp** 是设备端 LLM 的事实标准。它在 CPU（AVX2、NEON、I8MM）、Apple GPU（Metal）、NVIDIA GPU（CUDA）、AMD GPU（ROCm/Vulkan）甚至手机（通过 Android 上的 Termux）上运行。

## 联邦学习

- **联邦学习** 在不集中数据的前提下，跨多个设备训练模型。每个设备在本地数据上训练，计算梯度更新，只把更新（而非数据）发给中央服务器聚合。

- **算法**（FedAvg）：
    1. 服务器把当前模型发给 $K$ 个被选中的设备。
    2. 每个设备在本地数据上微调模型若干步。
    3. 每个设备把更新后的模型（或差值）发回服务器。
    4. 服务器平均更新：$W_{\text{new}} = \frac{1}{K} \sum_{k=1}^{K} W_k$。
    5. 重复。

- **隐私**：原始数据从不离开设备。服务器只看到聚合的模型更新。**差分隐私** 给更新加噪声，使单个数据点无法从梯度反推。

- **通信效率**：模型更新很大（与模型同尺寸）。压缩技术可减小：**梯度量化**（发 INT8 梯度而非 FP32）、**稀疏化**（只发最大的梯度）、**梯度累积**（多做本地步、少发送）。

- **应用**：Google 键盘预测（Gboard）、Apple 语音识别、健康监测（在敏感健康数据上训练而不集中化）。

## Latency 优化

- 除了压缩，还有若干技术可降低端到端推理 latency：

- **提前退出**：在中间层加分类头。若模型在第 6 层（共 24 层）已自信，则不跑 7-24 层直接返回预测。简单输入早退，难输入用完整模型。对易难混合的任务，平均 latency 显著下降。

- **模型划分**：把模型在 NPU（擅长 matmul）、GPU（擅长不规则运算）和 CPU（兜底其余）之间切分。编译器基于 profiling 决定每个运算跑在哪里。

- **缓存**：对有重复查询的应用（自动补全、代码补全），缓存近期计算。若用户输入 "How do I" 而模型最近为 "How do I" 生成过补全，则可复用缓存的 KV-cache，完全跳过 prefill 阶段。

- **推测性预取**：预测用户下一步要做什么，在其开口前就开始推理。聊天应用可在用户阅读当前回答时，开始为可能的后续问题生成响应。

## 编程练习（使用 CoLab 或 notebook）

1. 模拟模型压缩流水线。从一个 float32 模型出发，依次应用 distillation（模拟）、pruning 和量化，跟踪每步的体积。
```python
def compression_pipeline(original_params_M, original_bits=32):
    size_mb = original_params_M * 1e6 * original_bits / 8 / 1e6

    print(f"Original: {original_params_M}M params, {original_bits}-bit → {size_mb:.0f} MB")

    # Step 1: Knowledge distillation (reduce params)
    distilled_params = original_params_M * 0.15  # 70B → ~10B equivalent
    size_mb = distilled_params * 1e6 * original_bits / 8 / 1e6
    print(f"After distillation ({distilled_params:.0f}M params): {size_mb:.0f} MB")

    # Step 2: Structured pruning (remove 30% of remaining)
    pruned_params = distilled_params * 0.7
    size_mb = pruned_params * 1e6 * original_bits / 8 / 1e6
    print(f"After pruning ({pruned_params:.0f}M params): {size_mb:.0f} MB")

    # Step 3: INT4 quantisation
    size_mb = pruned_params * 1e6 * 4 / 8 / 1e6
    print(f"After INT4 quantisation: {size_mb:.0f} MB")

    print(f"Total compression: {original_params_M * 1e6 * original_bits / 8 / 1e6 / size_mb:.0f}x")

print("=== Starting from 70B model ===")
compression_pipeline(70000)

print("\n=== Starting from 7B model ===")
compression_pipeline(7000)
```

2. 估计设备端推理 latency。给定模型的运算量和硬件规格，计算是否满足 latency 目标。
```python
def estimate_latency(model_name, params_M, bits, compute_tops, mem_bw_gbs, seq_len=256):
    """Estimate token generation latency for a memory-bandwidth-bound model."""
    # Model size in bytes
    model_bytes = params_M * 1e6 * bits / 8

    # Decode is memory-bound: must load entire model per token
    time_per_token_ms = model_bytes / (mem_bw_gbs * 1e9) * 1000

    # Tokens per second
    tokens_per_sec = 1000 / time_per_token_ms

    print(f"{model_name}: {params_M/1000:.1f}B params @ {bits}-bit = {model_bytes/1e9:.1f} GB")
    print(f"  Memory bandwidth: {mem_bw_gbs} GB/s")
    print(f"  Time per token: {time_per_token_ms:.1f} ms")
    print(f"  Tokens/sec: {tokens_per_sec:.0f}")
    print()

# Apple M2 Pro: 200 GB/s unified memory bandwidth
print("=== Apple M2 Pro (200 GB/s) ===")
estimate_latency("Llama-7B Q4", 7000, 4, 15.8, 200)
estimate_latency("Llama-7B Q8", 7000, 8, 15.8, 200)
estimate_latency("Llama-70B Q4", 70000, 4, 15.8, 200)

# Phone (Snapdragon 8 Gen 3): ~50 GB/s LPDDR5
print("=== Snapdragon 8 Gen 3 (50 GB/s) ===")
estimate_latency("Phi-3 Mini Q4", 3800, 4, 45, 50)
estimate_latency("Llama-3B Q4", 3000, 4, 45, 50)
```
