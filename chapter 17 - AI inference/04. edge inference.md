# Edge Inference

*Edge inference (инференс на периферийных устройствах) запускает модели на пользовательских устройствах (телефонах, ноутбуках, IoT-датчиках) без отправки данных в облако. В этом файле рассматриваются ограничения периферийных устройств, пайплайн сжатия моделей, рантаймы для устройств, стек компиляторов, целевое аппаратное обеспечение (NPU, Neural Engines), LLM на устройствах, федеративное обучение и оптимизация задержки.*

- Облачный инференс требует сетевого подключения, увеличивает задержку (круговой путь 50–200 мс), стоит денег за каждый запрос и отправляет пользовательские данные на сторонние серверы. **Edge inference** устраняет все четыре проблемы: модель работает локально, отвечает мгновенно, не требует оплаты за инференс и сохраняет конфиденциальность данных.

- Компромисс: периферийные устройства обладают в 100–1000 раз меньшими вычислительными мощностями и объемом памяти, чем серверные GPU. Чтобы модели работали в рамках этих ограничений, требуется агрессивная оптимизация на всех уровнях.

- **Cactus** ([github.com/cactus-compute/cactus](https://github.com/cactus-compute/cactus)) — это низкозадержечный AI-движок, специально созданный для мобильных и носимых устройств. Он демонстрирует многие методы, описанные в этом файле, в продакшене: пользовательские ядра ARM SIMD для внимания и матричных операций (глава 16), квантование KV-кеша (глава 17, файл 01), блочный префилл (chunked prefill), инференс с ускорением NPU на чипах Apple и Qualcomm, отображение памяти с нулевым копированием (zero-copy memory mapping) для снижения использования RAM в 10 раз и автоматический переход на облако, когда вычислительных мощностей устройства недостаточно. Cactus поддерживает мультимодальный инференс (LLM, зрение, речь) на iOS, Android, macOS и встроенном Linux, с SDK для Swift, Kotlin, Python, Flutter, React Native и Rust. Его бенчмарки показывают декодирование 100 токенов/с на M4 Pro и 48 токенов/с на iPhone 17 Pro для модели 1.2B в формате INT4 — это конкретный пример того, как выглядит оптимизированный edge inference.

## Edge Constraints

| Ресурс | Облачный GPU (H100) | Ноутбук (M4) | Телефон (Snapdragon 8 Gen 3) | IoT (ESP32) |
|----------|-----------------|-------------|---------------------------|-------------|
| RAM | 80 GB HBM3 | 16-36 GB unified | 8-12 GB LPDDR5 | 520 KB |
| Вычисления | 989 TFLOPS (FP8) | 38 TOPS (Neural Engine) | 45 TOPS (NPU) | 0.001 TOPS |
| Энергопотребление | 700 W | 15-30 W | 5-10 W | 0.1 W |
| Хранилище | TB | 256 GB-2 TB | 128-512 GB | 4 MB |

- Разрыв в вычислительной мощности между облачным GPU и NPU телефона составляет ~20 раз. Между GPU и микроконтроллером — ~1 000 000 раз. Разным устройствам требуются разные уровни сжатия и разные архитектуры моделей.

## The Model Compression Pipeline

- Для развертывания на периферии сжатие — это не один метод, а **пайплайн** из взаимодополняющих методов, применяемых последовательно:

```
Full model (FP32, 70B params)
    ↓ Knowledge distillation → smaller model (7B params)
    ↓ Structured pruning → remove redundant heads/layers (4B effective)
    ↓ Quantisation (INT4) → 4x smaller (2 GB)
    ↓ Compiler optimisation → fused kernels, optimised memory layout
    ↓ Runtime → on-device execution
```

- Каждый шаг уменьшает размер и задержку. Порядок имеет значение: сначала дистилляция (уменьшение архитектуры), затем прунинг (удаление структуры), затем квантование (снижение точности), затем компиляция (оптимизация под целевое оборудование). Дистилляция после квантования привела бы к попытке сжать уже модель с потерями.

## On-Device Runtimes

- **Рантайм** загружает модель, выделяет память и выполняет инференс на целевом оборудовании. У каждой платформы есть свой предпочтительный рантайм:

- **ONNX Runtime**: кроссплатформенный (Windows, Linux, macOS, iOS, Android). Поддерживает CPU, GPU (CUDA, DirectML, CoreML, NNAPI) и множество бэкендов ускорителей. Самый портативный вариант. Модели экспортируются в формат ONNX из PyTorch/TensorFlow.

- **TensorFlow Lite (TFLite)**: рантайм Google для периферийных устройств. Оптимизирован для ARM CPU и Android NPU. Крошечный бинарный файл (~1 МБ). Поддерживает INT8 и float16. Стандарт для развертывания на Android.

- **Core ML**: рантайм Apple для iOS/macOS. Автоматически использует Neural Engine, GPU или CPU в зависимости от характеристик модели. Модели конвертируются из PyTorch/TensorFlow с помощью `coremltools`. Тесная интеграция с оборудованием Apple (объединенная память, Neural Engine).

- **ExecuTorch**: новый рантайм Meta для PyTorch на устройствах. Разработан для развертывания на периферии с предварительной компиляцией (ahead-of-time) и делегированием операций на аппаратные ускорители. Преемник PyTorch Mobile.

- **TensorRT**: рантайм NVIDIA для оптимизации инференса на GPU (глава 15). Объединяет слои, выбирает оптимальные ядра и автоматически квантует. В 2–5 раз быстрее, чем режим eager в PyTorch на GPU NVIDIA.

- **llama.cpp**: однофайловый C++ движок инференса для LLM. Поддерживает квантование GGUF (Q4, Q5, Q8), CPU (AVX/NEON), Metal (Apple GPU), CUDA и Vulkan. Основной инструмент для запуска LLM на потребительском оборудовании.

## The Compiler Stack

- Между высокоуровневой моделью (граф PyTorch) и оборудованием (инструкции NPU) находится **стек компилятора**, который оптимизирует модель под конкретную цель:

```
PyTorch model
    ↓ Export (torch.export, ONNX, TorchScript)
Graph IR (intermediate representation)
    ↓ Graph optimisations
        - Constant folding (compute constant expressions at compile time)
        - Dead code elimination (remove unused operations)
        - Operator fusion (conv + bn + relu → single fused op)
        - Layout transformation (NCHW → NHWC for ARM, channels-last)
    ↓ Lowering
Hardware-specific IR
    ↓ Backend optimisations
        - Tiling and loop ordering (cache-friendly access patterns)
        - Vectorisation (SIMD, chapter 16)
        - Memory planning (reuse buffers to minimise peak memory)
        - Kernel selection (choose the best implementation per op)
    ↓ Code generation
Machine code / NPU instructions
```

- **Слияние операторов (operator fusion)** — самая эффективная оптимизация. Блок трансформера содержит ~20 операций (matmul, add, layernorm, softmax и т. д.). Без слияния каждая из них записывает свой вывод в память, а следующая считывает его обратно. При слиянии несколько операций объединяются в одно ядро, которое хранит данные в регистрах/кеше. Это может быть в 2–5 раз быстрее (глава 16, модель roofline).

- **Планирование памяти**: компилятор анализирует граф модели, чтобы определить, какие тензоры перекрываются по времени жизни и могут использовать один и тот же буфер памяти. Модели со 100 промежуточными тензорами может потребоваться память только для 10, так как большинство из них потребляются и освобождаются до того, как создаются другие. Это критически важно на устройствах с ограниченным объемом RAM.

## Целевое аппаратное обеспечение

### Мобильные GPU

- **Qualcomm Adreno** (Android): поддерживает OpenCL, Vulkan compute (глава 16) и проприетарный движок Qualcomm SNPE (Snapdragon Neural Processing Engine). GPU Adreno имеют 256–1024 АЛУ с поддержкой FP16 и INT8.

- **ARM Mali** (Android): поддерживает OpenCL и Vulkan. GPU Mali используют тайловую архитектуру (отличную от десктопных GPU), что влияет на оптимальные паттерны доступа к памяти.

- **Apple GPU** (iOS/macOS): доступ осуществляется через Metal (графический API Apple). Архитектура унифицированной памяти означает отсутствие накладных расходов на копирование CPU↔GPU. Metal Performance Shaders (MPS) предоставляют оптимизированные примитивы машинного обучения.

### Нейронные процессоры (NPU)

- NPU — это специализированные ускорители с фиксированной логикой, разработанные специально для инференса моделей машинного обучения. Они гораздо энергоэффективнее GPU при выполнении стандартных операций машинного обучения (матричное умножение, свёртка, активация).

- **Apple Neural Engine**: 16 ядер, ~38 TOPS (INT8). Доступ осуществляется через Core ML. Отлично подходит для моделей компьютерного зрения и диффузионных моделей на устройстве. Не может выполнять произвольный код — только операции, поддерживаемые Core ML.

- **Qualcomm Hexagon NPU**: интегрирован в SoC Snapdragon. Поддерживает инференс в форматах INT8 и INT4. Доступ осуществляется через SNPE или ONNX Runtime с бэкендом QNN. Обеспечивает работу функций на устройстве, таких как размытие фона, распознавание речи и перевод в реальном времени.

- **Google Edge TPU**: компактная, энергоэффективная версия облачного TPU. 4 TOPS, 2 Вт. Используется в устройствах Coral для инференса на устройстве. Поддерживает только модели TFLite, квантованные до INT8.

- **Паттерн делегирования**: среда выполнения разделяет граф модели между NPU (для поддерживаемых операций) и CPU (для неподдерживаемых). Максимизация доли операций, выполняемых на NPU, является ключом к производительности и энергоэффективности.

## LLM на устройствах

- Запуск LLM на телефонах и ноутбуках стал возможен благодаря небольшим моделям и агрессивному квантованию:

| Модель | Параметры | Размер (квантованный) | Целевое устройство | Производительность |
|-------|--------|---------------|---------------|-------------|
| Phi-3 Mini | 3.8 млрд | ~2 ГБ (Q4) | Телефон/Ноутбук | ~15 токенов/с на iPhone 15 |
| Gemma 2B | 2 млрд | ~1.5 ГБ (Q4) | Телефон | ~20 токенов/с на Pixel 8 |
| Llama 3.2 1B | 1 млрд | ~700 МБ (Q4) | Телефон | ~30 токенов/с |
| Llama 3.2 3B | 3 млрд | ~2 ГБ (Q4) | Телефон/Ноутбук | ~15 токенов/с |
| Llama 3.1 8B | 8 млрд | ~4.5 ГБ (Q4) | Ноутбук | ~20 токенов/с на M2 |

- **Проблемы**:
    - **Память**: модель 3B Q4 занимает 2 ГБ, но KV-кеш для длинных диалогов значительно увеличивает потребление. Длина контекста на телефонах обычно ограничена 2–4 тыс. токенов.
    - **Троттлинг**: длительный инференс нагревает телефон. После 30 секунд непрерывной генерации SoC снижает тактовую частоту во избежание перегрева, что уменьшает производительность на 30–50%.
    - **Аккумулятор**: запуск модели 3B со скоростью 15 токенов/с потребляет ~3–5 Вт. 30-минутный диалог расходует ~5% заряда типичного аккумулятора телефона. Это приемлемо для эпизодического использования, но проблематично для постоянно работающих приложений.

- **llama.cpp** — стандарт для запуска LLM на устройствах. Работает на CPU (AVX2, NEON, I8MM), Apple GPU (Metal), NVIDIA GPU (CUDA), AMD GPU (ROCm/Vulkan) и даже на телефонах (через Termux на Android).

## Федеративное обучение

- **Федеративное обучение** (federated learning) позволяет обучать модели на множестве устройств без централизации данных. Каждое устройство обучается на своих локальных данных, вычисляет градиентное обновление и отправляет только это обновление (а не данные) на центральный сервер, который агрегирует полученные результаты.

- **Алгоритм** (FedAvg):
    1. Сервер отправляет текущую модель на $K$ выбранных устройств.
    2. Каждое устройство дообучает модель на своих локальных данных в течение нескольких шагов.
    3. Каждое устройство отправляет обновленную модель (или разницу) обратно на сервер.
    4. Сервер усредняет обновления: $W_{\text{new}} = \frac{1}{K} \sum_{k=1}^{K} W_k$.
    5. Повторить.

- **Конфиденциальность**: исходные данные никогда не покидают устройство. Сервер видит только агрегированные обновления модели. **Дифференциальная приватность** добавляет шум к обновлениям, чтобы отдельные точки данных нельзя было восстановить по градиенту.

- **Эффективность коммуникации**: обновления модели имеют большой размер (равный размеру самой модели). Методы сжатия уменьшают его: **квантование градиентов** (отправка градиентов в INT8 вместо FP32), **разреживание** (отправка только самых больших градиентов) и **накопление градиентов** (выполнение большего числа локальных шагов при более редкой отправке).

- **Приложения**: предиктивный ввод текста в клавиатурах (Gboard), распознавание голоса от Apple, мониторинг здоровья (обучение на чувствительных медицинских данных без их централизации).

## Оптимизация задержки

- Помимо сжатия, существует несколько методов уменьшения задержки инференса:

- **Ранний выход** (early exit): добавление классификационных «голов» к промежуточным слоям. Если модель уверена в ответе на 6-м слое (из 24), результат возвращается без выполнения слоев 7–24. Простые входные данные обрабатываются быстро, сложные — всей моделью. Средняя задержка значительно снижается для задач со смесью простых и сложных входных данных.

- **Разделение модели**: распределение модели между NPU (эффективен для матричного умножения), GPU (эффективен для нерегулярных операций) и CPU (обрабатывает всё остальное). Компилятор решает, какие операции где выполнять, основываясь на профилировании.

- **Кеширование**: для приложений с повторяющимися запросами (автодополнение текста, кода) кешируются недавние вычисления. Если пользователь вводит «Как мне», а модель недавно генерировала варианты для «Как мне», можно повторно использовать KV-кеш, полностью пропустив фазу префилла.

- **Спекулятивное предсказание**: предсказание действий пользователя и запуск инференса до того, как он отправит запрос. Чат-бот может начать генерировать ответ на вероятный уточняющий вопрос, пока пользователь еще читает текущий ответ.

## Задачи по программированию (используйте CoLab или ноутбук)

1. Смоделируйте пайплайн сжатия модели. Начните с модели в формате float32, примените дистилляцию (имитацию), прунинг и квантование, отслеживая размер на каждом этапе.
```python
def compression_pipeline(original_params_M, original_bits=32):
    size_mb = original_params_M * 1e6 * original_bits / 8 / 1e6

    print(f"Original: {original_params_M}M params, {original_bits}-bit → {size_mb:.0f} MB")

    # Step 1: Knowledge distillation (reduce params)
    distilled_params = original_params_M * 0.15  # 70B → ~10B equivalent
    size_mb = distilled_params * 1e6 * original_bits / 8 / 1e6
    print(f"After distillation ({distilled_params:.0f}M params): {size_mb:.0f} MB")

    # Step 2: Structured pruning (remove 30% of remaining)
    pruned_params = distilled_params * 0.7
    size_mb = pruned_params * 1e6 * original_bits / 8 / 1e6
    print(f"After pruning ({pruned_params:.0f}M params): {size_mb:.0f} MB")

    # Step 3: INT4 quantisation
    size_mb = pruned_params * 1e6 * 4 / 8 / 1e6
    print(f"After INT4 quantisation: {size_mb:.0f} MB")

    print(f"Total compression: {original_params_M * 1e6 * original_bits / 8 / 1e6 / size_mb:.0f}x")

print("=== Starting from 70B model ===")
compression_pipeline(70000)

print("\n=== Starting from 7B model ===")
compression_pipeline(7000)
```

def estimate_latency(model_name, params_M, bits, compute_tops, mem_bw_gbs, seq_len=256):
    """Estimate token generation latency for a memory-bandwidth-bound model."""
    # Model size in bytes
    model_bytes = params_M * 1e6 * bits / 8

    time_per_token_ms = model_bytes / (mem_bw_gbs * 1e9) * 1000
