# Проектирование ML-систем

*Проектирование ML-систем применяет инфраструктурные паттерны из глав 01-03 к специфическим задачам машинного обучения. В этом файле рассматриваются жизненный цикл ML, управление данными, инфраструктура обучения, оценка моделей, стратегии инференса, проектирование признаков, ML-пайплайны и мониторинг.*

- Вопрос на собеседовании по проектированию систем, например «Спроектируйте рекомендательную систему для YouTube», не требует от вас описания алгоритма рекомендаций. Он требует спроектировать **всю систему целиком**: пайплайны данных, проектирование признаков, обучение моделей, оценку, инференс, мониторинг и итерации. Этот файл предоставляет необходимый фреймворк.

## Жизненный цикл ML-системы

![ML system lifecycle: problem framing through deployment and monitoring, with continuous iteration](../images/ml_lifecycle.svg)

- Каждая ML-система проходит через один и тот же жизненный цикл, будь то классификатор спама или фундаментальная модель:

```
Problem Framing → Data → Features → Training → Evaluation → Deployment → Monitoring → Iteration
       ↑                                                                                    │
       └────────────────────────────────────────────────────────────────────────────────────┘
```

### Постановка задачи

- Прежде чем приступать к данным или моделям, определите:
    - **Что** вы предсказываете? (вероятность клика, следующий токен, ограничивающая рамка объекта)
    - **Кто** является пользователями? (конечные пользователи, внутренние аналитики, другие ML-модели)
    - **Каковы ограничения?** (задержка < 100мс, допустим офлайн-батч, должно работать на устройстве)
    - **Какова бизнес-метрика?** (выручка, вовлеченность, точность) и как ML-метрика связана с ней?
    - **Каково базовое решение (baseline)?** (эвристика, система на основе правил, существующая модель) — вы должны превзойти его, чтобы оправдать использование ML-системы.

- **Распространенная ошибка**: переход к архитектуре модели до понимания проблемы. Фраза «Мы должны использовать трансформер» не является ответом в проектировании систем. Ответ «Нам нужно предсказывать вероятность клика для 10 млн кандидатов в течение 200 мс, поэтому нам нужна двухэтапная система: быстрый поиск (retrieval), а затем небольшая модель ранжирования» — является.

## Управление данными

### Сбор данных и разметка

- **Явные метки**: люди размечают данные (клик/нет клика, ограничивающие рамки объектов, оценки качества диалога). Это дорого (~$0.02-$10 за метку в зависимости от сложности), медленно и субъективно.

- **Неявные метки**: получение меток из поведения пользователей. Клики, время просмотра, покупки, пропуски. Дешево и в больших объемах, но зашумлено (клик не означает удовлетворенность; пропуск не означает неприязнь).

- **Программная разметка** (Snorkel): написание функций разметки (эвристики, регулярные выражения, существующие модели), которые голосуют за каждый пример. Статистическое агрегирование голосов для получения вероятностных меток. Масштабируется до миллионов примеров с умеренной точностью.

- **Активное обучение**: модель определяет примеры, в которых она наиболее не уверена, и запрашивает для них разметку человеком. Это максимизирует эффективность разметки: 1000 меток, выбранных активно, могут быть эквивалентны 10 000 случайных меток.

### Качество данных

- **Валидация данных**: проверка каждого батча входящих данных на нарушение схемы (отсутствующие поля, неверные типы), сдвиги распределения (средние значения значительно изменились) и аномалии объема (ожидался 1 млн строк, получено 500 тыс.).

- **Great Expectations** и **TFX Data Validation** — это инструменты, которые определяют ожидания (expectations) относительно данных и оповещают при их нарушении.

- **Версионирование данных**: каждый запуск обучения должен быть воспроизводимым. **DVC** (глава 15) отслеживает файлы данных вместе с кодом. Каждая версия датасета получает хеш; конфигурация обучения ссылается на этот хеш.

### Хранилища признаков (Feature Stores)

![Feature store: same feature computation feeds both the offline store (for training) and online store (for serving), preventing training-serving skew](../images/feature_store.svg)

- **Хранилища признаков** (глава 15) обеспечивают согласованность признаков для обучения и инференса. Ключевые концепции:

    - **Офлайн-признаки**: вычисляются с помощью пакетных пайплайнов (Spark), хранятся в хранилищах данных. Используются во время обучения и для пакетного инференса. Примеры: средняя длительность сессии пользователя за 30 дней, общее количество покупок товара.

    - **Онлайн-признаки**: вычисляются в реальном времени или предварительно вычисляются и подаются из низкозадержечного хранилища (Redis, DynamoDB). Используются во время инференса в реальном времени. Примеры: последние 5 действий пользователя, текущее содержимое корзины.

    - **Training-serving skew (расхождение между обучением и инференсом)**: если вычисление признаков различается при обучении и инференсе, модель видит при инференсе другие значения признаков, чем те, на которых она обучалась. Хранилища признаков устраняют это, используя одни и те же вычисления для обоих случаев.

## Инфраструктура обучения

- Для аудитории этой книги распределенное обучение было подробно рассмотрено в главе 6 (параллелизм данных, параллелизм моделей, смешанная точность, законы масштабирования). Здесь мы сосредоточимся на **системных** аспектах:

- **Отслеживание экспериментов** (W&B, MLflow — глава 15): каждый запуск обучения логирует гиперпараметры, метрики, git-коммит, версию данных и оборудование. Это ML-эквивалент контроля версий для моделей.

- **Настройка гиперпараметров**: автоматизированный поиск по гиперпараметрам. Методы: сеточный поиск (исчерпывающий, дорогой), случайный поиск (удивительно эффективен), байесовская оптимизация (моделирование целевой функции, выборка там, где вероятно улучшение) и **ASHA** (Asynchronous Successive Halving: запуск множества попыток, досрочное прекращение неэффективных).

- **Оркестрация пайплайнов обучения** (Airflow, Kubeflow — глава 15): автоматизация последовательности: подготовка данных → обучение → оценка → регистрация. Планирование ежедневного переобучения. Оповещение об ошибках.

## Оценка моделей

### Офлайн-оценка

- **Отложенная тестовая выборка**: оценка на данных, которые модель никогда не видела во время обучения. Стандартный подход, но может вводить в заблуждение, если тестовая выборка не представляет данные из продакшена.

- **Оценка по срезам (slice-based evaluation)**: оценка на подгруппах (по демографии пользователей, типу контента, языку, периоду времени). Модель с 95% общей точности может иметь 70% точности для конкретной малочисленной группы — это неприемлемо.

- **Бэктестинг**: для временных рядов или последовательных предсказаний — оценка на исторических данных в хронологическом порядке. Обучение на данных до момента $t$, оценка на данных от $t$ до $t + \Delta t$. Позволяет избежать утечки данных, возникающей при использовании будущих данных для обучения.

### Онлайн-оценка

![A/B-тестирование: случайное разделение пользователей на контрольную (старая модель) и экспериментальную (новая модель) группы, сравнение метрик со статистической значимостью](../images/ab_testing.svg)

- **A/B-тестирование**: случайное разделение реального трафика на контрольную (старая модель) и экспериментальную (новая модель) группы. Сравнение бизнес-метрик (выручка, вовлеченность, удержание) со статистической значимостью. Это «золотой стандарт» оценки изменений в ML.

    - **Размер выборки**: вам нужно достаточно данных, чтобы обнаружить ожидаемый размер эффекта. Улучшение показателя кликабельности (CTR) на 0,1% требует миллионов показов для статистически значимого подтверждения.

    - **Длительность**: проводите тест как минимум в течение одного полного цикла (1–2 недели для большинства продуктов), чтобы учесть влияние дней недели.

    - **Защитные метрики (guardrail metrics)**: отслеживайте метрики, которые НЕ должны меняться (время загрузки страницы, частота ошибок, частота сбоев), наряду с целевой метрикой. Модель, которая увеличивает выручку, но при этом вызывает рост числа сбоев, в конечном итоге вредна.

- **Теневое развертывание (shadow deployment)**: запуск новой модели параллельно со старой в продакшене. Обе получают одни и те же запросы, но пользователям возвращаются только предсказания старой модели. Сравнение результатов позволяет выявить баги и проблемы с качеством без риска для пользователей.

- **Интерливинг (interleaving)**: для задач ранжирования — объединение результатов старой и новой моделей в единый список. Пользователи взаимодействуют с этим списком, а вы измеряете, результаты какой модели получают больше вовлеченности. Требует меньше пользователей для достижения статистической значимости, чем A/B-тестирование.

## Инференс моделей

### Пакетный (batch) и потоковый (real-time) инференс

- **Пакетный инференс (batch inference)**: предварительное вычисление предсказаний для всех возможных входных данных. Сохранение в базе данных или кеше. Выдача результатов из кеша. Подходит, когда: пространство входных данных конечно (рекомендации для всех пользователей каждую ночь), свежесть данных не критична (ежедневных предсказаний достаточно), а требования к задержке (latency) невысоки.

- **Потоковый инференс (real-time inference)**: вычисление предсказаний по запросу для каждого обращения. Подходит, когда: пространство входных данных бесконечно (любой запрос пользователя), свежесть данных важна (предсказание для конкретного запроса прямо сейчас), а задержка должна быть минимальной.

- Многие системы используют **оба подхода**: пакетный инференс предварительно вычисляет набор кандидатов (дешево, покрывает 80% трафика), а потоковый обрабатывает остальное (дорого, покрывает редкие запросы и новых пользователей).

### Версионирование моделей и реестр

- **Реестр моделей (model registry)** (MLflow, W&B, SageMaker) хранит обученные модели вместе с метаданными:
    - Номер версии и дата обучения.
    - Конфигурация обучения и версия данных.
    - Метрики оценки (точность, задержка, использование памяти).
    - Стадия: разработка → стейджинг → продакшн → архив.

- **Откат (rollback)**: если новая модель ухудшает метрики в продакшене, немедленно вернитесь к предыдущей версии. Реестр позволяет сделать это в один клик.

## Проектирование признаков (feature engineering)

- **Проектирование признаков** преобразует «сырые» данные в признаки, необходимые модели. Это часто самая эффективная деятельность в ML: качественные признаки улучшают любую модель, в то время как возможности даже самых лучших моделей ограничены качеством подаваемых на вход признаков.

### Онлайн- и офлайн-признаки

- **Офлайн-признаки** вычисляются заранее и меняются медленно (демография пользователей, агрегаты за 30 дней). Вычисляются пакетными пайплайнами (Spark) и хранятся в хранилище признаков (feature store).

- **Онлайн-признаки** отражают текущее состояние и меняются быстро (товары в корзине, последнее действие, текущее местоположение). Вычисляются в реальном времени из потоков событий или извлекаются из быстрого хранилища.

- **Свежесть признаков**: некоторые признаки должны быть актуальны с точностью до секунд (обнаружение мошенничества: является ли эта транзакция аномальной с учетом последних 5 транзакций?). Другие могут быть устаревшими на несколько часов (рекомендации: какие жанры предпочитает пользователь на основе его истории?). Более свежие признаки дороже в вычислении и хранении.

### Распространенные паттерны признаков

- **Признаки-счетчики**: количество событий за временное окно (покупки за последние 7 дней, входы в систему за последние 24 часа).
- **Эмбеддинги**: обученные эмбеддинги для категориальных переменных (эмбеддинг пользователя, эмбеддинг товара, эмбеддинг запроса). Они являются входными данными для двухбашенных (two-tower) моделей и подобных архитектур.
- **Кросс-признаки**: комбинации двух и более признаков (user_age × item_category). Позволяют уловить взаимодействия, которые не видны при анализе признаков по отдельности.
- **Временные признаки**: время с момента последнего действия, день недели, час дня. Позволяют уловить временные закономерности.
- **Агрегированные признаки**: среднее, медиана, минимум, максимум, стандартное отклонение числового признака по группе (средний рейтинг товаров этого продавца).

## ML-пайплайны

- ML-пайплайн организует весь рабочий процесс от данных до развернутой модели:

```
Data ingestion → Validation → Feature engineering → Training → Evaluation → Registration → Deployment → Monitoring
```

- Каждый шаг является задачей в оркестраторе (Airflow, Kubeflow, Metaflow — глава 15). Пайплайн:
    - Запускается по расписанию (ежедневное переобучение) или по триггеру (при появлении новых данных).
    - Идемпотентен (повторный запуск дает тот же результат).
    - Имеет логику повторных попыток (если вычисление признаков не удалось, повторить 3 раза с экспоненциальной задержкой).
    - Создает артефакты (обученная модель, отчет об оценке, статистика признаков), которые версионируются и сохраняются.

- **Metaflow** (Netflix/Outerbounds) особенно хорошо подходит для ML: он версионирует код, данные и модели вместе, поддерживает локальную разработку и облачное выполнение с использованием одного и того же кода, а также интегрируется с K8s и AWS.

## Мониторинг

- Мы рассмотрели основы мониторинга в главе 15 (Prometheus, Grafana, алерты). Здесь мы сосредоточимся на **мониторинге, специфичном для ML**:

### Дрейф данных (data drift)

- **Дрейф данных** возникает, когда распределение входящих данных меняется относительно обучающей выборки. Модель, обученная на летних данных, может плохо работать на зимних (другое поведение пользователей, другая доступность товаров).

- **Обнаружение**: сравнение распределения входящих признаков с распределением обучающей выборки с помощью статистических тестов:
    - **Критерий Колмогорова-Смирнова (KS test)**: сравнивает два эмпирических распределения. Проверяет, взяты ли они из одного и того же базового распределения.
    - **PSI (Population Stability Index)**: измеряет степень смещения распределения. PSI < 0,1 — стабильно, 0,1–0,25 — умеренное смещение, > 0,25 — значительное смещение.
    - **Дрейф эмбеддингов**: сравнение распределения эмбеддингов входящих запросов с обучающей выборкой с использованием расстояния между центроидами или MMD (Maximum Mean Discrepancy).

### Дрейф концепции (Concept Drift)

- **Дрейф концепции** возникает, когда меняется взаимосвязь между входными данными и выходными результатами. Признаки выглядят так же, как и раньше, но правильный прогноз становится другим. Пример: предпочтения пользователей меняются после культурного события, пандемии или изменения продукта.

- Дрейф концепции сложнее обнаружить, чем дрейф данных, так как для этого требуются размеченные данные. Отслеживайте прокси-метрики: показатель кликабельности (CTR), коэффициент конверсии, оценки удовлетворенности пользователей. Устойчивое снижение этих показателей указывает на дрейф концепции.

### Деградация модели

- Модели со временем деградируют по ряду причин: дрейф данных, дрейф концепции, ошибки в пайплайне признаков (признак начинает возвращать `null`) и изменения во входящих данных (сторонний API меняет формат ответа).

- **Реагирование**: при обнаружении деградации действия зависят от серьезности проблемы:
    - Незначительная: переобучение на свежих данных (это решается плановым переобучением).
    - Умеренная: исследование первопричины (какой признак изменился? какой сегмент пользователей затронут?).
    - Серьезная: немедленный откат к предыдущей версии модели, затем исследование.

### Петли обратной связи

- Системы машинного обучения создают **петли обратной связи**: прогнозы модели влияют на поведение пользователей, которое затем становится обучающей выборкой для следующей версии модели. Эти петли могут быть как полезными, так и вредными.

- **Положительная петля обратной связи** (опасная): рекомендательная модель показывает в основном популярные объекты → пользователи кликают на популярные объекты (потому что видят только их) → модель узнает, что популярные объекты стали еще популярнее → разнообразие падает. Модель создает данные, которые подтверждают ее собственные искажения.

- **Отрицательная петля обратной связи** (также опасная): модель детекции мошенничества ловит все мошенничество типа А → мошенничество типа А не попадает в обучающую выборку → следующая модель не учится распознавать тип А → мошенничество типа А появляется снова.

- **Методы смягчения**:
    - **Исследование (Exploration)**: показывайте некоторые объекты, в которых модель не уверена (epsilon-greedy, сэмплирование Томпсона). Это генерирует разнообразные обучающие данные.
    - **Контрфактическое логирование**: записывайте то, что модель *предсказала бы*, а не только то, что увидел пользователь. Обучайтесь на контрфактических данных для устранения искажений.
    - **Отложенные выборки (Holdout sets)**: случайно выбирайте часть трафика для обслуживания без фильтрации моделью. Неотфильтрованные данные предоставляют эталонную разметку для оценки качества модели.
    - **Отложенные метки**: дождитесь истинного результата, прежде чем использовать данные для обучения. Рекомендация, на которую кликнули сегодня, может вызвать сожаление завтра. Прогноз мошенничества должен дождаться периода чарджбэка (30–90 дней).

### Управление таблицами эмбеддингов

- Крупномасштабные системы машинного обучения часто имеют таблицы эмбеддингов с более чем 100 млн записей (по одному эмбеддингу на пользователя, объект, объявление или сущность). Управление ими в масштабе — это системная задача:

- **Хранение**: 100 млн сущностей × 256-размерность × `float16` = 50 ГБ. Это не помещается в память GPU. Решения: хранение в памяти CPU с кешированием на стороне GPU, шардирование по нескольким машинам или использование **хеш-эмбеддингов** (хеширование сущностей в таблицу фиксированного размера с допущением коллизий).

- **Обновления**: эмбеддинги меняются по мере переобучения модели. Развертывание новой таблицы эмбеддингов для инференса требует: загрузки 50 ГБ в память без прерывания текущего трафика, проверки корректности и отката в случае деградации метрик. Используйте blue-green развертывание для таблиц эмбеддингов.

- **Устаревание**: у недавно созданного пользователя нет эмбеддинга (проблема «холодного старта»). Решения: использование эмбеддинга по умолчанию, получение эмбеддинга из признаков пользователя через модель «признаки-в-эмбеддинг» или переход к неперсонализированной модели.

### Справедливость и предвзятость

- Системы машинного обучения могут систематически относиться к разным группам по-разному, часто отражая предвзятость в обучающих данных. **Мониторинг справедливости** — это обязанность, а не опциональная функция.

- **Метрики для мониторинга**:
    - **Демографический паритет**: различается ли доля положительных прогнозов между группами (пол, этническая принадлежность, возраст)?
    - **Равенство возможностей**: различается ли доля истинно положительных результатов (TPR) между группами? (Модель для найма должна одинаково хорошо определять квалифицированных кандидатов из всех групп.)
    - **Калибровка**: если модель говорит $P(\text{qualified}) = 0.7$ для группы А, действительно ли 70% группы А квалифицированы? И верно ли это для группы B?

- **Практические шаги**:
    - Оценивайте производительность модели на срезах (подгруппах), а не только по общим метрикам.
    - Включайте метрики справедливости в пайплайн оценки модели (модель, которая улучшает общую точность, но ухудшает показатели для конкретной группы, не должна развертываться без проверки).
    - Документируйте известные ограничения и режимы сбоев.
    - Установите процесс проверки для моделей, развертываемых в чувствительных областях (найм, кредитование, уголовное правосудие, здравоохранение).
