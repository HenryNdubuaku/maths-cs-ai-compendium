# Основы проектирования систем

*Проектирование систем — это то, как вы создаете программное обеспечение, которое надежно работает при масштабировании. Этот файл охватывает архитектуру клиент-сервер, сетевые протоколы, DNS, прокси, балансировку нагрузки, кеширование, базы данных, очереди сообщений, модели согласованности и паттерны отказоустойчивости.*

- Каждая система машинного обучения в продакшене — это распределенная система. Система рекомендаций — это не просто модель, это API-сервер, хранилище признаков (feature store), реестр моделей, слой кеширования, очередь сообщений и стек мониторинга, которые взаимодействуют по сети. Понимание проектирования систем — это то, что отличает «я обучил модель» от «я создал продукт».

- Собеседования по проектированию систем в ведущих технологических компаниях (Google, Meta, Amazon, OpenAI) проверяют, умеете ли вы проектировать такие системы. Эта глава дает вам базовые блоки (данный файл), облачную инфраструктуру (файл 02), паттерны масштабирования (файл 03), проектирование специфичных для ML систем (файл 04) и практические примеры (файл 05).

## Архитектура клиент-сервер

- Фундаментальный паттерн: **клиент** отправляет запрос, **сервер** обрабатывает его и возвращает ответ. Ваш браузер (клиент) отправляет HTTP-запрос на google.com (сервер), который возвращает HTML.

- **Модель запрос-ответ**: синхронная. Клиент ожидает ответа. Это просто, но создает «узкое место»: клиент простаивает в ожидании, а сервер должен обработать запрос, прежде чем переходить к следующему.

- **Серверы без сохранения состояния (stateless)**: сервер не запоминает предыдущие запросы. Каждый запрос содержит всю информацию, необходимую для его обработки. Это упрощает масштабирование: любой сервер может обработать любой запрос, поэтому вы можете добавлять больше серверов за балансировщиком нагрузки.

- **Серверы с сохранением состояния (stateful)**: сервер поддерживает состояние между запросами (например, сессию пользователя). Их сложнее масштабировать, так как запросы от одного и того же пользователя должны попадать на один и тот же сервер (привязка к сессии). Современные системы избегают хранения состояния на стороне сервера, сохраняя его в базе данных или кеше (Redis).

## Сетевые протоколы

- Мы рассматривали сети в главе 13 (уровни TCP/IP, сокеты). Здесь мы сосредоточимся на протоколах прикладного уровня, используемых при проектировании систем:

- **HTTP/HTTPS**: протокол интернета и большинства API. Методы запросов: GET (чтение), POST (создание/предсказание), PUT (обновление), DELETE (удаление). HTTPS добавляет TLS-шифрование (глава 13, безопасность). REST API (глава 15, файл 03) построены на HTTP.

- **WebSockets**: постоянное двунаправленное соединение между клиентом и сервером. В отличие от HTTP (запрос → ответ → соединение закрывается), WebSocket поддерживает соединение открытым для потоковой передачи в реальном времени. Используется для: потоковой передачи токенов LLM (отправка токенов по мере их генерации), живых дашбордов, чат-приложений.

- **gRPC**: RPC-фреймворк от Google. Использует Protocol Buffers (бинарная сериализация, примерно в 10 раз меньше и быстрее, чем JSON) поверх HTTP/2. Поддерживает потоковую передачу (на стороне сервера, на стороне клиента, двунаправленную). Используется для внутреннего взаимодействия между сервисами, где важна производительность. Triton Inference Server (глава 15) и TensorFlow Serving используют gRPC.

- **Protocol Buffers**: определяют схемы сообщений в файлах `.proto`:

```protobuf
message PredictRequest {
    repeated float features = 1;
    string model_version = 2;
}

message PredictResponse {
    float prediction = 1;
    float confidence = 2;
}

service ModelService {
    rpc Predict(PredictRequest) returns (PredictResponse);
}
```

- Схема компилируется в код клиента и сервера на любом языке (Python, C++, Go, Java). Безопасность типов, обратная совместимость и производительность предоставляются «из коробки».

## DNS

- **DNS** (Domain Name System) преобразует понятные человеку имена в IP-адреса (глава 13). Для проектирования систем DNS также обеспечивает:

- **Балансировку нагрузки через DNS**: возврат разных IP-адресов для одного и того же доменного имени, распределяя трафик между несколькими серверами. Это просто, но дает грубую точность (результаты DNS кешируются на время от минут до часов, поэтому трафик не перераспределяется быстро).

- **Географическую маршрутизацию**: возврат IP-адреса ближайшего дата-центра на основе местоположения клиента. Пользователь в Токио получает IP японского дата-центра, пользователь в Лондоне — европейского.

- **Отказоустойчивость (failover)**: если сервер выходит из строя, DNS перестает возвращать его IP. Новые клиенты направляются на исправные серверы. Однако из-за кеширования DNS-записей некоторые клиенты продолжают обращаться к неработающему серверу в течение нескольких минут (проблема TTL).

## Прокси

- **Прокси** — это посредник между клиентом и сервером:

- **Обратный прокси (reverse proxy)** (перед серверами): клиенты подключаются к прокси, который перенаправляет запросы на серверы бэкенда. Клиент не знает, какой именно сервер обработал запрос. **Nginx** и **HAProxy** — стандартные обратные прокси. Они обеспечивают: балансировку нагрузки (распределение запросов), SSL-терминацию (расшифровка HTTPS на прокси, отправка обычного HTTP на бэкенды), кеширование, ограничение частоты запросов (rate limiting) и сжатие.

- **API-шлюз (API gateway)**: специализированный обратный прокси для API. Обрабатывает аутентификацию, ограничение частоты запросов, маршрутизацию запросов (разные пути → разные сервисы) и версионирование API. **Kong**, **AWS API Gateway** и **Envoy** — популярные варианты.

- Для ML-инференса: API-шлюз располагается перед серверами моделей. Он проверяет ключи API, ограничивает частоту запросов для пользователей бесплатного тарифа, направляет `/v1/predict` на сервер модели A, а `/v2/predict` — на сервер модели B, и собирает метрики использования.

## Балансировка нагрузки

- Когда у вас есть несколько серверов, **балансировщик нагрузки** распределяет входящие запросы между ними.

![Балансировщик нагрузки распределяет входящие запросы между несколькими серверами бэкенда](../images/load_balancer.svg)

- **Алгоритмы**:
    - **Round robin**: отправка запросов на серверы по порядку (1, 2, 3, 1, 2, 3...). Просто, справедливо, но не учитывает нагрузку на сервер.
    - **Least connections**: отправка на сервер с наименьшим количеством активных соединений. Лучше подходит для запросов с переменным временем обработки (некоторые запросы к LLM генерируют 10 токенов, другие — 1000).
    - **Weighted round robin**: серверы с большей мощностью получают больше запросов. Сервер с 80 ГБ памяти GPU обрабатывает в 2 раза больше запросов, чем сервер с 40 ГБ.
    - **Consistent hashing**: хеширование ключа запроса для выбора конкретного сервера. Один и тот же ключ всегда попадает на один и тот же сервер. Полезно для: кеширования (запросы одного и того же пользователя попадают в один и тот же кеш), привязки к сессии и префиксного кеширования (глава 17: запросы с одинаковым системным промптом направляются на сервер, у которого есть KV-кеш для этого промпта).

- **Балансировка нагрузки L4 против L7**:
    - **L4** (транспортный уровень): маршрутизация на основе IP и порта. Работает быстро, но не может анализировать содержимое запроса.
    - **L7** (прикладной уровень): маршрутизация на основе HTTP-пути, заголовков или содержимого тела запроса. Позволяет направлять `/api/chat` на серверы чата, а `/api/embed` — на серверы эмбеддингов. Работает медленнее, но обладает большей гибкостью.

## Кэширование

- **Кэширование** — это хранение часто запрашиваемых данных в быстром слое памяти (RAM) для предотвращения их повторного вычисления или получения.

![Паттерн Cache-aside: сначала проверка кэша, при промахе — получение данных из базы и сохранение в кэш для последующего использования](../images/cache_aside_pattern.svg)

- **Паттерны кэширования**:
    - **Cache-aside** (ленивая загрузка): приложение сначала проверяет кэш. При промахе оно запрашивает данные из базы, сохраняет их в кэш и возвращает результат. Самый распространенный паттерн.
    - **Write-through**: каждая запись одновременно попадает и в кэш, и в базу данных. Гарантирует актуальность кэша, но замедляет операции записи.
    - **Write-back**: запись происходит только в кэш; кэш асинхронно сбрасывает данные в базу. Самый быстрый способ записи, но существует риск потери данных, если кэш выйдет из строя до завершения сброса.

- **Политики вытеснения** (когда кэш заполнен):
    - **LRU** (Least Recently Used): вытеснение элемента, к которому дольше всего не было обращений. Самая распространенная политика.
    - **LFU** (Least Frequently Used): вытеснение элемента, к которому обращались реже всего. Лучше подходит, если некоторые элементы стабильно популярны.
    - **TTL** (Time To Live): элементы удаляются по истечении фиксированного времени. Используется для данных, которые устаревают (предсказания модели кэшируются на 5 минут, значения признаков — на 1 час).

- **CDN** (Content Delivery Network): глобально распределенный кэш для статического контента (изображения, JavaScript, CSS). Серверы в более чем 100 точках по всему миру отдают кэшированный контент из ближайшей к пользователю локации. Для ML: веса моделей могут кэшироваться в CDN для быстрой загрузки.

- **Redis**: стандартный кэш/база данных в оперативной памяти. Поддерживает строки, списки, множества, сортированные множества, хеши и потоки. Задержка менее миллисекунды. Используется для: кэширования предсказаний моделей, хранения данных сессий, ограничения частоты запросов (подсчет запросов на пользователя в минуту) и обслуживания признаков в реальном времени.

- Для ML-инференса: кэшируйте предсказания для повторяющихся входных данных. Если многие пользователи спрашивают «Какова столица Франции?», вычислите ответ один раз и отдавайте кэшированный результат. Частота попаданий в кэш (cache hit rate) в 20–40% типична для чат-ботов, что пропорционально снижает затраты на GPU.

## Базы данных

### SQL (реляционные)

- **SQL-базы данных** (PostgreSQL, MySQL) хранят данные в таблицах со строками и столбцами. Связи между таблицами выражаются через внешние ключи. Запросы выполняются с помощью SQL. Гарантии **ACID**:

    - **Атомарность**: транзакция либо выполняется полностью, либо полностью откатывается. Частичные обновления невозможны.
    - **Согласованность**: база данных переходит из одного корректного состояния в другое. Ограничения (уникальные ключи, внешние ключи) всегда соблюдаются.
    - **Изоляция**: параллельные транзакции не влияют друг на друга.
    - **Долговечность**: зафиксированные данные сохраняются после сбоев (записываются на диск перед подтверждением).

- SQL-базы данных отлично подходят для: структурированных данных со связями, сложных запросов (соединения, агрегации), строгих требований к согласованности и целостности данных.

### NoSQL

- **NoSQL-базы данных** жертвуют некоторыми гарантиями ACID ради масштабируемости и гибкости:

    - **Key-value хранилища** (Redis, DynamoDB): простейшая модель. Быстрый поиск по ключу. Используются для кэширования, хранения сессий и хранилищ признаков.
    - **Документоориентированные хранилища** (MongoDB, Firestore): хранят JSON-подобные документы. Гибкая схема (каждый документ может иметь разные поля). Используются для профилей пользователей, каталогов товаров и конфигураций.
    - **Колоночные хранилища** (Cassandra, HBase): оптимизированы для интенсивной записи и временных рядов. Используются для логирования событий, метрик и аналитики.
    - **Графовые базы данных** (Neo4j): хранят узлы и ребра. Оптимизированы для запросов на обход графа. Используются для социальных сетей, графов знаний и рекомендательных систем.
    - **Векторные базы данных** (Pinecone, Milvus, Weaviate, FAISS): хранят высокоразмерные эмбеддинги и поддерживают поиск приближенных ближайших соседей (ANN). Необходимы для семантического поиска, RAG (поисково-генеративной генерации) и рекомендательных систем.

### Теорема CAP

- В распределенной базе данных можно обеспечить не более двух из трех свойств:

    - **Согласованность**: каждое чтение возвращает самую последнюю запись.
    - **Доступность**: каждый запрос получает ответ (даже если некоторые узлы недоступны).
    - **Устойчивость к разделению**: система продолжает работать, несмотря на сетевые разделения (когда узлы не могут обмениваться данными).

![Теорема CAP: выбирайте CP (согласованность) или AP (доступность), так как сетевые разделения неизбежны](../images/cap_theorem.svg)

- Поскольку сетевые разделения в распределенных системах неизбежны, реальный выбор стоит между **CP** (согласованность, но возможна недоступность во время разделения — например, PostgreSQL) и **AP** (доступность, но возможен возврат устаревших данных во время разделения — например, Cassandra, DynamoDB).

- Для ML: хранилища признаков обычно выбирают AP (немного устаревшее значение признака лучше, чем отсутствие предсказания). Реестры моделей выбирают CP (обслуживание неверной версии модели катастрофично).

### Шардирование

- **Шардирование** разделяет базу данных между несколькими машинами. Каждый шард содержит подмножество данных.

- **Хеш-шардирование**: хеширование ключа для определения шарда. `shard = hash(user_id) % num_shards`. Обеспечивает равномерное распределение, но делает невозможными диапазонные запросы.

- **Диапазонное шардирование**: каждый шард содержит диапазон ключей (пользователи A-G на шарде 1, H-N на шарде 2). Позволяет выполнять диапазонные запросы, но может создавать «горячие точки» (если у многих пользователей имена начинаются на «S»).

- **Проблема перешардирования**: добавление шарда делает хеш-отображение невалидным. **Консистентное хеширование** минимизирует перемещение данных: при добавлении n-го шарда нужно переместить только ~1/n ключей.

### Индексирование баз данных

- **Индекс** — это структура данных, которая ускоряет запросы ценой дополнительного места в хранилище и замедления записи. Без индекса запрос сканирует каждую строку (**O(n)**). С индексом он находит цель за **O(log n)**.

- **B-дерево (B-tree)** (по умолчанию): сбалансированное дерево, где каждый узел содержит несколько ключей и указателей. B-деревья эффективны для кэша (широкие узлы помещаются в строки кэша) и поддерживают диапазонные запросы (`WHERE age BETWEEN 20 AND 30`). Большинство SQL-баз данных используют B-деревья.

- **Хеш-индекс**: сопоставляет ключи с расположением строк с помощью хеш-функции. Поиск выполняется за $O(1)$, но не поддерживает диапазонные запросы. Используется для поиска по точному совпадению (`WHERE id = 12345`).

- **Составной индекс**: индекс по нескольким столбцам. `CREATE INDEX ON users(country, city)` ускоряет запросы с фильтрацией по стране или по стране + городу, но НЕ по одному городу (самый левый столбец должен присутствовать в запросе).

- **Компромисс**: каждый индекс ускоряет чтение, но замедляет запись (индекс должен обновляться при каждой операции вставки/обновления/удаления) и требует места для хранения (~10-30% от размера таблицы на каждый индекс). Не индексируйте всё подряд — индексируйте только те столбцы, по которым часто выполняются запросы.

- **Для систем машинного обучения**: онлайн-базе данных хранилища признаков (feature store) нужны индексы по ключам сущностей (user_id, item_id) для быстрого поиска признаков. Базе данных для отслеживания экспериментов нужны индексы по (experiment_id, metric_name) для запросов к дашбордам.

### Проектирование API

- Системы взаимодействуют через API. Хорошее проектирование API делает систему удобной, расширяемой и пригодной для отладки:

- **Соглашения REST**: используйте существительные для ресурсов (`/users`, `/models`), HTTP-методы для действий (GET = чтение, POST = создание, PUT = обновление, DELETE = удаление) и коды состояния для результатов (200 = OK, 201 = создано, 400 = неверный запрос, 404 = не найдено, 429 = превышен лимит запросов, 500 = ошибка сервера).

- **Пагинация**: для эндпоинтов, возвращающих списки, никогда не возвращайте все результаты сразу. Используйте пагинацию на основе курсора (`GET /items?cursor=abc&limit=50`) или на основе смещения (`GET /items?offset=100&limit=50`). Пагинация на основе курсора эффективнее для больших датасетов (пагинация со смещением требует пропуска строк).

- **Версионирование**: добавляйте версию в начало путей API (`/v1/predict`, `/v2/predict`). Это позволяет развивать API, не ломая существующие клиенты. Клиенты переходят на v2 в своем темпе; v1 считается устаревшей, но не удаляется, пока трафик не упадет.

- **Ответы об ошибках**: возвращайте структурированные ошибки с информацией, достаточной для отладки:

```json
{
    "error": {
        "code": "INVALID_INPUT",
        "message": "Feature 'user_age' must be a positive integer",
        "details": {"field": "user_age", "value": -5}
    }
}
```

## Очереди сообщений

- **Очереди сообщений** отделяют производителей (сервисы, которые генерируют работу) от потребителей (сервисов, которые ее обрабатывают). Производитель отправляет сообщение в очередь; потребитель забирает его, когда готов.

- **Зачем нужны очереди**: без очереди, если потребитель медленный или недоступен, производитель блокируется. С очередью производитель отправляет сообщение и забывает о нем; очередь буферизует сообщения, пока потребитель не будет готов.

- **Apache Kafka**: распределенная, персистентная очередь сообщений с высокой пропускной способностью. Сообщения хранятся в **топиках**, каждый из которых секционирован между несколькими брокерами. Потребители читают из партиций, отслеживая свою позицию (**offset**). Kafka гарантирует порядок внутри партиции и позволяет повторно считывать сообщения (лог персистентен).

- **Pub/sub**: издатели отправляют сообщения в топик; все подписчики этого топика получают копию. Используется для архитектуры, управляемой событиями: «новая модель развернута» запускает сервис мониторинга, сервис A/B-тестирования и сервис логирования одновременно.

- Для ML: запрос на предсказание поступает через HTTP, помещается в очередь Kafka, обрабатывается GPU-воркером, а результат возвращается через callback или WebSocket. Очередь буферизует всплески трафика и гарантирует, что запросы не будут потеряны, если GPU-воркер упадет.

## Модели согласованности

- В распределенной системе разные узлы могут иметь разное представление о данных. **Модели согласованности** определяют, какие гарантии предоставляет система:

- **Строгая согласованность**: после записи все последующие операции чтения (с любого узла) видят новое значение. Легко обосновать, но медленно (требует координации между узлами).

- **Согласованность в конечном счете**: после записи чтение может видеть устаревшие данные в течение некоторого времени, но в конечном итоге увидит новое значение. Быстро (без координации), но требует, чтобы приложение обрабатывало устаревшие данные.

- **Причинная согласованность**: если операция A причинно предшествует B (например, «запись X, затем чтение X»), система гарантирует, что B увидит результат A. Но несвязанные операции могут наблюдаться в любом порядке.

- **Согласованность «чтение своих записей»**: пользователь всегда видит свои записи немедленно, даже если другие пользователи видят устаревшие данные. Минимальная согласованность, необходимая большинству приложений.

## Паттерны отказоустойчивости

- **Ограничение частоты запросов (rate limiting)**: ограничение количества запросов на пользователя за временное окно. Защищает от злоупотреблений и обеспечивает справедливый доступ. Реализуется с помощью алгоритма «маркерная корзина» (token bucket) или счетчика скользящего окна в Redis.

- **Автоматический выключатель (circuit breaker)**: если нижестоящий сервис начинает давать сбои (частота ошибок превышает порог), выключатель «размыкается» и перестает отправлять к нему запросы (немедленно возвращая резервный ответ). После таймаута он переходит в состояние «полуоткрыт» и отправляет тестовый запрос. Если тест успешен, он замыкается (нормальная работа). Это предотвращает каскадные сбои: если хранилище признаков недоступно, сервер моделей возвращает предсказания без признаков, а не ждет таймаута на каждом запросе.

- **Обратное давление (backpressure)**: когда система перегружена, она сигнализирует вышестоящим компонентам о необходимости замедлиться. Вместо того чтобы принимать запросы и выдавать ошибки, она отклоняет избыточные запросы на раннем этапе (с кодом 429 или 503). Клиент повторяет попытку с экспоненциальной задержкой.

- **Повтор с экспоненциальной задержкой**: если запрос не удался, подождите 1 секунду и повторите. Если снова не удался, подождите 2 секунды. Затем 4, 8 и т. д. Добавьте джиттер (случайную задержку), чтобы предотвратить одновременную повторную отправку запросов всеми клиентами (проблема «стадного чувства»).

- **Идемпотентность**: операция является идемпотентной, если ее повторное выполнение дает тот же результат, что и однократное. `PUT /user/123 {"name": "Alice"}` идемпотентна (установка имени «Alice» дважды — это нормально). `POST /payments` — нет (платить дважды плохо). Обеспечение идемпотентности операций делает повторные попытки безопасными.
