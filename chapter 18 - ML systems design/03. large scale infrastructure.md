# Масштабируемая инфраструктура

*Создание систем, обслуживающих миллионы пользователей, требует большего, чем один сервер. В этом файле рассматриваются паттерны масштабируемости, основы распределенных систем, микросервисы, пайплайны данных, масштабирование баз данных, поисковые и векторные системы, наблюдаемость, надежность (reliability engineering) и CI/CD.*

- Модель, обрабатывающая 1 запрос в секунду, может работать на ноутбуке. Обслуживание 100 000 запросов в секунду с доступностью 99,9% требует использования распределенных систем, автоматического переключения при сбоях (failover) и тщательно спроектированных пайплайнов данных. В этом файле рассматриваются паттерны, которые позволяют преодолеть этот разрыв.

## Масштабируемость

- **Вертикальное масштабирование** (scale up): получение более мощной машины. Больше CPU, больше RAM, более мощный GPU. Просто, но имеет жесткие ограничения (максимально доступная машина) и является единой точкой отказа.

- **Горизонтальное масштабирование** (scale out): добавление большего количества машин. Каждая обрабатывает часть трафика. Нет ограничений одной машины, но требуются: балансировка нагрузки (файл 01), партиционирование данных и управление распределенным состоянием.

- **Stateless-сервисы** (сервисы без сохранения состояния) по умолчанию масштабируются горизонтально. Добавьте больше экземпляров за балансировщиком нагрузки. Сервер инференса модели, который загружает веса при запуске и обрабатывает запросы независимо, является stateless — любой экземпляр может обработать любой запрос.

- **Stateful-сервисы** (базы данных, KV-кеши, хранилища признаков) масштабировать сложнее. Состояние должно быть распределено по машинам (шардирование, файл 01) и реплицировано для обеспечения отказоустойчивости.

- **Уравнение масштабируемости**: для системы с $n$ серверами:
    - **Идеал**: пропускная способность масштабируется линейно ($n$ серверов → $n\times$ пропускная способность).
    - **Реальность**: накладные расходы на координацию, балансировку нагрузки и передачу данных означают, что пропускная способность масштабируется сублинейно. Применим закон Амдала (глава 13): последовательная часть (разделяемое состояние, координация) ограничивает ускорение.

## Распределенные системы

- **Распределенная система** — это группа машин, которые координируют свои действия для предоставления сервиса. Фундаментальные проблемы:

- **Сетевые разделы (network partitions)**: машины не всегда могут обмениваться данными. Сетевой кабель перерезан, коммутатор вышел из строя, в дата-центре отключилось питание. Система должна уметь обрабатывать частичные сбои.

- **Дрейф часов (clock skew)**: у машин разные часы. «Событие А произошло в 10:00:01 на машине 1» и «Событие B произошло в 10:00:01 на машине 2» не означает, что они произошли одновременно. **Логические часы** (метки времени Лампорта, векторные часы) устанавливают порядок без опоры на физические часы.

- **Консенсус**: как несколько машин приходят к согласию относительно значения (например, кто является лидером)? **Raft** — стандартный алгоритм консенсуса. Кластер узлов выбирает лидера. Лидер обрабатывает все записи. Если лидер выходит из строя, оставшиеся узлы выбирают нового. Требует большинства (3 из 5 узлов) для работы, поэтому выдерживает $\lfloor(n-1)/2\rfloor$ сбоев.

- **Распределенные блокировки**: гарантируют, что только одна машина выполняет критическую операцию. **Redlock** (на базе Redis) захватывает блокировки на нескольких экземплярах Redis. Если большинство экземпляров предоставляют блокировку, она считается захваченной. Используется для: предотвращения дублирования развертываний моделей, обеспечения того, чтобы только одно задание обучения записывало чекпоинт.

## Микросервисы

![Архитектура ML на микросервисах: API-шлюз направляет запросы к сервисам признаков, моделей и логирования, каждый из которых имеет собственную базу данных и связан через очередь сообщений](../images/microservices_architecture.svg)

- **Микросервисы** декомпозируют систему на небольшие, независимо развертываемые сервисы. Каждый сервис владеет одним доменом:

```
┌─────────────┐  ┌──────────────┐  ┌─────────────┐
│ API Gateway │→ │ Feature Svc  │→ │ Feature DB  │
└─────────────┘  └──────────────┘  └─────────────┘
       │
       ├────────→ ┌──────────────┐  ┌─────────────┐
       │          │ Model Svc    │→ │ Model Store │
       │          └──────────────┘  └─────────────┘
       │
       └────────→ ┌──────────────┐  ┌─────────────┐
                  │ Logging Svc  │→ │ Log Store   │
                  └──────────────┘  └─────────────┘
```

- **Преимущества**: независимое развертывание (обновление сервиса моделей без затрагивания сервиса признаков), независимое масштабирование (масштабирование серверов моделей в зависимости от нагрузки, серверов признаков — в зависимости от скорости чтения из хранилища признаков), свобода выбора технологий (сервис моделей на Python, сервис признаков на Go).

- **Недостатки**: сетевые накладные расходы (каждый вызов сервиса — это сетевой запрос туда-обратно), сложность (отладка охватывает несколько сервисов), согласованность данных (отсутствие транзакций между сервисами).

- **Service discovery**: как API-шлюз находит сервис моделей? Варианты: на основе DNS (каждый сервис регистрирует DNS-имя), сервисы K8s (встроенные) или реестр сервисов (Consul, Eureka).

- **Паттерн Saga**: для операций, охватывающих несколько сервисов (создание пользователя + выделение ресурсов + отправка приветственного письма), используйте сагу: последовательность локальных транзакций с компенсирующими действиями, если какой-либо шаг завершается неудачей.

## Пайплайны данных

- ML-системы потребляют огромные объемы данных. **Пайплайны данных** перемещают, преобразуют и предоставляют эти данные:

### Пакетная обработка (Batch Processing)

- Обработка больших объемов данных через регулярные интервалы (ежечасно, ежедневно).

- **MapReduce**: оригинальная парадигма пакетной обработки. Map (преобразование каждой записи независимо) → Shuffle (группировка по ключу) → Reduce (агрегация по группам). Концептуально прост, но громоздок в реализации.

- **Apache Spark**: современный движок пакетной обработки. Обработка в оперативной памяти (в 100 раз быстрее MapReduce для итеративных алгоритмов). Поддерживает SQL, DataFrames и ML-пайплайны. Стандарт для проектирования признаков в масштабе.

- **Пример**: вычисление признаков пользователей для рекомендательной системы. Вход: 1 млрд событий активности пользователей за последние 30 дней. Выход: 100 млн векторов признаков пользователей. Запускается ежедневно как Spark-задание, результат записывается в хранилище признаков.

### Потоковая обработка (Stream Processing)

- Обработка данных в реальном времени по мере их поступления (задержка менее секунды).

- **Apache Flink**: ведущий движок потоковой обработки. Обработка ровно один раз (exactly-once), обработка по времени события (обработка событий по времени их возникновения, а не поступления), работа с окнами (фиксированные, скользящие, сессионные окна).

- **Kafka Streams**: легковесная потоковая обработка, встроенная в Kafka. Хорошо подходит для простых преобразований (фильтрация, агрегация) без развертывания отдельного кластера.

- **Пример**: обнаружение мошенничества в реальном времени. Каждая транзакция по кредитной карте — это событие в Kafka. Задание Flink вычисляет текущую статистику (частоту транзакций, смену местоположения) и помечает аномалии в течение 100 мс.

### Лямбда-архитектура (Lambda Architecture)

- Объединяет пакетную обработку и потоковую обработку. **Пакетный слой** (batch layer) предоставляет точные и полные результаты (но с задержкой). **Скоростной слой** (speed layer) предоставляет приближенные результаты в реальном времени. **Обслуживающий слой** (serving layer) объединяет оба результата.

- На практике многие команды сейчас используют **Каппа-архитектуру** (Kappa architecture): только потоковая обработка, где поток рассматривается как единственный источник истины. Поток можно воспроизвести (Kafka сохраняет события), поэтому пакетную обработку можно имитировать путем повторного воспроизведения потока.

## Инфраструктура обучения моделей машинного обучения

- Обучение передовой модели (100 млрд+ параметров) — это крупномасштабная инфраструктурная задача: тысячи GPU, работающие месяцами, потребляющие мегаватты энергии, генерирующие петабайты данных и стоящие десятки миллионов долларов. Инфраструктура определяет, будет ли обучение успешным или провальным.

### Кластеры GPU

- Обучающий кластер — это совокупность серверов с GPU, соединенных высокоскоростными сетями. Ключевые компоненты:

![GPU cluster: each node has 8 GPUs connected by NVLink, nodes connected by InfiniBand in a fat tree topology, scaling from 64 to 16,000+ GPUs](../images/gpu_cluster_topology.svg)

- **Серверы GPU (узлы)**: каждый сервер имеет 4–8 GPU. Типичная конфигурация: 8 × H100 GPU, 2 × AMD EPYC CPU, 2 ТБ RAM, 30 ТБ NVMe SSD. GPU внутри узла соединены через **NVLink** (900 ГБ/с на GPU для H100), что в 30 раз быстрее, чем PCIe.

- **Размеры кластеров**: небольшой обучающий кластер насчитывает 64–256 GPU (8–32 узла). Кластер для обучения передовой модели насчитывает 4000–32 000 GPU (500–4000 узлов). Модель Llama 3 от Meta обучалась на 16 384 GPU H100. Google проводит обучение на TPU-подах с 8000+ чипами.

- **Грубая оценка**: обучение модели на 70 млрд параметров требует ~$2M in compute. Training a 400B+ frontier model requires ~$50-100M. Стоимость оборудования самого кластера составляет ~$500M-$1B при ценах на H100 ($30K per GPU × 16,000 GPUs = $480M).

### Сетевая топология

- Сеть между узлами GPU является наиболее критическим компонентом инфраструктуры. Если GPU не могут обмениваться градиентами достаточно быстро, они простаивают в ожидании завершения передачи данных.

- **InfiniBand** — это стандарт для сетей кластеров GPU. NVIDIA **Quantum-2 InfiniBand** обеспечивает 400 Гбит/с на порт. Каждый узел обычно имеет 8 портов InfiniBand (по одному на каждый GPU), что дает общую пропускную способность бисекции 400 ГБ/с на узел.

- **RDMA** (Remote Direct Memory Access): InfiniBand поддерживает RDMA, который передает данные напрямую между памятью GPU на разных узлах без участия CPU. Это сокращает задержку с ~100 мкс (TCP) до ~1 мкс и является необходимым условием для эффективного all-reduce градиентов (глава 6).

- **Топология сети имеет значение**: **толстое дерево** (fat tree, сеть Clos) обеспечивает полную пропускную способность бисекции (любой GPU может обмениваться данными с любым другим на полной скорости). Более дешевые топологии (**rail-optimised**, **3D torus**) обеспечивают меньшую пропускную способность, но стоят дешевле. Топология должна соответствовать стратегии параллелизма:
    - **Параллелизм данных**: all-reduce по всем GPU → требуется высокая пропускная способность бисекции (толстое дерево).
    - **Тензорный параллелизм**: обмен данными внутри узла → это обеспечивает NVLink (сеть не требуется).
    - **Конвейерный параллелизм**: обмен данными между соседними стадиями конвейера → требуется пропускная способность только между конкретными парами узлов (подходит rail-optimised).

- **Альтернативы на базе Ethernet**: **RoCE v2** (RDMA over Converged Ethernet) обеспечивает RDMA поверх стандартной инфраструктуры Ethernet. Дешевле, чем InfiniBand, но с более высокой задержкой и большим количеством перегрузок. Google использует RoCE для некоторых сетей TPU-подов. Консорциум Ultra Ethernet разрабатывает Ethernet без потерь для рабочих нагрузок ИИ.

### Хранилище для обучения

- Для обучения требуются три уровня хранения:

    - **Хранилище датасетов**: обучающий корпус (1–100 ТБ текста или петабайты мультимодальных данных). Хранится в распределенных файловых системах или объектных хранилищах. Должно поддерживать последовательное чтение с высокой пропускной способностью (загрузчик данных считывает данные большими батчами). **Lustre** и **GPFS** — распространенные файловые системы HPC; облачные альтернативы включают **FSx for Lustre** (AWS) и **Filestore** (GCP).

    - **Хранилище чекпоинтов**: состояние обучения (веса модели + состояние оптимизатора + состояние планировщика), сохраняемое периодически. Для модели на 70 млрд параметров в смешанной точности с оптимизатором Adam: ~560 ГБ на чекпоинт (70 млрд × 4 байта × 2 для оптимизатора). Сохранение каждый час в течение 3-месячного цикла = ~2000 чекпоинтов = 1,1 ПБ. На практике сохраняются только последние N чекпоинтов, а старые удаляются. Процесс должен быть достаточно быстрым, чтобы сохранение чекпоинтов не замедляло обучение.

    - **Логирование и метрики**: данные отслеживания экспериментов (кривые потерь, графики скорости обучения, нормы градиентов). Относительно небольшие объемы, но должны записываться в реальном времени. Этим занимаются W&B, MLflow или TensorBoard.

- **Узкое место хранилища**: кластер из 16 000 GPU при загрузке обучающего батча должен непрерывно считывать ~100 ГБ/с данных. Если файловая система не может поддерживать такую пропускную способность, GPU простаивают в ожидании данных. Оптимизация конвейера данных (префетчинг, кеширование, оптимизация формата с использованием WebDataset или Mosaic Streaming) критически важна.

### Планировщик заданий

- Кластер GPU обслуживает множество команд и проектов. **Планировщик заданий** (job scheduler) распределяет GPU для задач обучения:

- **SLURM**: стандартный планировщик заданий в HPC. Пользователи отправляют задания, указывая количество GPU, объем памяти и ограничение по времени. SLURM распределяет ресурсы и управляет очередью. Поддерживает планирование на основе приоритетов, вытеснение (preemption) и справедливое распределение ресурсов между командами.

- **Kubernetes с планированием GPU** (глава 18, файл 02): облачно-ориентированный подход. Плагины устройств GPU в K8s предоставляют GPU как планируемые ресурсы. **Volcano** и **Run:ai** добавляют функции планирования, специфичные для машинного обучения: групповое планирование (gang scheduling — выделение всех GPU для задания одновременно, а не по одному), очереди с приоритетами и совместное использование GPU по времени.

- **Проблемы планирования**:
    - **Фрагментация**: в кластере с 1000 GPU может быть 200 свободных, но распределенных по 50 узлам (по 4 свободных на узел). Задание, требующее 128 последовательных GPU, не может быть запущено, хотя общее количество GPU достаточно. **Дефрагментация** (миграция заданий для консолидации свободных GPU) или **планирование с учетом топологии** (выделение хорошо связанных GPU) решают эту проблему.
    - **Приоритет и вытеснение**: срочные эксперименты должны вытеснять задания с более низким приоритетом. Однако вытеснение задания, которое обучалось 2 дня, приводит к потере вычислительных ресурсов. Планировщик должен балансировать между приоритетом и эффективностью.
    - **Справедливое распределение (fair share)**: команды должны получать свою выделенную долю вычислительных мощностей с течением времени, даже если одна команда отправляет больше заданий, чем ей положено.

### Отказоустойчивость

- При масштабах в тысячи GPU, работающих месяцами, аппаратные сбои — это не исключение, а рутина. Среднее время наработки на отказ для кластера из 16 000 GPU измеряется часами, а не месяцами.

- **Типичные сбои**: ошибки памяти GPU (исправимые и неисправимые с помощью ECC), сбои NVLink (связь между GPU внутри узла), сбои каналов InfiniBand (связь между узлами), падения узлов (kernel panic, отказ блока питания) и сбои хранилища (отказ диска или контроллера).

- **Чекпоинтинг** — основное средство защиты. Сохраняйте полное состояние обучения (модель, оптимизатор, позицию загрузчика данных) каждые N шагов. В случае сбоя: определите вышедший из строя узел, замените или удалите его, перезапустите обучение с последнего чекпоинта. Стоимость сбоя — это объем вычислений между последним чекпоинтом и моментом сбоя.

- **Компромисс частоты чекпоинтов**: частые чекпоинты (каждые 10 минут) позволяют тратить меньше вычислений при сбое, но замедляют обучение (сохранение 560 ГБ требует времени). Редкие чекпоинты (каждые 2 часа) быстрее, но при сбое теряется до 2 часов вычислений. Большинство команд делают чекпоинты каждые 20–60 минут.

- **Эластичное обучение**: современные фреймворки (PyTorch Elastic, DeepSpeed) поддерживают изменение размера кластера во время обучения без перезапуска. Если из 500 узлов вышли из строя 2, обучение продолжается на 498 узлах. Неисправные узлы заменяются, и обучение автоматически подключает их, когда они возвращаются в строй.

- **Мониторинг состояния**: непрерывный мониторинг всех GPU (температура, ошибки памяти, пропускная способность вычислений), сетевых соединений (потеря пакетов, задержка) и хранилища (пропускная способность, частота ошибок). Автоматическое оповещение об аномалиях. Некоторые кластеры проводят периодические проверки работоспособности GPU (короткий вычислительный тест), чтобы заблаговременно выявить деградирующее оборудование до того, как оно выйдет из строя.

- **В масштабе**: при обучении Llama 3 от Meta (16 384 H100, 54 дня) произошло около 466 прерываний работы. Эффективное время обучения составило лишь ~90% от общего времени — 10% было потеряно из-за сбоев и восстановления. Инфраструктура, позволяющая достичь 90% (а не 50% или 70%), — это то, что отличает организации, способные обучать передовые модели, от тех, кто не может.

### Стоимость и эффективность

- Стоимость инфраструктуры обучения определяется часами работы GPU:

| Компонент | % от общей стоимости |
|-----------|----------------|
| Вычисления на GPU | 70-80% |
| Сеть (InfiniBand) | 10-15% |
| Хранилище | 5-10% |
| Охлаждение и питание | 5-10% |

- **Использование GPU** (Model FLOPs Utilisation, MFU) измеряет, какая часть теоретической пиковой производительности GPU фактически используется для полезных вычислений. Пиковая производительность H100 составляет 989 TFLOPS (FP8). Достижение 40–50% MFU — это хороший результат; 50–60% — отличный. Разрыв обусловлен: накладными расходами на коммуникацию (all-reduce, «пузыри» в конвейере), ограничениями пропускной способности памяти и временем простоя во время чекпоинтинга и загрузки данных.

- **Повышение MFU**: совмещение вычислений и коммуникации (глава 6), использование эффективного внимания (Flash Attention, глава 16), оптимизация загрузки данных (предотвращение «голодания» GPU), снижение накладных расходов на чекпоинтинг (асинхронный чекпоинтинг, сохранение сначала на быстрый NVMe, затем фоновое копирование в постоянное хранилище).

- **Собственная инфраструктура или облако**: при малых масштабах (<256 GPU) облако дешевле (нет первоначальных затрат, оплата по часам). При больших масштабах (>1000 GPU, постоянное использование в течение 6+ месяцев) владение оборудованием дешевле (~в 2–3 раза ниже совокупная стоимость владения за 3 года). Большинство AI-компаний используют гибридный подход: собственные кластеры для постоянного обучения, облако — для пиковых нагрузок и экспериментов.

## Масштабирование баз данных

- **Реплики для чтения**: перенаправление запросов на чтение к репликам основной базы данных. Основная база обрабатывает запись, реплики — чтение. Поскольку большинство рабочих нагрузок ориентированы на чтение (95%+), это масштабирует пропускную способность чтения линейно количеству реплик.

- **Партиционирование** (шардирование, из файла 01): разделение данных между несколькими базами данных. Каждый раздел независим, что позволяет выполнять параллельное чтение и запись. Сложность заключается в запросах между разделами (объединение данных из разных шардов).

- **Пул соединений**: базы данных имеют ограниченную емкость соединений. Пул соединений (например, PgBouncer для PostgreSQL) повторно использует соединения между запросами, предотвращая исчерпание лимита соединений, когда сотни экземпляров сервиса пытаются подключиться одновременно.

## Поиск и векторные системы

### Текстовый поиск

- **Инвертированный индекс**: основа текстового поиска. Для каждого слова хранится список документов, содержащих это слово. Запрос пересекает списки для каждого слова запроса. **Elasticsearch** — стандарт индустрии: распределенный, работающий в реальном времени, поддерживает полнотекстовый поиск, агрегации и геопространственные запросы.

- **BM25**: стандартная функция оценки релевантности при текстовом поиске. Оценивает документы на основе частоты терминов, обратной частоты документа и нормализации длины документа. Простой, но эффективный метод — до сих пор конкурентоспособен по сравнению с нейросетевыми методами для запросов, насыщенных ключевыми словами.

### Векторный поиск

- **Векторные базы данных** хранят эмбеддинги (многомерные векторы) и поддерживают быстрый поиск **приблизительных ближайших соседей (ANN)**. По заданному вектору запроса находят $k$ наиболее похожих сохраненных эмбеддингов.

- **FAISS** (Facebook AI Similarity Search): библиотека (не база данных) для ANN-поиска. Поддерживает несколько типов индексов:
    - **Flat**: точный поиск, $O(n)$. Используется для небольших датасетов или в качестве эталонной разметки.
    - **IVF** (Inverted File): разбиение векторов на кластеры, поиск только по ближайшим кластерам. $O(n/k)$ на запрос.
    - **HNSW** (Hierarchical Navigable Small World): на основе графов. Построение иерархического графа, навигация от грубого к точному. Чрезвычайно быстрый и точный, выбор по умолчанию для большинства приложений.
    - **Product Quantisation (PQ)**: сжатие векторов в компактные коды для поиска с эффективным использованием памяти. Обмен точности на память.

- **Управляемые векторные базы данных**: Pinecone, Weaviate, Milvus, Qdrant. Они берут на себя масштабирование, репликацию и обновления в реальном времени, чего нет в FAISS.

- **Для RAG** (Retrieval-Augmented Generation): запрос пользователя → эмбеддинг с помощью текстового энкодера → поиск релевантных документов в векторной базе данных → добавление найденных документов в промпт LLM. Качество поиска напрямую определяет качество ответа LLM.

## Наблюдаемость (Observability)

- **Наблюдаемость** — это способность понимать, что происходит внутри вашей системы, исходя из её внешних выходных данных. Три столпа:

### Логирование

- **Структурированные логи** (JSON) пригодны для поиска и парсинга. Неструктурированные логи ("ERROR: something failed") — нет. Всегда логируйте: временную метку, имя сервиса, ID запроса (для трассировки между сервисами), уровень критичности и соответствующий контекст.

- **Стек ELK** (Elasticsearch, Logstash, Kibana): стандартный пайплайн логирования. Logstash собирает и преобразует логи, Elasticsearch индексирует их, Kibana визуализирует и позволяет выполнять поиск.

### Метрики

- **Метрики** — это численные измерения во времени: частота запросов, частота ошибок, перцентили задержки, использование GPU, глубина очереди. **Prometheus** собирает метрики с сервисов; **Grafana** визуализирует их на дашбордах с настройкой алертинга.

- **Метод RED** для сервисов: **R**ate (частота запросов в секунду), **E**rrors (частота ошибок), **D**uration (задержка). Отслеживайте эти показатели для каждого сервиса.

- **Метод USE** для ресурсов: **U**tilisation (% использования), **S**aturation (глубина очереди), **E**rrors. Отслеживайте эти показатели для каждого ресурса (CPU, GPU, память, диск, сеть).

### Трассировка

- **Распределенная трассировка** отслеживает один запрос через множество сервисов. Пользовательский запрос попадает на API-шлюз → сервис признаков → сервис модели → постпроцессинг. **Трейс** записывает время каждого перехода, показывая, на что уходит задержка.

- **OpenTelemetry**: открытый стандарт для трейсов, метрик и логов. Инструментируйте код один раз и экспортируйте данные в любой бэкенд (Jaeger, Zipkin, Datadog).

## Надежность

- **SLO** (Service Level Objective): целевой уровень надежности. "99,9% запросов выполняются менее чем за 200 мс". Это дает конкретный бюджет ошибок: 0,1% запросов (около 43 минут в месяц) могут быть медленными или завершиться ошибкой.

- **SLI** (Service Level Indicator): показатель надежности. "99-й перцентиль задержки за последние 5 минут".

- **SLA** (Service Level Agreement): договорное обязательство с последствиями. "Если доступность падает ниже 99,95%, клиент получает компенсацию".

- **Бюджет ошибок**: если ваш SLO составляет 99,9%, а текущий показатель — 99,99%, у вас есть бюджет для рискованных изменений (развертывание новой модели, миграция баз данных). Если показатель 99,85%, заморозьте все изменения и сосредоточьтесь на надежности. Бюджеты ошибок превращают надежность из абстрактной цели в измеримый ресурс.

- **Хаос-инженерия**: преднамеренное внедрение сбоев (отключение сервера, добавление сетевой задержки, повреждение данных) для проверки того, правильно ли система их обрабатывает. Chaos Monkey от Netflix случайным образом завершает работу инстансов в продакшне. Если система продолжает работать, она устойчива. Если она падает, вы нашли баг раньше пользователей.

## CI/CD

- **Continuous Integration (непрерывная интеграция)**: автоматическая сборка и тестирование каждого изменения кода. Каждый пуш запускает: линтинг, проверку типов, модульные тесты, интеграционные тесты. Если что-то не проходит, изменение отклоняется. Это позволяет находить баги до того, как они попадут в продакшн.

- **Continuous Deployment (непрерывное развертывание)**: автоматическое развертывание изменений, прошедших CI. Стратегии развертывания:

    - **Blue-green**: запуск двух идентичных сред (blue = текущая, green = новая). Мгновенное переключение трафика с blue на green. Если green дает сбой, переключение обратно на blue (мгновенный откат).

    - **Canary**: перенаправление небольшой части трафика (1-5%) на новую версию. Мониторинг ошибок. Постепенное увеличение трафика, если метрики в норме. Это ограничивает радиус поражения при неудачном развертывании.

    - **Feature flags**: развертывание нового кода, скрытого за флагом. Включение флага для подмножества пользователей (внутренние тестировщики, затем бета-пользователи, затем все остальные). Разделение развертывания (код запущен) и релиза (пользователи видят функцию).

- Для ML: CI/CD включает специфические для моделей шаги. Изменение модели запускает: модульные тесты (проверка размерностей, проверка градиентов), оценку на отложенной выборке (точность не должна снизиться), теневое развертывание (запуск новой модели параллельно со старой, сравнение выходов) и постепенное внедрение (canary от 1% до 100%).
