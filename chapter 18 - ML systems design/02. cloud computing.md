# Облачные вычисления

*Облачные вычисления предоставляют инфраструктуру по требованию для рабочих нагрузок машинного обучения без необходимости владения оборудованием. В этом файле рассматриваются сервисные модели, основные облачные провайдеры, контейнеры и Kubernetes, хранилища, облачные сети, бессерверные вычисления, управление затратами и инфраструктура как код.*

- Обучение пограничной модели (frontier model) требует тысяч GPU в течение месяцев. Ни один стартап не владеет таким оборудованием. Облачные вычисления позволяют арендовать его почасово, масштабировать ресурсы вверх для обучения, масштабировать вниз для инференса и платить только за то, что вы используете. Понимание облачной инфраструктуры необходимо каждому, кто создает системы машинного обучения, выходящие за рамки одного ноутбука.

## Сервисные модели облаков

![Слои облачных сервисов: IaaS дает вам максимум контроля, SaaS — минимум](../images/cloud_service_layers.svg)

- Облачные сервисы распределены по слоям в зависимости от того, какую часть инфраструктуры управляет провайдер:

| Модель | Вы управляете | Провайдер управляет | Пример |
|-------|-----------|-----------------|---------|
| **IaaS** (Инфраструктура) | ОС, среда выполнения, приложение | Оборудование, виртуализация, сети | AWS EC2, GCP Compute Engine |
| **PaaS** (Платформа) | Приложение, данные | ОС, среда выполнения, масштабирование, патчи | AWS SageMaker, GCP Vertex AI |
| **SaaS** (ПО) | Ничем (просто используете) | Всем | OpenAI API, Weights & Biases |
| **FaaS** (Функция) | Отдельными функциями | Всем остальным | AWS Lambda, GCP Cloud Functions |

- **Для машинного обучения**: большинство команд используют комбинацию моделей. IaaS для кастомного обучения (полный контроль над GPU-инстансами), PaaS для управляемого обучения и инференса (SageMaker, Vertex AI берут на себя оркестрацию) и SaaS для инструментов (W&B для отслеживания экспериментов, OpenAI API для сравнения с базовыми решениями).

## Основные провайдеры

### AWS (Amazon Web Services)

- Крупнейший облачный провайдер (~32% доли рынка). Ключевые ML-сервисы:
    - **EC2**: виртуальные машины. GPU-инстансы: p4d (A100), p5 (H100), g5 (A10G для инференса).
    - **S3**: объектное хранилище. Стандарт для хранения датасетов и весов моделей. Практически неограниченная емкость, ~$0.023/ГБ в месяц.
    - **SageMaker**: управляемая ML-платформа. Обрабатывает обучение, подбор гиперпараметров, развертывание и мониторинг.
    - **EKS**: управляемый Kubernetes.
    - **Lambda**: бессерверные функции. Не подходят для GPU-нагрузок, но полезны для предобработки и оркестрации.

### GCP (Google Cloud Platform)

- Облако Google (~11% доли рынка). Ключевые ML-сервисы:
    - **Compute Engine**: виртуальные машины. GPU-инстансы с A100, H100. **TPU VMs** для доступа к TPU.
    - **GCS**: объектное хранилище (аналог S3).
    - **Vertex AI**: управляемая ML-платформа. Нативная поддержка JAX/TPU.
    - **GKE**: управляемый Kubernetes (самое зрелое решение K8s, так как Google создала Kubernetes).
    - **Cloud TPUs**: эксклюзивно для GCP. v5e и v5p для крупномасштабного обучения.

### Azure (Microsoft)

- Облако Microsoft (~23% доли рынка). Ключевые ML-сервисы:
    - **Azure VMs**: GPU-инстансы с A100, H100.
    - **Azure Blob Storage**: объектное хранилище.
    - **Azure ML**: управляемая ML-платформа.
    - **AKS**: управляемый Kubernetes.
    - **OpenAI Service**: эксклюзивный доступ к моделям OpenAI через API Azure.

## Контейнеры и Kubernetes

- Мы рассмотрели контейнеры (Docker) и Kubernetes концептуально в главе 13 (ОС) и практически в главе 15 (развертывание). Здесь мы сосредоточимся на паттернах, **специфичных для облака**:

### Kubernetes для машинного обучения

- **Kubernetes (K8s)** оркестрирует контейнеры в масштабе. Ключевые концепции:

    - **Pod (Под)**: наименьшая единица развертывания. Содержит один или несколько контейнеров, разделяющих сеть и хранилище. Под для инференса модели может содержать: контейнер сервера модели + sidecar-контейнер для сбора метрик.

    - **Deployment (Развертывание)**: управляет набором идентичных подов. Указывает желаемое количество реплик. Если под падает, K8s автоматически создает замену.

    - **Service (Сервис)**: стабильная сетевая точка доступа для набора подов. Клиенты подключаются к сервису; K8s перенаправляет запрос на работоспособный под. Типы: ClusterIP (внутренний), NodePort (внешний через порт узла), LoadBalancer (внешний через облачный балансировщик).

    - **StatefulSet**: как Deployment, но для нагрузок с состоянием. Каждый под получает постоянный идентификатор и стабильное хранилище. Используется для баз данных и распределенного обучения (каждому воркеру нужен стабильный идентификатор для связи).

    - **DaemonSet**: запускает один под на каждом узле. Используется для: агентов мониторинга (Prometheus node exporter), сборщиков логов (Fluentd), плагинов устройств GPU (NVIDIA device plugin).

- **Планирование GPU в K8s**: плагин устройств NVIDIA предоставляет GPU как ресурс K8s. Поды запрашивают GPU:

```yaml
resources:
  limits:
    nvidia.com/gpu: 2  # this pod needs 2 GPUs
```

- K8s планирует под на узел с 2 доступными GPU. Именно так облачные ML-платформы распределяют GPU для обучения и инференса.

### Автомасштабирование

- **Horizontal Pod Autoscaler (HPA)**: масштабирует количество подов на основе метрик (загрузка CPU, частота запросов, пользовательские метрики, такие как использование GPU или глубина очереди).

- **Cluster Autoscaler**: масштабирует количество узлов. Если поды не могут быть запланированы из-за нехватки узлов, Cluster Autoscaler запрашивает новые виртуальные машины у облачного провайдера. Когда узлы недогружены, он освобождает их и завершает работу.

- **KEDA** (Kubernetes Event-Driven Autoscaling): масштабирует на основе внешних источников событий (глубина очереди Kafka, частота HTTP-запросов). Идеально подходит для инференса: масштабирование серверов моделей при росте очереди запросов и уменьшение масштаба, когда она пуста.

## Хранилище

| Тип | Характеристики | Вариант использования | Пример |
|------|----------------|----------|---------|
| **Блочное** | Низкая задержка, привязано к одной ВМ | Диски ОС, базы данных | AWS EBS, GCP Persistent Disk |
| **Объектное** | Неограниченная емкость, HTTP-доступ | Датасеты, веса моделей, логи | AWS S3, GCS, Azure Blob |
| **Файловое** | Общий доступ для ВМ, POSIX | Общие данные для обучения | AWS EFS, GCP Filestore, NFS |
| **Data lake** | Schema-on-read, «сырые» данные | Аналитика, генерация признаков | Delta Lake, Iceberg, Hudi |

- **Для обучения ML**: датасеты хранятся в объектном хранилище (S3/GCS). Скрипты обучения считывают данные из объектного хранилища в RAM. Для быстрого произвольного доступа (загрузка перемешанных данных) используйте один из вариантов: (1) скачивание датасета на локальный SSD перед обучением, (2) использование высокопроизводительной файловой системы (Lustre, FSx) или (3) использование библиотеки загрузки данных, которая эффективно выполняет потоковую передачу и кеширование (WebDataset, FFCV).

- **Веса моделей**: хранятся в объектном хранилище с версионированием. Модель 70B в формате FP16 весит ~140 ГБ. Загрузка из S3 со скоростью 1 ГБ/с занимает ~2.5 минуты. Кеширование на локальном SSD сокращает время «холодного старта» для инференса.

## Облачные сети

- **VPC** (Virtual Private Cloud): изолированная сеть внутри облака. Ваши ВМ, базы данных и сервисы взаимодействуют внутри VPC. Внешний трафик поступает через балансировщики нагрузки или шлюзы.

- **Подсети (Subnets)**: разделяют VPC на сегменты. Публичные подсети имеют доступ к интернету (для API-серверов). Приватные подсети — нет (для баз данных, GPU-воркеров). Это сетевой эквивалент принципа наименьших привилегий в безопасности.

- **Группы безопасности (Security groups)** (AWS) / **Правила брандмауэра (Firewall rules)** (GCP): контролируют, какой трафик разрешен. «Разрешить входящий HTTP на порт 80 отовсюду. Разрешить входящий SSH на порт 22 только с моего IP. Блокировать всё остальное». Неправильно настроенные группы безопасности — причина номер один инцидентов облачной безопасности.

- **Service mesh** (Istio, Envoy): управляет межсервисным взаимодействием внутри K8s. Обеспечивает: шифрование mTLS (каждый вызов между сервисами зашифрован), маршрутизацию трафика (A/B-тестирование: направлять 10% трафика на новую модель), повторные попытки, тайм-ауты, разрыв цепи (circuit breaking) и наблюдаемость (какой сервис вызвал какой, сколько времени это заняло).

## Serverless

- **Serverless** (AWS Lambda, GCP Cloud Functions): вы загружаете функцию, а облачный провайдер запускает её при возникновении события. Нет серверов для управления, нет масштабирования для настройки. Вы платите за вызов (обычно $0.20 за 1 млн вызовов + время вычислений).

- **Холодные старты (Cold starts)**: первый вызов после периода бездействия занимает больше времени (провайдер должен выделить контейнер и загрузить ваш код). Холодные старты длятся 0.5–5 секунд, что делает serverless непригодным для ML-инференса, чувствительного к задержкам.

- **Для ML**: serverless полезен для: предобработки (изменение размера изображений перед отправкой в модель), постпроцессинга (форматирование вывода модели, отправка уведомлений), оркестрации (запуск пайплайна обучения при поступлении новых данных) и легкого инференса (небольшие модели, допускающие холодные старты).

- Serverless **не** подходит для: GPU-инференса (отсутствие поддержки GPU в большинстве serverless-платформ), длительных задач обучения (тайм-аут 15 минут в Lambda) или stateful-сервисов (отсутствие постоянного состояния между вызовами).

## Управление затратами

- Облачные расходы — главная операционная проблема для ML-команд. Один инстанс H100 стоит ~$8/hour. Обучение на 64 GPU стоит ~$500/hour. Месячный цикл обучения стоит ~$360,000. Оптимизация затрат — это инженерная задача, а не бухгалтерская.

- **Spot/preemptible-инстансы**: неиспользуемые облачные мощности, продаваемые со скидкой 60–90%. Провайдер может отозвать их с уведомлением за 30 секунд – 2 минуты. Используйте для: отказоустойчивого обучения (частые чекпоинты, возобновление на новых инстансах), пакетного инференса, предобработки данных. Не используйте для: сервисов, чувствительных к задержкам (прерывание = простой).

- **Reserved-инстансы**: обязательство использовать ресурсы в течение 1–3 лет со скидкой 30–60%. Используйте для: инференса в установившемся режиме, где известна базовая нагрузка.

- **Автомасштабирование**: масштабирование в часы пик, уменьшение масштаба ночью/в выходные. Сервер модели, которому нужно 10 GPU в пик и 2 ночью, экономит ~60% за счет автомасштабирования по сравнению с работой 10 GPU 24/7.

- **Right-sizing**: не используйте H100 для модели 7B, которая отлично работает на A10G. Подбирайте GPU под рабочую нагрузку. Используйте профилирование (глава 16), чтобы определить, какой GPU подходит лучше всего.

- **Затраты на хранение**: объектное хранилище дешево ($0.023/GB/month for S3 Standard), но оно накапливается. Команда, сохраняющая каждый чекпоинт обучения (по 10 ГБ, 100 на эксперимент, 50 экспериментов), накапливает 50 ТБ = $1,150/month. Установите политики жизненного цикла для автоматического удаления старых чекпоинтов.

## Развертывание в нескольких регионах

- Для глобальных ML-систем (обслуживающих пользователей по всему миру) развертывание в одном регионе означает высокую задержку для удаленных пользователей (пользователь в Токио, обращающийся к серверу в США, добавляет ~150 мс на сетевой round trip) и единую точку отказа (если регион падает, весь сервис недоступен).

- **Паттерны для нескольких регионов**:

    - **Активный-пассивный (Active-passive)**: один основной регион обрабатывает весь трафик. Вторичный регион находится в режиме «горячего» ожидания (реплицированные данные, готовность принять трафик). При сбое основного региона DNS переключается на вторичный. Время простоя при переключении: от 30 секунд до нескольких минут.

    - **Активный-активный (Active-active)**: оба региона обрабатывают трафик одновременно. Пользователи направляются в ближайший регион. Оба региона имеют актуальные данные (реплицируются асинхронно или синхронно). Нет простоя при сбое одного региона — трафик перенаправляется автоматически.

- **Репликация данных**: сложная часть. Веса моделей можно легко реплицировать (скопировать в S3 в каждом регионе). Данные в хранилище признаков (feature store) должны реплицироваться с допустимым уровнем устаревания. К пользовательским данным могут предъявляться **требования к месту хранения данных** (GDPR: данные европейских пользователей должны оставаться в Европе).

- **Сравнение цен на GPU в облаке** (приблизительно, 2026 г.):

| GPU | AWS | GCP | Azure | Типичное использование |
|-----|-----|-----|-------|-------------|
| A10G (24 GB) | $1.00/hr (g5) | $0.90/hr | $0.90/hr | Инференс небольших моделей |
| A100 (80 GB) | $4.10/hr (p4d) | $3.70/hr | $3.40/hr | Обучение, масштабный инференс |
| H100 (80 GB) | $8.00/hr (p5) | $7.50/hr | $7.00/hr | Обучение передовых моделей |
| TPU v5e | n/a | $1.20/hr | n/a | Масштабное обучение в JAX |

- Спотовые (прерываемые) инстансы обычно стоят на 60-70% дешевле этих тарифов. Цены варьируются в зависимости от региона и доступности.

## Инфраструктура как код (IaC)

- **IaC** определяет инфраструктуру (виртуальные машины, сети, базы данных, кластеры K8s) в конфигурационных файлах, находящихся под контролем версий. Вместо того чтобы нажимать кнопки в консоли AWS, вы пишете код, который описывает желаемый результат, а инструмент создает его.

- **Terraform** (HashiCorp): стандартный инструмент IaC. Работает со всеми основными облачными провайдерами. Декларативный подход: вы описываете желаемое состояние, а Terraform определяет, что нужно создать, изменить или удалить для его достижения.

```hcl
# main.tf — create a GPU VM for inference
resource "aws_instance" "model_server" {
  ami           = "ami-0abcdef1234567890"  # Deep Learning AMI
  instance_type = "g5.xlarge"               # A10G GPU

  tags = {
    Name = "model-server-prod"
  }
}

resource "aws_s3_bucket" "model_weights" {
  bucket = "my-model-weights-prod"

  versioning {
    enabled = true
  }
}
```

```bash
terraform init      # download provider plugins
terraform plan      # show what will change
terraform apply     # create the infrastructure
terraform destroy   # tear it all down
```

- **Почему IaC важна**: воспроизводимость (возможность воссоздать всю инфраструктуру из кода), аудит (история git показывает, кто и что изменил), аварийное восстановление (возможность пересобрать инфраструктуру в другом регионе из той же конфигурации) и идентичность сред (dev, staging и prod используют одни и те же шаблоны с разными параметрами).

- **Pulumi**: аналог Terraform, но использующий настоящие языки программирования (Python, TypeScript, Go) вместо HCL. Полезен, когда логика вашей инфраструктуры сложна (условные операторы, циклы, динамическая конфигурация).
