# 云计算

*云计算为 ML 工作负载提供按需的基础设施，无需自购硬件。本文件涵盖服务模型、主要云厂商、容器与 Kubernetes、存储、云网络、serverless 计算、成本管理与基础设施即代码*

- 训练一个前沿模型需要数千块 GPU 连续运行数月。没有哪家初创公司能拥有这些硬件。云计算让你按小时租用，训练时扩容、inference 时缩容，只为所用付费。对任何在笔记本之外构建 ML 系统的人而言，理解云基础设施都不可或缺。

## 云服务模型

![云服务分层：IaaS 给你最多控制权，SaaS 给你最少](../images/cloud_service_layers.svg)

- 云服务按厂商管理多少内容来分层：

| 模型 | 你管理的内容 | 厂商管理的内容 | 示例 |
|-------|-----------|-----------------|---------|
| **IaaS**（基础设施） | OS、运行时、应用 | 硬件、虚拟化、网络 | AWS EC2、GCP Compute Engine |
| **PaaS**（平台） | 应用、数据 | OS、运行时、扩展、补丁 | AWS SageMaker、GCP Vertex AI |
| **SaaS**（软件） | 无（直接用） | 全部 | OpenAI API、Weights & Biases |
| **FaaS**（函数） | 单个函数 | 其他一切 | AWS Lambda、GCP Cloud Functions |

- **对 ML**：多数团队混搭使用。IaaS 用于自定义训练（完全控制 GPU 实例），PaaS 用于托管训练与 serving（SageMaker、Vertex AI 负责编排），SaaS 用于工具（W&B 做实验追踪，OpenAI API 做基线对比）。

## 主要厂商

### AWS（Amazon Web Services）

- 最大的云厂商（约 32% 市场份额）。主要 ML 服务：
    - **EC2**：虚拟机。GPU 实例：p4d（A100）、p5（H100）、g5（A10G，用于 inference）。
    - **S3**：对象存储。存储数据集与模型权重的事实标准。容量近乎无限，约 $0.023/GB/月。
    - **SageMaker**：托管 ML 平台。处理训练、超参调优、部署与监控。
    - **EKS**：托管 Kubernetes。
    - **Lambda**：serverless 函数。不适合 GPU 工作负载，但可用于预处理与编排。

### GCP（Google Cloud Platform）

- Google 的云（约 11% 市场份额）。主要 ML 服务：
    - **Compute Engine**：VM。带 A100、H100 的 GPU 实例。**TPU VMs** 用于访问 TPU。
    - **GCS**：对象存储（类似 S3）。
    - **Vertex AI**：托管 ML 平台。原生支持 JAX/TPU。
    - **GKE**：托管 Kubernetes（最成熟的 K8s 方案，因为 Kubernetes 由 Google 创建）。
    - **Cloud TPUs**：GCP 独有。v5e 与 v5p 用于大规模训练。

### Azure（Microsoft）

- 微软的云（约 23% 市场份额）。主要 ML 服务：
    - **Azure VMs**：带 A100、H100 的 GPU 实例。
    - **Azure Blob Storage**：对象存储。
    - **Azure ML**：托管 ML 平台。
    - **AKS**：托管 Kubernetes。
    - **OpenAI Service**：经 Azure API 独家访问 OpenAI 模型。

## 容器与 Kubernetes

- 我们在第 13 章（OS）从概念上讲过容器（Docker）与 Kubernetes，并在第 15 章（部署）讲过实操。这里聚焦**云端特定**模式：

### 用于 ML 的 Kubernetes

- **Kubernetes（K8s）**在大规模下编排容器。核心概念：

    - **Pod**：最小可部署单元。包含一个或多个共享网络与存储的容器。一个模型 serving pod 可能包含：模型服务器容器 + 一个用于收集指标的 sidecar 容器。

    - **Deployment**：管理一组相同的 pod。指定期望的副本数。如果某个 pod 崩溃，K8s 自动创建替代。

    - **Service**：一组 pod 的稳定网络端点。客户端连接到 service；K8s 路由到健康 pod。类型：ClusterIP（内部）、NodePort（经节点端口对外）、LoadBalancer（经云 LB 对外）。

    - **StatefulSet**：类似 Deployment，但面向有状态工作负载。每个 pod 拥有持久身份与稳定存储。用于数据库与分布式训练（每个 worker 需要稳定身份以便通信）。

    - **DaemonSet**：在每个节点上跑一个 pod。用于：监控 agent（Prometheus node exporter）、日志收集器（Fluentd）、GPU 设备插件（NVIDIA device plugin）。

- **K8s 中的 GPU 调度**：NVIDIA device plugin 把 GPU 作为一种 K8s 资源暴露。pod 请求 GPU：

```yaml
resources:
  limits:
    nvidia.com/gpu: 2  # this pod needs 2 GPUs
```

- K8s 把该 pod 调度到有 2 块可用 GPU 的节点。云 ML 平台就是如此为训练与 inference 分配 GPU 的。

### autoscaling

- **Horizontal Pod Autoscaler（HPA）**：基于指标（CPU 使用率、请求率，或 GPU 利用率、队列深度等自定义指标）扩展 pod 数量。

- **Cluster Autoscaler**：扩展节点数量。如果因节点不足导致 pod 无法被调度，cluster autoscaler 会从云厂商开通新 VM。当节点利用率低时，它会排空并终止它们。

- **KEDA**（Kubernetes Event-Driven Autoscaling）：基于外部事件源（Kafka 队列深度、HTTP 请求率）扩展。非常适合 inference：请求队列增长时扩容模型服务器，队列空时缩容。

## 存储

| 类型 | 特点 | 用例 | 示例 |
|------|----------------|----------|---------|
| **Block** | 低 latency，挂载到单个 VM | OS 盘、数据库 | AWS EBS、GCP Persistent Disk |
| **Object** | 容量无限，HTTP 访问 | 数据集、模型权重、日志 | AWS S3、GCS、Azure Blob |
| **File** | 跨 VM 共享、POSIX | 共享训练数据 | AWS EFS、GCP Filestore、NFS |
| **Data lake** | 读时 schema、原始数据 | 分析、特征工程 | Delta Lake、Iceberg、Hudi |

- **对 ML 训练**：数据集存于对象存储（S3/GCS）。训练脚本把数据从对象存储读入 RAM。要实现快速随机访问（打乱的数据加载），可以：(1) 训练前把数据集下载到本地 SSD；(2) 使用高吞吐文件系统（Lustre、FSx）；或 (3) 使用高效流式加载与缓存的数据加载库（WebDataset、FFCV）。

- **模型权重**：带版本管理地存于对象存储。一个 70B 模型在 FP16 下约 140 GB。以 1 GB/s 从 S3 加载约需 2.5 分钟。缓存到本地 SSD 可缩短 inference 的冷启动时间。

## 云网络

- **VPC**（Virtual Private Cloud，虚拟私有云）：云中一个隔离网络。你的 VM、数据库与服务在 VPC 内通信。外部流量经负载均衡器或网关进入。

- **子网**（subnets）：把 VPC 划分成若干段。公有子网可访问互联网（用于 API 服务器）；私有子网不可（用于数据库、GPU worker）。这是安全上最小权限原则在网络层面的对应。

- **安全组**（AWS）/ **防火墙规则**（GCP）：控制哪些流量被允许。“允许任来源经 80 端口入站 HTTP。仅允许我的 IP 经 22 端口入站 SSH。阻止其他一切。”配置错误的安全组是云安全事件的头号原因。

- **服务网格**（service mesh，Istio、Envoy）：管理 K8s 内部服务间通信。提供：mTLS 加密（每次服务间调用都加密）、流量路由（A/B testing：把 10% 流量路由到新模型）、重试、超时、熔断，以及可观测性（哪个服务调用了哪个服务、耗时多久）。

## Serverless

- **Serverless**（AWS Lambda、GCP Cloud Functions）：你上传一个函数，云厂商在触发时运行它。无需管理服务器，无需配置扩展。按调用付费（通常 $0.20/百万次调用 + 计算时间）。

- **冷启动**（cold starts）：在一段空闲之后的第一次调用耗时更长（厂商需要分配容器并加载你的代码）。冷启动在 0.5-5 秒之间，使 serverless 不适合对 latency 敏感的 ML inference。

- **对 ML**：serverless 适用于：预处理（送入模型前调整图像大小）、后处理（格式化模型输出、发送通知）、编排（新数据到达时触发训练流水线），以及轻量级 inference（能容忍冷启动的小模型）。

- Serverless **不**适用于：GPU inference（多数 serverless 平台不支持 GPU）、长时间运行的训练任务（Lambda 有 15 分钟超时），或需要持久状态的服务（调用之间无持久状态）。

## 成本管理

- 云成本是 ML 团队头号运营关切。单块 H100 实例约 $8/小时。64-GPU 训练约 $500/小时。为期一个月的训练约 $360,000。成本优化属于工程而非财务。

- **Spot/preemptible 实例**：以 60-90% 折扣出售的空闲云容量。厂商可在 30 秒到 2 分钟通知后回收。用于：容错训练（频繁 checkpoint，在新实例上恢复）、批量 inference、数据预处理。不用于：对 latency 敏感的 serving（中断 = 停机）。

- **预留实例**：承诺 1-3 年用量换 30-60% 折扣。用于：你清楚基线负载的稳态 inference serving。

- **autoscaling**：高峰时段扩容，夜间/周末缩容。一个峰值需 10 块 GPU、夜间只需 2 块的模型服务器，相比 24/7 跑 10 块 GPU，autoscaling 可省约 60%。

- **合理选型**（right-sizing）：别为能跑在 A10G 上的 7B 模型用 H100。让 GPU 与工作负载匹配。用 profiling（第 16 章）确定最合适的 GPU。

- **存储成本**：对象存储便宜（S3 Standard 为 $0.023/GB/月），但会累积。一个把每次训练 checkpoint 都保存下来（每个 10 GB、每次实验 100 个、50 次实验）的团队会累积 50 TB = $1,150/月。设置生命周期策略自动删除旧 checkpoint。

## 多区域部署

- 对面向全球用户的 ML 系统而言，部署在单一区域意味着远端用户 latency 高（东京用户访问美国服务器增加约 150ms 网络往返），以及单点故障（区域宕机则整服务下线）。

- **多区域模式**：

    - **主-备**（active-passive）：一个主区域处理全部流量。一个次区域持有热备用（数据已复制、随时可接流量）。主区故障时 DNS 切到次区。故障转移期间停机时间：30 秒到数分钟。

    - **双活**（active-active）：两个区域同时处理流量。用户被路由到最近区域。两区都有最新数据（异步或同步复制）。单区域故障时不停机——流量被自动重路由。

- **数据复制**：难点所在。模型权重易于复制（在每个区域复制到 S3）。feature store 数据必须以可接受的 staleness 复制。用户数据可能有**数据驻留要求**（GDPR：欧洲用户数据必须留在欧洲）。

- **GPU 云定价对比**（约数，2026 年）：

| GPU | AWS | GCP | Azure | 典型用途 |
|-----|-----|-----|-------|-------------|
| A10G (24 GB) | $1.00/hr (g5) | $0.90/hr | $0.90/hr | 小模型 inference |
| A100 (80 GB) | $4.10/hr (p4d) | $3.70/hr | $3.40/hr | 训练、大模型 inference |
| H100 (80 GB) | $8.00/hr (p5) | $7.50/hr | $7.00/hr | 前沿训练 |
| TPU v5e | n/a | $1.20/hr | n/a | 大规模 JAX 训练 |

- Spot/preemptible 定价通常比上述低 60-70%。价格随区域与可用性而异。

## 基础设施即代码

- **IaC**（Infrastructure as Code，基础设施即代码）把基础设施（VM、网络、数据库、K8s 集群）定义在带版本控制的配置文件里。你不再在 AWS 控制台点按钮，而是写代码描述你想要什么，由工具去创建。

- **Terraform**（HashiCorp）：标准 IaC 工具。兼容所有主流云厂商。声明式：你描述期望状态，Terraform 算出要创建/修改/删除什么来达到该状态。

```hcl
# main.tf — create a GPU VM for inference
resource "aws_instance" "model_server" {
  ami           = "ami-0abcdef1234567890"  # Deep Learning AMI
  instance_type = "g5.xlarge"               # A10G GPU

  tags = {
    Name = "model-server-prod"
  }
}

resource "aws_s3_bucket" "model_weights" {
  bucket = "my-model-weights-prod"

  versioning {
    enabled = true
  }
}
```

```bash
terraform init      # download provider plugins
terraform plan      # show what will change
terraform apply     # create the infrastructure
terraform destroy   # tear it all down
```

- **IaC 为何重要**：可复现（从代码重建整套基础设施）、可审计（git 历史显示谁改了什么）、灾难恢复（用同一份配置在另一区域重建）、环境一致性（dev、staging、prod 使用相同模板、不同参数）。

- **Pulumi**：类似 Terraform，但用真实编程语言（Python、TypeScript、Go）而非 HCL。当你的基础设施逻辑复杂（条件、循环、动态配置）时很有用。
