# ML 系统设计

*ML 系统设计把第 01-03 篇的基础设施模式应用到机器学习的特有挑战上。本文件涵盖 ML 生命周期、数据管理、训练基础设施、模型评估、serving 策略、特征工程、ML 流水线与监控。*

- 像“为 YouTube 设计推荐系统”这样的系统设计面试题，不是要你描述推荐算法。它要你设计**整个系统**：数据流水线、特征工程、模型训练、评估、serving、监控与迭代。本文件提供这一框架。

## ML 系统生命周期

![ML 系统生命周期：从问题定义到部署与监控，持续迭代](../images/ml_lifecycle.svg)

- 每个 ML 系统都遵循同一生命周期，无论它是垃圾分类器还是基础模型：

```
Problem Framing → Data → Features → Training → Evaluation → Deployment → Monitoring → Iteration
       ↑                                                                                    │
       └────────────────────────────────────────────────────────────────────────────────────┘
```

### 问题定义

- 在碰数据或模型之前，先定义：
    - **预测什么**？（点击概率、下一个 token、物体边界框）
    - **用户是谁**？（终端用户、内部分析师、其他 ML 模型）
    - **约束是什么**？（latency < 100ms、离线 batch 可以、必须跑在设备端）
    - **业务指标是什么**？（收入、参与度、准确率）以及 ML 指标与之如何关联？
    - **基线是什么**？（启发式、规则系统、现有模型）——你必须胜过它才能证明 ML 系统的合理性。

- **常见错误**：在理解问题前就跳到模型架构。“我们应该用 transformer”不是系统设计答案。“我们要在 200ms 内对 10M 个候选预测点击概率，因此需要两阶段系统：快速召回 + 小型排序模型”才是。

## 数据管理

### 数据采集与标注

- **显式标签**：人工标注数据（点击/未点击、物体边界框、对话质量评分）。昂贵（依复杂度约 $0.02-$10 每条）、慢且主观。

- **隐式标签**：从用户行为中派生标签。点击、停留时长、购买、跳过。便宜且海量但有噪声（点击不等于满意；跳过不等于不喜欢）。

- **程序化标注**（Snorkel）：编写标注函数（启发式、正则、已有模型）对每条样本投票。用统计方法聚合投票以生成概率标签。可扩展到数百万条样本，精度中等。

- **主动学习**（active learning）：模型识别自己最不确定的样本并请求人工标注。这最大化标注效率：1000 条主动挑选的标签可媲美 10,000 条随机标签。

### 数据质量

- **数据校验**：检查每批入站数据的 schema 违规（缺失字段、类型错误）、分布漂移（均值显著变化）与体量异常（期望 1M 行，实际 500K）。

- **Great Expectations** 与 **TFX Data Validation** 是定义数据期望并在被违反时告警的工具。

- **数据版本管理**：每次训练运行都应可复现。**DVC**（第 15 章）把数据文件与代码一起追踪。每个数据集版本拿到一个 hash；训练配置引用该 hash。

### Feature Store

![Feature store：同一份特征计算同时供离线 store（训练）与在线 store（serving），消除 training-serving skew](../images/feature_store.svg)

- **Feature store**（第 15 章）为训练与 serving 提供一致的特征。关键概念：

    - **离线特征**：由批流水线（Spark）计算，存于数据仓库。用于训练与批量 inference。例如：用户 30 天的平均会话时长、商品的总购买数。

    - **在线特征**：实时计算或预计算后从低 latency store（Redis、DynamoDB）serving。用于实时 inference。例如：用户最近 5 个动作、当前购物车内容。

    - **Training-serving skew**：如果特征计算在训练与 serving 间不一致，模型在 inference 时看到的特征值与训练时不同。feature store 通过对两者使用同一份计算来消除此问题。

## 训练基础设施

- 对本书读者而言，分布式训练已在第 6 章深入讲解（数据并行、模型并行、混合精度、scaling law）。这里聚焦**系统**方面：

- **实验追踪**（W&B、MLflow——第 15 章）：每次训练运行记录超参、指标、git commit、数据版本与硬件。这是模型领域中等价于版本控制的东西。

- **超参调优**：在超参上自动搜索。方法：grid search（穷举、昂贵）、random search（出奇地有效）、贝叶斯优化（对目标建模，在可能有改进处采样），以及 **ASHA**（Asynchronous Successive Halving：启动大量 trial，早早杀掉表现差的）。

- **训练流水线编排**（Airflow、Kubeflow——第 15 章）：自动化“数据准备 → 训练 → 评估 → 注册”序列。调度每日重训。失败时告警。

## 模型评估

### 离线评估

- **留出测试集**：在模型训练时从未见过的数据上评估。标准做法，但如果测试集不代表生产数据则可能误导。

- **分片评估**：在子组上评估（按用户人口统计、内容类型、语言、时间段）。一个总体准确率 95% 的模型可能对某个少数群体只有 70%——不可接受。

- **回测**（backtesting）：对时序或序列预测，按时间顺序在历史数据上评估。用截至时刻 $t$ 的数据训练，在 $t$ 到 $t + \Delta t$ 的数据上评估。避免用未来数据训练造成的泄漏。

### 在线评估

![A/B testing：随机把用户分成对照组（旧模型）与实验组（新模型），带统计显著性比较指标](../images/ab_testing.svg)

- **A/B testing**：把线上流量随机分成对照组（旧模型）与实验组（新模型）。带统计显著性地比较业务指标（收入、参与度、留存）。评估 ML 变更的金标准。

    - **样本量**：需要足够数据才能检测到预期效应量。点击率 0.1% 的提升需要数百万次曝光才能显著检出。

    - **时长**：至少跑一个完整周期（多数产品 1-2 周）以捕捉星期效应。

    - **护栏指标**：在目标指标之外监控那些**不应**变化的指标（页面加载时间、错误率、崩溃率）。一个既提高收入又增加崩溃的模型净效果为负。

- **影子部署**（shadow deployment）：在生产中把新模型与旧模型并行运行。两者收到相同请求，但只有旧模型的预测会提供给用户。对比输出。这能在不给用户带来风险的情况下发现 bug 与质量问题。

- **交错**（interleaving）：对排序问题，把新旧模型的结果交错在单个列表中。用户与交错列表交互，你衡量哪个模型的结果参与度更高。比 A/B testing 需要更少用户即可达到显著性。

## 模型 serving

### 批量 vs 实时

- **批量 inference**（batch inference）：对所有可能输入预计算预测。存到数据库/缓存。从缓存 serving。适用情形：输入空间有限（每晚为所有用户生成推荐）、新鲜度非关键（每日预测即可）、对 latency 容忍度高。

- **实时 inference**：按每个请求按需计算预测。适用情形：输入空间无限（任意用户查询）、新鲜度重要（此刻就为该特定查询预测）、latency 必须低。

- 许多系统**两者并用**：批量预计算一组候选（便宜，覆盖 80% 流量），实时处理其余（昂贵，覆盖长尾查询与新用户）。

### 模型版本与注册

- **Model registry**（MLflow、W&B、SageMaker）带元数据存储已训练模型：
    - 版本号与训练日期。
    - 训练配置与数据版本。
    - 评估指标（准确率、latency、内存占用）。
    - 阶段：development → staging → production → archived。

- **回滚**（rollback）：如果新模型在生产中使指标下降，立即回退到上一版本。注册表让这成为一键操作。

## 特征工程

- **特征工程**（feature engineering）把原始数据变换为模型所需的输入。它往往是 ML 中杠杆最高的活动：更好的特征能改善每个模型，而更好的模型则受限于它收到的特征。

### 在线 vs 离线特征

- **离线特征**预计算且变化缓慢（用户人口统计、30 天聚合）。由批流水线（Spark）计算，存于 feature store。

- **在线特征**反映当前状态且变化迅速（购物车中的商品、上一个动作、当前位置）。从事件流实时计算或从快速 store 查找。

- **特征新鲜度**：某些特征需要秒级新鲜（欺诈检测：结合最近 5 笔交易看当前交易是否异常？）。另一些可接受数小时陈旧（推荐：基于历史看该用户偏好哪些类型？）。越新的特征计算与 serving 越贵。

### 常见特征模式

- **计数特征**：时间窗口内的事件计数（最近 7 天的购买、最近 24 小时的登录）。
- **Embedding 特征**：为类别变量学习的 embedding（用户 embedding、商品 embedding、查询 embedding）。它们是双塔模型及类似架构的输入。
- **交叉特征**：两个或多个特征的组合（user_age × item_category）。捕捉单个特征遗漏的交互。
- **时间特征**：距上次动作的时间、星期几、小时。捕捉时间模式。
- **聚合特征**：某个数值特征在某个分组上的均值、中位数、最小、最大、标准差（该卖家商品的平均评分）。

## ML 流水线

- ML 流水线编排从数据到已部署模型的整个工作流：

```
Data ingestion → Validation → Feature engineering → Training → Evaluation → Registration → Deployment → Monitoring
```

- 每一步都是编排器（Airflow、Kubeflow、Metaflow——第 15 章）中的一个任务。该流水线：
    - 按调度运行（每日重训）或按触发运行（有新数据可用）。
    - 幂等（重跑产出相同结果）。
    - 带重试逻辑（若特征计算失败，带退避重试 3 次）。
    - 产出带版本并存储的制品（已训练模型、评估报告、特征统计）。

- **Metaflow**（Netflix/Outerbounds）特别适合 ML：它把代码、数据与模型一同版本化，支持用同一份代码在本地开发与云端执行，并与 K8s、AWS 集成。

## 监控

- 我们在第 15 章讲过监控基础（Prometheus、Grafana、告警）。这里聚焦 **ML 专属监控**：

### 数据漂移

- **数据漂移**（data drift）指入站数据分布相对训练数据发生变化。在夏季数据上训练的模型可能在冬季数据上表现差（用户行为不同、商品可得性不同）。

- **检测**：用统计检验比较入站特征分布与训练分布：
    - **KS 检验**（Kolmogorov-Smirnov）：比较两个经验分布。检验它们是否来自同一底层分布。
    - **PSI**（Population Stability Index，群体稳定性指数）：衡量分布漂移程度。PSI < 0.1 稳定，0.1-0.25 中等漂移，> 0.25 显著。
    - **Embedding 漂移**：用质心距离或 MMD（Maximum Mean Discrepancy，最大均值差异）比较入站查询的 embedding 分布与训练集。

### 概念漂移

- **概念漂移**（concept drift）指输入与输出之间的关系发生变化。特征看起来一样，但正确预测却不同。例如：文化事件、疫情或产品变更之后用户偏好转变。

- 概念漂移比数据漂移更难检测，因为它需要带标签数据。监控代理指标：点击率、转化率、用户满意度评分。持续下滑暗示概念漂移。

### 模型退化

- 模型会随时间退化，原因多种：数据漂移、概念漂移、特征流水线 bug（某个特征开始返回 null），以及上游数据变化（第三方 API 改了响应格式）。

- **响应**：检测到退化时，行动依严重程度而定：
    - 轻微：在近期数据上重训（计划性重训即可处理）。
    - 中等：调查根因（哪个特征变了？哪个用户分段受影响？）。
    - 严重：立即回滚到上一模型版本，再调查。

### 反馈回路

- ML 系统会形成**反馈回路**（feedback loops）：模型的预测影响用户行为，而用户行为又成为下一版模型的训练数据。这些回路可能良性也可能恶性。

- **正反馈回路**（危险）：推荐模型主要展示热门商品 → 用户点击热门商品（因为只看到这些）→ 模型学到热门商品更热门 → 多样性崩溃。模型制造出印证自身偏差的数据。

- **负反馈回路**（同样危险）：欺诈检测模型抓住所有 A 类欺诈 → 没有 A 类欺诈进入训练数据 → 下一版模型学不到检测 A 类 → A 类欺诈重现。

- **缓解措施**：
    - **探索**（exploration）：展示一些模型不确定的商品（epsilon-greedy、Thompson sampling）。这产生多样化的训练数据。
    - **反事实日志**（counterfactual logging）：记录模型*本会*预测什么，而非仅记录用户看到的。在反事实数据上训练以去偏。
    - **留出集**（holdout sets）：随机让一部分流量不经模型过滤直接 serving。未过滤的数据为评估模型质量提供真值。
    - **延迟标签**（delayed labels）：等到真实结果出现后再用该数据训练。今天被点击的推荐明天可能被后悔。欺诈预测必须等到拒付窗口（30-90 天）。

### Embedding 表管理

- 大规模 ML 系统常有 1 亿+ 条目的 embedding 表（每个用户、商品、广告或实体一个 embedding）。在大规模下管理它们是一个系统挑战：

- **存储**：1 亿实体 × 256 维 × float16 = 50 GB。放不进 GPU 显存。方案：存于 CPU 内存并在 GPU 侧缓存、跨多机分片，或使用 **hash embedding**（把实体哈希到固定大小表，接受冲突）。

- **更新**：模型重训时 embedding 会变。把新 embedding 表部署到 serving 需要：在不中断线上流量的情况下把 50 GB 加载进内存、校验正确性，并在指标下降时回滚。对 embedding 表使用蓝绿部署。

- **陈旧**：新创建的用户没有 embedding（冷启动问题）。方案：用默认 embedding、通过 feature-to-embedding 模型从用户特征派生 embedding，或回退到非个性化模型。

### 公平性与偏差

- ML 系统可能系统性地对不同群体区别对待，这往往反映训练数据中的偏差。**公平性监控**是一项责任，而非可选特性。

- **需监控的指标**：
    - **人口统计平价**（demographic parity）：不同群体（性别、种族、年龄）的正预测率是否不同？
    - **机会均等**（equal opportunity）：不同群体的真正例率是否不同？（招聘模型应在所有群体中同等擅长识别合格候选人。）
    - **校准**（calibration）：若模型说 P(qualified) = 0.7（群体 A），是否真有 70% 的群体 A 合格？群体 B 是否也如此？

- **实践步骤**：
    - 在分片（子组）上评估模型表现，而不只是看总体指标。
    - 把公平性指标纳入模型评估流水线（一个提升总体准确率却让某群体退化的模型，未经评审不应部署）。
    - 记录已知限制与失败模式。
    - 为部署于敏感领域（招聘、放贷、刑事司法、医疗）的模型建立评审流程。
