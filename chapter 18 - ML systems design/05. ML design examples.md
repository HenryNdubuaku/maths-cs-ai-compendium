# Примеры проектирования систем машинного обучения

*Лучший способ изучить проектирование систем машинного обучения — это разбор практических примеров. В этом файле рассматриваются семь полноценных проектов: рекомендательные системы, ранжирование результатов поиска, предсказание кликов в рекламе, обнаружение мошенничества, модерация контента, разговорный ИИ и крупномасштабный поиск изображений.*

- Каждый пример следует единому фреймворку:
    1. **Постановка задачи**: что мы создаем, кто наши пользователи, каковы ограничения?
    2. **Данные**: какие данные у нас есть, как они собираются, как размечаются?
    3. **Признаки**: какие признаки нужны модели?
    4. **Модель**: какая архитектура и подход к обучению?
    5. **Инференс**: как модель развертывается и обслуживается?
    6. **Оценка**: как мы измеряем успех?
    7. **Итерация**: какие улучшения мы внесем со временем?

## 1. Рекомендательная система (например, YouTube, Netflix, Spotify)

### Постановка задачи

- **Цель**: показывать пользователям контент, который им понравится, максимизируя вовлеченность (время просмотра, прослушивания, клики).
- **Масштаб**: 1 млрд+ пользователей, 100 млн+ элементов, 10 тыс.+ рекомендаций в секунду.
- **Задержка**: <200 мс для всего пайплайна рекомендаций.
- **Ключевая проблема**: пространство кандидатов огромно (100 млн элементов). Невозможно оценить все элементы для всех пользователей в режиме реального времени.

### Архитектура: двухэтапный пайплайн

![Recommendation pipeline: 100M items narrowed to 1000 by candidate generation, ranked to 100, re-ranked to 20 shown items](../images/recommendation_pipeline.svg)

```
100M items → Candidate Generation (fast, coarse) → 1000 candidates
          → Ranking (slow, precise) → 100 ranked items
          → Re-ranking (business rules) → 20 shown to user
```

### Генерация кандидатов

- **Цель**: сократить 100 млн элементов до ~1000 кандидатов. Должно работать быстро (<50 мс).
- **Двухбашенная модель (two-tower model)**: кодирует пользователей и элементы в одно и то же пространство эмбеддингов. Эмбеддинг пользователя отражает предпочтения; эмбеддинг элемента — характеристики контента. Оценка = скалярное произведение эмбеддингов пользователя и элемента.
- **Обучение**: контрастивное обучение на тройках (пользователь, положительный_элемент, отрицательные_элементы). Положительные = элементы, с которыми взаимодействовал пользователь. Отрицательные = случайные элементы + «сложные» отрицательные примеры (популярные элементы, с которыми пользователь не взаимодействовал).
- **Инференс**: предварительное вычисление всех эмбеддингов элементов. Во время запроса: вычисление эмбеддинга пользователя, ANN-поиск (HNSW в векторной базе данных) для нахождения 1000 ближайших эмбеддингов элементов.

### Ранжирование

- **Цель**: точно оценить 1000 кандидатов. Допустимо ~100 мс.
- **Модель**: глубокая нейронная сеть (MLP или трансформер), которая использует богатый набор признаков: признаки пользователя (демография, история, контекст), признаки элемента (контент, популярность, свежесть) и кросс-признаки (история взаимодействий пользователя с элементом, контекстная релевантность).
- **Выход**: предсказанная вероятность взаимодействия (клик, просмотр 50%+, лайк, репост). Можно объединить несколько целевых функций: $\text{score} = w_1 \cdot P(\text{click}) + w_2 \cdot P(\text{watch}) + w_3 \cdot P(\text{like})$.

### Переранжирование

- Применение бизнес-правил: разнообразие (не показывать 5 видео от одного автора), свежесть (продвигать новый контент), безопасность (фильтровать помеченный контент) и персонализированное исследование (показывать некоторые менее релевантные элементы, которые пользователь мог бы открыть для себя).

### Оценочные расчеты (Back-of-Envelope)

- **Индекс эмбеддингов элементов**: 100 млн элементов × 256-размерность × float16 = 50 ГБ. Индекс HNSW добавляет ~2x накладных расходов → ~100 ГБ. Помещается на одной машине со 128 ГБ ОЗУ или шардируется на 4 машины по 32 ГБ.

- **Вычисление эмбеддинга пользователя**: ~5 мс на пользователя (небольшая MLP на признаках пользователя). При 10 тыс. QPS нужно ~50 реплик модели для обработки нагрузки.

- **ANN-поиск**: ~2 мс для топ-1000 из 100 млн векторов с помощью HNSW. При 10 тыс. QPS каждая реплика индекса обрабатывает ~500 QPS → нужно 20 реплик.

- **Модель ранжирования**: 1000 кандидатов × ~0,1 мс на кандидата = 100 мс на запрос. При 10 тыс. QPS нужно 1000 GPU-секунд в секунду → ~10 GPU A10G только для ранжирования.

- **Общая инфраструктура**: ~20 реплик индекса эмбеддингов + ~50 серверов для эмбеддингов пользователей + ~10 GPU для ранжирования + кеширование + балансировщики нагрузки. Стоимость: ~$50K-$100K/месяц по облачным ценам.

### Холодный старт

- **Новые пользователи** (нет истории): использование демографических признаков, контекста устройства/местоположения и рекомендаций на основе популярности. После 5–10 взаимодействий переключение на персонализированную модель.

- **Новые элементы** (нет данных о взаимодействиях): использование контентных признаков (заголовок, описание, эмбеддинги миниатюр). Выделение бюджета на исследование: показ новых элементов части пользователей для быстрого сбора данных о взаимодействии. Элементы без взаимодействий после периода продвижения понижаются в выдаче.

- **Холодный старт — это системная проблема**: хранилище признаков должно корректно обрабатывать отсутствующие признаки (возвращать значения по умолчанию, а не ошибки). Модель должна обучаться с учетом отсутствующих признаков (dropout на признаках истории пользователя во время обучения имитирует новых пользователей).

### Оценка

- **Офлайн**: NDCG (Normalized Discounted Cumulative Gain), recall@K, precision@K на отложенной выборке.
- **Онлайн**: A/B-тестирование с измерением времени просмотра, DAU, удержания (retention). Долгосрочные A/B-тесты (недели) для выявления эффектов удержания пользователей, которые пропускают короткие тесты.

## 2. Ранжирование результатов поиска (например, Google, Bing)

### Постановка задачи

- **Цель**: по запросу пользователя вернуть наиболее релевантные результаты из корпуса, состоящего из миллиардов документов.
- **Задержка**: <500 мс всего (100 мс поиск + 200 мс ранжирование + 100 мс рендеринг + накладные расходы).

### Архитектура: понимание запроса → поиск → ранжирование

### Понимание запроса

- Перед поиском обработайте исходный запрос для улучшения результатов:

- **Исправление опечаток**: "reccomendation systm" → "recommendation system". Используйте модель расстояния редактирования или sequence-to-sequence модель, обученную на парах (с опечаткой, исправлено) из логов поиска.

- **Расширение запроса**: добавление связанных терминов для улучшения полноты (recall). "Python ML" → "Python machine learning scikit-learn pytorch". Используйте словари синонимов, word embeddings или LLM для генерации расширений.

- **Классификация интента**: определение того, что хочет пользователь. "купить кроссовки Nike" — **транзакционный** (показывать страницы товаров). "Как работает обратное распространение ошибки" — **информационный** (показывать статьи). "facebook.com" — **навигационный** (перейти прямо на сайт). Разные интенты должны запускать разные стратегии поиска и макеты результатов.

- **Распознавание сущностей**: извлечение сущностей из запроса. "лучшие рестораны рядом с Таймс-сквер" → локация: "Таймс-сквер", тип сущности: "рестораны". Маршрутизация в пайплайн поиска с учетом местоположения.

### Поиск (Retrieval)

- **BM25** (традиционный): поиск на основе сопоставления терминов с использованием инвертированного индекса. Быстрый, эффективный для ключевых слов. Отсутствует семантическое понимание ("собачий корм" не совпадает с "питание для псовых").

- **Плотный поиск (Dense retrieval)**: кодирование запросов и документов в виде эмбеддингов (с использованием bi-encoder, таких как DPR или ColBERT). Поиск через ANN. Улавливает семантическое сходство ("собачий корм" совпадает с "питание для псовых"). Медленнее, чем BM25, но лучше подходит для запросов на естественном языке.

- **Гибридный поиск**: объединение BM25 и плотного поиска. BM25 находит точные совпадения по ключевым словам; плотный поиск находит семантические соответствия. Объединение и дедупликация. Лучшее из обоих миров.

### Ранжирование (Ranking)

- **Обучение ранжированию (Learning to rank)**: модель присваивает оценку каждой паре (запрос, документ). Три подхода:
    - **Поточечный (Pointwise)**: предсказание оценки релевантности для каждого документа независимо. Просто, но игнорирует относительный порядок.
    - **Попарный (Pairwise)**: предсказание того, какой из двух документов более релевантен. Классический подход — LambdaMART (градиентный бустинг над деревьями).
    - **Списочный (Listwise)**: оптимизация всего ранжированного списка напрямую для метрики уровня списка (NDCG). Более сложный, но дает лучшие результаты.

- **Cross-encoder**: трансформер, который принимает `[query, document]` на вход и выдает оценку релевантности. Точнее, чем bi-encoder (которые кодируют запрос и документ независимо), поскольку улавливает тонкие взаимодействия. Однако слишком медленный для всего корпуса — используется только для переранжирования топ-100–1000 кандидатов после этапа поиска.

### Признаки (Features)

- **Признаки запроса**: длина запроса, язык, классификация интента (навигационный, информационный, транзакционный).
- **Признаки документа**: PageRank, свежесть, оценка качества контента, авторитетность домена.
- **Признаки пары запрос-документ**: оценка BM25, сходство эмбеддингов, количество точных совпадений, CTR для этой пары (запрос, документ) в исторических логах.

## 3. Предсказание кликабельности рекламы (Ads Click Prediction)

### Постановка задачи

- **Цель**: предсказать вероятность того, что пользователь кликнет по рекламе. Это определяет размер ставки в аукционах реального времени.
- **Масштаб**: 100 тыс.+ аукционов в секунду, каждый требует предсказания в течение 10 мс.
- **Влияние на доход**: улучшение точности предсказания клика на 0,1% превращается в миллионы дополнительного дохода.

### Архитектура

- **Генерация признаков (Feature engineering)** — ядро рекламных систем. Признаки включают:
    - **Признаки пользователя**: демография, история просмотров, история покупок, устройство, местоположение, время суток.
    - **Признаки объявления**: креатив (изображение/текст), рекламодатель, категория, исторический CTR, размер ставки.
    - **Контекстные признаки**: контент страницы, позиция объявления, тип устройства, скорость соединения.
    - **Кросс-признаки**: взаимодействие user_category × ad_category, взаимодействие user_region × ad_campaign.

- **Модель**: исторически логистическая регрессия (простая, быстрая, интерпретируемая). Современные системы используют глубокое обучение: **DLRM** (Deep Learning Recommendation Model) с таблицами эмбеддингов для категориальных признаков и MLP для плотных признаков.

- **Калибровка**: предсказанная вероятность должна быть точной (если модель говорит P(click) = 0.05, то 5% таких показов должны приводить к клику). Калибровка критически важна, так как предсказанная вероятность напрямую определяет размер ставки.

- **Исследование-эксплуатация (Exploration-exploitation)**: постоянный показ предсказанного лучшего объявления неоптимален в долгосрочной перспективе (вы никогда не узнаете, что новое объявление может быть лучше). Thompson sampling или $\epsilon$-greedy исследование гарантируют, что некоторая доля показов достанется менее определенным объявлениям для сбора данных.

### Торги в реальном времени (Real-Time Bidding)

- Когда пользователь загружает страницу, рекламный аукцион проходит менее чем за 100 мс:
    1. Паблишер отправляет запрос на ставку (информация о пользователе, контекст страницы) в несколько рекламных бирж.
    2. Сервер ставок каждого рекламодателя предсказывает CTR для своих объявлений.
    3. Ставка = CTR × value_per_click. Более высокие ставки выигрывают аукцион.
    4. Выигрышное объявление показывается; если по нему кликают, рекламодатель платит.

## 4. Детекция мошенничества (Fraud Detection)

### Постановка задачи

- **Цель**: обнаружение мошеннических транзакций в реальном времени (мошенничество с кредитными картами, захват аккаунта, фейковые отзывы).
- **Задержка**: <100 мс (транзакция должна быть одобрена или помечена до завершения платежа).
- **Ключевая проблема**: экстремальный дисбаланс классов (0,1% мошенничества). Ложноположительные срабатывания блокируют легитимных пользователей; ложноотрицательные приводят к потере денег.

### Архитектура

![Fraud detection pipeline: transaction → real-time feature pipeline → ML model → decision engine → allow/review/block, with human review feeding labels back for retraining](../images/fraud_detection_pipeline.svg)

### Признаки (Features)

- **Признаки транзакции**: сумма, валюта, категория мерчанта, время суток, является ли международной.
- **Признаки пользователя**: возраст аккаунта, средняя сумма транзакции, количество недавних транзакций, цифровой отпечаток устройства.
- **Признаки скорости (Velocity features)** (реальное время, из потокового пайплайна): количество транзакций за последние 5 минут, количество различных мерчантов за последний час, географическое расстояние от предыдущей транзакции.
- **Графовые признаки**: связан ли этот мерчант с известными мошенническими сетями? Используется ли это устройство с помеченными аккаунтами?

### Модель

- **Градиентный бустинг над деревьями** (XGBoost, LightGBM) — стандарт для табличной детекции мошенничества. Они работают со смешанными типами признаков, интерпретируемы (важность признаков) и быстро обучаются.

- **Работа с дисбалансом**: недосэмплирование мажоритарного класса, пересэмплирование миноритарного (SMOTE) или использование весов классов в функции потерь. Focal loss (глава 8) снижает веса для простых негативных примеров.

- **Матрица стоимости**: ложноположительное срабатывание (блокировка легитимной транзакции) имеет свою стоимость (недовольство пользователя, потерянная продажа). Ложноотрицательное (пропуск мошенничества) имеет другую стоимость (финансовый убыток). Порог принятия решения должен минимизировать общую ожидаемую стоимость, а не максимизировать точность.

### Человек в контуре (Human-in-the-Loop)

- Неоднозначные предсказания (уверенность модели от 0,3 до 0,7) отправляются на проверку людям. Решения проверяющих становятся метками для дообучения. Это создает цикл обратной связи: модель улучшается со временем, видя больше размеченных случаев мошенничества.

## 5. Модерация контента

### Постановка задачи

- **Цель**: автоматическое обнаружение и удаление вредоносного контента (язык вражды, насилие, дезинформация, CSAM) на платформе.
- **Масштаб**: миллиарды публикаций в день (текст, изображения, видео).
- **Сложность**: зависимость от контекста (сарказм, сатира, культурные нюансы). Необходимо соблюдать баланс между свободой выражения мнений и безопасностью.

### Архитектура

- **Мультимодальная классификация**: отдельные модели для текста, изображений и видео с уровнем слияния (fusion layer), который объединяет их сигналы.

- **Модерация текста**: дообученная языковая модель классифицирует текст по категориям (домогательства, язык вражды, дезинформация, спам). Мультиязычные модели поддерживают более 100 языков.

- **Модерация изображений**: модель компьютерного зрения обнаруживает: неприемлемый контент (нагота, насилие), текст на изображениях (OCR + текстовый классификатор) и известный вредоносный контент (хеш-сравнение с базами данных известных материалов CSAM).

- **Модерация видео**: выборка кадров через равные промежутки времени, запуск классификатора изображений на каждом кадре, объединение с транскрипцией аудио (ASR → текстовый классификатор).

- **Политика как код (Policy-as-code)**: политики модерации определяются в виде структурированных правил, которые сопоставляют выходные данные модели с действиями:

```python
if text_model.hate_speech_score > 0.9:
    action = "remove"
elif text_model.hate_speech_score > 0.7:
    action = "human_review"
else:
    action = "allow"
```

- Политики часто меняются (новые нормативные акты, меняющиеся нормы). Отделение политики от модели гарантирует, что изменения можно внедрять без переобучения.

### Проактивная и реактивная модерация

- **Проактивная** (до публикации): запуск классификаторов для контента до того, как он станет общедоступным. Нарушения с высокой степенью уверенности блокируются автоматически. Это предотвращает появление вредоносного контента, но увеличивает задержку публикации и создает риск ложноположительных срабатываний (блокировка легитимного контента).

- **Реактивная** (после публикации): контент становится доступным немедленно. Пользователи могут сообщать о нарушениях. Жалобы инициируют проверку классификатором + проверку человеком. Меньшая задержка для авторов, но вредоносный контент остается видимым до момента обнаружения.

- **Большинство платформ используют оба подхода**: проактивный для категорий высокой степени тяжести (CSAM: нулевая терпимость, блокировка до публикации) и реактивный для нюансированных категорий (дезинформация: требует человеческого суждения, проверка после жалоб).

### Хеш-сравнение

- Для известного вредоносного контента (CSAM, террористическая пропаганда) используется **перцептивное хеширование**: вычисление хеша изображения/видео, устойчивого к незначительным изменениям (обрезка, изменение размера, сжатие). Сравнение с базами данных известного вредоносного контента (база хешей NCMEC, общая база хешей GIFCT). Совпадение → немедленное удаление без необходимости использования классификатора.

- **PhotoDNA** (Microsoft) — стандартный перцептивный хеш для обнаружения CSAM. Во многих юрисдикциях это юридическое обязательство, а не просто технический выбор.

### Приблизительные оценки (Back-of-Envelope)

- **Масштаб**: 1 млрд публикаций в день ≈ 12 тыс. публикаций в секунду. Для каждой публикации требуется: классификация текста (~5 мс), классификация изображения (~20 мс), хеш-сравнение (~1 мс). При 12 тыс. QPS: нужно ~60 текстовых классификаторов, ~240 классификаторов изображений и ~12 узлов хеш-сравнения (плюс избыточность).

- **Проверка человеком**: если 2% публикаций отправляются на проверку = 20 млн в день. При 100 проверках на одного модератора в день требуется 200 тыс. модераторов (вот почему важна точность автоматизации: каждые 0,1% снижения ложноположительных срабатываний экономят 1 млн проверок в день).

- **Бюджет задержки**: проактивная модерация должна завершаться в рамках пайплайна публикации (~500 мс). Текст (5 мс) + изображение (20 мс) + хеш (1 мс) + накладные расходы = вполне укладывается в бюджет. Видео — исключение: даже выборка 1 кадра в секунду из 10-минутного видео требует 600 вызовов классификатора → обработка в асинхронном режиме.

### Рабочий процесс эскалации

- Автоматическое удаление → проверка апелляций человеком → проверка специалистами (юридические, культурные эксперты) → команда по политике для неоднозначных случаев. Каждый уровень обрабатывает меньше случаев с большей степенью нюансированности.

- **Обратная связь для модели**: решения, принятые при проверке человеком, являются наиболее качественными метками для дообучения. Разногласия между моделью и модераторами имеют приоритет для активного обучения (active learning) — они представляют собой случаи, с которыми модель справляется хуже всего.

## 6. Разговорный ИИ (чат-бот на основе RAG)

### Постановка задачи

- **Цель**: чат-бот, отвечающий на вопросы о продуктах компании с использованием её документации.
- **Требования**: точность (отсутствие галлюцинаций), цитирование источников, обработка уточняющих вопросов и работа строго в рамках домена продукта.

![RAG architecture: embed query, search vector DB for relevant chunks, rerank, feed to LLM with original query to generate grounded response](../images/rag_architecture.svg)

### Архитектура: Retrieval-Augmented Generation (RAG)

```
User query → Query Embedding → Vector Search (documentation) → Top-K chunks
                                                                      ↓
User query + Retrieved chunks → LLM → Response (with citations)
```

### Компоненты

- **Ингестия документов**: разбиение документов на фрагменты (chunking) и их эмбеддинг. **Стратегия разбиения** имеет существенное значение:

    - **Разбиение фиксированного размера**: разделение каждые N токенов (например, 500) с перекрытием в M токенов (например, 50). Простой, предсказуемый размер фрагментов, но может разрывать предложение или абзац, теряя контекст.

    - **Семантическое разбиение**: разделение по границам абзацев или разделов. Каждый фрагмент является связной единицей информации. Переменный размер (некоторые фрагменты по 100 токенов, другие по 800), что требует от системы поиска обработки переменной длины.

    - **Рекурсивное разбиение**: попытка разделения по границам абзацев. Если абзац слишком длинный, разделение по границам предложений. Если предложение слишком длинное, разделение по фиксированному размеру. Лучший баланс связности и согласованности размера.

    - **Эмбеддинг**: эмбеддинг каждого фрагмента с помощью текстового энкодера (например, E5, BGE, Cohere embed). Хранение в векторной базе данных.

- **Поиск (Retrieval)**: эмбеддинг запроса пользователя, поиск в векторной базе данных $k$ наиболее похожих фрагментов (обычно $k = 5$-$10$). Опционально — переранжирование с помощью кросс-энкодера для более высокой точности.

- **Генерация**: создание промпта с извлеченными фрагментами в качестве контекста:

```
System: You are a helpful assistant. Answer based ONLY on the provided context.
If the answer is not in the context, say "I don't know."

Context:
[chunk 1]
[chunk 2]
...

User: {question}
```

- **Guardrails**: предотвращают ответы LLM на вопросы вне предметной области продукта, генерацию вредоносного контента или противоречия извлеченному контексту. Реализуются через: фильтрацию входных данных (отклонение нерелевантных запросов), фильтрацию выходных данных (проверка ответа на соответствие извлеченному контексту) и конституциональный промптинг (инструктаж модели отказывать в определенных запросах).

- **Память диалога**: сохранение последних $n$ реплик разговора. Включение их в промпт, чтобы модель понимала уточняющие вопросы («А как насчет цены?» → требует предыдущего контекста о том, о каком продукте идет речь).

### Переписывание запросов (Query Rewriting)

- Пользователи часто задают неоднозначные уточняющие вопросы: «А как насчет цены?» (цена чего?). **Переписывание запроса** использует историю диалога для создания автономного запроса:

    - Вход: история диалога + «А как насчет цены?»
    - Переписанный запрос: «Какова цена корпоративного плана для продукта X?»

- Именно этот переписанный запрос преобразуется в эмбеддинг и ищется в векторной базе данных. Без переписывания поиск по слову «цена» выполнялся бы без контекста и возвращал нерелевантные фрагменты.

- Переписывание запроса можно выполнить с помощью вызова небольшой LLM (~50 мс) или дообученной модели sequence-to-sequence (~5 мс).

### Оценочные расчеты (Back-of-Envelope Numbers)

- **Корпус документации**: 10 тыс. страниц, в среднем по 2000 токенов каждая = 20 млн токенов. При 500 токенах на чанк с перекрытием 50 = ~44 тыс. чанков.
- **Индекс эмбеддингов**: 44 тыс. чанков × 768-размерность × float16 = ~65 МБ. Легко помещается в память. Даже 10 млн чанков составили бы ~15 ГБ.
- **Разбивка задержки**: эмбеддинг запроса (5 мс) + векторный поиск (2 мс) + кросс-энкодер для ранжирования (20 мс для топ-50) + генерация LLM (500–2000 мс) = ~600–2100 мс всего. LLM занимает основное время. Используйте стриминг для снижения воспринимаемой задержки.
- **Стоимость**: при $3/1M tokens (Claude/GPT-4 API), 1000 queries/day with ~2K tokens each = ~$6/day. При масштабировании (1 млн запросов в день) выгоднее развернуть собственную модель 7B на 2 GPU A10G (~$50/день), что даст 100-кратное снижение затрат.

### Оценка (Evaluation)

- **Качество поиска**: Recall@K (содержат ли топ-K чанков ответ?), MRR (Mean Reciprocal Rank).
- **Качество генерации**: фактическая точность (соответствует ли ответ извлеченному контексту?), обоснованность (ссылается ли ответ на правильные чанки?), релевантность ответа.
- **End-to-end**: удовлетворенность пользователей (лайки/дизлайки), частота перевода диалога на оператора-человека.

## 7. Поиск изображений в большом масштабе

### Постановка задачи

- **Цель**: по заданному изображению найти визуально похожие изображения в корпусе из 1 млрд+ изображений.
- **Применение**: обратный поиск изображений, поиск товаров (фото → подходящие товары), поиск дубликатов.
- **Задержка**: <500 мс, включая сетевые задержки.

### Архитектура

```
Query image → Embedding Model (ViT/CLIP) → 512-dim vector → ANN Search → Top-K results
```

### Извлечение эмбеддингов

- **Модель**: предобученный визуальный энкодер (ViT, визуальный энкодер CLIP, DINOv2). При необходимости дообучается на специфическом домене (мода, электронная коммерция, медицинские изображения).

- **Обучение**: контрастивное обучение (глава 10). Положительные пары = разные ракурсы одного и того же изображения (или изображение + соответствующий текст). Отрицательные пары = случайные изображения. Модель учится создавать похожие эмбеддинги для похожих изображений и непохожие — для разных.

### Индексация

- **Оффлайн**: создание эмбеддингов для всех 1 млрд изображений и построение ANN-индекса. Для HNSW (файл 03) построение индекса занимает часы, а сам индекс хранится в памяти (~128 ГБ для 1 млрд × 512-размерность × float16 + накладные расходы графа).

- **Шардирование**: разделение индекса между несколькими машинами. Каждая машина хранит свой шард. Во время запроса поиск выполняется параллельно по всем шардам, после чего результаты топ-K объединяются.

- **Инкрементальные обновления**: новые изображения (загрузки, новые товары) должны добавляться в индекс. HNSW поддерживает инкрементальную вставку без перестроения. Векторные базы данных (Milvus, Pinecone) поддерживают это нативно.

### Обслуживание (Serving)

- **Сервис эмбеддингов**: GPU-сервер, на котором работает модель ViT. Задержка: ~20 мс на изображение. Для повышения пропускной способности используйте батчинг запросов.

- **Сервис поиска**: сервер с ANN-индексом. Задержка: ~10 мс для поиска топ-100 по 1 млрд векторов (с использованием HNSW).

- **Кеширование**: кеширование результатов для популярных запросов. Для поиска дубликатов: кешируйте эмбеддинги недавно загруженных изображений и сравнивайте новые загрузки с кешем перед поиском по полному индексу.

### Оценка (Evaluation)

- **Precision@K**: действительно ли результаты в топ-K похожи?
- **Recall@K**: сколько из всех действительно похожих изображений в корпусе попало в топ-K?
- **Mean Average Precision (mAP)**: площадь под кривой точности-полноты.
- **Оценка человеком**: для субъективного сходства асессоры оценивают, являются ли найденные изображения релевантными.

## Фреймворк для собеседования

- При встрече с вопросом по системному дизайну следуйте этому фреймворку:

1. **Уточнение требований** (2–3 минуты): спросите о масштабе, задержках, требованиях к согласованности и граничных случаях. «Сколько пользователей? Какая задержка допустима? Что происходит при сбоях?»

2. **Высокоуровневый дизайн** (5–7 минут): нарисуйте основные компоненты и их взаимодействия. Начните с «счастливого пути» (happy path). Используйте паттерны из файлов 01–03.

3. **Детальный разбор** (15–20 минут): выберите наиболее интересный/сложный компонент и спроектируйте его детально. Здесь вы показываете глубину знаний. Для ML-системы это часто: архитектура модели, пайплайн признаков или архитектура обслуживания.

4. **Оценка и мониторинг** (3–5 минут): как измерить успех? Что может пойти не так? Как обнаруживать проблемы и реагировать на них?

5. **Итерация** (2–3 минуты): что бы вы улучшили при наличии большего времени/ресурсов? Это показывает, что вы понимаете компромиссы и умеете расставлять приоритеты.

- **На что смотрят интервьюеры**: структурированное мышление (не перескакивать сразу к решениям), понимание компромиссов (у каждого выбора есть цена), практические знания (вы действительно строили системы) и коммуникация (можете ли вы четко объяснить свой дизайн?).
