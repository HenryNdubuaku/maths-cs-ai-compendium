# Git и контроль версий

*Git — это инструмент, с помощью которого команды разработчиков работают вместе, не перезаписывая чужую работу. В этом файле рассматриваются ментальная модель Git, стратегии ветвления, слияние и перебазирование, разрешение конфликтов, pull-реквесты, а также управление специфическими для машинного обучения задачами, такими как работа с большими файлами и отслеживание экспериментов.*

- Любой серьезный программный проект использует контроль версий. **Git** — это доминирующая система, используемая практически всеми проектами с открытым исходным кодом и компаниями. Без git совместная работа — это пересылка zip-архивов по электронной почте с надеждой, что никто не перезапишет ваши изменения. С git каждое изменение отслеживается, может быть отменено и привязано к автору.

- Для инженеров по машинному обучению: git отслеживает ваш код, конфигурации и скрипты экспериментов. В сочетании с инструментами для отслеживания экспериментов это обеспечивает воспроизводимость: «какой именно код и конфигурация создали эту модель?»

## Ментальная модель

- Git отслеживает **снимки** (snapshots) вашего проекта. Каждый коммит — это полный снимок всех отслеживаемых файлов в данный момент, а не diff (внутренне git хранит diff-ы для эффективности, но концептуально каждый коммит — это полное состояние).

- Четыре «местоположения» для ваших файлов:

    1. **Working directory**: фактические файлы на диске. Вы редактируете их.
    2. **Staging area** (индекс): файлы, которые вы отметили для следующего коммита. `git add` перемещает изменения сюда.
    3. **Local repository**: ваша история коммитов, хранящаяся в `.git/`. `git commit` сохраняет состояние staging area как новый снимок.
    4. **Remote repository** (например, GitHub): общая копия. `git push` загружает ваши коммиты, `git pull` скачивает чужие.

```
Working Dir  →  git add  →  Staging  →  git commit  →  Local Repo  →  git push  →  Remote
                                                        ←  git pull  ←
```

- Staging area — это то, что делает git мощным инструментом. Вы можете отредактировать 10 файлов, но закоммитить только 3 из них, сохранив остальные изменения для отдельного коммита. Это позволяет делать чистые, сфокусированные коммиты.

### Основные команды

```bash
git init                          # create a new repository
git clone url                     # download a remote repository
git status                        # what has changed? (most-used command)
git add file.py                   # stage a specific file
git add .                         # stage all changes (use with caution)
git commit -m "descriptive msg"   # commit staged changes
git push                          # upload commits to remote
git pull                          # download + merge remote changes
git log --oneline                 # compact commit history
git diff                          # show unstaged changes
git diff --staged                 # show staged changes
```

## Ветвление

- **Ветвь** (branch) — это указатель на коммит. Ветвь по умолчанию называется `main` (или `master`). Создание ветви дает вам независимую линию разработки: вы можете вносить изменения, не затрагивая `main`.

```bash
git branch feature-x              # create a branch
git checkout feature-x            # switch to it
git checkout -b feature-x         # create and switch in one step
git branch -d feature-x           # delete branch (after merging)
git branch -a                     # list all branches (local + remote)
```

- **Когда создавать ветви**: всегда. Никогда не коммитьте напрямую в `main`. Каждая функция, исправление ошибки или эксперимент должны находиться в своей собственной ветви. Это сохраняет `main` стабильным и готовым к развертыванию.

### Стратегии ветвления

- **Feature branches** (наиболее распространены): каждая функция/исправление получает ветвь от `main`. Когда работа завершена, открывается pull request (PR) для слияния обратно. Просто и подходит для большинства команд.

- **Trunk-based development**: разработчики часто коммитят в `main` (несколько раз в день), используя feature flags, чтобы скрыть незавершенную работу. Предпочитается командами, которые используют непрерывное развертывание (Google, Facebook). Требует отличной CI/CD.

- **Gitflow**: отдельные ветви для функций, релизов и исправлений (hotfixes). Более сложная стратегия, лучше подходит для программного обеспечения с версионными релизами (мобильные приложения, упакованное ПО). Избыточна для большинства проектов по машинному обучению.

- Для команд по машинному обучению: **feature branches** с «короткоживущими» ветвями (слияние в течение 1-3 дней) — это оптимальный вариант. «Долгоживущие» ветви отклоняются от `main` и создают болезненные конфликты при слиянии.

## Слияние и перебазирование

- **Merge** создает новый «коммит слияния» (merge commit), который объединяет две ветви:

```bash
git checkout main
git merge feature-x
```

- Это сохраняет полную историю: вы можете видеть, что работа велась в ветви и когда она была слита. У коммита слияния два родителя.

- **Rebase** переносит коммиты вашей ветви поверх целевой ветви:

```bash
git checkout feature-x
git rebase main
```

- Это переписывает историю: коммиты вашей ветви получают новые хеши, как если бы вы начали свою работу с текущего кончика `main`. Результат — линейная история (без коммитов слияния), которую легче читать.

- **Когда что использовать**:
    - **Rebase** для обновления вашей feature-ветви последними изменениями из `main` (сохраняет вашу ветвь чистой и актуальной).
    - **Merge** для интеграции вашей feature-ветви в `main` (сохраняет историю ветви).
    - **Никогда не делайте rebase коммитов, которые были отправлены (pushed) и которыми пользуются другие**. Rebase переписывает историю; если кто-то другой основывал свою работу на исходных коммитах, rebase вызовет хаос.

## Разрешение конфликтов

- **Конфликт** возникает, когда две ветви изменяют одну и ту же строку одного и того же файла. Git не может автоматически решить, какое изменение оставить, и просит вас разрешить его вручную.

```
<<<<<<< HEAD
learning_rate = 0.001
=======
learning_rate = 0.0005
>>>>>>> feature-x
```

- Между `<<<<<<< HEAD` и `=======` находится версия из текущей ветви. Между `=======` и `>>>>>>> feature-x` — версия из вливаемой ветви. Вы решаете, что оставить (или объединяете их), удаляете маркеры, сохраняете файл и выполняете `git add` для разрешенного файла.

- **Ловушка**: не оставляйте маркеры конфликтов в закоммиченных файлах. Это обычный текст, который нарушит работу вашего кода. Всегда ищите `<<<<<<<` после разрешения конфликтов.

- **Уменьшение количества конфликтов**: делайте ветви короткоживущими, часто вливайте `main` в свою ветвь и избегайте одновременного редактирования одного и того же файла несколькими людьми.

## Написание хороших сообщений коммитов

- Сообщение коммита предназначено для вас в будущем и для ваших коллег. «fix bug» не говорит ни о чем. «Fix off-by-one in batch size calculation that caused OOM on 8-GPU training» говорит обо всем.

- **Формат**:

```
Short summary (50 chars or less, imperative mood)

Longer description if needed. Explain WHY, not WHAT
(the diff shows what changed). Wrap at 72 characters.

Fixes #123
```

- **Повелительное наклонение**: «Add feature», а не «Added feature» или «Adds feature». Читайте это как завершение предложения: «Если применить этот коммит, он **add feature**».

- **Атомарные коммиты**: каждый коммит должен делать что-то одно. «Add data loader» — это один коммит. «Add data loader and fix unrelated bug and update README» — это должно быть три коммита. Это делает возможным использование `git bisect` (поиск коммита, который внес баг).

## Pull Requests и Code Review

- **Pull request (PR)** предлагает слияние ветки в `main`. Это шлюз для code review: коллеги читают ваши изменения, предлагают улучшения и одобряют их перед слиянием.

- **Хорошие практики PR**:
    - Делайте PR небольшими (менее 400 строк изменений). Большие PR одобряются формально («rubber-stamped»), потому что никто не хочет проверять 2000 строк.
    - Пишите понятное описание: что изменилось, почему и как это протестировать.
    - Ссылайтесь на issue или тикет, который стал причиной изменений.
    - Оперативно отвечайте на комментарии к ревью.
    - Объединяйте (squash) тривиальные коммиты перед слиянием (чтобы в `main` была чистая история).

- **Code review — это не поиск багов** (для этого есть тесты). Это про: обмен знаниями (ревьюер изучает кодовую базу), обратную связь по дизайну (правильный ли это подход?) и поддержание стандартов (именование, стиль, архитектура).

## .gitignore

- Файл `.gitignore` указывает git, какие файлы исключить из отслеживания. Для проектов по машинному обучению:

```gitignore
# Python
__pycache__/
*.pyc
*.egg-info/
.venv/
env/

# Data and models (too large for git)
data/
*.csv
*.parquet
models/
*.pt
*.onnx
*.bin
checkpoints/

# Secrets
.env
*.pem
credentials.json

# IDE
.vscode/
.idea/
*.swp

# OS
.DS_Store
Thumbs.db

# Jupyter
.ipynb_checkpoints/

# Experiment outputs
wandb/
mlruns/
outputs/
logs/
```

- **Подвох**: добавление файла в `.gitignore` после того, как он был закоммичен, не удаляет его из репозитория. Вы также должны выполнить `git rm --cached file`, чтобы прекратить его отслеживание. Файл останется в истории навсегда, если вы не перепишете историю (что довольно сложно).

## Git для машинного обучения

- Машинное обучение создает проблемы, с которыми не сталкивается традиционная разработка:

- **Большие файлы**: датасеты и веса моделей весят гигабайты и более. Git спроектирован для текстовых файлов (исходного кода), а не для бинарных объектов. Решения:
    - **Git LFS** (Large File Storage): отслеживает указатели в git, хранит сами файлы на отдельном сервере. Просто, но имеет ограничения по хранилищу/трафику на GitHub.
    - **DVC** (Data Version Control): управляет файлами данных и моделей отдельно от git, используя удаленное хранилище (S3, GCS). Работает как git для данных: `dvc add data.csv`, `dvc push`, `dvc pull`.

- **Отслеживание экспериментов**: какой коммит + какие гиперпараметры + какие данные дали какие метрики? Git отслеживает код, но не полный контекст эксперимента.
    - **Weights & Biases (W&B)**: логирует метрики, гиперпараметры, системную информацию и ссылки на коммит в git. Предоставляет дашборды для сравнения запусков.
    - **MLflow**: open-source инструмент для отслеживания экспериментов с реестром моделей. Логирует параметры, метрики и артефакты.
    - **Простой подход**: логируйте git-хеш в вашем скрипте обучения: `git_hash = subprocess.check_output(['git', 'rev-parse', 'HEAD']).strip()`. Сохраняйте его вместе с результатами.

- **Чек-лист воспроизводимости** (что отслеживать для каждого эксперимента):
    - Хеш коммита git (точная версия кода)
    - Файл конфигурации / гиперпараметры
    - Случайные зерна (random seeds)
    - Версии Python и библиотек (`pip freeze`)
    - Версия данных (хеш DVC или тег версии датасета)
    - Оборудование (тип GPU, количество GPU)

```bash
# Quick reproducibility snapshot
echo "Commit: $(git rev-parse HEAD)" > experiment_info.txt
echo "Branch: $(git branch --show-current)" >> experiment_info.txt
echo "Dirty: $(git status --porcelain | wc -l) files" >> experiment_info.txt
pip freeze >> experiment_info.txt
nvidia-smi >> experiment_info.txt
```
