# Тестирование и обеспечение качества

*Тестирование — это способ убедиться, что ваш код работает не только сейчас, но и после каждого внесенного изменения. В этом файле рассматриваются пирамида тестирования, модульные тесты с использованием pytest, мокирование (mocking), тестирование специфического кода для машинного обучения, CI/CD-пайплайны, линтинг, форматирование и код-ревью — практики, которые позволяют выявлять ошибки до того, как они попадут в продакшн.*

- Код для машинного обучения печально известен недостаточным покрытием тестами. Преобладающее отношение: «Оно обучается, значит, оно работает». Это приводит к скрытым ошибкам: загрузчик данных, который выполняет перемешивание некорректно; функция потерь с ошибкой в знаке; этап предобработки, который отбрасывает 5% данных. Такие ошибки не приводят к аварийному завершению программы. Они просто делают вашу модель незаметно хуже, а вы тратите недели на отладку метрик, которые «должны быть выше».

- Тестирование — это не накладные расходы. Это самый быстрый способ двигаться вперед, ничего не ломая.

## Пирамида тестирования

- Тесты организованы по уровням, от быстрых и узконаправленных до медленных и комплексных:

    - **Модульные тесты** (основание): тестируют отдельные функции и классы в изоляции. Быстрые (миллисекунды), многочисленные (от сотен до тысяч). «Производит ли `normalise_image` значения в диапазоне [0, 1]?»

    - **Интеграционные тесты** (середина): тестируют совместную работу компонентов. Более медленные (секунды). «Производит ли загрузчик данных батчи в формате, который ожидает модель?»

    - **End-to-end тесты** (вершина): тестируют весь пайплайн от входных данных до результата. Медленные (минуты). «Завершается ли `python train.py --config test.yaml` без ошибок и создает ли он валидный чекпоинт?»

- Форма пирамиды означает следующее: пишите много модульных тестов, меньше интеграционных и небольшое количество end-to-end тестов. Модульные тесты выявляют большинство ошибок и выполняются за секунды. End-to-end тесты находят проблемы интеграции, но они медленные и хрупкие.

## Модульные тесты с pytest

- **pytest** — это стандартный фреймворк для тестирования в Python. Тест представляет собой функцию, имя которой начинается с `test_`, в файле, имя которого также начинается с `test_`:

```python
# tests/test_utils.py

def test_normalise_image():
    import numpy as np
    image = np.array([0, 128, 255], dtype=np.uint8)
    result = normalise_image(image, mean=128, std=128)
    assert result.min() >= -1.0
    assert result.max() <= 1.0
    assert abs(result[1]) < 1e-6  # 128 normalised by mean=128 should be ~0

def test_normalise_empty():
    import numpy as np
    image = np.array([], dtype=np.uint8)
    result = normalise_image(image, mean=128, std=128)
    assert len(result) == 0
```

```bash
pytest tests/                     # run all tests
pytest tests/test_utils.py        # run one file
pytest -v                         # verbose output
pytest -x                         # stop on first failure
pytest -k "normalise"             # run tests matching name pattern
pytest --tb=short                 # shorter tracebacks
```

### Фикстуры

- **Фикстуры** (fixtures) предоставляют возможность повторного использования настроек для тестов. Вместо того чтобы повторять код настройки в каждом тесте, определите его один раз:

```python
import pytest

@pytest.fixture
def sample_dataset():
    """Create a small dataset for testing."""
    return {
        "inputs": torch.randn(10, 3, 32, 32),
        "labels": torch.randint(0, 10, (10,))
    }

@pytest.fixture
def trained_model():
    """Load a small pretrained model."""
    model = SmallModel()
    model.load_state_dict(torch.load("tests/fixtures/small_model.pt"))
    return model

def test_model_output_shape(trained_model, sample_dataset):
    output = trained_model(sample_dataset["inputs"])
    assert output.shape == (10, 10)  # batch_size x num_classes
```

- Фикстуры могут иметь **области видимости (scopes)**: `scope="function"` (по умолчанию, создается заново для каждого теста), `scope="module"` (один раз на файл), `scope="session"` (один раз на сессию запуска тестов). Используйте `scope="session"` для ресурсоемких операций, таких как загрузка модели.

### Параметризованные тесты

- Тестируйте одну и ту же функцию с разными входными данными без дублирования кода:

```python
@pytest.mark.parametrize("input,expected", [
    ([1, 2, 3], 6),
    ([], 0),
    ([-1, 1], 0),
    ([1000000, 1000000], 2000000),
])
def test_sum(input, expected):
    assert sum(input) == expected
```

## Мокирование и патчинг

- **Мокирование** (mocking) заменяет реальную зависимость фиктивной во время тестирования. Это позволяет тестировать функцию в изоляции, без необходимости в базе данных, API или GPU.

```python
from unittest.mock import patch, MagicMock

def test_training_logs_metrics():
    mock_logger = MagicMock()

    with patch("my_project.training.trainer.wandb") as mock_wandb:
        trainer = Trainer(logger=mock_logger)
        trainer.train_one_epoch()

        # verify that the trainer logged metrics
        mock_logger.log.assert_called()
        # verify it logged a loss value
        call_args = mock_logger.log.call_args
        assert "loss" in call_args[1]
```

- **Когда использовать моки**: для внешних сервисов (API, базы данных, облачные хранилища), дорогостоящих операций (вычисления на GPU, ввод-вывод больших файлов) и недетерминированного поведения (генераторы случайных чисел, временные метки).

- **Когда НЕ использовать моки**: для собственного кода. Если вы мокируете всё, ваши тесты проверяют, что моки ведут себя ожидаемо, а не то, что ваш код работает правильно. Мокируйте на границах системы, тестируйте логику напрямую.

## Тестирование кода для машинного обучения

- Код для машинного обучения имеет уникальные сложности при тестировании: выходные данные вероятностны, обучение происходит медленно, а понятие «правильности» размыто.

### Детерминированные сиды

- Устанавливайте случайные сиды (seeds) везде, чтобы сделать тесты воспроизводимыми:

```python
import random
import numpy as np
import torch

def set_seed(seed=42):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    if torch.cuda.is_available():
        torch.cuda.manual_seed_all(seed)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False
```

### Численные допуски

- Сравнение чисел с плавающей запятой требует использования допусков (глава 13, IEEE 754):

```python
# BAD: exact comparison fails due to floating point
assert model_output == 0.5

# GOOD: approximate comparison
import numpy as np
assert np.isclose(model_output, 0.5, atol=1e-5)

# For tensors
assert torch.allclose(output, expected, atol=1e-4)
```

### Что тестировать в машинном обучении

- **Тесты размерностей (shape tests)**: проверяйте, что выходные данные имеют ожидаемые размерности.

```python
def test_model_output_shape():
    model = MyModel(d_model=256, n_classes=10)
    x = torch.randn(8, 32, 256)  # batch=8, seq=32, dim=256
    output = model(x)
    assert output.shape == (8, 10)
```

- **Градиентный поток (Gradient flow)**: убедитесь, что градиенты для обучаемых параметров не равны нулю.

```python
def test_gradients_flow():
    model = MyModel()
    x = torch.randn(4, 3, 32, 32)
    y = torch.randint(0, 10, (4,))

    output = model(x)
    loss = F.cross_entropy(output, y)
    loss.backward()

    for name, param in model.named_parameters():
        assert param.grad is not None, f"No gradient for {name}"
        assert param.grad.abs().sum() > 0, f"Zero gradient for {name}"
```

- **Переобучение на одном батче (Overfit on one batch)**: модель должна быть способна запомнить один батч. Если это не удается, значит, в архитектуре есть фундаментальная ошибка.

```python
def test_overfit_one_batch():
    model = MyModel()
    optimiser = torch.optim.Adam(model.parameters(), lr=1e-3)
    x, y = get_single_batch()

    for _ in range(100):
        loss = F.cross_entropy(model(x), y)
        loss.backward()
        optimiser.step()
        optimiser.zero_grad()

    assert loss.item() < 0.01, f"Cannot overfit one batch: loss={loss.item()}"
```

- **Валидация данных (Data validation)**: убедитесь, что загрузка данных выдает корректные результаты.

```python
def test_dataset_basics():
    dataset = MyDataset("tests/fixtures/small_data.csv")
    assert len(dataset) > 0
    x, y = dataset[0]
    assert x.shape == (3, 224, 224)
    assert 0 <= y < 10
    assert not torch.isnan(x).any()
    assert not torch.isinf(x).any()
```

- **Детерминизм (Determinism)**: одинаковые входные данные + одинаковое начальное состояние (seed) → одинаковый результат.

```python
def test_determinism():
    set_seed(42)
    output1 = model(input_data)
    set_seed(42)
    output2 = model(input_data)
    assert torch.allclose(output1, output2)
```

## CI/CD пайплайны

- **Непрерывная интеграция (Continuous Integration, CI)**: автоматический запуск тестов при каждом коммите или PR. Если тесты не проходят, PR не может быть слит (merged). Это предотвращает попадание нерабочего кода в ветку `main`.

- Пример **GitHub Actions** (`.github/workflows/ci.yml`):

```yaml
name: CI
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install -e ".[dev]"
      - run: ruff check src/
      - run: mypy src/
      - run: pytest tests/ -v --tb=short
```

- **Pre-commit хуки**: запуск проверок перед каждым коммитом (локально), что позволяет выявить проблемы до того, как они попадут в CI:

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.3.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
```

```bash
pip install pre-commit
pre-commit install    # now hooks run on every git commit
```

## Линтинг и форматирование

- **Линтинг (Linting)** обнаруживает ошибки и проблемы со стилем без запуска кода. **Форматирование (Formatting)** автоматически обеспечивает единообразие стиля.

- **Ruff**: быстрый линтер и форматер для Python (заменяет flake8, isort и black в одном инструменте):

```bash
ruff check src/          # lint
ruff check --fix src/    # lint and auto-fix
ruff format src/         # format
```

- **mypy**: инструмент для статической проверки типов в Python. Выявляет ошибки типов до выполнения программы:

```bash
mypy src/
# src/model.py:42: error: Argument 1 to "forward" has incompatible type "int"; expected "Tensor"
```

- Аннотации типов делают код самодокументированным и помогают находить ошибки:

```python
def train(
    model: nn.Module,
    dataloader: DataLoader,
    optimiser: torch.optim.Optimizer,
    num_epochs: int = 10,
) -> float:
    """Train model and return final loss."""
    ...
```

## Лучшие практики код-ревью

- **Для автора**:
    - Проведите самопроверку своего diff перед запросом ревью. Вы обнаружите очевидные проблемы.
    - Старайтесь делать PR небольшими и сфокусированными. Одна задача на один PR.
    - Напишите четкое описание: что сделано, зачем и как протестировать.
    - Отвечайте на каждый комментарий (даже если просто "готово").

- **Для ревьюера**:
    - Будьте вежливы. Критикуйте код, а не человека. "Это можно сделать понятнее", а не "это запутано".
    - Различайте блокирующие проблемы (баги, безопасность) и предложения (стиль, именование). Используйте метки: "nit:", "suggestion:", "blocking:".
    - Задавайте вопросы вместо требований. "Что произойдет, если этот список пуст?" полезнее, чем "обработай случай с пустым списком".
    - Утверждайте PR оперативно. PR, ожидающий ревью несколько дней, блокирует автора и провоцирует создание больших, пакетных PR (которые сложнее проверять).
