# 代码库设计与模式

*良好的代码库设计是研究原型与生产软件的分水岭。本文件涵盖项目结构、整洁代码原则、与 ML 相关的设计模式、配置管理、logging、API 设计以及打包*

- 大多数 ML 代码最初都是 Jupyter notebook。notebook 不断膨胀、被复制、被修改、被共享，最终变成一团无法维护的全局变量、死单元格和魔法数字。**代码库设计**（codebase design）就是这样一门纪律：组织代码，使其在项目成长过程中仍然可理解、可修改。

- 这不是为了守规矩而守规矩。它的目的是缩短从“我想改 X”到“X 已改好且能用”之间的时间。在良好设计的代码库中，这段时间以分钟计；在糟糕设计的代码库中，则要在无文档的意大利面条式代码里做考古，耗时数天。

## 项目结构

- 一致的项目布局让任何人（包括未来的你）都能瞬间在代码库中导航。

```
my_project/
├── src/my_project/       # source code (importable package)
│   ├── __init__.py
│   ├── data/             # data loading and preprocessing
│   │   ├── __init__.py
│   │   ├── dataset.py
│   │   └── transforms.py
│   ├── models/           # model architectures
│   │   ├── __init__.py
│   │   ├── transformer.py
│   │   └── layers.py
│   ├── training/         # training loops, optimisers
│   │   ├── __init__.py
│   │   ├── trainer.py
│   │   └── losses.py
│   └── utils/            # shared utilities
│       ├── __init__.py
│       └── logging.py
├── configs/              # configuration files
│   ├── base.yaml
│   └── experiment_1.yaml
├── scripts/              # entry points (train, evaluate, serve)
│   ├── train.py
│   ├── evaluate.py
│   └── serve.py
├── tests/                # test files (mirrors src/ structure)
│   ├── test_dataset.py
│   ├── test_model.py
│   └── test_trainer.py
├── notebooks/            # exploration only (not production code)
├── pyproject.toml        # project metadata and dependencies
├── README.md
├── .gitignore
└── Dockerfile
```

- **`src/` 布局**：把源代码放在 `src/my_project/` 下，可以防止从当前目录意外导入（这种做法会掩盖那些本应在生产环境暴露出来的导入错误）。开发时使用 `pip install -e .` 安装。

- **Monorepo vs multi-repo**：**monorepo** 把所有相关项目放在一个仓库里（便于跨项目修改、共享 CI）。**multi-repo** 则每个项目一个仓库（边界更清晰、版本独立）。大多数 ML 团队起步时用 monorepo，需要时再拆分。

- **脚本与库**：把入口点（`train.py`、`evaluate.py`）放在 `scripts/`，把可复用逻辑放在 `src/`。一个训练脚本应该只有约 50 行：解析 config、构建 dataset、构建 model、构建 trainer、训练。所有复杂性都住在库里。

## 整洁代码原则

- **命名**：这是你能做的最具影响的一件事。一个名为 `x` 的变量需要你阅读周围代码才能理解。一个名为 `learning_rate` 的变量则是自解释的。

```python
# BAD
def proc(d, n, lr):
    for i in range(n):
        for k, v in d.items():
            v -= lr * g[k]

# GOOD
def update_parameters(parameters, num_steps, learning_rate):
    for step in range(num_steps):
        for name, param in parameters.items():
            param -= learning_rate * gradients[name]
```

- **单一职责原则**：每个函数/类只做一件事。一个名为 `load_data_and_train_model` 的函数做了两件事，应该拆开。这样每一段都能独立测试、复用和理解。

- **DRY（Don't Repeat Yourself）**——但不要过早。如果你把同一段代码复制粘贴了三次，就把它抽成一个函数。但对于只用过一次的代码，不要为它创造抽象。过早抽象比重复更糟：它在没有验证收益的前提下增加了复杂性。

```python
# Premature abstraction (one use case, over-engineered)
class AbstractDataTransformPipelineFactory:
    ...

# Just right (direct, clear, used in three places)
def normalise_image(image, mean, std):
    return (image - mean) / std
```

- **魔法数字**：永远不要使用未加解释的字面值。

```python
# BAD
if len(batch) > 32:
    split_batch(batch, 32)

# GOOD
MAX_BATCH_SIZE = 32
if len(batch) > MAX_BATCH_SIZE:
    split_batch(batch, MAX_BATCH_SIZE)
```

- **函数应该短小**：如果一个函数一屏放不下（约 30 行），它很可能做了太多事。把逻辑块抽成命名贴切的辅助函数，函数体读起来就像高层摘要。

## ML 的设计模式

- 设计模式是针对常见问题的可复用解决方案。以下是与 ML 代码库最相关的几种：

- **工厂模式（Factory pattern）**：在不指定具体类的情况下创建对象。当你的 config 写着 `model: "transformer"`、需要实例化正确的类时很有用：

```python
MODEL_REGISTRY = {
    "transformer": TransformerModel,
    "cnn": CNNModel,
    "mlp": MLPModel,
}

def build_model(config):
    model_cls = MODEL_REGISTRY[config["model"]]
    return model_cls(**config["model_params"])
```

- 这把训练脚本与具体 model 实现解耦。新增一个 model 只需在 registry 里加一行，不必修改训练循环。

- **策略模式（Strategy pattern）**：在运行时替换算法。适用于 loss、optimiser、scheduler：

```python
LOSS_FUNCTIONS = {
    "mse": nn.MSELoss,
    "cross_entropy": nn.CrossEntropyLoss,
    "focal": FocalLoss,
}

loss_fn = LOSS_FUNCTIONS[config["loss"]]()
```

- **观察者模式（Observer pattern）**（callbacks/hooks）：让模块在不紧耦合的情况下对事件做出反应。训练框架（PyTorch Lightning、Keras）大量使用 callbacks：

```python
class EarlyStopping:
    def __init__(self, patience=5):
        self.patience = patience
        self.best_loss = float('inf')
        self.counter = 0

    def on_epoch_end(self, epoch, val_loss):
        if val_loss < self.best_loss:
            self.best_loss = val_loss
            self.counter = 0
        else:
            self.counter += 1
            if self.counter >= self.patience:
                return "stop"
```

- **依赖注入（Dependency injection）**：把依赖项传入函数/类，而不是在内部创建。这让测试变简单（注入一个 mock），也让配置更灵活：

```python
# BAD: hard-coded dependency
class Trainer:
    def __init__(self):
        self.logger = WandbLogger()  # cannot test without W&B

# GOOD: injected dependency
class Trainer:
    def __init__(self, logger):
        self.logger = logger  # can inject any logger, including a mock
```

## 配置管理

- 把超参数、文件路径和 model 设置硬编码会让实验不可复现、修改变得痛苦。**把配置外置**到文件里。

- **YAML** 是 ML config 最常用的格式：

```yaml
# configs/experiment_1.yaml
model:
  name: transformer
  d_model: 512
  n_heads: 8
  n_layers: 6

training:
  batch_size: 64
  learning_rate: 3e-4
  max_epochs: 100
  early_stopping_patience: 10

data:
  train_path: /data/train.parquet
  val_path: /data/val.parquet
  max_seq_length: 512
```

- **Hydra**（Facebook 出品）是一个配置框架，支持组合（将 base config 与实验特定的覆盖合并）、命令行覆盖（`python train.py training.lr=1e-3`）以及多运行（对超参数做 sweep）。

- **argparse** 对于只有少量参数的脚本更简单：

```python
import argparse

parser = argparse.ArgumentParser()
parser.add_argument("--lr", type=float, default=3e-4)
parser.add_argument("--batch-size", type=int, default=64)
parser.add_argument("--config", type=str, default="configs/base.yaml")
args = parser.parse_args()
```

- **最佳实践**：准备一个包含所有默认值的 base config，再为每个实验准备只覆盖差异部分的实验 config。把每次实验的 config 与其结果一起记录。

## Logging 与可观测性

- `print` 语句用于调试。**Logging** 用于生产：

```python
import logging

logger = logging.getLogger(__name__)
logger.setLevel(logging.INFO)

logger.debug("Batch loaded: %d samples", len(batch))     # noisy, for debugging
logger.info("Epoch %d: loss=%.4f, lr=%.6f", epoch, loss, lr)  # normal operation
logger.warning("GPU memory >90%%, consider reducing batch size")
logger.error("Failed to load checkpoint: %s", path)       # recoverable error
logger.critical("CUDA out of memory, aborting")            # fatal
```

- **为什么不用 print**：logging 支持等级（在生产中过滤掉 debug 消息）、格式化（时间戳、模块名）以及处理器（写文件、发到监控系统），且无需改动 logging 调用。

- **结构化 logging** 在人类可读消息之外输出机器可解析格式（JSON）。这样可以基于特定字段进行搜索和告警：

```python
logger.info("training_step", extra={
    "epoch": 5, "step": 1200, "loss": 0.0342, "lr": 2.1e-4
})
```

## API 设计

- 如果你的 model 会被其他服务使用（web app、移动 app、另一个 ML pipeline），它就需要一个 **API**（Application Programming Interface）。

- **REST API** 使用 HTTP 方法：`GET` 读取、`POST` 创建/预测、`PUT` 更新、`DELETE` 删除。endpoint 采用基于资源的命名：

```
POST /api/v1/predict          # send input, get prediction
GET  /api/v1/models           # list available models
GET  /api/v1/models/{id}      # get model details
POST /api/v1/models/{id}/predict  # predict with a specific model
```

- **FastAPI** 是 ML serving 的首选 Python 框架：

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class PredictRequest(BaseModel):
    text: str

class PredictResponse(BaseModel):
    label: str
    confidence: float

@app.post("/predict", response_model=PredictResponse)
async def predict(request: PredictRequest):
    result = model.predict(request.text)
    return PredictResponse(label=result.label, confidence=result.score)
```

- FastAPI 自动生成 API 文档（Swagger UI 位于 `/docs`），用 Pydantic 模型校验输入/输出，并支持 async 以获得高 throughput。

- **gRPC** 在内部服务间通信方面比 REST 更快。它使用 Protocol Buffers（二进制序列化，比 JSON 更小更快）并支持流式传输。TensorFlow Serving、Triton Inference Server 以及许多 microservice 架构都在使用它。

## 打包与分发

- 把你的代码打包成可安装的 package，让别人（以及你自己的脚本）能干净地导入：

```toml
# pyproject.toml
[project]
name = "my-ml-project"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = [
    "torch>=2.0",
    "jax>=0.4",
    "pydantic>=2.0",
]

[project.optional-dependencies]
dev = ["pytest", "ruff", "mypy"]

[build-system]
requires = ["setuptools>=64"]
build-backend = "setuptools.backends._legacy:_Backend"
```

```bash
pip install -e ".[dev]"    # install in editable mode with dev dependencies
```

- **可编辑安装（Editable install）**（`-e`）：对源代码的修改会立即生效，无需重新安装。开发阶段不可或缺。

- **锁定依赖（Pinning dependencies）**：用精确版本（`torch==2.2.1`，而非 `torch>=2.0`）的 `requirements.txt` 保证可复现性。用 `pip freeze > requirements.txt` 捕获当前环境。更复杂的依赖管理可使用 `uv`、`poetry` 或 `pip-tools`。

## 与 AI 编码 Agent 协作

- AI 编码 agent（Claude Code、GitHub Copilot、Cursor 等）如今已是专业工程工作流的一部分。用得好，它们能极大加速开发；用得不好，它们会引入隐蔽 bug、侵蚀你对自己代码库的理解，并制造虚假的生产力感。

- 正确的心智模型：**agent 是一个快速但缺乏经验的结对编程伙伴**。它能快速写代码、懂语法和标准模式、读过的文档比你一辈子都多。但它不理解你具体的系统、你的业务约束、你的边界情况，也不理解你设计决策背后的 *为什么*。你是高级工程师，agent 是初级工程师。你来指挥、review，并承担责任。

### Agent 表现出色的场景

- **样板代码与脚手架**：生成 Dockerfile、CI config、test fixture、data class 定义、argparse 配置。这些遵循已知模式，手写又枯燥。让 agent 生成，然后 review 正确性。

- **写测试**：描述函数的行为，agent 生成测试用例。它常常能想到你会漏掉的边界情况（空输入、负值、Unicode）。一定要读生成的测试——它们验证的是你的假设，而不仅仅是你的代码。

- **重构**：“把这段抽成函数”、“把这个 class 改成用 dataclasses”、“给这个模块加 type hint”。意图清晰、细微出错风险低的机械化变换。

- **探索与原型**：“写个快速脚本 benchmark 推理 latency”或“给我看看怎么用 HuggingFace tokeniser API”。agent 比读文档更快地给你一个可用的起点。

- **文档与 docstring**：agent 能根据你的代码结构生成文档。要 review 准确性，但苦力活被自动化了。

- **调试辅助**：贴一段错误 traceback 问问诊断。agent 常能定位根因并给出修复建议，尤其针对常见问题（shape 不匹配、导入错误、CUDA out of memory）。

### 不要依赖 Agent 的场景

- **新颖架构决策**：如果你在设计一个新训练 pipeline，agent 会给出通用答案。它不知道你的数据约束、latency 要求或团队专长。用 agent 来实现你已经想清楚的设计。

- **安全关键代码**：认证、加密、输入消毒。agent 可能写出看起来正确但有隐蔽漏洞（SQL 注入、不安全的默认值、时序攻击）的代码。安全代码应由理解威胁模型的人编写，并由另一个人 review。

- **性能关键的内部循环**：agent 会写出正确但朴素的代码。对 GPU kernel、内存敏感的数据结构或 latency 敏感的 serving 路径，你需要理解硬件约束（第 13 章、第 16 章）并有意识地优化。

- **你不理解的代码**：如果 agent 生成了 200 行而你说不出每行在干什么，就别提交。你这是在维护自己不理解的代码，等它出问题时（一定会出），你无法调试。这是最常见也最危险的失败模式。

### Review 纪律

- 提交前**务必阅读每一行**生成的代码。这不是可选项。agent 的代码是草稿，不是成品。把它当作同事的 pull request 一样看待：批判性地 review。

- **要检查什么**：
    - **正确性**：它真的做了你要求的事吗？agent 常常解了一个与你意图略有不同的题。
    - **边界情况**：它处理了空输入、None 值、负数、超大输入吗？agent 常常省略边界处理。
    - **幻觉的 API**：agent 可能调用不存在的函数或使用不存在的参数，尤其针对较新或较冷门的库。验证每个 API 调用都是真的。
    - **过度工程**：agent 倾向于产出比必要更多的代码。用 50 行解决一个 10 行问题只是徒增复杂性而无收益。毫不留情地简化。
    - **安全**：硬编码的密钥、未消毒的用户输入、不安全的默认值。agent 不会从对抗角度思考。
    - **风格一致性**：生成的代码是否符合你项目的约定（命名、模式、错误处理）？

### 如何写好 Prompt

- agent 输出的质量与你的指令质量成正比。模糊的 prompt 得到模糊的代码。

- **差**：“写一个 data loader”
- **好**：“为含有 'text' 和 'label' 两列的 CSV 文件写一个 PyTorch DataLoader。用 HuggingFace tokeniser 'bert-base-uncased' 对 text 做分词，max_length=512。返回 input_ids、attention_mask 和 label 作为 tensor。当 CSV 的 label 列有缺失值时跳过这些行。”

- **提供上下文**：告诉 agent 你的项目结构、现有代码、约束和约定。上下文越多，输出越好。

- **指定约束**：“只用标准库”、“必须兼容 Python 3.10”、“不要使用全局变量”、“遵循 `src/models/transformer.py` 里的现有模式”。

- **要求解释**：“实现 X 并解释关键设计决策。”这迫使 agent 阐明自己的推理，让你更容易发现错误假设。

### 用质量门拦截 Agent 的错误

- 你既有的质量基础设施（第 04 篇）拦截 agent 的错误和拦截人的错误一样有效：

    - **类型检查（mypy）**：拦截幻觉的 API 签名和类型不匹配。
    - **Linting（ruff）**：拦截未使用的导入、未定义变量和风格违规。
    - **测试（pytest）**：如果 agent 的代码通过了你的测试套件，就更可能正确。如果你还没有测试，就在让 agent 实现功能 *之前* 先写测试（测试驱动开发与 agent 配合尤其好）。
    - **CI pipeline**：每次提交自动运行上述全部。

- “agent 写代码” + “质量门验证”的组合比单独任一个都更高效。agent 快但粗心；质量门细致但不会写代码。两者结合，你同时得到了速度和正确性。

### 生产力陷阱

- 编码 agent 最大的风险是**生产力的幻觉**。你能在 10 分钟内生成 500 行代码。但如果你因为不理解这 500 行而花 2 小时调试，那比你自己 30 分钟写 200 行还慢。

- 真正来自 agent 的生产力源于：
    1. **保持掌控**：你决定架构，agent 填充实现。
    2. **理解生成的内容**：如果你解释不了，就重写或让 agent 简化。
    3. **投入质量门**：测试、类型和 linting 把成本摊销到每一次与 agent 的交互中。
    4. **让 agent 补你的短板**：如果你擅长算法但写测试慢，就让 agent 写测试；如果你写 UI 代码快但不熟数据库查询，就让 agent 起草 SQL。发挥你的长处，委托你的短板。

- 从编码 agent 中获益最多的工程师，恰恰是那些本来编码就很好的人。agent 放大你现有的技能，而不是取代它。理解数据结构、算法、系统设计和软件工程（整章内容）才能让你有效地指挥 agent 并批判性地评估其输出。
