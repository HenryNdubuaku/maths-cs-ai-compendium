# Проектирование кодовой базы и паттерны

*Хорошее проектирование кодовой базы — это то, что отличает исследовательский прототип от программного обеспечения для продакшна. В этом файле рассматриваются структура проекта, принципы чистого кода, паттерны проектирования, актуальные для машинного обучения, управление конфигурациями, логирование, проектирование API и упаковка.*

- Большинство кода в машинном обучении начинается с Jupyter-ноутбуков. Ноутбук разрастается, его копируют, изменяют, передают другим, и в конечном итоге он превращается в не поддающийся поддержке клубок глобальных переменных, нерабочих ячеек и «магических чисел». **Проектирование кодовой базы** — это дисциплина организации кода таким образом, чтобы он оставался понятным и подлежащим изменению по мере роста проекта.

- Речь идет не о следовании правилам ради самих правил. Речь идет о сокращении времени между «я хочу изменить X» и «X изменен и работает». В хорошо спроектированной кодовой базе это время исчисляется минутами. В плохо спроектированной — днями археологических раскопок в недокументированном «спагетти-коде».

## Структура проекта

- Согласованная структура проекта позволяет любому (включая вас в будущем) мгновенно ориентироваться в кодовой базе.

```
my_project/
├── src/my_project/       # source code (importable package)
│   ├── __init__.py
│   ├── data/             # data loading and preprocessing
│   │   ├── __init__.py
│   │   ├── dataset.py
│   │   └── transforms.py
│   ├── models/           # model architectures
│   │   ├── __init__.py
│   │   ├── transformer.py
│   │   └── layers.py
│   ├── training/         # training loops, optimisers
│   │   ├── __init__.py
│   │   ├── trainer.py
│   │   └── losses.py
│   └── utils/            # shared utilities
│       ├── __init__.py
│       └── logging.py
├── configs/              # configuration files
│   ├── base.yaml
│   └── experiment_1.yaml
├── scripts/              # entry points (train, evaluate, serve)
│   ├── train.py
│   ├── evaluate.py
│   └── serve.py
├── tests/                # test files (mirrors src/ structure)
│   ├── test_dataset.py
│   ├── test_model.py
│   └── test_trainer.py
├── notebooks/            # exploration only (not production code)
├── pyproject.toml        # project metadata and dependencies
├── README.md
├── .gitignore
└── Dockerfile
```

- **Структура `src/`**: размещение исходного кода в `src/my_project/` предотвращает случайные импорты из текущей директории (которые маскируют ошибки импорта, проявляющиеся в продакшне). Для разработки используйте установку через `pip install -e .`.

- **Монорепозиторий против мультирепозитория**: **монорепозиторий** хранит все связанные проекты в одном репозитории (упрощает кросс-проектные изменения, общую CI). **Мультирепозиторий** предоставляет каждому проекту собственный репозиторий (более четкие границы, независимое версионирование). Большинство команд по машинному обучению начинают с монорепозитория и при необходимости разделяют его позже.

- **Скрипты против библиотеки**: храните точки входа (`train.py`, `evaluate.py`) в `scripts/`. Храните переиспользуемую логику в `src/`. Скрипт обучения должен занимать около 50 строк: парсинг конфигурации, создание датасета, создание модели, создание тренера, обучение. Вся сложность должна находиться в библиотеке.

## Принципы чистого кода

- **Именование**: это единственная вещь, которая оказывает наибольшее влияние. Переменная с именем `x` требует чтения окружающего кода для понимания. Переменная с именем `learning_rate` является самодокументируемой.

```python
# BAD
def proc(d, n, lr):
    for i in range(n):
        for k, v in d.items():
            v -= lr * g[k]

# GOOD
def update_parameters(parameters, num_steps, learning_rate):
    for step in range(num_steps):
        for name, param in parameters.items():
            param -= learning_rate * gradients[name]
```

- **Принцип единственной ответственности (Single Responsibility Principle)**: каждая функция или класс должны делать что-то одно. Функция с названием `load_data_and_train_model` делает две вещи, и ее следует разделить. Это делает каждую часть независимо тестируемой, переиспользуемой и понятной.

- **DRY (Don't Repeat Yourself — не повторяйся)** — но не преждевременно. Если вы копируете код трижды, вынесите его в функцию. Но не создавайте абстракцию для кода, который вы использовали только один раз. Преждевременная абстракция хуже дублирования: она добавляет сложность без доказанной пользы.

```python
# Premature abstraction (one use case, over-engineered)
class AbstractDataTransformPipelineFactory:
    ...

# Just right (direct, clear, used in three places)
def normalise_image(image, mean, std):
    return (image - mean) / std
```

- **Магические числа**: никогда не используйте неявные литеральные значения.

```python
# BAD
if len(batch) > 32:
    split_batch(batch, 32)

# GOOD
MAX_BATCH_SIZE = 32
if len(batch) > MAX_BATCH_SIZE:
    split_batch(batch, MAX_BATCH_SIZE)
```

- **Функции должны быть короткими**: если функция не помещается на одном экране (~30 строк), вероятно, она делает слишком много. Выносите логические блоки в вспомогательные функции с описательными именами. Тогда тело функции будет читаться как высокоуровневое резюме.

## Паттерны проектирования для машинного обучения

- Паттерны проектирования — это переиспользуемые решения распространенных проблем. Вот те из них, которые наиболее актуальны для кодовых баз машинного обучения:

- **Паттерн «Фабрика»**: создание объектов без указания конкретного класса. Полезно, когда ваша конфигурация говорит `model: "transformer"`, и вам нужно инстанцировать правильный класс:

```python
MODEL_REGISTRY = {
    "transformer": TransformerModel,
    "cnn": CNNModel,
    "mlp": MLPModel,
}

def build_model(config):
    model_cls = MODEL_REGISTRY[config["model"]]
    return model_cls(**config["model_params"])
```

- Это отделяет скрипт обучения от конкретных реализаций моделей. Добавление новой модели означает добавление одной строки в реестр, а не изменение цикла обучения.

- **Паттерн «Стратегия»**: подмена алгоритмов во время выполнения. Полезно для функций потерь, оптимизаторов, планировщиков:

```python
LOSS_FUNCTIONS = {
    "mse": nn.MSELoss,
    "cross_entropy": nn.CrossEntropyLoss,
    "focal": FocalLoss,
}

loss_fn = LOSS_FUNCTIONS[config["loss"]]()
```

- **Паттерн «Наблюдатель»** (колбэки/хуки): позволяет модулям реагировать на события без жесткой связности. Фреймворки для обучения (PyTorch Lightning, Keras) активно используют колбэки:

```python
class EarlyStopping:
    def __init__(self, patience=5):
        self.patience = patience
        self.best_loss = float('inf')
        self.counter = 0

    def on_epoch_end(self, epoch, val_loss):
        if val_loss < self.best_loss:
            self.best_loss = val_loss
            self.counter = 0
        else:
            self.counter += 1
            if self.counter >= self.patience:
                return "stop"
```

- **Внедрение зависимостей (Dependency injection)**: передавайте зависимости в функцию или класс вместо того, чтобы создавать их внутри. Это упрощает тестирование (можно внедрить заглушку/mock) и делает конфигурацию гибкой:

```python
# BAD: hard-coded dependency
class Trainer:
    def __init__(self):
        self.logger = WandbLogger()  # cannot test without W&B

# GOOD: injected dependency
class Trainer:
    def __init__(self, logger):
        self.logger = logger  # can inject any logger, including a mock
```

## Управление конфигурацией

- Жесткое кодирование гиперпараметров, путей к файлам и настроек модели делает эксперименты невоспроизводимыми, а внесение изменений — трудоемким. **Выносите конфигурацию** во внешние файлы.

- **YAML** — самый распространенный формат для конфигураций в машинном обучении:

```yaml
# configs/experiment_1.yaml
model:
  name: transformer
  d_model: 512
  n_heads: 8
  n_layers: 6

training:
  batch_size: 64
  learning_rate: 3e-4
  max_epochs: 100
  early_stopping_patience: 10

data:
  train_path: /data/train.parquet
  val_path: /data/val.parquet
  max_seq_length: 512
```

- **Hydra** (от Facebook) — это фреймворк для конфигураций, который поддерживает композицию (объединение базовой конфигурации с переопределениями для конкретного эксперимента), переопределение параметров через командную строку (`python train.py training.lr=1e-3`) и множественный запуск (перебор гиперпараметров).

- **argparse** проще использовать для скриптов с небольшим количеством параметров:

```python
import argparse

parser = argparse.ArgumentParser()
parser.add_argument("--lr", type=float, default=3e-4)
parser.add_argument("--batch-size", type=int, default=64)
parser.add_argument("--config", type=str, default="configs/base.yaml")
args = parser.parse_args()
```

- **Лучшая практика**: иметь базовую конфигурацию со всеми значениями по умолчанию и конфигурации для каждого эксперимента, которые переопределяют только то, что меняется. Отслеживайте конфигурацию каждого эксперимента вместе с его результатами.

## Логирование и наблюдаемость

- Инструкции `print` предназначены для отладки. **Логирование** предназначено для продакшена:

```python
import logging

logger = logging.getLogger(__name__)
logger.setLevel(logging.INFO)

logger.debug("Batch loaded: %d samples", len(batch))     # noisy, for debugging
logger.info("Epoch %d: loss=%.4f, lr=%.6f", epoch, loss, lr)  # normal operation
logger.warning("GPU memory >90%%, consider reducing batch size")
logger.error("Failed to load checkpoint: %s", path)       # recoverable error
logger.critical("CUDA out of memory, aborting")            # fatal
```

- **Почему не стоит использовать print**: логирование поддерживает уровни (позволяет отфильтровать отладочные сообщения в продакшене), форматирование (метки времени, имена модулей) и обработчики (запись в файл, отправка в систему мониторинга) без изменения самих вызовов логирования.

- **Структурированное логирование** выводит данные в машиночитаемых форматах (JSON) наряду с сообщениями для людей. Это позволяет выполнять поиск и настраивать оповещения по конкретным полям:

```python
logger.info("training_step", extra={
    "epoch": 5, "step": 1200, "loss": 0.0342, "lr": 2.1e-4
})
```

## Проектирование API

- Если ваша модель будет использоваться другими сервисами (веб-приложением, мобильным приложением, другим ML-пайплайном), ей необходим **API** (интерфейс прикладного программирования).

- **REST API** использует методы HTTP: `GET` для чтения, `POST` для создания/предсказания, `PUT` для обновления, `DELETE` для удаления. Эндпоинты следуют именованию на основе ресурсов:

```
POST /api/v1/predict          # send input, get prediction
GET  /api/v1/models           # list available models
GET  /api/v1/models/{id}      # get model details
POST /api/v1/models/{id}/predict  # predict with a specific model
```

- **FastAPI** — основной Python-фреймворк для развертывания моделей:

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class PredictRequest(BaseModel):
    text: str

class PredictResponse(BaseModel):
    label: str
    confidence: float

@app.post("/predict", response_model=PredictResponse)
async def predict(request: PredictRequest):
    result = model.predict(request.text)
    return PredictResponse(label=result.label, confidence=result.score)
```

- FastAPI автоматически генерирует документацию API (Swagger UI по адресу `/docs`), проверяет входные/выходные данные с помощью моделей Pydantic и поддерживает асинхронность для высокой пропускной способности.

- **gRPC** быстрее, чем REST, для внутреннего взаимодействия между сервисами. Он использует Protocol Buffers (бинарная сериализация, меньше и быстрее, чем JSON) и поддерживает потоковую передачу. Используется в TensorFlow Serving, Triton Inference Server и многих архитектурах микросервисов.

## Упаковка и дистрибуция

- Если сделать ваш код устанавливаемым как пакет, другие (и ваши собственные скрипты) смогут импортировать его в чистом виде:

```toml
# pyproject.toml
[project]
name = "my-ml-project"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = [
    "torch>=2.0",
    "jax>=0.4",
    "pydantic>=2.0",
]

[project.optional-dependencies]
dev = ["pytest", "ruff", "mypy"]

[build-system]
requires = ["setuptools>=64"]
build-backend = "setuptools.backends._legacy:_Backend"
```

```bash
pip install -e ".[dev]"    # install in editable mode with dev dependencies
```

- **Редактируемая установка** (`-e`): изменения в исходном коде сразу вступают в силу без переустановки. Это критически важно при разработке.

- **Фиксация зависимостей**: `requirements.txt` с точными версиями (например, `torch==2.2.1`, а не `torch>=2.0`) обеспечивает воспроизводимость. Используйте `pip freeze > requirements.txt` для фиксации текущего окружения. Для более сложного управления зависимостями используйте `uv`, `poetry` или `pip-tools`.

## Работа с AI-агентами для написания кода

- AI-агенты для написания кода (Claude Code, GitHub Copilot, Cursor и др.) стали частью профессионального инженерного рабочего процесса. При правильном использовании они значительно ускоряют разработку. При неправильном — привносят скрытые ошибки, снижают понимание собственной кодовой базы и создают ложное чувство продуктивности.

- Правильная ментальная модель: **агент — это быстрый, но неопытный парный программист**. Он может быстро писать код, знает синтаксис и стандартные паттерны, а также прочитал больше документации, чем вы когда-либо сможете. Но он не понимает вашу конкретную систему, бизнес-ограничения, граничные случаи или *причины* ваших проектных решений. Вы — старший инженер, агент — младший. Вы направляете, проверяете и несете ответственность.

### Когда агенты эффективны

- **Шаблонный код и каркасы**: генерация `Dockerfile`, конфигураций CI, тестовых фикстур, определений классов данных, настроек `argparse`. Они следуют хорошо известным шаблонам, и писать их вручную утомительно. Позвольте агенту сгенерировать их, а затем проверьте на корректность.

- **Написание тестов**: опишите поведение функции, и агент сгенерирует тестовые случаи. Он часто находит граничные условия, которые вы могли бы упустить (пустой ввод, отрицательные значения, Unicode). Всегда читайте сгенерированные тесты — они проверяют ваши предположения, а не только ваш код.

- **Рефакторинг**: "извлеки этот блок в функцию", "преобразуй этот класс в `dataclasses`", "добавь аннотации типов в этот модуль". Механические преобразования, где намерение ясно, а риск скрытых ошибок невелик.

- **Исследование и прототипирование**: "напиши быстрый скрипт для замера задержки инференса" или "покажи, как использовать API токенизатора HuggingFace". Агент даст вам рабочий отправной пункт быстрее, чем чтение документации.

- **Документация и докстринги**: агент может генерировать документацию на основе структуры вашего кода. Проверяйте на точность, но рутинная работа будет автоматизирована.

- **Помощь в отладке**: вставьте трассировку ошибки и попросите диагностику. Агент часто может определить первопричину и предложить исправление, особенно для распространенных проблем (несоответствие размерностей, ошибки импорта, нехватка памяти CUDA).

### Когда НЕ стоит полагаться на агентов

- **Принятие архитектурных решений**: если вы проектируете новый пайплайн обучения, агент даст вам общий ответ. Он не знает ограничений ваших данных, требований к задержке или квалификации команды. Используйте агента для реализации дизайна, который вы уже продумали.

- **Критически важный для безопасности код**: аутентификация, шифрование, очистка входных данных. Агент может сгенерировать код, который выглядит корректным, но имеет скрытые уязвимости (SQL-инъекции, небезопасные значения по умолчанию, атаки по времени). Код, связанный с безопасностью, должен писать тот, кто понимает модель угроз, и проверять кто-то другой.

- **Критически важные для производительности внутренние циклы**: агент напишет корректный, но наивный код. Для ядер GPU, критичных к памяти структур данных или чувствительных к задержкам путей обслуживания, вам нужно понимать аппаратные ограничения (глава 13, глава 16) и оптимизировать осознанно.

- **Код, который вы не понимаете**: если агент генерирует 200 строк, а вы не можете объяснить, что делает каждая из них, не коммитьте это. Теперь вы поддерживаете код, который не понимаете, и когда он сломается (а он сломается), вы не сможете его отладить. Это самый распространенный и самый опасный режим отказа.

### Дисциплина проверки

- **Всегда читайте каждую строку** сгенерированного кода перед коммитом. Это не опционально. Код агента — это черновик, а не готовый продукт. Относитесь к нему точно так же, как к пул-реквесту от коллеги: проверяйте критически.

- **Что проверять**:
    - **Корректность**: делает ли он действительно то, что вы просили? Агенты часто решают немного другую задачу, чем та, которую вы задумывали.
    - **Граничные условия**: обрабатывает ли он пустые входные данные, значения `None`, отрицательные числа, очень большие входные данные? Агенты часто опускают обработку граничных условий.
    - **Галлюцинируемые API**: агент может вызывать функции или использовать параметры, которых не существует, особенно для новых или менее распространенных библиотек. Проверяйте, что каждый вызов API реален.
    - **Избыточное усложнение**: агенты склонны создавать больше кода, чем необходимо. Решение из 50 строк для задачи из 10 строк добавляет сложность без пользы. Упрощайте безжалостно.
    - **Безопасность**: жестко закодированные секреты, неочищенный ввод пользователя, небезопасные значения по умолчанию. Агент не мыслит как злоумышленник.
    - **Согласованность стиля**: соответствует ли сгенерированный код соглашениям вашего проекта (именование, шаблоны, обработка ошибок)?

### Как писать хорошие промпты

- Качество вывода агента прямо пропорционально качеству вашей инструкции. Расплывчатые промпты дают расплывчатый код.

- **Плохо**: "напиши загрузчик данных"
- **Хорошо**: "напиши `DataLoader` на PyTorch для CSV-файла со столбцами 'text' и 'label'. Токенизируй текст, используя токенизатор HuggingFace 'bert-base-uncased' с max_length=512. Верни input_ids, attention_mask и label в виде тензоров. Обработай случай, когда в CSV отсутствуют значения в столбце label, пропуская эти строки."

- **Предоставляйте контекст**: расскажите агенту о структуре вашего проекта, существующем коде, ограничениях и соглашениях. Чем больше контекста, тем лучше результат.

- **Указывайте ограничения**: "используй только стандартную библиотеку", "должно работать с Python 3.10", "не используй глобальные переменные", "следуй существующему шаблону в `src/models/transformer.py`".

- **Просите объяснения**: "реализуй X и объясни ключевые проектные решения". Это заставляет агента сформулировать свои рассуждения, что облегчает вам обнаружение ошибочных предположений.

### Использование барьеров качества для выявления ошибок агента

- Ваша существующая инфраструктура качества (файл 04) выявляет ошибки агента так же хорошо, как и ошибки человека:

    - **Проверка типов (mypy)**: выявляет галлюцинируемые сигнатуры API и несоответствия типов.
    - **Линтинг (ruff)**: выявляет неиспользуемые импорты, неопределенные переменные и нарушения стиля.
    - **Тесты (pytest)**: если код агента проходит ваш набор тестов, он с большей вероятностью корректен. Если у вас нет тестов, напишите их *до* того, как просить агента реализовать функционал (разработка через тестирование особенно хорошо работает с агентами).
    - **CI-пайплайн**: автоматически запускает все вышеперечисленное при каждом коммите.

- Комбинация "агент пишет код" + "барьеры качества проверяют его" продуктивнее, чем что-то одно. Агент быстр, но небрежен; барьеры тщательны, но не пишут код. Вместе вы получаете скорость и корректность.

### Ловушка продуктивности

- Самый большой риск использования агентов для написания кода — **иллюзия продуктивности**. Вы можете сгенерировать 500 строк кода за 10 минут. Но если вы потратите 2 часа на отладку этих 500 строк, потому что не понимали их, вы работали медленнее, чем если бы написали 200 строк самостоятельно за 30 минут.

- Настоящая продуктивность при работе с агентами достигается за счет:
    1. **Сохранения контроля**: вы определяете архитектуру, агент заполняет реализацию.
    2. **Понимания того, что сгенерировано**: если вы не можете объяснить код, перепишите его или попросите агента упростить.
    3. **Инвестиций в барьеры качества**: тесты, типы и линтинг окупают свою стоимость при каждом взаимодействии с агентом.
    4. **Использования агента для компенсации ваших слабых сторон**: если вы сильны в алгоритмах, но медленно пишете тесты, позвольте агенту писать тесты. Если вы быстры в UI-коде, но не знакомы с SQL-запросами, позвольте агенту набросать SQL. Играйте на своих сильных сторонах, делегируйте свои пробелы.

- Инженеры, которые извлекают максимум пользы из использования кодинг-агентов, — это те, кто уже умеет хорошо программировать. Агент усиливает ваши текущие навыки, а не заменяет их. Понимание структур данных, алгоритмов, системного проектирования и программной инженерии (всей этой главы) — это то, что позволяет вам эффективно управлять агентом и критически оценивать результаты его работы.
