# Git 与版本控制

*Git 是软件团队协作而不互相覆盖工作的方式。本文件涵盖 Git 心智模型、分支策略、merge 与 rebase、冲突解决、pull request,以及管理 ML 特有的挑战如大文件和实验追踪。*

- 每一个严肃的软件项目都使用版本控制。**Git** 是占据主导地位的系统,几乎所有开源项目和公司都在使用。没有 git,协作就是邮件传 zip 文件并祈祷没人覆盖你的改动。有了 git,每一次改动都被追踪、可回溯、可归因。

- 对 ML 工程师而言:git 追踪你的代码、配置和实验脚本。与实验追踪工具结合,它给你可复现性:"究竟是哪份代码和配置产生了这个模型?"

## The Mental Model

- Git 追踪你项目的**快照**。每个 commit 是那一刻所有被追踪文件的完整快照,而不是一个 diff(git 内部出于效率存储 diff,但概念上每个 commit 是一个完整状态)。

- 文件的四个"位置":

    1. **Working directory**: 磁盘上的实际文件。你编辑这些。
    2. **Staging area**(index): 你为下一次 commit 标记的文件。`git add` 把改动移到这里。
    3. **Local repository**: 你的 commit 历史,存储在 `.git/`。`git commit` 把 staging area 保存为新的快照。
    4. **Remote repository**(例如 GitHub): 共享副本。`git push` 上传你的 commit,`git pull` 下载他人的 commit。

```
Working Dir  →  git add  →  Staging  →  git commit  →  Local Repo  →  git push  →  Remote
                                                        ←  git pull  ←
```

- staging area 正是 git 强大的原因。你可以编辑 10 个文件却只 commit 其中 3 个,把其余改动留给单独的 commit。这使得干净、聚焦的 commit 成为可能。

### Essential Commands

```bash
git init                          # create a new repository
git clone url                     # download a remote repository
git status                        # what has changed? (most-used command)
git add file.py                   # stage a specific file
git add .                         # stage all changes (use with caution)
git commit -m "descriptive msg"   # commit staged changes
git push                          # upload commits to remote
git pull                          # download + merge remote changes
git log --oneline                 # compact commit history
git diff                          # show unstaged changes
git diff --staged                 # show staged changes
```

## Branching

- **branch**(分支)是指向某个 commit 的指针。默认分支是 `main`(或 `master`)。创建分支给你一条独立的开发线:你可以在不影响 `main` 的情况下做改动。

```bash
git branch feature-x              # create a branch
git checkout feature-x            # switch to it
git checkout -b feature-x         # create and switch in one step
git branch -d feature-x           # delete branch (after merging)
git branch -a                     # list all branches (local + remote)
```

- **何时开分支**: 始终开分支。永远不要直接 commit 到 `main`。每个功能、bug 修复或实验都有自己的分支。这让 `main` 保持稳定、可部署。

### Branching Strategies

- **Feature branches**(最常见): 每个功能/修复从 `main` 开一个分支。完成后,发起 pull request(PR)合回主干。简单,适合大多数团队。

- **Trunk-based development**: 开发者频繁地向 `main` commit(每天多次),用 feature flag 隐藏未完成的工作。受持续 deploy 的团队偏爱(Google、Facebook)。要求优秀的 CI/CD。

- **Gitflow**: 为功能、发布、hotfix 分别设立分支。更复杂,更适合有版本化发布的软件(移动应用、打包软件)。对大多数 ML 项目是大材小用。

- 对 ML 团队: **feature branches** + 短生命周期分支(1-3 天内合并)是最佳平衡点。长生命周期分支会偏离 `main`,造成痛苦的 merge 冲突。

## Merging and Rebasing

- **Merge** 创建一个新的"merge commit"来合并两个分支:

```bash
git checkout main
git merge feature-x
```

- 这保留完整历史:你能看到工作发生在分支上以及何时被合并。merge commit 有两个父节点。

- **Rebase** 把你分支的 commit 重新放到目标分支的顶部播放:

```bash
git checkout feature-x
git rebase main
```

- 这会改写历史:你分支的 commit 得到新的 hash,就好像你是从 `main` 当前 tip 开始工作的。结果是一条线性的历史(没有 merge commit),读起来更干净。

- **何时使用哪个**:
    - **Rebase** 用于用最新的 `main` 改动更新你的功能分支(保持分支干净、最新)。
    - **Merge** 用于把你的功能分支并入 `main`(保留分支历史)。
    - **永远不要 rebase 已经推送并共享给他人的 commit**。rebase 会改写历史;如果他人已经基于原 commit 开展工作,rebase 会引发混乱。

## Resolving Conflicts

- 当两个分支修改同一个文件的同一行时,就会发生**冲突**。Git 无法自动决定保留哪个改动,于是请求你手动解决。

```
<<<<<<< HEAD
learning_rate = 0.001
=======
learning_rate = 0.0005
>>>>>>> feature-x
```

- `<<<<<<< HEAD` 与 `=======` 之间是当前分支的版本。`=======` 与 `>>>>>>> feature-x` 之间是进入分支的版本。你决定保留哪个(或合并二者),删除标记,保存,并对已解决的文件执行 `git add`。

- **陷阱**: 不要把冲突标记留在已 commit 的文件里。它们是字面文本,会破坏你的代码。解决后一定要搜索 `<<<<<<<`。

- **减少冲突**: 让分支短生命周期、频繁把 `main` merge 进你的分支、避免多人同时编辑同一个文件。

## Writing Good Commit Messages

- commit message 是写给你未来的自己与队友的。"fix bug" 什么也没告诉你。"Fix off-by-one in batch size calculation that caused OOM on 8-GPU training" 告诉你一切。

- **格式**:

```
Short summary (50 chars or less, imperative mood)

Longer description if needed. Explain WHY, not WHAT
(the diff shows what changed). Wrap at 72 characters.

Fixes #123
```

- **祈使语气**(imperative mood): 写 "Add feature" 而不是 "Added feature" 或 "Adds feature。"把它读作完成这个句子:"If applied, this commit will **add feature**."

- **原子 commit**: 每个 commit 只做一件事。"Add data loader" 是一个 commit。"Add data loader and fix unrelated bug and update README" 应该是三个 commit。这让 `git bisect`(找出哪个 commit 引入了 bug)成为可能。

## Pull Requests and Code Review

- **pull request(PR)** 提议把一个分支合并进 `main`。它是 code review 的关口:队友阅读你的改动、提出改进建议,并在合并前批准。

- **良好的 PR 实践**:
    - 让 PR 小而聚焦(改动少于 400 行)。大型 PR 会被橡皮图章通过,因为没人想 review 2000 行。
    - 写清晰的描述:改了什么、为什么、如何测试。
    - 链接到触发此改动的 issue 或 ticket。
    - 及时回应 review 评论。
    - 合并前 squash 掉琐碎的 commit(让 `main` 历史干净)。

- **code review 不是为了找 bug**(那是测试的事)。它的意义在于:知识共享(reviewer 学习代码库)、设计反馈(这是正确的做法吗?)、维护标准(命名、风格、架构)。

## .gitignore

- `.gitignore` 文件告诉 git 哪些文件不纳入追踪。对 ML 项目:

```gitignore
# Python
__pycache__/
*.pyc
*.egg-info/
.venv/
env/

# Data and models (too large for git)
data/
*.csv
*.parquet
models/
*.pt
*.onnx
*.bin
checkpoints/

# Secrets
.env
*.pem
credentials.json

# IDE
.vscode/
.idea/
*.swp

# OS
.DS_Store
Thumbs.db

# Jupyter
.ipynb_checkpoints/

# Experiment outputs
wandb/
mlruns/
outputs/
logs/
```

- **陷阱**: 在文件已被 commit 之后再把它加入 `.gitignore` 并不会从仓库移除它。你还必须执行 `git rm --cached file` 才能停止追踪。除非改写历史(这很麻烦),该文件会永远留在历史中。

## Git for ML

- ML 引入了传统软件不会遇到的挑战:

- **大文件**: 数据集和模型权重是 GB 级甚至更大。git 为文本文件(源代码)设计,而非二进制大 blob。解决方案:
    - **Git LFS**(Large File Storage): 在 git 中追踪指针,把实际文件存储在单独的服务器上。简单,但在 GitHub 上有存储/带宽限制。
    - **DVC**(Data Version Control): 把数据和模型文件与 git 分开管理,使用远程存储(S3、GCS)。对数据而言工作方式像 git: `dvc add data.csv`、`dvc push`、`dvc pull`。

- **实验追踪**: 哪个 commit + 哪些超参数 + 哪份数据产生了哪些指标? Git 追踪代码,但不追踪完整的实验上下文。
    - **Weights & Biases (W&B)**: 记录指标、超参数、系统信息,并链接到 git commit。提供用于比较多次 run 的 dashboard。
    - **MLflow**: 开源的实验追踪,带 model registry。记录参数、指标和 artifact。
    - **简单做法**: 在训练脚本中记录 git hash: `git_hash = subprocess.check_output(['git', 'rev-parse', 'HEAD']).strip()`。把它与你的结果一起存储。

- **可复现性清单**(每次实验要追踪的内容):
    - Git commit hash(精确的代码版本)
    - 配置文件 / 超参数
    - 随机种子
    - Python 和库版本(`pip freeze`)
    - 数据版本(DVC hash 或数据集版本 tag)
    - 硬件(GPU 类型、GPU 数量)

```bash
# Quick reproducibility snapshot
echo "Commit: $(git rev-parse HEAD)" > experiment_info.txt
echo "Branch: $(git branch --show-current)" >> experiment_info.txt
echo "Dirty: $(git status --porcelain | wc -l) files" >> experiment_info.txt
pip freeze >> experiment_info.txt
nvidia-smi >> experiment_info.txt
```
