# Развёртывание и DevOps

*Развёртывание — это этап, на котором ваша модель перестает быть исследовательским артефактом и становится продуктом. В этом файле рассматриваются Docker для машинного обучения, инференс моделей (model serving), отслеживание экспериментов, воспроизводимость, мониторинг в продакшене, хранилища признаков (feature stores) и оркестрация пайплайнов — инфраструктура, которая превращает обученную модель из ноутбука в сервис для миллионов пользователей.*

- Модель, которая работает только на вашем ноутбуке, — это прототип. Модель, которая надежно работает при высоких нагрузках, выдает предсказания за миллисекунды, восстанавливается после сбоев и может обновляться без простоя, — это продукт. Разрыв между ними заполняется **развёртыванием и DevOps**.

- Большинство ML-инженеров тратят больше времени на развёртывание, мониторинг и отладку проблем в продакшене, чем на обучение моделей. Понимание этой инфраструктуры обязательно для любого, кто создает реальные системы машинного обучения.

## Docker для машинного обучения

- Мы рассматривали контейнеры концептуально в главе 13 (ОС). Здесь мы сосредоточимся на практической стороне: написании Dockerfile для задач машинного обучения.

- **Dockerfile** — это рецепт сборки образа контейнера:

```dockerfile
# Start from an official CUDA base image
FROM nvidia/cuda:12.1.0-cudnn8-runtime-ubuntu22.04

# System dependencies
RUN apt-get update && apt-get install -y \
    python3.11 python3-pip git \
    && rm -rf /var/lib/apt/lists/*

# Python dependencies (install separately for caching)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy source code (changes frequently, so this layer is last)
COPY src/ /app/src/
COPY configs/ /app/configs/
WORKDIR /app

# Entry point
CMD ["python3", "src/scripts/serve.py", "--config", "configs/serve.yaml"]
```

- **Кеширование слоев**: Docker кеширует каждый слой. Если `requirements.txt` не изменился, `pip install` пропускается при повторной сборке. Размещайте редко изменяемые слои (системные пакеты, pip install) перед часто изменяемыми (исходный код). Это превращает 10-минутную сборку в 10-секундную пересборку.

- **Доступ к GPU**: используйте базовые образы `nvidia/cuda` и запускайте с помощью `docker run --gpus all`. Пакет `nvidia-container-toolkit` обеспечивает проброс GPU из хостовой системы в контейнер.

- **Многоэтапная сборка (multi-stage builds)** уменьшает размер образа за счет отделения среды сборки от среды выполнения:

```dockerfile
# Build stage: install build tools, compile dependencies
FROM python:3.11 AS builder
COPY requirements.txt .
RUN pip install --user -r requirements.txt

# Runtime stage: only runtime dependencies
FROM nvidia/cuda:12.1.0-cudnn8-runtime-ubuntu22.04
COPY --from=builder /root/.local /root/.local
COPY src/ /app/src/
ENV PATH=/root/.local/bin:$PATH
```

- Итоговый образ содержит только библиотеки для выполнения, без компиляторов, заголовочных файлов и инструментов сборки. Образ для сборки размером 5 ГБ превращается в образ для выполнения размером 2 ГБ.

- **Docker Compose** запускает многоконтейнерные конфигурации (сервер моделей + балансировщик нагрузки + мониторинг):

```yaml
# docker-compose.yml
services:
  model:
    build: .
    ports:
      - "8080:8080"
    deploy:
      resources:
        reservations:
          devices:
            - capabilities: [gpu]
  prometheus:
    image: prom/prometheus
    ports:
      - "9090:9090"
```

## Инференс моделей (Model Serving)

- **Инференс моделей** — это запуск предсказаний как сервиса: получение запросов, выполнение модели, возврат предсказаний.

- **FastAPI** (рассмотрен в файле 03) — самый простой подход для низкой и средней пропускной способности. Для высокой пропускной способности и инференса, оптимизированного под GPU, используйте специализированные инструменты:

- **Triton Inference Server** (NVIDIA): обслуживает модели в форматах TensorRT, ONNX, PyTorch и TensorFlow. Особенности:
    - **Динамический батчинг (dynamic batching)**: собирает отдельные запросы и объединяет их в батчи для эффективной работы GPU. Поток одиночных запросов объединяется в группы по 32, что значительно увеличивает пропускную способность.
    - **Ансамбли моделей**: объединение нескольких моделей (препроцессор → модель → постпроцессор) в рамках одного запроса.
    - **Мультимодельный инференс**: обслуживание нескольких моделей на одном GPU с совместным использованием ресурсов.
    - **Параллельное выполнение моделей**: запуск нескольких запросов на инференс параллельно на одном GPU.

- **TorchServe** (PyTorch): обслуживает модели PyTorch с использованием REST/gRPC API. Поддерживает версионирование моделей, A/B-тестирование и пользовательские обработчики.

- **vLLM**: специализированное решение для инференса LLM. Реализует PagedAttention (эффективное управление KV-кешем), непрерывный батчинг и тензорный параллелизм между GPU. Достигает пропускной способности в 10–20 раз выше, чем при обычном инференсе больших языковых моделей.

- **Cactus** ([github.com/cactus-compute/cactus](https://github.com/cactus-compute/cactus)): движок ИИ с низкой задержкой для инференса на мобильных устройствах и периферии (edge). Cactus предоставляет **API, совместимый с OpenAI** (завершение чата, стриминг, вызов инструментов, транскрипция, эмбеддинги, RAG, зрение), который работает полностью на устройстве, с автоматическим **переключением на облако**, когда локальная модель не может обработать запрос. Эта гибридная архитектура означает, что ваш код приложения использует один и тот же API независимо от того, выполняется ли инференс локально или в облаке — движок принимает решение на основе уверенности модели и возможностей устройства. Доступны SDK для Python, Swift, Kotlin, Flutter, React Native и Rust, а предобученные веса моделей доступны на HuggingFace. Поддерживает мультимодальный инференс (LLM, зрение, речь) с использованием пользовательских ядер ARM SIMD для максимально быстрого инференса на ARM-процессорах и отображение памяти с нулевым копированием (zero-copy) для 10-кратного снижения использования оперативной памяти (глава 16, глава 17).

- **Оптимизация формата моделей**:
    - **ONNX**: открытый формат для интероперабельности. Экспортируйте из PyTorch/TensorFlow и запускайте где угодно.
    - **TensorRT**: оптимизатор от NVIDIA. Объединяет слои, выбирает оптимальные ядра, квантует веса. Обычно в 2–5 раз быстрее, чем PyTorch на GPU NVIDIA.
    - **GGUF/GGML**: форматы для эффективного инференса на CPU, популярны для запуска LLM на потребительском оборудовании.

## Отслеживание экспериментов

- Без отслеживания экспериментов исследования в области машинного обучения превращаются в: «Мне кажется, модель с прошлого вторника с тем конфигом, в котором я что-то изменил, была лучшей, но я не помню, что именно я менял».

- **Weights & Biases (W&B)**: самый популярный инструмент для отслеживания экспериментов. Логируйте что угодно из вашего скрипта обучения:

```python
import wandb

wandb.init(project="my-project", config={
    "model": "transformer",
    "lr": 3e-4,
    "batch_size": 64,
})

for epoch in range(num_epochs):
    train_loss = train_one_epoch()
    val_loss = validate()

    wandb.log({
        "train/loss": train_loss,
        "val/loss": val_loss,
        "epoch": epoch,
    })

    # Log model as artifact
    if val_loss < best_loss:
        wandb.save("best_model.pt")

wandb.finish()
```

- W&B предоставляет: дашборды для сравнения запусков, инструменты для подбора гиперпараметров, реестр моделей, версионирование датасетов и инструменты для командной работы.

- **MLflow**: open-source альтернатива. Работает локально или на сервере:

```python
import mlflow

mlflow.set_experiment("my-experiment")

with mlflow.start_run():
    mlflow.log_params({"lr": 3e-4, "batch_size": 64})
    mlflow.log_metric("val_loss", 0.042, step=epoch)
    mlflow.pytorch.log_model(model, "model")
```

- **Реестр моделей (Model registry)**: центральное хранилище обученных моделей с поддержкой версионирования, стадий (разработка → стейджинг → продакшн) и метаданных. И W&B, и MLflow предоставляют такие реестры. Реестр отвечает на вопросы: «какая модель сейчас в продакшне, кто её обучил, какова её точность на валидации и какой код/данные использовались для её создания?»

## Воспроизводимость

- Воспроизводимость означает: при использовании того же кода, данных и конфигурации получается та же самая модель. В машинном обучении это на удивление сложно из-за недетерминизма операций на GPU, перемешивания данных и накопления ошибок вычислений с плавающей запятой.

- **Чек-лист воспроизводимости**:

| Что | Как |
|------|-----|
| Версия кода | Хеш коммита Git |
| Конфигурация / гиперпараметры | Файл конфигурации (версионируется в git или логируется в W&B) |
| Случайные сиды | Установить и залогировать все сиды (Python, NumPy, PyTorch, CUDA) |
| Версия данных | Хеш DVC, тег версии датасета или версия объекта S3 |
| Зависимости | `pip freeze`, хеш Docker-образа или lock-файл |
| Оборудование | Тип GPU, количество GPU, версия CUDA |
| Недетерминизм | `torch.backends.cudnn.deterministic = True` (медленнее, но воспроизводимо) |

- **Фиксация всего**: `pip install torch==2.2.1`, а не `torch>=2.0`. Обновление минорной версии может изменить численные результаты, реализации оптимизаторов или значения гиперпараметров по умолчанию.

- **Docker для воспроизводимости**: Docker-образ фиксирует ОС, системные библиотеки, версию Python и пакеты pip. Хеш образа — это полный «отпечаток» окружения. Если вы можете воспроизвести Docker-образ, вы можете воспроизвести обучение.

## Мониторинг в продакшне

- Развёртывание модели — это не конец, а начало нового набора проблем. Модели деградируют со временем, когда меняется реальный мир (**дрейф концепции**) и когда меняется распределение входных данных (**дрейф данных**).

- **Что мониторить**:

    - **Задержка (Latency)**: сколько времени занимает инференс? Отслеживайте p50 (медиану), p95 и p99. Значение p99 в 500 мс означает, что 1 из 100 пользователей ждет полсекунды, что может быть неприемлемо.

    - **Пропускная способность (Throughput)**: сколько запросов в секунду? Справляется ли система с нагрузкой?

    - **Частота ошибок**: какая доля запросов завершается ошибкой (исключения, таймауты, некорректные входные данные)?

    - **Метрики модели**: точность, полнота, recall на отложенной выборке. Если в продакшне доступны размеченные данные (например, исправления пользователей), отслеживайте онлайн-метрики.

    - **Дрейф данных**: изменилось ли распределение входящих данных? Модель, обученная на дневных фотографиях, может не работать на ночных. Статистические тесты (тест Колмогорова-Смирнова, PSI) сравнивают распределение обучающей выборки с распределением «живых» данных.

    - **Дрейф признаков**: изменились ли распределения отдельных признаков? Признак, который при обучении был распределен нормально, а теперь стал бимодальным, сигнализирует о проблеме в пайплайне данных.

- **Инструменты**:
    - **Prometheus** + **Grafana**: стандарт для мониторинга инфраструктуры. Prometheus собирает метрики, Grafana визуализирует их на дашбордах с оповещениями.
    - **Evidently AI**: open-source инструмент для мониторинга ML. Генерирует отчеты о дрейфе данных, качестве модели и качестве данных.

- **Оповещения**: не ограничивайтесь дашбордами — настройте автоматические оповещения. «Если p99 задержка превышает 200 мс в течение 5 минут, отправить уведомление в Slack». «Если показатель дрейфа данных превышает порог, вызвать дежурного инженера».

## Хранилища признаков (Feature Stores)

- **Хранилище признаков** — это централизованный репозиторий предварительно вычисленных признаков, общий для обучения и инференса. Оно решает две проблемы:

    - **Расхождение обучения и инференса (Training-serving skew)**: признаки, используемые при обучении, должны быть идентичны тем, что используются при инференсе. Если при обучении `user_age_at_signup` вычисляется одним способом, а при инференсе — другим, предсказания модели будут неверными без явных ошибок.

    - **Повторное использование признаков**: несколько моделей часто используют одни и те же признаки (демография пользователей, эмбеддинги объектов, агрегированная статистика). Вычисление их один раз и совместное использование позволяет избежать дублирования и несогласованности.

- **Feast** — самое популярное open-source хранилище признаков. Оно управляет онлайн-признаками (низкая задержка, выдача из Redis или DynamoDB) и офлайн-признаками (батч-обработка, хранение в хранилищах данных для обучения).

- Хранилища признаков критически важны для рекомендательных систем, систем обнаружения мошенничества и любых приложений, где признаки вычисляются на основе пайплайнов обработки сырых данных.

## Оркестрация пайплайнов

- ML-система в продакшне — это не просто модель. Это **пайплайн**: получение данных → предобработка → вычисление признаков → обучение → оценка → развёртывание → мониторинг. Каждый шаг зависит от предыдущего, может завершиться ошибкой независимо и может требовать запуска по разному расписанию.

- **Оркестраторы** управляют этими пайплайнами:

- **Apache Airflow**: стандарт для оркестрации пайплайнов данных. DAG (направленные ациклические графы) определяют зависимости задач. Каждая задача выполняется независимо, может быть перезапущена при сбое и отслеживается через веб-интерфейс.

```python
# airflow DAG example (simplified)
from airflow import DAG
from airflow.operators.python import PythonOperator

dag = DAG("training_pipeline", schedule="@daily")

preprocess = PythonOperator(task_id="preprocess", python_callable=preprocess_data, dag=dag)
train = PythonOperator(task_id="train", python_callable=train_model, dag=dag)
evaluate = PythonOperator(task_id="evaluate", python_callable=evaluate_model, dag=dag)
deploy = PythonOperator(task_id="deploy", python_callable=deploy_model, dag=dag)

preprocess >> train >> evaluate >> deploy
```

- **Kubeflow Pipelines**: оркестрация для ML на базе Kubernetes. Каждый шаг выполняется в контейнере, ресурсы GPU выделяются по запросу, а эксперименты отслеживаются автоматически.

- **Prefect** и **Dagster**: современные альтернативы Airflow с лучшим опытом разработки, нативными Python API и встроенным отслеживанием происхождения данных (data lineage).

- **Когда использовать оркестрацию**: когда ваш пайплайн состоит более чем из 2–3 шагов, выполняется по расписанию, задействует несколько команд или сервисов, либо требует автоматического восстановления после сбоев. Для задачи обучения, запускаемой одним скриптом, оркестратор не нужен. А вот для пайплайна ежедневного переобучения, который собирает данные из 5 источников, обучает 3 модели, оценивает их и развёртывает лучшую, он абсолютно необходим.
