# 测试与质量保证

*测试是你确认代码能用——不仅是当下能用，更是每次改动后都能用——的手段。本文件涵盖测试金字塔、用 pytest 做单元测试、mocking、ML 特有代码的测试、CI/CD pipeline、linting、格式化以及 code review，这些在 bug 触达生产之前就把它们拦截下来的实践。*

- ML 代码的测试严重不足是出了名的。“它能训练，所以它能用”是主流态度。这会导致隐性 bug：data loader 错误地 shuffle、loss function 符号写反、preprocessing 步骤丢掉 5% 的数据。这些 bug 不会让你的程序崩溃，只会让你的 model 悄悄变差，你却在浪费数周调试那些“本该更高”的指标。

- 测试不是开销。它是快速迭代又不弄坏东西的最快途径。

## 测试金字塔

- 测试分为若干层，从又快又窄到又慢又宽：

    - **单元测试（Unit tests）**（底层）：隔离地测试单个函数和类。快（毫秒级），数量多（数百到数千）。“`normalise_image` 的输出是否落在 [0, 1]？”

    - **集成测试（Integration tests）**（中层）：测试组件能否协同工作。较慢（秒级）。“data loader 产出的 batch 是否符合 model 期望的格式？”

    - **端到端测试（End-to-end tests）**（顶层）：测试从输入到输出的完整 pipeline。慢（分钟级）。“`python train.py --config test.yaml` 能否无错误地完成并产出有效的 checkpoint？”

- 金字塔形状意味着：写大量单元测试、较少集成测试、少量端到端测试。单元测试能拦下大多数 bug，且几秒就跑完。端到端测试能抓到集成问题，但既慢又脆弱。

## 用 pytest 做单元测试

- **pytest** 是 Python 的标准测试框架。一个测试就是位于以 `test_` 开头的文件中、以 `test_` 开头的函数：

```python
# tests/test_utils.py

def test_normalise_image():
    import numpy as np
    image = np.array([0, 128, 255], dtype=np.uint8)
    result = normalise_image(image, mean=128, std=128)
    assert result.min() >= -1.0
    assert result.max() <= 1.0
    assert abs(result[1]) < 1e-6  # 128 normalised by mean=128 should be ~0

def test_normalise_empty():
    import numpy as np
    image = np.array([], dtype=np.uint8)
    result = normalise_image(image, mean=128, std=128)
    assert len(result) == 0
```

```bash
pytest tests/                     # run all tests
pytest tests/test_utils.py        # run one file
pytest -v                         # verbose output
pytest -x                         # stop on first failure
pytest -k "normalise"             # run tests matching name pattern
pytest --tb=short                 # shorter tracebacks
```

### Fixture

- **Fixture** 为测试提供可复用的 setup。与其在每个测试里重复 setup 代码，不如定义一次：

```python
import pytest

@pytest.fixture
def sample_dataset():
    """Create a small dataset for testing."""
    return {
        "inputs": torch.randn(10, 3, 32, 32),
        "labels": torch.randint(0, 10, (10,))
    }

@pytest.fixture
def trained_model():
    """Load a small pretrained model."""
    model = SmallModel()
    model.load_state_dict(torch.load("tests/fixtures/small_model.pt"))
    return model

def test_model_output_shape(trained_model, sample_dataset):
    output = trained_model(sample_dataset["inputs"])
    assert output.shape == (10, 10)  # batch_size x num_classes
```

- Fixture 可以有 **scope**：`scope="function"`（默认，每个测试新建）、`scope="module"`（每个文件一次）、`scope="session"`（每次测试运行一次）。像加载 model 这种昂贵的 setup 用 `scope="session"`。

### 参数化测试

- 无需重复代码即可用多组输入测试同一函数：

```python
@pytest.mark.parametrize("input,expected", [
    ([1, 2, 3], 6),
    ([], 0),
    ([-1, 1], 0),
    ([1000000, 1000000], 2000000),
])
def test_sum(input, expected):
    assert sum(input) == expected
```

## Mocking 与 Patching

- **Mocking** 在测试期间用一个假依赖替换真依赖。这样就能隔离地测试一个函数，而不需要数据库、API 或 GPU。

```python
from unittest.mock import patch, MagicMock

def test_training_logs_metrics():
    mock_logger = MagicMock()

    with patch("my_project.training.trainer.wandb") as mock_wandb:
        trainer = Trainer(logger=mock_logger)
        trainer.train_one_epoch()

        # verify that the trainer logged metrics
        mock_logger.log.assert_called()
        # verify it logged a loss value
        call_args = mock_logger.log.call_args
        assert "loss" in call_args[1]
```

- **何时 mock**：外部服务（API、数据库、云存储）、昂贵操作（GPU 计算、大文件 I/O）、以及非确定性行为（随机数生成器、时间戳）。

- **何时不该 mock**：你自己的代码。如果你把一切都 mock 了，你的测试验证的是 mock 按预期表现，而不是你的代码能用。在边界处 mock，直接测试你的逻辑。

## 测试 ML 代码

- ML 代码有独特的测试挑战：输出是概率性的，训练很慢，而“正确”是模糊的。

### 确定性种子

- 到处设置随机种子，让测试可复现：

```python
import random
import numpy as np
import torch

def set_seed(seed=42):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    if torch.cuda.is_available():
        torch.cuda.manual_seed_all(seed)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False
```

### 数值容差

- 浮点比较需要容差（第 13 章，IEEE 754）：

```python
# BAD: exact comparison fails due to floating point
assert model_output == 0.5

# GOOD: approximate comparison
import numpy as np
assert np.isclose(model_output, 0.5, atol=1e-5)

# For tensors
assert torch.allclose(output, expected, atol=1e-4)
```

### ML 该测什么

- **Shape 测试**：验证输出具有预期的维度。

```python
def test_model_output_shape():
    model = MyModel(d_model=256, n_classes=10)
    x = torch.randn(8, 32, 256)  # batch=8, seq=32, dim=256
    output = model(x)
    assert output.shape == (8, 10)
```

- **梯度流动**：验证可训练参数的梯度非零。

```python
def test_gradients_flow():
    model = MyModel()
    x = torch.randn(4, 3, 32, 32)
    y = torch.randint(0, 10, (4,))

    output = model(x)
    loss = F.cross_entropy(output, y)
    loss.backward()

    for name, param in model.named_parameters():
        assert param.grad is not None, f"No gradient for {name}"
        assert param.grad.abs().sum() > 0, f"Zero gradient for {name}"
```

- **在一个 batch 上过拟合**：一个 model 应当能记住单个 batch。如果做不到，说明根本上有问题。

```python
def test_overfit_one_batch():
    model = MyModel()
    optimiser = torch.optim.Adam(model.parameters(), lr=1e-3)
    x, y = get_single_batch()

    for _ in range(100):
        loss = F.cross_entropy(model(x), y)
        loss.backward()
        optimiser.step()
        optimiser.zero_grad()

    assert loss.item() < 0.01, f"Cannot overfit one batch: loss={loss.item()}"
```

- **数据验证**：验证数据加载产出有效输出。

```python
def test_dataset_basics():
    dataset = MyDataset("tests/fixtures/small_data.csv")
    assert len(dataset) > 0
    x, y = dataset[0]
    assert x.shape == (3, 224, 224)
    assert 0 <= y < 10
    assert not torch.isnan(x).any()
    assert not torch.isinf(x).any()
```

- **确定性**：相同输入 + 相同种子 → 相同输出。

```python
def test_determinism():
    set_seed(42)
    output1 = model(input_data)
    set_seed(42)
    output2 = model(input_data)
    assert torch.allclose(output1, output2)
```

## CI/CD Pipeline

- **持续集成（Continuous Integration, CI）**：每次提交或 PR 自动运行测试。如果测试失败，PR 无法合并。这阻止了坏代码进入 `main`。

- **GitHub Actions** 示例（`.github/workflows/ci.yml`）：

```yaml
name: CI
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install -e ".[dev]"
      - run: ruff check src/
      - run: mypy src/
      - run: pytest tests/ -v --tb=short
```

- **Pre-commit hook**：在每次提交前（本地）运行检查，在问题触达 CI 之前就抓住它们：

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.3.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
```

```bash
pip install pre-commit
pre-commit install    # now hooks run on every git commit
```

## Linting 与格式化

- **Linting** 不运行代码即可抓出 bug 和风格问题。**格式化** 自动强制风格一致。

- **Ruff**：快速的 Python linter 和 formatter（一个工具替代 flake8、isort 和 black）：

```bash
ruff check src/          # lint
ruff check --fix src/    # lint and auto-fix
ruff format src/         # format
```

- **mypy**：Python 的静态类型检查器。在运行前抓出类型错误：

```bash
mypy src/
# src/model.py:42: error: Argument 1 to "forward" has incompatible type "int"; expected "Tensor"
```

- 类型注解让代码自文档化并抓出 bug：

```python
def train(
    model: nn.Module,
    dataloader: DataLoader,
    optimiser: torch.optim.Optimizer,
    num_epochs: int = 10,
) -> float:
    """Train model and return final loss."""
    ...
```

## Code Review 最佳实践

- **给作者**：
    - 请求 review 前先自查你的 diff。你会抓到明显问题。
    - 保持 PR 小而聚焦。一个 PR 只解决一件事。
    - 写清楚的描述：是什么、为什么、怎么测。
    - 回应每一条评论（哪怕只回一句“done”）。

- **给 reviewer**：
    - 友善。批评代码，不批评人。说“这里可以更清晰”，别说“这写得让人困惑”。
    - 区分阻塞性问题（bug、安全）和建议（风格、命名）。用 label：“nit:”、“suggestion:”、“blocking:”。
    - 用提问代替命令。“如果这个 list 是空的会怎样？”比“处理一下空情况”更有帮助。
    - 及时 approve。一个 PR 等好几天才被 review 会阻塞作者，并鼓励那种又大又批量、更难 review 的 PR。
