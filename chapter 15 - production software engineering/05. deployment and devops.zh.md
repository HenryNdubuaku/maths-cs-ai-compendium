# Deployment 与 DevOps

*Deployment 是你的 model 从研究产物变成产品的转折点。本文件涵盖面向 ML 的 Docker、model serving、实验追踪、可复现性、生产监控、feature store 以及 pipeline 编排，这些基础设施把一个训练好的 model 从 notebook 送向数百万用户。*

- 只能在你笔记本上运行的 model 是原型。能大规模可靠运行、以毫秒级 serving 预测、能从故障中恢复、并且能零停机更新的，才是产品。两者之间的鸿沟就是 **deployment 与 DevOps**。

- 大多数 ML 工程师花在 deployment、监控和调试生产问题上的时间比花在训练 model 上的还多。对任何构建真实 ML 系统的人来说，理解这套基础设施不是可选项。

## 面向 ML 的 Docker

- 我们在第 13 章（OS）里概念性地讲过 container。这里聚焦实践面：为 ML 工作负载写 Dockerfile。

- **Dockerfile** 是构建 container image 的配方：

```dockerfile
# Start from an official CUDA base image
FROM nvidia/cuda:12.1.0-cudnn8-runtime-ubuntu22.04

# System dependencies
RUN apt-get update && apt-get install -y \
    python3.11 python3-pip git \
    && rm -rf /var/lib/apt/lists/*

# Python dependencies (install separately for caching)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy source code (changes frequently, so this layer is last)
COPY src/ /app/src/
COPY configs/ /app/configs/
WORKDIR /app

# Entry point
CMD ["python3", "src/scripts/serve.py", "--config", "configs/serve.yaml"]
```

- **层缓存**：Docker 缓存每一层。如果 `requirements.txt` 没变，重新构建时会跳过 `pip install`。把很少变的层（系统包、pip install）放在频繁变化的层（源代码）之前。这能把 10 分钟的构建变成 10 秒的重建。

- **GPU 访问**：用 `nvidia/cuda` 基础镜像，并用 `docker run --gpus all` 运行。`nvidia-container-toolkit` 提供从宿主到 container 的 GPU 直通。

- **多阶段构建（Multi-stage build）** 通过把构建环境与运行时分开来缩小 image 体积：

```dockerfile
# Build stage: install build tools, compile dependencies
FROM python:3.11 AS builder
COPY requirements.txt .
RUN pip install --user -r requirements.txt

# Runtime stage: only runtime dependencies
FROM nvidia/cuda:12.1.0-cudnn8-runtime-ubuntu22.04
COPY --from=builder /root/.local /root/.local
COPY src/ /app/src/
ENV PATH=/root/.local/bin:$PATH
```

- 最终 image 只包含运行时库，不含编译器、头文件或构建工具。5 GB 的构建 image 变成 2 GB 的运行时 image。

- **Docker Compose** 运行多 container 组合（model server + load balancer + 监控）：

```yaml
# docker-compose.yml
services:
  model:
    build: .
    ports:
      - "8080:8080"
    deploy:
      resources:
        reservations:
          devices:
            - capabilities: [gpu]
  prometheus:
    image: prom/prometheus
    ports:
      - "9090:9090"
```

## Model Serving

- **Model serving** 就是把推理作为服务运行：接收请求、运行 model、返回预测。

- **FastAPI**（见第 03 篇）是低到中等 throughput 场景下最简单的方案。对于高 throughput 和 GPU 优化的 serving，使用专门工具：

- **Triton Inference Server**（NVIDIA）：以 TensorRT、ONNX、PyTorch 和 TensorFlow 格式 serving model。特性：
    - **动态 batching（dynamic batching）**：收集单个请求并 batch 它们以提高 GPU 效率。单请求流被 batch 成 32 个一组，大幅提升 throughput。
    - **Model ensembles**：在单次请求中串联多个 model（preprocessor → model → postprocessor）。
    - **多 model serving**：在同一 GPU 上 serving 多个 model，共享资源。
    - **并发 model 执行**：在同一 GPU 上并行运行多个推理请求。

- **TorchServe**（PyTorch）：以 REST/gRPC API serving PyTorch model。支持 model 版本管理、A/B 测试和自定义 handler。

- **vLLM**：专为 LLM serving 打造。实现 PagedAttention（高效的 KV cache 管理）、连续 batching 和跨 GPU 的 tensor 并行。对大语言模型相比朴素 serving 可获得 10-20 倍的 throughput。

- **Cactus**（[github.com/cactus-compute/cactus](https://github.com/cactus-compute/cactus)）：面向移动与边缘端 on-device serving 的低 latency AI 引擎。Cactus 提供 **OpenAI 兼容 API**（chat completion、流式、tool calling、转写、embeddings、RAG、vision），完全在设备上运行，并在本地 model 无法处理某请求时自动**回退到云（cloud fallback）**。这种混合架构意味着你的应用代码无论推理在本地还是云端运行都用同一套 API——引擎会根据 model 置信度和设备能力做决定。提供 Python、Swift、Kotlin、Flutter、React Native 和 Rust 的 SDK，HuggingFace 上有预转换的 model 权重。支持多模态推理（LLM、vision、speech），并针对 ARM CPU 编写了自定义 ARM SIMD kernel 以获得最快的推理速度，使用零拷贝内存映射以降低 10 倍的 RAM 占用（第 16 章、第 17 章）。

- **Model 格式优化**：
    - **ONNX**：用于互操作的开放格式。从 PyTorch/TensorFlow 导出，到处运行。
    - **TensorRT**：NVIDIA 的优化器。融合层、选择最优 kernel、量化权重。在 NVIDIA GPU 上通常比 PyTorch 快 2-5 倍。
    - **GGUF/GGML**：面向 CPU 高效推理的格式，在消费级硬件上运行 LLM 时很流行。

## 实验追踪

- 没有实验追踪，ML 研究就会退化成：“我觉得上周二那个、被我改了点什么的 config 的 model 是最好的，但我不记得改了啥。”

- **Weights & Biases (W&B)**：最流行的实验追踪器。从你的训练脚本里 log 任何东西：

```python
import wandb

wandb.init(project="my-project", config={
    "model": "transformer",
    "lr": 3e-4,
    "batch_size": 64,
})

for epoch in range(num_epochs):
    train_loss = train_one_epoch()
    val_loss = validate()

    wandb.log({
        "train/loss": train_loss,
        "val/loss": val_loss,
        "epoch": epoch,
    })

    # Log model as artifact
    if val_loss < best_loss:
        wandb.save("best_model.pt")

wandb.finish()
```

- W&B 提供：用于比较运行的 dashboard、超参数 sweep 工具、model registry、数据集版本管理以及团队协作。

- **MLflow**：开源替代品。可在本地或服务器上运行：

```python
import mlflow

mlflow.set_experiment("my-experiment")

with mlflow.start_run():
    mlflow.log_params({"lr": 3e-4, "batch_size": 64})
    mlflow.log_metric("val_loss", 0.042, step=epoch)
    mlflow.pytorch.log_model(model, "model")
```

- **Model registry**：训练好的 model 的集中存储，带版本管理、staging（dev → staging → production）和元数据。W&B 和 MLflow 都提供 registry。registry 回答：“当前生产里跑的是哪个 model、是谁训练的、它的验证准确率是多少、产出它的代码/数据是什么？”

## 可复现性

- 可复现性意味着：给定相同的代码、数据和 config，产出相同的 model。这在 ML 中出奇地难，因为 GPU 操作、数据 shuffle 和浮点累加都存在非确定性。

- **可复现性清单**：

| 内容 | 方法 |
|------|-----|
| 代码版本 | Git commit hash |
| Config / 超参数 | Config 文件（在 git 里版本管理或 log 到 W&B） |
| 随机种子 | 设置并记录所有种子（Python、NumPy、PyTorch、CUDA） |
| 数据版本 | DVC hash、数据集版本 tag 或 S3 object version |
| 依赖 | `pip freeze`、Docker image hash 或 lockfile |
| 硬件 | GPU 类型、GPU 数量、CUDA 版本 |
| 非确定性 | `torch.backends.cudnn.deterministic = True`（更慢但可复现） |

- **锁定一切**：`pip install torch==2.2.1` 而非 `torch>=2.0`。一个小版本升级就可能改变数值行为、optimiser 实现或默认超参数。

- **用 Docker 做可复现性**：Docker image 锁定了 OS、系统库、Python 版本和 pip 包。image hash 是完整的环境指纹。如果你能复现 Docker image，就能复现训练。

## 生产监控

- 部署一个 model 并不是终点——而是一连串新问题的开始。随着真实世界变化，model 会随时间退化（**concept drift**），输入数据分布也会漂移（**data drift**）。

- **监控什么**：

    - **Latency**：推理要多久？追踪 p50（中位数）、p95 和 p99。p99 为 500ms 意味着每 100 个用户就有 1 个等半秒，这可能是不可接受的。

    - **Throughput**：每秒多少请求？系统是否跟得上需求？

    - **错误率**：多少比例的请求失败（异常、超时、无效输入）？

    - **Model 指标**：在 holdout 集上的准确率、precision、recall。如果生产中有标签数据（如用户纠错），追踪在线指标。

    - **Data drift**：输入数据的分布变了吗？在白天照片上训练的 model 可能在夜间照片上失效。统计检验（KS 检验、PSI）把训练分布与线上分布做比较。

    - **Feature drift**：单个 feature 的分布变了吗？某个训练时服从正态分布、现在却变成双峰的 feature，暗示数据 pipeline 有问题。

- **工具**：
    - **Prometheus** + **Grafana**：基础设施监控的标准组合。Prometheus 采集指标，Grafana 在 dashboard 中可视化并配告警。
    - **Evidently AI**：开源 ML 监控。生成关于 data drift、model 性能和数据质量的报告。

- **告警**：别只做 dashboard——设置自动告警。“如果 p99 latency 超过 200ms 持续 5 分钟，发 Slack 通知。”“如果 data drift 分数超阈值，呼叫值班工程师。”

## Feature Store

- **Feature store** 是预计算 feature 的集中式仓库，在训练和 serving 之间共享。它解决两个问题：

    - **Training-serving skew**：训练时使用的 feature 必须与 serving 时使用的完全一致。如果训练用一种方式计算 `user_age_at_signup` 而 serving 用另一种方式计算，model 的预测会悄无声息地出错。

    - **Feature 复用**：多个 model 常用相同的 feature（用户人口统计、item embedding、聚合统计）。算一次并共享，避免重复和不一致。

- **Feast** 是最流行的开源 feature store。它管理在线 feature（低 latency，从 Redis 或 DynamoDB 提供）和离线 feature（批量，存在数据仓库中供训练）。

- Feature store 对推荐系统、欺诈检测以及任何从原始数据 pipeline 计算 feature 的应用都至关重要。

## Pipeline 编排

- 生产 ML 系统不只是一个 model。它是一条 **pipeline**：数据摄入 → preprocessing → feature 计算 → 训练 → 评估 → deployment → 监控。每一步都依赖前一步，可能独立失败，也可能需要按不同节奏运行。

- **编排器（Orchestrator）** 管理这些 pipeline：

- **Apache Airflow**：数据 pipeline 编排的事实标准。DAG（有向无环图）定义任务依赖。每个任务独立运行、失败可重试，并通过 web UI 监控。

```python
# airflow DAG example (simplified)
from airflow import DAG
from airflow.operators.python import PythonOperator

dag = DAG("training_pipeline", schedule="@daily")

preprocess = PythonOperator(task_id="preprocess", python_callable=preprocess_data, dag=dag)
train = PythonOperator(task_id="train", python_callable=train_model, dag=dag)
evaluate = PythonOperator(task_id="evaluate", python_callable=evaluate_model, dag=dag)
deploy = PythonOperator(task_id="deploy", python_callable=deploy_model, dag=dag)

preprocess >> train >> evaluate >> deploy
```

- **Kubeflow Pipelines**：在 Kubernetes 上的 ML 专用编排。每步在 container 中运行，GPU 资源按需分配，实验自动追踪。

- **Prefect** 与 **Dagster**：Airflow 的现代替代品，有更好的开发者体验、原生 Python API 和内建数据血缘。

- **何时编排**：当你的 pipeline 超过 2-3 步、按节奏运行、涉及多个团队或服务、或需要从故障中自动恢复时。单脚本训练任务不需要编排器。一个每天从 5 个数据源摄入数据、训练 3 个 model、评估并部署其中最优者的重训练 pipeline，绝对需要。
